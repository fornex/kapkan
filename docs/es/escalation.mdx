# Escalado progresivo

Un único método de `mitigation` dispara la misma respuesta en el instante en que se detecta un
ataque. Un escalado progresivo, en cambio, **intensifica la respuesta cuanto más persiste un
ataque**, de forma declarativa y sin scripts de callback.

A la mayoría de los despliegues les basta con un único método `mitigation:`. Recurre a un escalado
solo cuando quieras una respuesta más suave primero (por ejemplo, alerta y luego filtrado
quirúrgico) antes de pasar a un [blackhole](/docs/glossary) completo.

```yaml
escalation:                         # supersedes `mitigation` when present
  - { after_seconds: 0,   action: none }       # alert only at first
  - { after_seconds: 30,  action: flowspec }   # still under attack after 30s → surgical drop
  - { after_seconds: 90,  action: divert }     # still under attack after 90s → scrub
  - { after_seconds: 300, action: blackhole }  # still under attack after 300s → blackhole
flowspec:
  action: discard
scrubbing:
  next_hop: "192.0.2.100"   # scrubbing center; see Traffic diversion
  community: "65000:200"
```

Se requiere un [bloque `bgp`](/docs/mitigation#configuración-de-bgp) para que cualquier peldaño
pueda anunciar; los bloques `flowspec` y `scrubbing` de arriba solo hacen falta para los
peldaños `flowspec` y `divert`.

## Cómo se aplican los peldaños

El valor `after_seconds` de cada peldaño se mide desde el inicio del ataque. Un peldaño se aplica
una vez que ha transcurrido ese tiempo **y el bloqueo sigue activo**. (Un bloqueo solo se retira
después de que el tráfico se mantenga por debajo del umbral durante el periodo de enfriamiento
configurado, `ban.unban_hysteresis_seconds`; consulta [Configuración](/docs/configuration).) El
reloj del peldaño sigue corriendo desde el inicio del ataque en todo momento.

- El primer peldaño debe estar en `0` segundos.
- Un escalado puede tener como máximo **5 peldaños**: es un límite estricto; el demonio se niega a
  arrancar con un escalado más largo.
- `action` es `none` (solo alerta), `flowspec`, `divert` ([scrubbing](/docs/scrubbing)) o
  `blackhole`.
- Un escalado solo puede **mantener o reforzar** la respuesta: `none` < `flowspec` < `divert` <
  `blackhole`. Desescalar entre peldaños es un error de configuración.

## Make-before-break

Subir a un peldaño se hace con la estrategia make-before-break (establecer antes de cortar): primero
se anuncia el nuevo peldaño y solo después de que eso tiene éxito se retira el anterior, de modo que
la víctima nunca queda desprotegida ni por un instante durante el cambio. Si el anuncio falla, el
bloqueo mantiene el peldaño en funcionamiento y reintenta en el siguiente ciclo. Si varios peldaños
vencen a la vez (un ataque de larga duración, o el demonio poniéndose al día tras una pausa), el
bloqueo salta directamente al peldaño vencido más alto y nunca anuncia los peldaños que omite.

Un peldaño `flowspec` o `divert` cuyo anuncio rechace el peer ascendente recae en blackhole (a menos
que establezcas `ban.fallback: none` en [Configuración](/docs/configuration)); la API y las
notificaciones lo reflejan mediante el campo `fell_back_from`. Los peldaños de blackhole y de solo
alerta no tienen mecanismo de recaída.

La única excepción al baile de anunciar-luego-retirar es `divert → blackhole`: ambos anuncian la
misma ruta de host (el `/32` o `/128` de la víctima), de modo que el anuncio del blackhole reemplaza
atómicamente la ruta de divert, sin retirada aparte y sin hueco. Los cambios entre familias (reglas
FlowSpec frente a una ruta de host) siguen el caso de make-before-break descrito arriba.

## Ciclo de vida y compatibilidad

El escalado es **anulable por hostgroup** y comparte el resto del ciclo de vida del bloqueo: la
retirada automática por TTL, el límite `max_active_bans`, los límites de radio de impacto
(`max_banned_fraction`, `max_bans_per_window`), la lista blanca absoluta y el dry-run (que avanza
por el escalado y registra cada peldaño pero nunca anuncia).

Cuando no se establece ningún bloque `escalation`, el método único de `mitigation` se comporta
exactamente como un escalado de un solo peldaño en `0` segundos: compatibilidad total con versiones
anteriores. Cada bloqueo en `/api/v1/bans` expone el escalado completo configurado (`escalation`),
el índice del peldaño actual (`escalation_step`) y el método del peldaño activo (`method`).
