# Referencia de CLI

Kapkan es un único binario. Todo lo que hace se selecciona mediante opciones de línea de comandos
al arrancar; no hay subcomandos más allá de los modos de utilidad que se describen abajo. Todos
los registros van a **stderr**; la API REST y las métricas de Prometheus se sirven por HTTP en
`api.listen` (predeterminado `127.0.0.1:8080`), mientras que las flags de utilidad (`-version`,
`-dump-schema`) imprimen su salida en **stdout**.

```sh
kapkan [flags]
```

## Flags

| Flag | Predeterminado | Descripción |
| --- | --- | --- |
| `-config <path>` | `configs/dev.yaml` | Ruta al [archivo de configuración](/docs/configuration) YAML. |
| `-log-format <fmt>` | `json` | Formato de salida de los registros: `json` (estructurado, para recolectores de logs) o `text` (legible para humanos, para ejecuciones locales). Cualquier valor distinto de `text` se trata como `json`. |
| `-log-level <lvl>` | `info` | Nivel mínimo de registro: `debug`, `info`, `warn`, `error`. Un valor no reconocido recae en `info`. |
| `-check-config <path>` | — | Valida el archivo de configuración en `<path>` y termina; **no** inicia el demonio. Consulta [más abajo](#validar-una-configuración). |
| `-dump-schema` | `false` | Imprime el esquema JSON de la configuración en stdout y termina. Consulta [más abajo](#volcar-el-esquema). |
| `-version` | `false` | Imprime la versión de compilación y termina. La misma versión se expone en tiempo de ejecución mediante `/api/v1/status` y la [métrica](/docs/metrics) `kapkan_build_info`. |
| `-check-update` | `false` | Ejecuta la [comprobación de actualizaciones](/docs/configuration) opcional una vez y termina: `0` actualizado, `10` hay una actualización disponible, `1` ante un error. Consulta la API de Releases de GitHub; envía solo la propia solicitud, nunca la identidad del nodo ni la configuración. |
| `-pid-file <path>` | `/run/kapkan/kapkan.pid` | Dónde escribe el demonio su id de proceso al arrancar (se elimina al apagarse limpiamente); lo lee `-s` para localizar el demonio en ejecución. Un fallo de escritura no es fatal: el demonio sigue ejecutándose, solo que `-s` no está disponible. |
| `-s <signal>` | — | Envía una señal a un demonio en ejecución y termina, leyendo `-pid-file`: `reload`, `stop` o `quit`. **No** inicia un demonio. Consulta [más abajo](#controlar-un-demonio-en-ejecución). |

<Callout type="info" title="¿json o text?">
Usa `-log-format text` para el desarrollo local; es el formato que emplea la
[Guía rápida](/docs/quickstart). Mantén el valor predeterminado `json` en producción para
que un recolector de logs pueda analizar los campos estructurados; la [unidad de systemd](/docs/deployment)
se entrega con `-log-format json -log-level info`.
</Callout>

## Ejecutar el demonio

Sin ninguna flag de utilidad, Kapkan abre sus listeners de flow, la API REST y (cuando no está en
modo dry-run) el hablante BGP, y luego se ejecuta hasta que recibe `SIGINT` o `SIGTERM`, momento en
el que se apaga limpiamente. Solo termina con un código distinto de cero ante un error fatal de
arranque o de ejecución.

```sh
# Local, human-readable logs against the bundled dev config:
kapkan -config configs/dev.yaml -log-format text

# Production-style: structured logs at info level:
kapkan -config /etc/kapkan/config.yaml -log-format json -log-level info
```

El archivo de configuración se puede recargar en tiempo de ejecución sin reiniciar: ejecuta
[`kapkan -s reload`](#controlar-un-demonio-en-ejecución), envía `SIGHUP` directamente
(`systemctl reload kapkan`) o `POST /api/v1/config/reload`. Consulta la
[referencia de configuración](/docs/configuration) para saber qué se recarga y qué queda fijado al arrancar.

## Controlar un demonio en ejecución

```sh
kapkan -s reload    # re-read the config file (SIGHUP)
kapkan -s stop      # graceful shutdown (SIGTERM)
```

`-s` lee el id de proceso de `-pid-file` (predeterminado `/run/kapkan/kapkan.pid`, escrito por el
demonio al arrancar) y entrega la señal correspondiente a ese proceso: el mismo control local al
estilo nginx que `nginx -s reload`. No necesita conexión de red ni token de API, así que funciona
incluso si la API REST está caída; sí necesita permiso para enviar señales al demonio, así que
ejecútalo como `root` o como el usuario `kapkan`.

| Valor de `-s` | Señal | Efecto |
| --- | --- | --- |
| `reload` | `SIGHUP` | Recarga en caliente el archivo de configuración. Igual que `systemctl reload kapkan` o `POST /api/v1/config/reload`. |
| `stop` / `quit` | `SIGTERM` | Apagado graceful. Igual que `systemctl stop kapkan`. Los dos son alias: el apagado de Kapkan siempre es graceful (pide a los peers BGP que conserven las rutas de mitigación mediante [Graceful Restart](/docs/mitigation)). |

Si el pid file falta o está obsoleto (no existe tal proceso), o si no tienes permiso para enviarle
señales, `-s` imprime el motivo en stderr y termina con `1`.

<Callout type="info" title="Dónde vive el pid file">
La [unidad de systemd](/docs/deployment) incluida establece `RuntimeDirectory=kapkan`, de modo que
`/run/kapkan` existe y el usuario `kapkan` puede escribir en él; el demonio escribe ahí
`/run/kapkan/kapkan.pid` y lo elimina al detenerse limpiamente. En una ejecución de desarrollo local
ese directorio puede no existir: el demonio registra una advertencia y continúa, y `-s` informará de
"no pid file". Pasa `-pid-file /tmp/kapkan.pid` tanto al demonio como a `-s` si quieres el atajo en local.
</Callout>

## Flags de utilidad

Estas flags terminan antes de que arranque el demonio; nunca abren un listener ni envían un
anuncio de ruta, así que son seguras de ejecutar en un host de producción. (`-check-update` realiza
una única solicitud saliente a la API de releases; las demás son totalmente offline.)

### Validar una configuración

```sh
kapkan -check-config /etc/kapkan/config.yaml
```

Ejecuta exactamente el mismo análisis y la misma validación del motor —incluidas las reglas entre
campos que un esquema estático no puede expresar— y luego imprime la configuración resuelta (modo,
redes protegidas y la mitigación efectiva por cada [hostgroup](/docs/hostgroups)) y termina:

| Código de salida | Significado |
| --- | --- |
| `0` | La configuración es válida. |
| `1` | La configuración no es válida; el error exacto se imprime en stderr. |

La división limpia 0/1 encaja directamente en CI o en una comprobación previa al despliegue. Consulta
[Validar la configuración](/docs/configuration#validar-la-configuración).

### Volcar el esquema

```sh
kapkan -dump-schema
```

Imprime el esquema JSON de la configuración en stdout y termina con `0` (o con `1` ante un error
interno). El [generador de configuración](/config) usa este esquema para validar los campos en tu navegador.

### Imprimir la versión

```sh
kapkan -version
```

Imprime la versión de compilación (la etiqueta de la release más la revisión corta de VCS cuando se
conoce) en stdout y termina. La misma cadena se sirve en tiempo de ejecución en `/api/v1/status` y
la transporta la [métrica](/docs/metrics) `kapkan_build_info`, así que puedes confirmar la versión de
un binario localmente sin ninguna salida de red.

### Comprobar actualizaciones

```sh
kapkan -check-update
```

Ejecuta la comprobación [`update_check`](/docs/configuration) una vez bajo demanda y termina: `0` si
está actualizado, `10` si hay una release más reciente disponible, `1` ante un error. Consulta la API
de Releases de GitHub y envía solo la propia solicitud (tu IP y un User-Agent genérico), nunca la
identidad del nodo, la configuración ni datos de ataques. La comprobación periódica en proceso está
**desactivada de forma predeterminada**; esta flag funciona independientemente de ese ajuste.
