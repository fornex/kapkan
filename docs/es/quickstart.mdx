# Inicio rápido

Esto te lleva desde cero hasta una instancia de Kapkan en ejecución, en modo dry-run, que detecta
ataques contra un prefijo de prueba. No hay paso de compilación — Kapkan se distribuye como un
único binario preconstruido y firmado — y nada de lo que se hace aquí puede anunciar una ruta,
porque dry-run es el modo predeterminado.

<Callout type="info" title="Lo que necesitas">
Un host Linux (`amd64` o `arm64`). Para ver detecciones *reales* necesitas un router que exporte
telemetría de flow — NetFlow, IPFIX o sFlow (registros de quién envía tráfico a quién; consulta el
[glosario](/docs/glossary)). Sin router y con una copia del código fuente, aún puedes confirmar que
la detección funciona ejecutando la batería de pruebas — consulta la
[última sección](#5-opcional-no-tienes-router-valida-la-detección-desde-el-código-fuente). Solo necesitas un toolchain
de Go si tomas ese camino, o si
[compilas desde el código fuente](/docs/deployment#compilar-desde-el-código-fuente) en lugar de descargar.
</Callout>

## 1. Instala Kapkan

Elige la opción que se adapte a tu host — cada versión se distribuye preconstruida y firmada para
`linux` `amd64`/`arm64`, de modo que nunca compilas nada.

<Callout type="info" title="Usa la última versión — reemplaza v1.0.0">
Antes de copiar y pegar, reemplaza `v1.0.0` en cada comando de abajo por la etiqueta más reciente de
la [página de versiones](https://github.com/fornex/kapkan/releases/latest) (copiar `v1.0.0` tal cual
dará un 404 si no es la versión actual), y `amd64` por `arm64` en hosts ARM. Los nombres del archivo
y del paquete eliminan la `v` inicial (p. ej. `kapkan_1.0.0_linux_amd64.tar.gz`).
</Callout>

### Debian / Ubuntu (`.deb`)

```sh
VER=v1.0.0
curl -fLO "https://github.com/fornex/kapkan/releases/download/$VER/kapkan_${VER#v}_linux_amd64.deb"
sudo apt install "./kapkan_${VER#v}_linux_amd64.deb"
```

El paquete instala el binario en `/usr/local/bin/kapkan`, crea un usuario `kapkan` sin privilegios,
prepara `/etc/kapkan` con un `config.yaml` en dry-run e instala la unidad systemd reforzada —
dejada **detenida** para que puedas revisar primero la configuración. La página de
[Despliegue en producción](/docs/deployment) cubre el servicio a partir de ahí. RHEL / Fedora es
igual con el `.rpm`: `sudo dnf install ./kapkan_<ver>_linux_amd64.rpm`.

### Tarball (cualquier distro)

```sh
VER=v1.0.0
curl -fLO "https://github.com/fornex/kapkan/releases/download/$VER/kapkan_${VER#v}_linux_amd64.tar.gz"
tar xzf "kapkan_${VER#v}_linux_amd64.tar.gz"
```

Eso desempaqueta el binario `kapkan` más un directorio `deploy/` (la unidad systemd, una
configuración de ejemplo y `update.sh`). El resto de esta página usa el tarball. Antes de
ejecutarlo en producción conviene [verificar la descarga](/docs/upgrading#verificar-una-descarga-manualmente)
— la firma de cosign y el SHA-256.

## 2. Ejecuta en modo dry-run

Desde el tarball desempaquetado, inicia Kapkan con la configuración de ejemplo incluida y registros
(logs) legibles para humanos:

```sh
./kapkan -config deploy/config.example.yaml -log-format text
```

Se ejecuta en primer plano e imprime los registros en tu terminal; pulsa Ctrl-C para detenerlo de
forma limpia.

`-config` apunta a la configuración YAML y `-log-format text` cambia los registros del formato JSON
estructurado predeterminado a una forma legible para humanos; consulta la
[Referencia de CLI](/docs/cli) para conocer cada flag y sus valores admitidos.

En dry-run, Kapkan no necesita privilegios especiales y puede ejecutarse como un usuario normal.
(La puesta en producción enlaza BGP al puerto 179, lo que requiere root o la capacidad
`CAP_NET_BIND_SERVICE` — la unidad systemd incluida se encarga de esto; consulta
[Despliegue en producción](/docs/deployment)).

La [`deploy/config.example.yaml`](/docs/configuration) incluida tiene `dry_run: true`, redes
protegidas de prueba de `203.0.113.0/24` y `2001:db8::/32`, una `protected_whitelist`, y escucha
sFlow en `:6343` y NetFlow/IPFIX en `:2055`. Como dry-run está activado, Kapkan registra los
blackholes hipotéticos pero nunca anuncia una ruta.

<Callout type="info" title="¿Instalaste el paquete?">
Si instalaste el `.deb`/`.rpm`, el mismo daemon se ejecuta como servicio contra
`/etc/kapkan/config.yaml` en su lugar — `sudo systemctl start kapkan`, y luego
`journalctl -u kapkan -f`. Consulta [Despliegue en producción](/docs/deployment).
</Callout>

## 3. Apunta tus exportadores a Kapkan

Configura los exportadores de flow de tus routers para que envíen a los puertos de escucha de Kapkan:

| Protocolo | Puerto predeterminado |
| --- | --- |
| sFlow v5 | `:6343` |
| NetFlow v5/v9 + IPFIX | `:2055` |

NetFlow e IPFIX comparten el mismo socket UDP. Debe configurarse al menos un listener.

## 4. Míralo en acción

Kapkan expone su API REST y sus métricas en `127.0.0.1:8080` de forma predeterminada:

```sh
# Current mode, protected networks, thresholds, active attack/ban counts
curl -s localhost:8080/api/v1/status | jq

# Active attacks plus the last 100 that ended, with samples and classification
curl -s localhost:8080/api/v1/attacks | jq

# Prometheus metrics
curl -s localhost:8080/metrics | grep kapkan_
```

También puedes abrir el [dashboard](/docs/dashboard) integrado en
[http://localhost:8080](http://localhost:8080) en un navegador.

## 5. Opcional: ¿no tienes router? Valida la detección desde el código fuente

<Callout type="info" title="Este es un camino para desarrolladores, no el del binario">
Este paso **no** usa el binario preconstruido que descargaste arriba. Ejecuta la batería de pruebas
de Kapkan, así que necesita una copia del código fuente y un [toolchain de Go](https://go.dev/dl/).
Si solo tienes el `.deb`/`.rpm`/tarball y todavía no tienes router, omítelo — apunta un router a los
puertos de escucha (paso 3) cuando tengas uno. La herramienta de ataques sintéticos (`pkg/flowgen`)
es un paquete de la batería de pruebas, no un comando que exponga el binario incluido.
</Callout>

Lo que esto demuestra: Kapkan detecta un ataque y habría incluido a la víctima en su lista negra —
todo ello sin enviar nada a un router real, ya que dry-run solo registra el bloqueo hipotético (un
*ban virtual* — una ruta de [blackhole](/docs/glossary) que Kapkan expone a través de la API pero
nunca anuncia).

La prueba envía una falsa inundación de amplificación NTP a una instancia en dry-run a través de un
socket UDP real y comprueba que el ataque aparece en la API, con un ban virtual que más tarde expira
automáticamente por su TTL. Ejecútala desde una copia del código fuente:

```sh
git clone https://github.com/fornex/kapkan && cd kapkan/engine
go test ./internal/app -run TestEndToEndNTPAmplification -v
```
