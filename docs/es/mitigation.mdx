# Mitigación RTBH

Cuando el motor reporta un ataque, Kapkan lo mitiga anunciando una ruta de **blackhole activado
de forma remota (RTBH, remotely-triggered blackhole)** para el host atacado: un prefijo `/32`
(IPv4) o `/128` (IPv6) transportado a través de un hablante [GoBGP](https://github.com/osrg/gobgp)
embebido, etiquetado con tu community RTBH y apuntado a un next-hop de descarte. Tus routers de
borde hacen coincidir la community y descartan todo el tráfico hacia ese host, deteniendo la
inundación antes de que alcance el núcleo de tu red.

La mitigación se basa en el destino: el blackhole elimina el tráfico *hacia* el objetivo. Para una
inundación entrante esto detiene el ataque; para un [ataque saliente](/docs/detection) desde un
host comprometido deja ese host fuera de línea (lo que normalmente detiene el abuso) y detiene la
inundación saliente solo donde tu borde también descarta orígenes en prefijos sometidos a blackhole,
por ejemplo con uRPF.

<Callout type="warning" title="Dry-run de forma predeterminada">
Hasta que establezcas explícitamente `dry_run: false`, cada posible blackhole se calcula, se
registra y se expone a través de la API, pero **nunca se anuncia** a tus routers. Valida la
detección y el peering BGP frente a la telemetría de producción antes de que pueda enviarse
cualquier ruta. Consulta [Puesta en producción](/docs/going-live).
</Callout>

## Métodos de mitigación

`blackhole` (RTBH) es el método predeterminado y el tema del resto de esta página. Hay otros
métodos disponibles a través de la clave `mitigation`, cada uno sobreescribible por
[hostgroup](/docs/hostgroups):

- **[FlowSpec](/docs/flowspec)** (`mitigation: flowspec`) — descarta solo el vector de ataque con
  reglas BGP FlowSpec (RFC 8955/8956) en lugar de someter a blackhole a la víctima completa.
- **[Desvío de tráfico](/docs/scrubbing)** (`mitigation: divert`) — anuncia la víctima hacia un
  centro de scrubbing para que su tráfico se limpie y se reinyecte en lugar de descartarse.
- **[Escaleras de escalado](/docs/escalation)** (`escalation:`) — intensifica la respuesta cuanto
  más persiste un ataque, desde la alerta a FlowSpec, luego a divert y luego a blackhole.

Todos comparten el mismo ciclo de vida del baneo, el TTL, la histéresis de desbaneo y el límite
`max_active_bans` que se describen a continuación.

## Configuración de BGP

El bloque `bgp` define la identidad BGP de Kapkan, los next-hops de blackhole, la community RTBH y
los pares eBGP a los que anuncia.

```yaml
bgp:
  local_asn: 65000               # Kapkan's local ASN
  router_id: "198.51.100.1"      # must be a valid IPv4 address
  next_hop: "192.0.2.1"          # IPv4 discard next-hop
  next_hop6: "100::1"            # IPv6 discard next-hop (optional)
  community: "65000:666"         # RTBH community, ASN:value
  # communities: ["65000:666", "65000:777"]  # full set; overrides `community`
  # local_pref: 100              # optional LOCAL_PREF for iBGP peers; 0 = omit
  neighbors:
    - address: "198.51.100.254"
      remote_asn: 65001
      port: 179                  # optional; defaults to 179, override for testing
```

| Clave | Significado |
| --- | --- |
| `local_asn` | El ASN local de Kapkan para las sesiones eBGP. Obligatorio. |
| `router_id` | El ID de router BGP. Debe ser una dirección **IPv4** válida. |
| `next_hop` | El next-hop de descarte IPv4 anunciado con cada blackhole `/32`. |
| `next_hop6` | El next-hop de descarte IPv6 anunciado con cada blackhole `/128`. Opcional; cuando no se establece, Kapkan recurre a `100::1` (el prefijo de descarte de la RFC 6666). |
| `community` | La community RTBH en formato `ASN:value` (por ejemplo `65000:666`). Se analiza una sola vez al cargar y se convierte en el valor de cable enviado con cada ruta. |
| `communities` | Lista opcional de communities que **sobreescribe** `community` cuando se establece, para upstreams que esperan un conjunto completo. |
| `local_pref` | Atributo de ruta `LOCAL_PREF` opcional adjuntado a los blackholes (significativo para los pares iBGP). El valor predeterminado `0` lo omite. |
| `neighbors[]` | Los pares eBGP a los que anuncia Kapkan. Cada entrada toma una `address`, un `remote_asn` y un `port` opcional (predeterminado `179`, destinado a pruebas). |

`community`, `communities`, `next_hop`, `next_hop6` y `local_pref` pueden sobreescribirse todos por
hostgroup — consulta [Atributos BGP por hostgroup](#atributos-bgp-por-hostgroup).

La ruta de blackhole que Kapkan construye para un objetivo se lee como `<prefix> next-hop <next_hop> community
<community>` — exactamente la cadena que aparece en el campo `route` del [objeto de baneo](#el-objeto-de-baneo).
Los objetivos IPv6 usan `next_hop6` (o el valor de reserva `100::1`) en lugar de `next_hop`.

<Callout type="info" title="El peering ocurre en dry-run">
El hablante BGP establece peering con tus vecinos incluso mientras `dry_run` es true; solo los
anuncios de rutas están condicionados por la bandera. Esto te permite confirmar que las sesiones
alcanzan `ESTABLISHED` (registrado como `bgp peer state`) y validar la conectividad antes de
anunciar una sola ruta.
</Callout>

## Atributos BGP por hostgroup

El bloque `bgp` global establece los next-hops de blackhole y la community RTBH predeterminados. Un
[hostgroup](/docs/hostgroups) puede sobreescribir cualquiera de ellos, de modo que distintos
clientes señalicen sus propios upstreams:

```yaml
bgp:
  next_hop: "192.0.2.1"
  community: "65000:666"                          # the default RTBH community

hostgroups:
  - name: customer-a
    networks: ["203.0.113.64/26"]
    bgp:
      communities: ["65000:100", "65001:200"]     # customer-A's own blackhole signal
      next_hop: "192.0.2.50"                       # and discard next-hop
      local_pref: 250
```

Cada campo que se deja sin establecer hereda el valor `bgp` global, de modo que un grupo puede
sobreescribir solo su community mientras comparte el next-hop global. Los atributos resueltos se
**congelan en cada baneo cuando este se crea**: una recarga de configuración cambia únicamente los
baneos futuros, nunca la ruta que un baneo activo ya anunció. Los `next_hop`, `community` y
`local_pref` por baneo son visibles en `/api/v1/bans` y en el campo `route`.

## El ciclo de vida del baneo

Un baneo es la unidad que Kapkan rastrea para cada decisión de blackhole. Su ciclo de vida está
acotado en cada paso por el [modelo de seguridad](/docs/safety) — no existe ninguna vía hacia un
baneo permanente o descontrolado.

1. **Anuncio al detectar.** Cuando el motor reporta un nuevo ataque por host y la política del host
   permite el baneo, Kapkan anuncia la ruta de blackhole (o, en dry-run, registra la ruta que se
   habría anunciado) y el baneo entra en el estado `active`. Un segundo ataque — por ejemplo una
   inundación entrante y otra saliente sobre el mismo host a la vez — comparte la única ruta RTBH en
   lugar de anunciarla dos veces.
2. **Retirada automática por TTL.** Cada anuncio lleva un TTL. Después de `ban.ttl_seconds` la ruta
   se retira automáticamente incluso si el ataque sigue en curso; un ataque que persiste refresca el
   TTL, pero nunca más allá de un TTL nuevo a partir de ahora. **No hay baneos permanentes.**
3. **Desbaneo con histéresis.** Un baneo impulsado por un ataque se retira solo después de que el
   tráfico permanezca por debajo del umbral durante `ban.unban_hysteresis_seconds`. Este retardo
   anti-oscilación evita que un ataque límite anuncie y retire rápidamente la misma ruta.
4. **Límite estricto.** `ban.max_active_bans` es un techo absoluto para los baneos activos
   simultáneos. Una vez superado el límite, los nuevos baneos se **rechazan** (se devuelven como un
   baneo en el estado `rejected` con la razón `max_active_bans reached`) y se alerta a los operadores,
   de modo que Kapkan nunca pueda someter a blackhole media red en un ataque amplio.

```yaml
ban:
  ttl_seconds: 1800            # every announcement auto-withdraws after this
  unban_hysteresis_seconds: 60 # traffic must stay below threshold this long before withdraw
  max_active_bans: 100         # hard cap on simultaneous bans
```

Un baneo también se retira anticipadamente si una recarga de configuración elimina su objetivo de
las `networks` protegidas — Kapkan no mantendrá una ruta activa para un espacio de direcciones que
ya no protege.

## Reserva cuando se rechaza un método

Los upstreams varían en lo que aceptan — un par puede rechazar de plano un anuncio FlowSpec o de
divert. De forma predeterminada Kapkan no deja a la víctima indefensa en ese caso: `ban.fallback` es
`blackhole`, de modo que un anuncio `flowspec`/`divert` rechazado **degrada a una ruta RTBH** en lugar
de hacer fallar el baneo. Establece `ban.fallback: none` para desactivarlo y rechazar el baneo en su
lugar.

```yaml
ban:
  fallback: blackhole   # degrade a rejected flowspec/divert announce to RTBH (default); none disables
```

Cuando ocurre una reserva, el baneo registra `fell_back_from` (el método que el par rechazó) y la
[métrica](/docs/metrics) `kapkan_mitigate_fallback_total` se incrementa — una serie `from="flowspec"`
distinta de cero señala los upstreams que no aceptan FlowSpec. Una caída total de BGP hace fallar
ambos métodos, por lo que el baneo se rechaza igual que antes; la reserva solo cambia el caso en que
específicamente el método quirúrgico no está soportado.

## Dry-run

En modo dry-run (`dry_run: true`, el valor predeterminado) Kapkan hace todo *excepto* enviar la
ruta. El prefijo objetivo, el next-hop, la community y la cadena `route` completa se calculan, la
decisión se registra como `DRY-RUN: would announce blackhole route (not sent)`, y el baneo se
rastrea en la API con `dry_run: true` y los estados en vivo `active`/`withdrawn`. La expiración del
TTL y la histéresis se ejecutan exactamente igual que en producción, por lo que todo el ciclo de
vida es observable antes de que cualquier ruta llegue a un router.

No se anuncia nada hasta que estableces `dry_run: false` y recargas (`SIGHUP` o
`POST /api/v1/config/reload`). Como [el peering ocurre en dry-run](#configuración-de-bgp), el único
comportamiento que cambia al alternar la bandera es el propio anuncio/retirada.

<Callout type="info" title="Valida primero">
Ejecuta frente a la telemetría de producción en dry-run, confirma que las detecciones se disparan
sobre los prefijos correctos y que los campos `route` son correctos, luego desactiva dry-run. El
procedimiento completo está en [Puesta en producción](/docs/going-live).
</Callout>

## Baneos manuales

Los operadores pueden someter a blackhole o liberar un host directamente a través de la
[API REST](/docs/api):

```http
POST /api/v1/ban
Content-Type: application/json

{"ip": "203.0.113.66"}
```

```http
POST /api/v1/unban
Content-Type: application/json

{"ip": "203.0.113.66"}
```

Un baneo manual se crea con `manual: true` y sigue el mismo TTL e histéresis que uno automático.
Respeta todas las reglas de seguridad sin excepciones: un objetivo en lista blanca se rechaza
(devuelve `409`, razón `whitelisted`), un objetivo fuera de las `networks` configuradas se rechaza
(devuelve `409`, razón `outside configured networks`), y superar `max_active_bans` se rechaza
(devuelve `409`). Ambos endpoints POST requieren el tipo de contenido `application/json`. Consulta la
[referencia de la API](/docs/api) para conocer las formas completas de solicitud y respuesta.

## El objeto de baneo

La API devuelve cada baneo como un objeto JSON. Estos son los campos exactamente tal como los
serializa el mitigador:

| Campo | Tipo | Significado |
| --- | --- | --- |
| `target` | string | La dirección del host sometido a blackhole (IPv4 o IPv6). |
| `prefix` | string | El prefijo de host anunciado: `/32` para IPv4, `/128` para IPv6. |
| `metric` | string | La métrica que disparó el baneo impulsado por el ataque (se omite en los baneos manuales). |
| `rate` | number | La tasa observada en el momento del baneo, en las unidades de la métrica (se omite cuando es cero). |
| `threshold` | number | El umbral que la tasa cruzó (se omite cuando es cero). |
| `next_hop` | string | El next-hop de descarte anunciado con la ruta. |
| `community` | string | La community RTBH en formato `ASN:value`. |
| `local_pref` | number | El `LOCAL_PREF` adjuntado a la ruta (se omite cuando es cero). |
| `route` | string | La ruta completa como `<prefix> next-hop <next_hop> community <community>`, o un resumen `flowspec: ...` para baneos FlowSpec. |
| `state` | string | Uno de `active`, `withdrawn` o `rejected`. |
| `dry_run` | boolean | Si el baneo se realizó en modo dry-run (la ruta no se envió realmente). |
| `manual` | boolean | `true` para baneos solicitados por el operador, `false` para los automáticos. |
| `started_at` | timestamp | Cuándo se creó el baneo. |
| `expires_at` | timestamp | Cuándo el TTL retira automáticamente la ruta. |
| `withdrawn_at` | timestamp | Cuándo se retiró la ruta (se omite mientras está activa). |
| `reason` | string | Por qué se retiró o se rechazó el baneo (se omite mientras está activa). |
| `method` | string | El método de mitigación que produjo el baneo: `blackhole`, `flowspec` o `divert`. |
| `flowspec` | array | Las reglas [FlowSpec](/docs/flowspec) generadas, cuando el método es `flowspec`. |
| `escalation` | array | La [escalera de escalado](/docs/escalation) configurada, cuando hay una establecida. |
| `escalation_step` | number | El índice del peldaño actual de la escalera. |

Un baneo `rejected` nunca se anuncia — registra un rechazo (en lista blanca, fuera de las `networks`
o límite alcanzado) para que la decisión sea auditable en la API y en las notificaciones.
