# Introducción

**Kapkan es una solución gratuita y de código abierto de detección de DDoS (denegación de servicio distribuida) y mitigación mediante [RTBH](/docs/glossary) para ISP y proveedores de hosting.**

Es un único binario en Go que ingiere telemetría de flujos — NetFlow v5/v9, IPFIX y sFlow v5 —
desde tus routers, detecta en segundos los ataques volumétricos contra los [prefijos](/docs/glossary) (rangos
de IP en forma CIDR, p. ej. `203.0.113.0/24`) que proteges y activa una mitigación
[BGP](/docs/glossary) (Border Gateway Protocol — cómo los routers intercambian rutas)
[RTBH](/docs/glossary) (remotely-triggered blackhole, agujero negro activado de forma remota)
automatizada. Incluye una API REST, un panel web integrado, métricas de Prometheus y
notificaciones por Telegram / Slack / email / webhook / exec-hook.

Kapkan es un reemplazo gratuito de las funciones por las que cobran los productos comerciales de DDoS basados en flow.

<Callout type="info" title="¿Nuevo en BGP, RTBH o NetFlow?">
No pasa nada — el [Inicio rápido](/docs/quickstart) te guía paso a paso, el
[Glosario](/docs/glossary) explica cada término y Kapkan no tocará ni una sola ruta hasta
que tú lo indiques. Kapkan gestiona ataques *volumétricos* (inundaciones que intentan saturar tu enlace o
host con volumen bruto de paquetes y bits), no ataques de capa de aplicación.
</Callout>

<Callout type="warning" title="Dry-run de forma predeterminada">
Hasta que establezcas explícitamente `dry_run: false`, cada posible blackhole se registra y se expone
a través de la API, pero **nunca se anuncia** a tus routers. Puedes ejecutar Kapkan contra
telemetría de producción y validar la detección antes de que pueda tocar una sola ruta. Consulta el
[Modelo de seguridad](/docs/safety).
</Callout>

## Qué hace

- **Ingiere** — sFlow v5, NetFlow v5/v9 e IPFIX sobre UDP, en modo biblioteca (sin sidecar),
  mediante [goflow2](https://github.com/netsampler/goflow2).
- **Detecta** — ataques volumétricos por destino utilizando umbrales de pps / Mbps /
  flows por segundo corregidos por muestreo sobre una ventana deslizante, además de límites
  opcionales por protocolo, [líneas base aprendidas](/docs/baselines) y detección de ataques salientes.
- **Clasifica** — cada ataque se etiqueta a partir de su muestra de flow: amplificación
  (NTP / DNS / CLDAP / memcached / SSDP / chargen), inundaciones SYN / UDP / TCP / ICMP / de fragmentos.
- **Mitiga** — anuncia rutas blackhole `/32` y `/128` mediante un speaker
  [GoBGP](https://github.com/osrg/gobgp) integrado, descarta únicamente el vector de ataque con
  reglas quirúrgicas de [BGP FlowSpec](/docs/flowspec) (RFC 8955/8956) o desvía a la víctima a un
  [centro de limpieza (scrubbing)](/docs/scrubbing) en lugar de descartar su tráfico — escalado opcionalmente
  con el tiempo mediante [escalado progresivo](/docs/escalation). Kapkan registra cada decisión de
  mitigación como un **baneo**; para el método predeterminado, un baneo *es* una ruta blackhole (RTBH).
- **Observa** — una API REST, un panel web integrado, `/metrics` de Prometheus y
  notificaciones por Telegram, Slack, email, webhook y un exec hook.
- **Persiste (opcional)** — conserva el historial de ataques y tráfico en
  [ClickHouse](/docs/storage) para análisis forense y generación de informes.

## Para quién está pensado

Operadores de red — ISP, proveedores de hosting y cualquiera que ejecute BGP — que necesiten
mitigación automatizada de ataques volumétricos sin una licencia comercial (incluidos los usuarios de
FastNetMon Community que buscan una alternativa de código abierto).

## Por qué Kapkan

| | Kapkan |
| --- | --- |
| **Licencia** | Apache 2.0, totalmente de código abierto — incluida la lógica de detección |
| **Huella** | Un único binario estático en Go; sin MongoDB, sin servicio web aparte |
| **Interfaz web** | Panel integrado incluido, no un complemento de pago |
| **Configuración** | YAML declarativo que puedes guardar en git y comparar con diff |
| **Seguridad** | Dry-run de forma predeterminada; TTL, histéresis y límites de baneo aplicados en el código |

## Qué no hace (todavía)

Esto está en la hoja de ruta
pero **no** en la versión actual:

- Deduplicación de alta disponibilidad integrada entre instancias
- Captura por port-mirror (SPAN / AF_PACKET / XDP) como fuente de ingesta
- Enrutamiento de notificaciones por tenant (hoy las notificaciones son globales; la acotación por tenant se aplica solo a la [API](/docs/multi-tenancy))
