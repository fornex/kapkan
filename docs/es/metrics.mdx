# Métricas

Kapkan expone métricas de Prometheus bajo el espacio de nombres `kapkan_` en `GET /metrics` en el
listener de la API (`api.listen`, por defecto `127.0.0.1:8080`). Cubren todo el pipeline —
ingesta de flows, el motor de detección, la mitigación RTBH y las notificaciones — para que puedas
recopilar (scrape) el throughput, la latencia, los ataques activos y las rutas anunciadas en tu
monitorización existente.

<Callout type="info" title="/metrics permanece abierto">
El endpoint `/metrics` siempre está sin autenticar, incluso cuando configuras un token de API con
`api.token_env`. Solo los endpoints de datos `/api/v1` requieren `Authorization: Bearer <token>`.
Vincula (bind) el listener en consecuencia — consulta [Autenticación](/docs/authentication).
</Callout>

## Métricas disponibles

| Métrica | Etiquetas | Significado |
| --- | --- | --- |
| `kapkan_ingest_flows_total` | `proto` | Total de flows ingeridos, por protocolo de flow. |
| `kapkan_ingest_packets_total` | `exporter`, `proto` | Total de datagramas de telemetría recibidos, por exportador de origen (con cardinalidad acotada; consulta [`flow_sources`](/docs/configuration)) y protocolo. |
| `kapkan_ingest_decode_errors_total` | `proto` | Datagramas de telemetría que no se pudieron decodificar, por protocolo. |
| `kapkan_ingest_dropped_flows_total` | — | Flows descartados porque la cola de entrada del motor estaba llena. |
| `kapkan_engine_active_attacks` | — | Gauge de ataques actualmente en curso. |
| `kapkan_engine_attacks_total` | — | Contador de ataques detectados desde el arranque. |
| `kapkan_engine_process_latency_seconds` | — | Histograma de la latencia de procesamiento en la ruta crítica (hot-path), observada una vez por lote de ingesta (no por flow). |
| `kapkan_engine_tracked_hosts` | — | Gauge de hosts actualmente rastreados por el motor. |
| `kapkan_mitigate_announced_routes` | `mode` | Gauge de rutas de blackhole anunciadas actualmente, separadas por el modo `real` / `dry_run`. |
| `kapkan_mitigate_flowspec_rules` | `mode` | Gauge de reglas de [FlowSpec](/docs/flowspec) anunciadas actualmente, separadas por el modo `real` / `dry_run`. Vigílalo frente al límite de rutas FlowSpec de tus routers. |
| `kapkan_mitigate_bans_rejected_total` | `reason` | Bloqueos (bans) rechazados por una regla de seguridad, por `reason`: `max_active_bans`, `blast_radius_fraction`, `blast_radius_rate`, o `max_active_prefix_bans` (carpet). |
| `kapkan_mitigate_fallback_total` | `from`, `to` | Anuncios de mitigación que degradaron a un método de fallback porque el peer rechazó el primario, por `from`/`to`. Una serie con `from="flowspec"` distinta de cero señala upstreams que no respetan FlowSpec. |
| `kapkan_notify_notifications_total` | `channel`, `result` | Notificaciones intentadas, por canal (telegram, slack, email, webhook, exec) y `result`. |
| `kapkan_storage_rows_total` | `table`, `result` | Filas entregadas al [almacenamiento opcional en ClickHouse](/docs/storage), por tabla y `written` / `dropped` / `error`. |
| `kapkan_build_info` | `version`, `revision`, `goversion`, `goos`, `goarch` | Gauge informativo constante en `1` que lleva el build en ejecución en sus etiquetas (el idiom de `node_exporter`). Consulta la dispersión de versiones de la flota con `count by (version)(kapkan_build_info)` — sin phone-home. |
| `kapkan_update_available` | `latest_version`, `security` | `1` cuando la comprobación opcional [`update_check`](/docs/configuration) encuentra una versión más reciente (ausente en caso contrario); `security="true"` señala una versión relevante para la seguridad. |

La etiqueta `mode` en `kapkan_mitigate_announced_routes` es la forma más rápida de confirmar el estado
de dry-run de un vistazo: mientras `dry_run: true`, cada ruta que se habría anunciado se cuenta bajo
`mode="dry_run"` y nada bajo `mode="real"`. Consulta [Mitigación](/docs/mitigation) y el
[Modelo de seguridad](/docs/safety).

## Scraping

Añade Kapkan como destino de scrape en tu configuración de Prometheus:

```yaml
scrape_configs:
  - job_name: kapkan
    metrics_path: /metrics
    static_configs:
      - targets: ["127.0.0.1:8080"]
```

Usa el host y el puerto de tu valor `api.listen`. Si expones el listener más allá de
localhost, configura un token de API — `/metrics` sigue siendo accesible sin él, por lo que tu scraper no
necesita credenciales.

Para inspeccionar la salida en bruto directamente:

```sh
curl -s localhost:8080/metrics | grep kapkan_
```

## Grafana dashboard

Un dashboard oficial de Grafana para el conjunto `kapkan_*` está disponible para descargar directamente en
[`kapkan-overview.json`](/kapkan-overview.json) (su código fuente reside en el repositorio bajo
[`engine/deploy/grafana/`](https://github.com/fornex/kapkan/tree/main/engine/deploy/grafana)).
Cubre todo el pipeline — tasas de ingesta de flows/datagramas/decodificación y flows descartados, ataques
activos e iniciados, hosts rastreados, cuantiles de latencia en la ruta crítica, rutas anunciadas y reglas
FlowSpec por `mode`, bloqueos rechazados por `reason`, fallbacks de mitigación, y resultados de notificación
y almacenamiento.

En Grafana, ve a **Dashboards → New → Import**, sube el JSON y elige tu fuente de datos
Prometheus cuando se te solicite (está parametrizada como `DS_PROMETHEUS`). El dashboard se dirige a
Grafana 10+ y está verificado para importación contra Grafana 11. Consulta el
[README](https://github.com/fornex/kapkan/blob/main/engine/deploy/grafana/README.md) junto a
él para el aprovisionamiento.
