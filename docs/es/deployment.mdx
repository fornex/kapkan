# Despliegue en producción

En el directorio [`engine/deploy/`](https://github.com/fornex/kapkan/tree/main/engine/deploy) del repositorio
se incluyen una unidad systemd reforzada y un ejemplo de configuración de producción:
`engine/deploy/kapkan.service` y `engine/deploy/config.example.yaml`. Esta página explica cómo instalar
el binario como servicio del sistema, dónde reside cada archivo, cómo mantener los secretos
fuera del YAML y cómo recargar la configuración sin reiniciar.

<Callout type="info" title="Compila primero">
Los pasos siguientes asumen que ya dispones de un binario `kapkan`. Si todavía no lo has
compilado, consulta la [Guía rápida](/docs/quickstart) o [Compilar desde el código fuente](#compilar-desde-el-código-fuente).
</Callout>

## Instalación

Ejecuta lo siguiente como root desde una copia de trabajo del repositorio (para que
`engine/deploy/` esté disponible). Instala el binario, crea un usuario del sistema sin privilegios,
prepara `/etc/kapkan`, instala la configuración de producción, escribe el archivo de entorno
de secretos con permisos restrictivos, instala la unidad systemd y habilita el servicio.

```sh
sudo install -m 0755 kapkan /usr/local/bin/kapkan
sudo useradd --system --no-create-home --shell /usr/sbin/nologin kapkan
sudo install -d -o kapkan -g kapkan /etc/kapkan
sudo install -m 0640 -o kapkan -g kapkan engine/deploy/config.example.yaml /etc/kapkan/config.yaml
echo 'KAPKAN_API_TOKEN=a-long-random-string' | sudo install -m 0600 /dev/stdin /etc/kapkan/kapkan.env
sudo install -m 0644 engine/deploy/kapkan.service /etc/systemd/system/kapkan.service
sudo systemctl daemon-reload && sudo systemctl enable --now kapkan
```

El archivo de entorno anterior solo escribe `KAPKAN_API_TOKEN`, ya que se recomienda un token
de API antes de exponer el listener (consulta [Autenticación](/docs/authentication)). Escribir el
archivo de entorno no habilita por sí solo la autenticación: la configuración incluida enlaza
`127.0.0.1` con `api.token_env` comentado, por lo que el token permanece inactivo. Descomenta
`api.token_env` (o añade una lista `api.tokens`) para activar la autenticación. Todos los canales
de notificación son opcionales — añade `KAPKAN_TG_TOKEN`, las credenciales de SMTP o ClickHouse
solo si realmente usas esas funciones. La lista completa de claves `*_env` está más abajo en
[Secretos](#secretos).

La unidad se ejecuta como el usuario `kapkan` con `ExecStart=/usr/local/bin/kapkan -config
/etc/kapkan/config.yaml -log-format json -log-level info`, se reinicia ante fallos y está
fuertemente aislada en un sandbox: `NoNewPrivileges`, `ProtectSystem=strict`, `PrivateTmp`,
`ReadOnlyPaths=/etc/kapkan`, `RestrictAddressFamilies=AF_INET AF_INET6`,
`MemoryDenyWriteExecute` y más. También declara `StateDirectory=kapkan` (modo `0700`),
lo que proporciona al daemon un `/var/lib/kapkan` escribible para [`ban.state_file`](/docs/mitigation)
mientras que `ProtectSystem=strict` mantiene el resto del sistema de archivos en modo solo lectura.

<Callout type="info" title="Enlazar BGP al puerto 179">
Kapkan no necesita privilegios especiales en modo dry-run ni cuando establece el peering
desde un puerto local alto. Si enlazas el hablante BGP al puerto bien conocido 179, concede
la capacidad añadiendo `AmbientCapabilities=CAP_NET_BIND_SERVICE` a la sección `[Service]`
de la unidad.
</Callout>

Una vez en ejecución, comprueba el estado y sigue los registros:

```sh
sudo systemctl status kapkan
sudo journalctl -u kapkan -f
```

## Disposición de archivos

| Ruta | Modo | Propósito |
| --- | --- | --- |
| `/usr/local/bin/kapkan` | `0755` | El único binario estático del daemon. |
| `/etc/kapkan/config.yaml` | `0640` | La configuración YAML, propiedad de `kapkan:kapkan`. |
| `/etc/kapkan/kapkan.env` | `0600` | Archivo de entorno con secretos (tokens, credenciales SMTP). |
| `/etc/systemd/system/kapkan.service` | `0644` | La unidad systemd reforzada. |
| `/var/lib/kapkan/` | `0700` | Directorio de estado escribible (`StateDirectory=kapkan`); contiene `ban.state_file` cuando la persistencia de baneos está habilitada. |

Todo el directorio `/etc/kapkan` se monta en modo solo lectura para el proceso mediante
`ReadOnlyPaths=/etc/kapkan`, de modo que el daemon puede leer su configuración y sus secretos
pero no puede modificarlos.

## Secretos

Ninguna credencial debe ir en `config.yaml`. El YAML referencia cada secreto por el nombre de
una **variable de entorno**, y la unidad carga esas variables desde el archivo de entorno
mediante `EnvironmentFile=-/etc/kapkan/kapkan.env`. Mantén ese archivo en modo `0600`.

Las claves de configuración que leen del entorno son las claves `*_env`:

- `notify.telegram.token_env` — p. ej. `KAPKAN_TG_TOKEN`, el token del bot de Telegram.
- `notify.email.username_env` / `notify.email.password_env` — credenciales SMTP.
- `api.token_env` — el token bearer de la API (consulta [Autenticación](/docs/authentication)).
- `storage.clickhouse.username_env` / `password_env` — credenciales de ClickHouse, si
  habilitas el almacén de histórico opcional.

Por ejemplo, el YAML nombra la variable y el archivo de entorno proporciona su valor:

```yaml
notify:
  telegram:
    token_env: "KAPKAN_TG_TOKEN"   # name of the env var, never the token itself
    chat_id: "-1001234567890"
```

```text
# /etc/kapkan/kapkan.env  (mode 0600)
KAPKAN_TG_TOKEN=123456:abcdef
KAPKAN_API_TOKEN=a-long-random-string
```

<Callout type="success" title="Por qué esto importa">
Como los secretos residen únicamente en el entorno del proceso, tu `config.yaml` se puede
mantener y comparar (diff) de forma segura en git. El sandbox de systemd mantiene
`/etc/kapkan` en modo solo lectura y ejecuta el daemon como un usuario sin privilegios, por
lo que el archivo de entorno nunca es escribible por el servicio.
</Callout>

## Recarga de configuración

Kapkan recarga su configuración en caliente al recibir `SIGHUP`: sin reinicio y sin pérdida
del estado de flows. La unidad systemd conecta esto mediante `ExecReload=/bin/kill -HUP $MAINPID`,
de modo que puedes recargar con:

```sh
sudo systemctl reload kapkan
```

De forma equivalente, puedes desencadenar la misma recarga a través de la API:

```sh
curl -s -X POST localhost:8080/api/v1/config/reload \
  -H 'Content-Type: application/json' \
  -H "Authorization: Bearer $KAPKAN_API_TOKEN"
```

`Content-Type: application/json` es obligatorio en cada `POST` mutante (de lo contrario el
servidor devuelve `415`), y cuando la autenticación está habilitada la recarga necesita un token
Bearer sin ámbito (de administrador) — un token con ámbito de inquilino obtiene un `403`.

La mayoría de las claves se recargan en vivo, incluidas `dry_run`, los umbrales, `networks`,
los hostgroups, las líneas base y los ajustes de notificación. Algunos parámetros de
dimensionamiento requieren un reinicio, en particular el dimensionamiento del ring buffer
`samples.buffer_flows` / `flows_per_attack`.

## Compilar desde el código fuente

La compilación requiere **Go 1.22 o más reciente**. El `Makefile` ofrece los objetivos
estándar:

```sh
make build  # static binary
make test   # go test -race ./...
make lint   # golangci-lint run
make bench  # engine hot-path benchmarks
```

`make build` produce el único binario estático `kapkan` que instalas en los pasos anteriores.
Las pruebas usan datagramas NetFlow/sFlow sintéticos generados por `pkg/flowgen` (formato de
cable real) y un peer GoBGP en proceso, de modo que nunca se contacta con routers reales.

<Callout type="warning" title="Antes de exponer Kapkan">
Mantén `dry_run: true` hasta que hayas validado la detección contra telemetría en vivo;
consulta [Puesta en producción](/docs/going-live). Y establece un token de API antes de
enlazar el listener más allá de `127.0.0.1`, o de lo contrario la API y el panel quedarán
accesibles sin autenticación; consulta [Autenticación](/docs/authentication).
</Callout>
