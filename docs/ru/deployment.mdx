# Развёртывание в продакшене

Защищённый юнит systemd и пример продакшен-конфигурации поставляются в каталоге
[`engine/deploy/`](https://github.com/fornex/kapkan/tree/main/engine/deploy) репозитория: `engine/deploy/kapkan.service`
и `engine/deploy/config.example.yaml`. На этой странице описано, как установить бинарный файл как
системный сервис, где располагается каждый файл, как хранить секреты вне YAML и как
перезагрузить конфигурацию без перезапуска.

<Callout type="info" title="Сборка не требуется">
Самый быстрый путь — готовый пакет: `apt install ./kapkan_*.deb` (или `.rpm`) делает за вас
всё, что описано в этом разделе, — см. [Установка пакета](#install-the-package) ниже. Ручные
шаги после него предназначены для [release-tarball](/docs/quickstart#1-установка-kapkan). В любом
случае ничего компилировать не нужно; [Сборка из исходного кода](#building-from-source) — только
если вам этого хочется.
</Callout>

## Install

### Install the package

На Debian/Ubuntu (`.deb`) или RHEL/Fedora (`.rpm`) пакет делает всё то же, что и ручные шаги
ниже: устанавливает бинарный файл в `/usr/local/bin/kapkan`, создаёт непривилегированного
пользователя `kapkan`, формирует структуру `/etc/kapkan` с конфигурацией `config.yaml` в режиме
dry-run на основе примера, создаёт доступный для записи каталог состояния и устанавливает
защищённый юнит systemd (в `/usr/lib/systemd/system`).

```sh
VER=v1.0.0
curl -fLO "https://github.com/fornex/kapkan/releases/download/$VER/kapkan_${VER#v}_linux_amd64.deb"
sudo apt install "./kapkan_${VER#v}_linux_amd64.deb"   # RHEL/Fedora: dnf install ./kapkan_<ver>_linux_amd64.rpm
```

Сервис устанавливается, но **остаётся остановленным** — демон митигации не должен запускаться
прежде, чем вы посмотрите его конфигурацию. Поставляемая `config.yaml` содержит `dry_run: true`.
Просмотрите её, (рекомендуется) [задайте API-токен](#secrets) в `/etc/kapkan/kapkan.env`, затем
включите автозапуск и запустите его сейчас:

```sh
sudoedit /etc/kapkan/config.yaml          # adapt networks, BGP and thresholds
sudo systemctl enable --now kapkan
```

Обновляйтесь более новым пакетом (`apt install ./kapkan_<newer>.deb`) или через
[`update.sh`](/docs/upgrading) — оба сохраняют вашу отредактированную `/etc/kapkan/config.yaml`.
`apt purge kapkan` удаляет все следы, включая конфигурацию и состояние.

### Manual install (tarball)

Предпочитаете размещать файлы самостоятельно или работаете на дистрибутиве без `apt`/`dnf`? После
распаковки [release-tarball](/docs/quickstart#1-установка-kapkan) (чтобы каталог `deploy/` оказался
рядом с бинарником `kapkan`) выполните эквивалент от имени root. Он устанавливает бинарный файл,
создаёт непривилегированного системного пользователя, формирует структуру `/etc/kapkan`,
устанавливает продакшен-конфигурацию, записывает файл с секретами с ограничительными правами,
устанавливает юнит systemd и включает сервис.

```sh
sudo install -m 0755 kapkan /usr/local/bin/kapkan
sudo useradd --system --no-create-home --shell /usr/sbin/nologin kapkan
sudo install -d -o kapkan -g kapkan /etc/kapkan
sudo install -m 0640 -o kapkan -g kapkan deploy/config.example.yaml /etc/kapkan/config.yaml
echo 'KAPKAN_API_TOKEN=a-long-random-string' | sudo install -m 0600 /dev/stdin /etc/kapkan/kapkan.env
sudo install -m 0644 deploy/kapkan.service /etc/systemd/system/kapkan.service
sudo systemctl daemon-reload && sudo systemctl enable --now kapkan
```

В env-файл выше записывается только `KAPKAN_API_TOKEN`, поскольку API-токен рекомендуется
задать до того, как вы откроете доступ к слушателю (см. [Аутентификация](/docs/authentication)).
Запись env-файла сама по себе не включает аутентификацию: поставляемая конфигурация привязывается
к `127.0.0.1` с закомментированным `api.token_env`, поэтому токен остаётся неактивным.
Раскомментируйте `api.token_env` (или добавьте список `api.tokens`), чтобы включить
аутентификацию. Все каналы уведомлений опциональны — добавляйте `KAPKAN_TG_TOKEN`, учётные данные
SMTP или ClickHouse только если действительно используете эти возможности. Полный список ключей
`*_env` приведён ниже в разделе [Секреты](#secrets).

Юнит работает от имени пользователя `kapkan` с `ExecStart=/usr/local/bin/kapkan -config
/etc/kapkan/config.yaml -log-format json -log-level info`, перезапускается при сбое и сильно
изолирован: `NoNewPrivileges`, `ProtectSystem=strict`, `PrivateTmp`,
`ReadOnlyPaths=/etc/kapkan`, `RestrictAddressFamilies=AF_INET AF_INET6`,
`MemoryDenyWriteExecute` и другие настройки. Он также объявляет `StateDirectory=kapkan`
(режим `0700`), предоставляя демону доступный для записи `/var/lib/kapkan` для
[`ban.state_file`](/docs/mitigation), в то время как `ProtectSystem=strict` оставляет
остальную часть файловой системы доступной только для чтения. Защита от цикла перезапусков
(`StartLimitIntervalSec=60`, `StartLimitBurst=4`) переводит юнит в состояние `failed`, если
бинарный файл не запускается четыре раза за минуту, поэтому неудачное обновление становится
заметным, а не зацикливается — [скрипт обновления](/docs/upgrading) опирается на это вместе с
проверкой готовности `/healthz`.

<Callout type="info" title="Привязка BGP к порту 179">
Kapkan не требует особых привилегий в режиме dry-run или при пиринге с высокого локального порта.
Если вы привязываете BGP-спикер к общеизвестному порту 179, предоставьте соответствующую
возможность, добавив `AmbientCapabilities=CAP_NET_BIND_SERVICE` в секцию `[Service]` юнита.
</Callout>

После запуска проверьте статус и следите за логами:

```sh
sudo systemctl status kapkan
sudo journalctl -u kapkan -f
```

## File layout

| Путь | Режим | Назначение |
| --- | --- | --- |
| `/usr/local/bin/kapkan` | `0755` | Единственный статический бинарный файл демона. |
| `/etc/kapkan/config.yaml` | `0640` | Конфигурация в YAML, владелец `kapkan:kapkan`. |
| `/etc/kapkan/kapkan.env` | `0600` | Файл секретов в окружении (токены, учётные данные SMTP). |
| `/etc/systemd/system/kapkan.service` | `0644` | Защищённый юнит systemd. Установка из `.deb`/`.rpm` помещает его в `/usr/lib/systemd/system/kapkan.service` (расположение поставщика). |
| `/var/lib/kapkan/` | `0700` | Доступный для записи каталог состояния (`StateDirectory=kapkan`); хранит `ban.state_file`, когда включено сохранение банов. |

Весь каталог `/etc/kapkan` монтируется для процесса только для чтения через
`ReadOnlyPaths=/etc/kapkan`, поэтому демон может читать свою конфигурацию и секреты, но не может
их изменять.

## Secrets

Никакие учётные данные не должны находиться в `config.yaml`. YAML ссылается на каждый секрет по
имени **переменной окружения**, а юнит загружает эти переменные из файла окружения через
`EnvironmentFile=-/etc/kapkan/kapkan.env`. Держите этот файл в режиме `0600`.

Ключи конфигурации, читающие значения из окружения, — это ключи `*_env`:

- `notify.telegram.token_env` — например `KAPKAN_TG_TOKEN`, токен бота Telegram.
- `notify.email.username_env` / `notify.email.password_env` — учётные данные SMTP.
- `api.token_env` — bearer-токен API (см. [Аутентификацию](/docs/authentication)).
- `storage.clickhouse.username_env` / `password_env` — учётные данные ClickHouse, если вы
  включаете опциональное хранилище истории.

Например, YAML задаёт имя переменной, а файл окружения предоставляет её значение:

```yaml
notify:
  telegram:
    token_env: "KAPKAN_TG_TOKEN"   # name of the env var, never the token itself
    chat_id: "-1001234567890"
```

```text
# /etc/kapkan/kapkan.env  (mode 0600)
KAPKAN_TG_TOKEN=123456:abcdef
KAPKAN_API_TOKEN=a-long-random-string
```

<Callout type="success" title="Почему это важно">
Поскольку секреты находятся только в окружении процесса, ваш `config.yaml` можно безопасно хранить
в git и сравнивать через diff. Песочница systemd оставляет `/etc/kapkan` доступным только для
чтения и запускает демон от имени непривилегированного пользователя, поэтому файл окружения никогда
не доступен сервису для записи.
</Callout>

## Config reload

Kapkan перезагружает конфигурацию «на лету» по сигналу `SIGHUP` — без перезапуска и без потери
состояния flow. Юнит systemd настраивает это через `ExecReload=/bin/kill -HUP $MAINPID`, поэтому
перезагрузка выполняется так:

```sh
sudo systemctl reload kapkan
```

Эквивалентно, ту же перезагрузку можно запустить через API:

```sh
curl -s -X POST localhost:8080/api/v1/config/reload \
  -H 'Content-Type: application/json' \
  -H "Authorization: Bearer $KAPKAN_API_TOKEN"
```

`Content-Type: application/json` обязателен для каждого изменяющего `POST` (иначе сервер
возвращает `415`), а при включённой аутентификации перезагрузка требует bearer-токена без
ограничения области (admin) — токен, ограниченный тенантом, получает `403`.

Большинство ключей перезагружается без перезапуска, включая `dry_run`, пороги, `networks`,
hostgroups, базовые линии (baselines), настройки уведомлений и `sampling.default_rate`. Шесть
параметров фиксируются при запуске, и перезагрузка, изменяющая любой из них, отклоняется
(требуется перезапуск): адреса `listen`, идентичность BGP (`bgp.local_asn` / `router_id`),
`api.listen`, настройки `samples`, настройки `storage` и настройки `geoip`. См.
[справочник по конфигурации](/docs/configuration), а для смены бинарного файла без потери
митигации — [Обновление](/docs/upgrading).

## Building from source

Для сборки требуется **Go 1.22 или новее**. `Makefile` предоставляет стандартные цели:

```sh
make build  # static binary
make test   # go test -race ./...
make lint   # golangci-lint run
make bench  # engine hot-path benchmarks
```

`make build` создаёт единственный статический бинарный файл `kapkan`, который вы устанавливаете на
шагах выше. Тесты используют синтетические датаграммы NetFlow/sFlow, формируемые `pkg/flowgen`
(реальный формат на проводе), и внутрипроцессный пир GoBGP, поэтому к настоящим маршрутизаторам
обращений никогда не происходит.

<Callout type="warning" title="Прежде чем открывать доступ к Kapkan">
Держите `dry_run: true`, пока не проверите детектирование на живой телеметрии — см.
[Запуск в боевом режиме](/docs/going-live). И задайте токен API, прежде чем привязывать слушатель
за пределами `127.0.0.1`, иначе API и дашборд будут доступны без аутентификации — см.
[Аутентификацию](/docs/authentication).
</Callout>
