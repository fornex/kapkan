# Справочник конфигурации

<Callout type="info" title="Нужна пошаговая форма?">
[Конструктор конфигурации](/config) соберёт корректный конфиг прямо в браузере — с проверкой
полей и безопасными значениями по умолчанию (dry-run включён). Ничего из введённого не покидает страницу.
</Callout>

Kapkan настраивается одним YAML-файлом, который передаётся через `-config`. В репозитории есть
две отправные точки: [`configs/dev.yaml`](/docs/quickstart) для локальной разработки (тестовые
префиксы, привязка к loopback, `dry_run: true`) и [`engine/deploy/config.example.yaml`](/docs/deployment)
для промышленной среды — в нём собраны авторитетные встроенные комментарии и значения по умолчанию.
Скопируйте этот пример в `/etc/kapkan/config.yaml` и адаптируйте его под свою сеть.

<Callout type="info" title="Впервые сталкиваетесь с BGP / RTBH / FlowSpec?">
Эта страница предполагает, что вы знаете, что означают эти термины. Если нет — сначала прочитайте
[Как это работает](/docs/how-it-works) или держите [глоссарий](/docs/glossary) открытым в соседней
вкладке: каждый акроним ниже там определён. Коротко: **BGP** (Border Gateway Protocol) — это то, как
маршрутизаторы обмениваются маршрутами, а Kapkan выступает в роли BGP-спикера, который сообщает вашим
маршрутизаторам, какие маршруты установить, чтобы подавить атаку.
</Callout>

Файл можно перечитывать во время работы. Отправьте `SIGHUP` (`systemctl reload kapkan`) или
`POST /api/v1/config/reload` — и Kapkan перечитает файл и применит новые настройки, не прерывая
приём телеметрии потоков. Почти всё перечитывается на лету, включая `dry_run`, пороги, `networks`,
hostgroup, базовые линии (baseline), настройки уведомлений и `sampling.default_rate`. Шесть
вещей фиксируются при запуске, и перезагрузка, меняющая любую из них, **отклоняется** (а не молча
игнорируется), так что для их изменения нужно перезапустить демон:

- адреса `listen`;
- идентичность BGP (`bgp.local_asn`, `bgp.router_id`);
- `api.listen`;
- настройки `samples`;
- настройки `storage`;
- настройки `geoip`.

<Callout type="warning" title="Dry-run по умолчанию включён">
`dry_run` по умолчанию равен `true`, а отсутствие ключа `dry_run` трактуется как `true`. Оставляйте
это значение, пока не проверите обнаружение на живой телеметрии. См. [Запуск в боевом режиме](/docs/going-live).
</Callout>

## Проверка конфигурации

Перед перезагрузкой или запуском в бой проверьте файл собственным средством проверки движка:

```sh
kapkan -check-config /etc/kapkan/config.yaml
```

Он выполняет ровно тот же разбор и проверку, что и демон, — включая кросс-полевые правила, которые
форма выразить не может (никаких пересекающихся `networks`, префиксы hostgroup внутри защищаемой сети,
монотонные лестницы [эскалации](/docs/escalation)) — затем печатает разрешённую конфигурацию (режим,
защищаемые сети и эффективное подавление по каждой [hostgroup](/docs/hostgroups)) и завершается с кодом
`0`. При любой проблеме он печатает точную ошибку и завершается с кодом `1`, так что легко встраивается
в CI или предрелизную проверку.

`kapkan -dump-schema` печатает JSON-схему конфигурации в stdout; [Конструктор конфигурации](/config)
использует эту схему, чтобы проверять поля прямо в браузере по мере ввода. Оба флага, как и все
остальные, которые принимает бинарник, перечислены в [Справочнике CLI](/docs/cli).

## Секреты и переменные окружения

Секреты никогда не пишутся в файл конфигурации. Каждый из них указывает на переменную окружения,
и Kapkan читает значение из окружения процесса при загрузке:

- `notify.telegram.token_env` — токен бота Telegram.
- `notify.email.username_env` / `notify.email.password_env` — учётные данные SMTP.
- `storage.clickhouse.username_env` / `password_env` — учётные данные ClickHouse.
- `api.token_env` (или каждый `api.tokens[].token_env`) — bearer-токен(ы) REST API / панели управления.

В systemd-юните они находятся в `EnvironmentFile`, например `/etc/kapkan/kapkan.env`,
с правами `0600`:

```sh
KAPKAN_TG_TOKEN=123456:abc...
KAPKAN_API_TOKEN=a-long-random-string
```

## Ключи верхнего уровня

Каждый ключ верхнего уровня, его смысл и значение по умолчанию. Для более крупных подсистем
однострочная сводка ведёт на отдельную страницу.

<Callout type="info" title="Минимальная конфигурация для запуска">
Большинство ключей ниже опциональны. Чтобы демон вообще запустился, нужно лишь: хотя бы один адрес
`listen`, `sampling.default_rate` (`>= 1`), хотя бы один защищаемый префикс в `networks` и
`thresholds.pps` / `.mbps` / `.flows_per_sec` (все `> 0`). По умолчанию `api.listen` равен
`127.0.0.1:8080`, так что его можно опустить. Чтобы реально **анонсировать** blackhole (то есть после
того, как вы отключите `dry_run`), вам также понадобится блок `bgp`. Всё остальное — `carpet`, `baseline`,
`flowspec.source_anchored`, `flow_sources`, `geoip`, `escalation` — это опциональное уточнение.
</Callout>

| Ключ | Смысл |
| --- | --- |
| `dry_run` | Когда `true` (значение по умолчанию, в том числе при отсутствии ключа), blackhole логируются и отслеживаются, но **никогда не анонсируются**. |
| `listen.sflow` / `listen.netflow` | UDP-адреса прослушивания. NetFlow v5/v9 и IPFIX используют общий сокет `netflow`. Нужен хотя бы один слушатель. См. [Как это работает](/docs/how-it-works). |
| `sampling.default_rate` | Частота выборки, применяемая, когда экспортёр не сообщает свою. Должна быть `>= 1`. |
| `sampling.boundary` | Опциональный подсчёт по границе интерфейсов на каждый экспортёр, который дедуплицирует поток, наблюдаемый в нескольких точках выборки (избыточные экспортёры, выборка на ingress+egress, транзитные каналы). Каждая запись задаёт `external_ifindexes` экспортёра (и опциональный `egress_sampling`); тогда поток учитывается только при пересечении внешнего интерфейса. Экспортёры без записи учитывают каждую выборку. См. «Коррекция выборки» ниже. |
| `sampling.boundary_debug` | Когда `true`, экспортирует `kapkan_engine_boundary_debug_bytes_total` (байты на экспортёр и интерфейс), чтобы помочь определить внешние интерфейсы. Кардинальность не ограничена — включайте ненадолго, затем отключайте. |
| `flow_sources` | Опциональный белый список доверенных адресов-источников экспортёров. Телеметрия приходит по неаутентифицированному UDP, поэтому, когда он задан, только перечисленные источники помечаются по экспортёру в метриках (всё остальное складывается в `other`), ограничивая кардинальность меток метрик. Пустой список вместо этого применяет автоматический предел. На обнаружение не влияет. |
| `networks` | Защищаемые [префиксы](/docs/glossary) (диапазоны IP в форме CIDR, например `203.0.113.0/24`). Обнаружение применяется **только** к назначениям внутри них, и они не должны пересекаться. |
| `protected_whitelist` | Адреса, которые **никогда** не блокируются, независимо от трафика. См. [Модель безопасности](/docs/safety). |
| `thresholds.pps` / `.mbps` / `.flows_per_sec` | Пороги на одно назначение после коррекции выборки. Все должны быть `> 0`. См. [Обнаружение](/docs/detection). |
| `thresholds.tcp_pps` / `udp_pps` / `icmp_pps` / `tcp_syn_pps` / `frag_pps` (и каждый вариант `_mbps`) | Опциональные лимиты по протоколам. Значение `0` или отсутствие ключа отключает этот лимит. Срабатывает любой превышенный порог (семантика OR). |
| `thresholds_outgoing` | Опционально. Обнаруживает атаки, **исходящие от** защищаемых хостов (скомпрометированные машины). Те же ключи, что и у `thresholds`; должен быть задан хотя бы один. При отсутствии исходящий трафик не учитывается. См. [Обнаружение](/docs/detection). |
| `hostgroups[]` | Опциональные именованные группы префиксов с собственными порогами и политикой подавления, включая расчёт `per_host` или `total`. Каждая может нести метку `tenant`. См. [Hostgroups](/docs/hostgroups). |
| `tenant` | Опциональная метка tenant для неявной глобальной/резервной группы. См. [Мультиарендность](/docs/multi-tenancy). |
| `samples.enabled` / `buffer_flows` / `flows_per_attack` | Непрерывный буфер трафика для образцов атак (по умолчанию: `true` / `65536` / `20`; максимум `buffer_flows` — `1048576`). Изменение размеров требует перезапуска. |
| `baseline` | Опциональные непрерывно обучаемые пороги на хост (EWMA — экспоненциально взвешенное скользящее среднее), переопределяемые на уровне hostgroup. См. [Базовые линии](/docs/baselines). |
| `carpet` | Опциональное обнаружение [ковровой бомбардировки](/docs/glossary) (распределение по подсети: объём, тонко размазанный по множеству адресов в префиксе): поднимает атаку на префикс, когда трафик, агрегированный по `/24` (`aggregation_prefix_v4`/`v6`), превышает `carpet.thresholds` на не менее чем `min_hosts` различных хостах — ловит флуд, остающийся под каждым порогом на хост. Работает только на оповещение, пока не задан `carpet.mitigation` (`flowspec`/`blackhole`), который отказывает префиксу, содержащему адрес из белого списка, и ограничен `carpet.max_active_prefix_bans`. |
| `ban.ttl_seconds` | Сколько длится блокировка без повторного обнаружения — непрекращающаяся атака продолжает обновлять её, поэтому маршрут автоматически отзывается через указанное число секунд после прекращения обнаружения. Постоянных блокировок нет. |
| `ban.unban_hysteresis_seconds` | Трафик должен оставаться ниже порога это время, прежде чем блокировка будет снята, что предотвращает «дребезг» (flapping). |
| `ban.max_active_bans` | Жёсткий предел одновременных блокировок. Новые блокировки сверх лимита отклоняются. |
| `ban.fallback` | Что делать, когда peer **отклоняет** анонс `flowspec`/`divert`: `blackhole` (по умолчанию) деградирует до [RTBH](/docs/glossary)-маршрута (Remotely-Triggered Blackhole — маршрут, который отправляет весь трафик к IP жертвы в «никуда»), чтобы жертва всё же была защищена; `none` отклоняет блокировку. `blackhole` — терминальный. |
| `ban.max_banned_fraction` | Отклонять новые блокировки, когда в blackhole отправлено больше этой доли (`0`–`1`, на семейство адресов) защищаемого пространства — ограничивает «радиус поражения» отравленного baseline или шторма с подменой источников. `0` (по умолчанию) отключает. |
| `ban.max_bans_per_window` / `ban_window_seconds` | Защита от шторма: не более стольких **новых** блокировок за окно (в секундах). `max_bans_per_window: 0` (по умолчанию) отключает; окно обязательно, когда задана скорость. |
| `ban.state_file` | Опциональный путь с правом записи (например, `/var/lib/kapkan/bans.json`), куда сохраняются активные блокировки, чтобы при перезапуске они восстанавливались и **повторно анонсировались**, а не сбрасывали подавление до повторного обнаружения. Работает в паре с `bgp.graceful_restart`. Пустое значение (по умолчанию) отключает сохранение. См. [Подавление атак](/docs/mitigation). |
| `mitigation` | Метод подавления: `blackhole` (по умолчанию), `flowspec` ([FlowSpec](/docs/glossary) — расширение BGP, которое фильтрует не только по IP назначения, но и по порту/протоколу и т. п., так что можно отбрасывать только атакующий трафик) или `divert`. Переопределяется на уровне hostgroup. См. [FlowSpec](/docs/flowspec) и [Отвод трафика](/docs/scrubbing). |
| `flowspec.action` / `rate_mbps` | Действие правила FlowSpec: `discard` или `rate_limit` (с `rate_mbps`). См. [FlowSpec](/docs/flowspec). |
| `flowspec.source_anchored` / `min_source_concentration` | Когда включено, правило FlowSpec закрепляет **и** жертву (назначение), **и** доминирующий источник-атакующего из образца атаки — отбрасывая только атакующих и щадя легитимных клиентов — но лишь когда образец достаточно концентрирован (по умолчанию `0.8`); иначе откат к привязке по жертве. По умолчанию выключено. |
| `escalation[]` | Опциональная лестница ступеней `{ after_seconds, action }` (`none` / `flowspec` / `divert` / `blackhole`), которая имеет приоритет над `mitigation`. См. [Лестницы эскалации](/docs/escalation). |
| `bgp.local_asn` / `router_id` / `next_hop` / `next_hop6` / `community` | Идентичность BGP, next-hop blackhole для IPv4/IPv6 и RTBH-[community](/docs/glossary) — тег маршрута в форме `ASN:value` (например, `65000:666`), который ваш вышестоящий провайдер сопоставляет, чтобы запустить blackhole. `router_id` должен быть IPv4 в формате dotted-quad. `next_hop6` опционален даже когда в `networks` есть IPv6-префикс; если его не задать, IPv6-blackhole используют next-hop отбрасывания `100::1` по RFC 6666. См. [Подавление атак](/docs/mitigation). |
| `bgp.communities` / `local_pref` | Опциональный **список** community (переопределяет `community`) и `LOCAL_PREF` (атрибут предпочтения маршрута BGP) для [iBGP](/docs/glossary)-пиров (внутренний BGP) — `LOCAL_PREF` распространяется только внутри вашей собственной сети, а не к внешним ([eBGP](/docs/glossary)) пирам, поэтому имеет смысл только для ваших внутренних соседей. Все атрибуты `bgp` переопределяются на уровне hostgroup. |
| `bgp.neighbors[]` | [eBGP](/docs/glossary) (внешний BGP) пиры: `address`, `remote_asn` и опциональный `port` для тестирования. |
| `bgp.graceful_restart` | BGP Graceful Restart (RFC 4724), **включён по умолчанию**. Поддерживающий его peer удерживает маршруты подавления Kapkan как устаревшие, пока сессия не работает (например, при обновлении), вместо того чтобы сбрасывать их. Ключи: `enabled` (по умолчанию `true`), `restart_seconds` (по умолчанию `120`, диапазон `0`–`4095`, где `0` означает «использовать значение по умолчанию 120»), `long_lived` (LLGR — Long-Lived Graceful Restart, по умолчанию `false`) и `long_lived_stale_seconds` (по умолчанию `3600`, `≤ 86400`). См. [Подавление атак](/docs/mitigation). |
| `scrubbing.next_hop` / `next_hop6` / `community` / `communities` / `local_pref` | Центр очистки (scrubbing), на который действие `divert` перенаправляет трафик. `next_hop` обязателен для отвода (`next_hop6` тоже, когда защищается IPv6); community опционально. Переопределяется на уровне hostgroup. См. [Отвод трафика](/docs/scrubbing). |
| `notify.*` | Каналы Telegram, Slack, email, webhook и exec-hook. См. [Уведомления](/docs/notifications). |
| `storage.clickhouse.*` | Опциональное сохранение в ClickHouse истории атак и трафика (`url`, `database`, `username_env`, `password_env`, `ttl_days`, …). См. [Хранилище](/docs/storage). |
| `geoip` | Опциональная атрибуция источников образцов атак по ASN (и стране) через MaxMind GeoLite2/GeoIP2 (`enabled`, `asn_database`, `country_database`). По умолчанию выключено; фиксируется при запуске. |
| `api.listen` | Адрес прослушивания REST API и метрик. По умолчанию `127.0.0.1:8080`. |
| `api.dashboard` | Отдаёт встроенный веб-интерфейс на `/` по адресу слушателя API. По умолчанию `true`. См. [Панель управления](/docs/dashboard). |
| `api.token_env` | Указывает переменную окружения с одним operator-bearer-токеном. Обязателен перед открытием слушателя за пределы localhost. См. [Аутентификация](/docs/authentication). |
| `api.tokens[]` | Токены с ролями: у каждой записи есть `name`, `token_env`, `role` (`viewer` или `operator`) и опциональная область `tenant`. Имеет приоритет над `token_env` (взаимоисключающи). См. [Аутентификация](/docs/authentication) и [Мультиарендность](/docs/multi-tenancy). |
| `update_check` | Опциональная, **подключаемая вручную** проверка более новой версии Kapkan — **по умолчанию выключена**, поэтому Kapkan никогда не «звонит домой», пока вы её не включите. Ключи: `enabled` (по умолчанию `false`), `interval_seconds` (по умолчанию `21600` = 6 ч; должна быть `>= 3600` / 1 ч, иначе конфигурация **отклоняется** при загрузке), `channel` (`stable` или `prerelease`), `notify` (также отправлять «доступно обновление» в ваши каналы уведомлений при первом обнаружении новой версии; по умолчанию `false`) и опциональное переопределение зеркала `url`. Когда включена, она опрашивает GitHub Releases API и показывает результат в `/api/v1/status` и [метрике](/docs/metrics) `kapkan_update_available`. Текущая версия всегда доступна локально независимо от этого. Можно также запустить `kapkan -check-update`, чтобы выполнить проверку один раз по требованию, не включая опрос (код выхода `10` = доступно обновление, `0` = всё актуально, `1` = ошибка). |

### Подробнее о порогах

Три базовых порога — `pps`, `mbps`, `flows_per_sec` — обязательны и все должны быть `> 0`.
Лимиты по протоколам — это опциональные уточнения: `tcp_syn_pps` считает чистые SYN-пакеты (флаг
SYN установлен, ACK снят), а `frag_pps` считает не-первые IP-фрагменты. Обнаружение использует
семантику OR — срабатывание атаки вызывает любой отдельно превышенный порог. Все значения
выражены в реальных, невыбранных единицах (см. [Коррекция выборки](#коррекция-выборки)).

`flows_per_sec` имеет смысл только для экспортёров NetFlow/IPFIX. sFlow экспортирует одну выборку на
пакет (без агрегации в потоки), поэтому хосты, источником которых является sFlow, всегда сообщают
`flows_per_sec = 0`, и этот порог для них никогда не срабатывает — используйте вместо него
`pps`/`tcp_syn_pps`/`udp_pps`. См. [Обнаружение](/docs/detection).

## Полный пример

Реалистичный конфиг для промышленной среды, адаптированный из `engine/deploy/config.example.yaml`:

```yaml
dry_run: true                       # keep until detection is validated

listen:
  sflow: ":6343"
  netflow: ":2055"                  # NetFlow v5/v9 + IPFIX share this socket

sampling:
  default_rate: 1000                # used only when an exporter omits its own rate
  # boundary: optional interface-boundary counting (deduplicates a flow seen at
  # several vantage points). List each exporter's external/uplink interfaces;
  # set egress_sampling only if calibration shows that ifIndex double-counts (see
  # Sampling correction) — not just because the device samples on egress.
  # boundary_debug: false
  # boundary:
  #   - exporter: "10.1.32.2"
  #     external_ifindexes: [1000025]
  #     egress_sampling: false   # true only if boundary_debug shows ~2x

networks:                           # detection applies ONLY inside these prefixes
  - "203.0.113.0/24"
  - "2001:db8::/32"

protected_whitelist:                # never banned, regardless of traffic
  - "203.0.113.1"                   # gateway / router
  - "203.0.113.2"                   # authoritative nameserver

thresholds:                         # per destination host, after sampling correction
  pps: 80000
  mbps: 1000
  flows_per_sec: 35000
  tcp_syn_pps: 5000                 # optional per-protocol; 0/absent disables
  udp_pps: 60000

thresholds_outgoing:                # detect compromised hosts attacking outward
  pps: 50000
  udp_pps: 20000

ban:
  ttl_seconds: 600                  # every ban auto-withdraws after this
  unban_hysteresis_seconds: 120     # stay below threshold this long before unban
  max_active_bans: 50               # refuse new bans past this cap
  # fallback: blackhole             # degrade a rejected flowspec/divert announce to RTBH (default)
  # max_banned_fraction: 0.05       # never blackhole more than 5% of protected space (per family)
  # max_bans_per_window: 100        # storm guard: at most 100 new bans per window
  # ban_window_seconds: 60          # the window for max_bans_per_window

bgp:
  local_asn: 65001
  router_id: "10.0.0.1"             # must be an IPv4 dotted-quad
  next_hop: "192.0.2.1"             # IPv4 blackhole (discard) next-hop
  next_hop6: "100::1"               # IPv6 blackhole next-hop
  community: "65000:666"            # RTBH community your upstream honors
  neighbors:
    - address: "10.0.0.254"
      remote_asn: 65000

notify:
  telegram:
    token_env: "KAPKAN_TG_TOKEN"    # token read from this env var, never the file
    chat_id: "-1001234567890"
  slack:
    webhook_url: ""                 # optional Slack incoming webhook
  email:
    smtp_host: ""                   # "mail.example.com:587"; empty disables
    from: ""
    to: []
    username_env: "KAPKAN_SMTP_USER"
    password_env: "KAPKAN_SMTP_PASS"
    require_tls: false              # STARTTLS auto-required when credentials are set

api:
  listen: "127.0.0.1:8080"          # default localhost bind needs no auth
  dashboard: true                   # embedded web UI at /; false = JSON API only
  # token_env: "KAPKAN_API_TOKEN"   # REQUIRED before exposing beyond localhost
```

Блоки `hostgroups`, `baseline` и `samples` здесь опущены для краткости. Полные схемы
смотрите на отдельных страницах:

- [Hostgroups](/docs/hostgroups) — пороги на префикс, `ban: false` и `calculation: total`.
- [Базовые линии](/docs/baselines) — `factor`, `half_life_seconds`, `warmup_seconds`, `floor`.
- Блок `samples`: `enabled` (по умолчанию `true`), `buffer_flows` (по умолчанию `65536`, максимум
  `1048576`) и `flows_per_attack` (по умолчанию `20`). Они управляют непрерывным кольцом потоков,
  которое привязывает доминирующие источники, порты и протоколы к каждой атаке в момент её
  срабатывания. Изменение размеров требует перезапуска.

## Коррекция выборки

Телеметрия потоков выбирается — маршрутизатор экспортирует одну запись на каждые N пакетов.
Kapkan умножает каждую наблюдаемую скорость на частоту выборки экспортёра, чтобы пороги
выражались в реальных, невыбранных единицах трафика. Частота берётся из пакета потока, когда
экспортёр её сообщает, иначе — из `sampling.default_rate`.

Это означает, что порог вроде `pps: 80000` — это реальная скорость атаки, на которую вы хотите
реагировать, а не выбранный счётчик. Если ваш экспортёр выбирает с частотой `1:1000` и не
объявляет её, задайте `sampling.default_rate: 1000`, чтобы один выбранный пакет в секунду
корректно считался за 1000 реальных пакетов в секунду.

<Callout type="warning" title="Настройте выборку правильно до запуска в бой">
Неверный `sampling.default_rate` масштабирует каждый порог на один и тот же коэффициент.
Установите его в соответствии с тем, что реально экспортируют ваши маршрутизаторы, а затем
убедитесь в режиме dry-run, что обнаружение срабатывает на ожидаемых скоростях.
</Callout>

### Учитываем каждый пакет один раз

Коррекция предполагает, что каждый пакет наблюдается один раз. Когда один и тот же пакет
выбирается более чем в одной точке наблюдения — избыточные экспортёры (например, пара MLAG),
выборка на ingress **и** egress на одном устройстве (Arista `sflow sample output`) или
транзитные/пиринговые каналы — Kapkan иначе суммировал бы каждую копию и завышал бы `pps`,
`mbps` и `flows_per_sec` на постоянный коэффициент.

Подсчёт по границе интерфейсов исправляет это без изменения экспортёра. Классифицируйте внешние
(uplink/border) интерфейсы каждого экспортёра; тогда поток учитывается для защищаемого хоста только
при пересечении границы — входящий на внешнем input-интерфейсе, исходящий на внешнем output-интерфейсе.
Экспортёры без записи `boundary` продолжают учитывать каждую выборку, так что включать это можно по
одному экспортёру за раз.

```yaml
sampling:
  default_rate: 2500
  boundary:
    - exporter: "10.1.32.2"          # one entry per flow exporter (sampler/agent IP)
      external_ifindexes: [1000025]  # the uplink/border interface ifIndex values
      egress_sampling: false         # only if calibration shows this ifIndex double-counts (see below)
```

Чтобы выяснить, какие интерфейсы внешние, ненадолго задайте `sampling.boundary_debug: true` и читайте
`kapkan_engine_boundary_debug_bytes_total` в разбивке по экспортёру и интерфейсу.

<Callout type="warning" title="Калибруйте egress_sampling — не задавайте его наугад">
`egress_sampling: true` уменьшает вдвое учитываемую по границе скорость, чтобы отменить двойной учёт
in+out, но **`sflow sample output` сам по себе не означает, что он вам нужен.** Действительно ли копия
с egress завышает границу, зависит от топологии: на Arista MLAG копия входящего пакета с egress
выбирается на access-порту с *другим* input-интерфейсом, поэтому она никогда не попадает во входной
интерфейсный фильтр границы — пакет уже учтён один раз, и `egress_sampling: true` тогда *занизит* его
вдвое. Откалибруйте: сравните скорость `boundary_debug` внешнего интерфейса с реальным счётчиком этого
интерфейса (например, `show interfaces`). Задавайте `egress_sampling: true`, только если граничный
интерфейс показывает ~2x реальной скорости; иначе оставьте `false` (значение по умолчанию).
</Callout>
