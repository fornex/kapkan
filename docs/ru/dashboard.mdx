# Панель управления

Откройте **http://localhost:8080** в браузере, запущенном на узле с Kapkan. Когда вы под
атакой, в первую очередь смотрите на баннер **Live mode** (Kapkan действительно анонсирует
или только наблюдает?) и на список **Active attacks** (что атакуют и насколько сильно).

Веб-интерфейс встроен прямо в сам бинарный файл `kapkan`, поэтому ничего дополнительно
устанавливать или разворачивать не нужно — один и тот же процесс обслуживает панель
управления, JSON API и метрики Prometheus.

<Callout type="info" title="Один бинарный файл, один слушатель">
Панель управления, JSON API и `/metrics` обслуживаются одним и тем же процессом по одному
и тому же адресу. Нет отдельного веб-сервиса, который нужно разворачивать или поддерживать
в синхронном состоянии.
</Callout>

## Где она находится

Панель управления обслуживается по пути `/` по тому же адресу, что и REST API — это значение
`api.listen` из вашей конфигурации. При значении по умолчанию `127.0.0.1:8080` вы открываете
её по адресу [http://localhost:8080](http://localhost:8080).

Поскольку по умолчанию привязка идёт к `127.0.0.1`, панель управления доступна только с самого
узла Kapkan — если открыть `localhost:8080` на собственном ноутбуке, вы ничего не увидите. Чтобы
посмотреть её со своей рабочей станции, пробросьте порт через SSH, а не меняйте привязку
`api.listen` на публичный адрес:

```bash
ssh -L 8080:127.0.0.1:8080 user@kapkan-host
```

Затем откройте [http://localhost:8080](http://localhost:8080) на своей рабочей станции.
(Чтобы привязать слушатель к публичному адресу, сначала нужен API-токен — см. оговорку об
[аутентификации](#аутентификация) ниже.)

Чтобы обслуживать только JSON API и метрики без интерфейса, отключите его:

```yaml
api:
  listen: "127.0.0.1:8080"
  dashboard: false   # serve the JSON API and /metrics only
```

При `api.dashboard: false` конечные точки `/api/v1` и `/metrics` продолжают работать — перестаёт
обслуживаться только оболочка встроенного интерфейса.

## Что она показывает

Панель управления опрашивает JSON API и отображает всё, что Kapkan отслеживает в памяти:

- **Live mode** — запущен ли экземпляр в режиме dry-run или в боевом (live), чтобы вы могли
  с первого взгляда понять, будет ли обнаруженная атака действительно анонсирована.
- **Активные и недавние атаки** — атаки, происходящие в данный момент, а также недавно
  завершившиеся, каждая со своей классификацией типа атаки (например, усиление через NTP/DNS
  или SYN-флуд — см. страницу [Обнаружение](/docs/detection) о том, что означает каждый тип) и
  доминирующими источниками, портами и протоколами, зафиксированными по выборкам потоков.
- **Топ источников (Top talkers)** — снимок отслеживаемых хостов с показателями по каждому
  направлению и обученными базовыми уровнями (baselines), чтобы вы могли видеть, кто шумит
  громче всех и как это соотносится с нормой.
- **Hostgroups** — ваши именованные группы префиксов и их состояние.
- **Таблица банов** — все баны, активные и исторические.
- **Трафик / Отчёты (Traffic / Reports)** — графики входящего/исходящего трафика в реальном
  времени, построенные на буферах скоростей в памяти, плюс раздел **Исторические отчёты
  (Historical reports)**, который читает сохранённые таблицы ClickHouse, когда хранение включено.
- **Управление** — те же действия, что предлагает API: **ручной бан / разбан** адреса и
  **перезагрузка конфигурации** (эквивалент отправки `SIGHUP`).

Раздел **Исторические отчёты (Historical reports)** на представлении Трафик / Отчёты — это
единственная часть, которой нужно хранение: он заполняется только тогда, когда включено
[хранение в ClickHouse](/docs/storage), а в противном случае отображает помеченный плейсхолдер
с пояснением, как его включить.

<Callout type="success" title="База данных не требуется">
Панель управления полностью строится на состоянии движка, хранящемся в памяти. Историческое
хранение потоков — это отдельная, опциональная возможность; она не нужна для просмотра текущих
атак, топа источников или банов.
</Callout>

## Аутентификация

По умолчанию API и панель управления не требуют аутентификации, что безопасно только потому,
что значение `api.listen` по умолчанию привязано к `127.0.0.1`. Когда вы задаёте API-токен,
панель управления адаптируется: статическая оболочка интерфейса по-прежнему загружается, но
данные за ней не появляются, пока вы не предоставите токен.

Когда аутентификация включена, панель управления запрашивает токен и хранит его в
`sessionStorage` браузера, добавляя его в виде заголовка bearer к каждому запросу к API, который
она выполняет. Каждый запрос `/api/v1` авторизуется с помощью `Authorization: Bearer <token>`;
`/metrics` и статическая оболочка остаются открытыми, а актуальные данные и элементы управления
банами / перезагрузкой — нет.

<Callout type="warning" title="Задайте токен перед тем, как открывать слушатель">
Прежде чем привязывать `api.listen` к чему-либо за пределами `127.0.0.1`, настройте API-токен.
См. [Аутентификация](/docs/authentication) о том, как задать `api.token_env` и как проверяются
запросы.
</Callout>
