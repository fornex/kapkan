# Справочник CLI

Kapkan — это единый бинарник. Всё, что он делает, выбирается флагами командной строки при запуске —
никаких подкоманд, кроме двух служебных режимов ниже, нет. Все логи идут в **stderr**, поэтому
REST API и метрики на stdout остаются чистыми для перенаправления по конвейеру.

```sh
kapkan [flags]
```

## Флаги

| Флаг | По умолчанию | Описание |
| --- | --- | --- |
| `-config <path>` | `configs/dev.yaml` | Путь к YAML-[файлу конфигурации](/docs/configuration). |
| `-log-format <fmt>` | `json` | Формат вывода логов: `json` (структурированный, для сборщиков логов) или `text` (читаемый, для локальных запусков). Любое значение, кроме `text`, трактуется как `json`. |
| `-log-level <lvl>` | `info` | Минимальный уровень логов: `debug`, `info`, `warn`, `error`. Нераспознанное значение откатывается к `info`. |
| `-check-config <path>` | — | Проверить файл конфигурации по пути `<path>` и завершиться — демон **не** запускается. См. [ниже](#проверка-конфигурации). |
| `-dump-schema` | `false` | Напечатать JSON-схему конфигурации в stdout и завершиться. См. [ниже](#вывод-схемы). |

<Callout type="info" title="json или text?">
Используйте `-log-format text` для локальной разработки — это формат, который применяют Быстрый старт и
цель [`make run-dev`](/docs/quickstart). В production оставляйте `json` по умолчанию, чтобы сборщик логов
мог разбирать структурированные поля; [systemd-юнит](/docs/deployment) поставляется с
`-log-format json -log-level info`.
</Callout>

## Запуск демона

Без служебного флага Kapkan открывает свои flow-listener'ы, REST API и (когда не в режиме dry-run) BGP-спикер,
затем работает, пока не получит `SIGINT` или `SIGTERM`, после чего корректно завершается. Он завершается с
ненулевым кодом только при фатальной ошибке запуска или выполнения.

```sh
# Local, human-readable logs against the bundled dev config:
kapkan -config configs/dev.yaml -log-format text

# Production-style: structured logs at info level:
kapkan -config /etc/kapkan/config.yaml -log-format json -log-level info
```

Файл конфигурации можно перечитывать во время работы без перезапуска — отправьте `SIGHUP`
(`systemctl reload kapkan`) или `POST /api/v1/config/reload`. См.
[справочник конфигурации](/docs/configuration), чтобы узнать, что перечитывается, а что фиксируется при запуске.

## Служебные подкоманды

Эти два флага завершаются до запуска демона — они никогда не открывают listener и не отправляют
анонс, поэтому их безопасно запускать на production-хосте.

### Проверка конфигурации

```sh
kapkan -check-config /etc/kapkan/config.yaml
```

Выполняет ровно тот же разбор и валидацию, что и движок, — включая кросс-полевые правила, которые
статическая схема выразить не может, — затем печатает разрешённую конфигурацию (режим, защищаемые сети и
эффективную митигацию по каждой [hostgroup](/docs/hostgroups)) и завершается:

| Код выхода | Значение |
| --- | --- |
| `0` | Конфигурация корректна. |
| `1` | Конфигурация некорректна; точная ошибка печатается в stderr. |

Чёткое разделение 0/1 без проблем встраивается в CI или предрелизную проверку. См.
[Проверку конфигурации](/docs/configuration#проверка-конфигурации).

### Вывод схемы

```sh
kapkan -dump-schema
```

Печатает JSON-схему конфигурации в stdout и завершается с кодом `0` (или `1` при внутренней ошибке).
[Конструктор конфигурации](/config) использует эту схему, чтобы проверять поля прямо в вашем браузере.
