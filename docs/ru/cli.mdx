# Справочник CLI

Kapkan — это единый бинарник. Всё, что он делает, выбирается флагами командной строки при запуске —
никаких подкоманд, кроме служебных режимов ниже, нет. Все логи идут в **stderr**; REST API
и метрики Prometheus отдаются по HTTP на `api.listen` (по умолчанию `127.0.0.1:8080`), а
служебные флаги (`-version`, `-dump-schema`) печатают свой вывод в **stdout**.

```sh
kapkan [flags]
```

## Флаги

| Флаг | По умолчанию | Описание |
| --- | --- | --- |
| `-config <path>` | `configs/dev.yaml` | Путь к YAML-[файлу конфигурации](/docs/configuration). |
| `-log-format <fmt>` | `json` | Формат вывода логов: `json` (структурированный, для сборщиков логов) или `text` (читаемый, для локальных запусков). Любое значение, кроме `text`, трактуется как `json`. |
| `-log-level <lvl>` | `info` | Минимальный уровень логов: `debug`, `info`, `warn`, `error`. Нераспознанное значение откатывается к `info`. |
| `-check-config <path>` | — | Проверить файл конфигурации по пути `<path>` и завершиться — демон **не** запускается. См. [ниже](#проверка-конфигурации). |
| `-dump-schema` | `false` | Напечатать JSON-схему конфигурации в stdout и завершиться. См. [ниже](#вывод-схемы). |
| `-version` | `false` | Напечатать версию сборки и завершиться. Та же версия доступна во время работы через `/api/v1/status` и [метрику](/docs/metrics) `kapkan_build_info`. |
| `-check-update` | `false` | Однократно выполнить опциональную [проверку обновлений](/docs/configuration) и завершиться — `0`, если версия актуальна, `10`, если доступно обновление, `1` при ошибке. Опрашивает GitHub Releases API; отправляет только сам запрос, никогда — идентичность узла или конфигурацию. |
| `-pid-file <path>` | `/run/kapkan/kapkan.pid` | Куда демон записывает свой идентификатор процесса при запуске (удаляется при корректном завершении); читается `-s`, чтобы найти работающий демон. Сбой записи не фатален — демон всё равно работает, недоступен только `-s`. |
| `-s <signal>` | — | Послать сигнал работающему демону и завершиться, прочитав `-pid-file`: `reload`, `stop` или `quit`. Демон при этом **не** запускается. См. [ниже](#управление-работающим-демоном). |

<Callout type="info" title="json или text?">
Используйте `-log-format text` для локальной разработки — это формат, который применяет
[Быстрый старт](/docs/quickstart). В production оставляйте `json` по умолчанию, чтобы сборщик логов
мог разбирать структурированные поля; [systemd-юнит](/docs/deployment) поставляется с
`-log-format json -log-level info`.
</Callout>

## Запуск демона

Без служебного флага Kapkan открывает свои flow-listener'ы, REST API и (когда не в режиме dry-run) BGP-спикер,
затем работает, пока не получит `SIGINT` или `SIGTERM`, после чего корректно завершается. Он завершается с
ненулевым кодом только при фатальной ошибке запуска или выполнения.

```sh
# Local, human-readable logs against the bundled dev config:
kapkan -config configs/dev.yaml -log-format text

# Production-style: structured logs at info level:
kapkan -config /etc/kapkan/config.yaml -log-format json -log-level info
```

Файл конфигурации можно перечитывать во время работы без перезапуска — выполните [`kapkan -s reload`](#управление-работающим-демоном),
отправьте `SIGHUP` напрямую (`systemctl reload kapkan`) или сделайте `POST /api/v1/config/reload`. См.
[справочник конфигурации](/docs/configuration), чтобы узнать, что перечитывается, а что фиксируется при запуске.

## Управление работающим демоном

```sh
kapkan -s reload    # re-read the config file (SIGHUP)
kapkan -s stop      # graceful shutdown (SIGTERM)
```

`-s` читает идентификатор процесса из `-pid-file` (по умолчанию `/run/kapkan/kapkan.pid`, записывается
демоном при запуске) и доставляет соответствующий сигнал этому процессу — то же локальное управление в стиле
nginx, что и `nginx -s reload`. Ему не нужны ни сетевое соединение, ни API-токен, поэтому он работает даже
если REST API недоступен; ему нужно право слать сигнал демону, так что запускайте его от `root` или
пользователя `kapkan`.

| Значение `-s` | Сигнал | Эффект |
| --- | --- | --- |
| `reload` | `SIGHUP` | Горячая перезагрузка файла конфигурации. То же, что `systemctl reload kapkan` или `POST /api/v1/config/reload`. |
| `stop` / `quit` | `SIGTERM` | Корректное завершение. То же, что `systemctl stop kapkan`. Эти два — псевдонимы: завершение Kapkan всегда корректное (он просит BGP-пиров сохранить маршруты митигации через [Graceful Restart](/docs/mitigation)). |

Если pid-файл отсутствует или устарел (нет такого процесса) либо у вас нет прав послать сигнал, `-s`
печатает причину в stderr и завершается с кодом `1`.

<Callout type="info" title="Где находится pid-файл">
Поставляемый [systemd-юнит](/docs/deployment) задаёт `RuntimeDirectory=kapkan`, так что `/run/kapkan`
существует и доступен для записи пользователю `kapkan`; демон записывает туда `/run/kapkan/kapkan.pid` и
удаляет его при корректной остановке. В локальном dev-запуске этого каталога может не быть — демон печатает
предупреждение и продолжает работу, а `-s` сообщит «no pid file». Передайте `-pid-file /tmp/kapkan.pid` и
демону, и `-s`, если хотите воспользоваться этим сокращением локально.
</Callout>

## Служебные флаги

Эти флаги завершаются до запуска демона — они никогда не открывают listener и не отправляют
анонс, поэтому их безопасно запускать на production-хосте. (`-check-update` делает один
исходящий запрос к releases API; остальные полностью офлайн.)

### Проверка конфигурации

```sh
kapkan -check-config /etc/kapkan/config.yaml
```

Выполняет ровно тот же разбор и валидацию, что и движок, — включая кросс-полевые правила, которые
статическая схема выразить не может, — затем печатает разрешённую конфигурацию (режим, защищаемые сети и
эффективную митигацию по каждой [hostgroup](/docs/hostgroups)) и завершается:

| Код выхода | Значение |
| --- | --- |
| `0` | Конфигурация корректна. |
| `1` | Конфигурация некорректна; точная ошибка печатается в stderr. |

Чёткое разделение 0/1 без проблем встраивается в CI или предрелизную проверку. См.
[Проверку конфигурации](/docs/configuration#проверка-конфигурации).

### Вывод схемы

```sh
kapkan -dump-schema
```

Печатает JSON-схему конфигурации в stdout и завершается с кодом `0` (или `1` при внутренней ошибке).
[Конструктор конфигурации](/config) использует эту схему, чтобы проверять поля прямо в вашем браузере.

### Вывод версии

```sh
kapkan -version
```

Печатает версию сборки (тег релиза плюс короткую VCS-ревизию, когда она известна) в stdout и завершается.
Та же строка отдаётся во время работы на `/api/v1/status` и переносится [метрикой](/docs/metrics) `kapkan_build_info` —
так что вы можете подтвердить версию бинарника локально без какого-либо исходящего трафика.

### Проверка обновлений

```sh
kapkan -check-update
```

Однократно по требованию выполняет [`update_check`](/docs/configuration) и завершается: `0`, если версия актуальна, `10`,
если доступен более новый релиз, `1` при ошибке. Запрашивает GitHub Releases API и отправляет только сам
запрос (ваш IP и обобщённый User-Agent) — никогда идентичность узла, конфигурацию или данные об
атаках. Периодическая проверка внутри процесса **выключена по умолчанию**; этот флаг работает независимо от
этой настройки.
