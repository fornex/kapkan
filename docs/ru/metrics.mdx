# Метрики

Kapkan экспортирует метрики Prometheus в пространстве имён `kapkan_` по адресу `GET /metrics` на
слушателе API (`api.listen`, по соглашению `127.0.0.1:8080` — значение из примера конфигурации; поле обязательное, поэтому задайте его). Они охватывают весь конвейер —
приём потоков, движок обнаружения, подавление атак через RTBH и уведомления — поэтому вы можете собирать
данные о пропускной способности, задержке, активных атаках и анонсированных маршрутах в свою
существующую систему мониторинга.

<Callout type="info" title="/metrics остаётся открытым">
Конечная точка `/metrics` всегда доступна без аутентификации, даже когда вы задаёте токен API через
`api.token_env`. Только конечные точки данных `/api/v1` требуют `Authorization: Bearer <token>`.
Привязывайте слушатель соответствующим образом — см. [Аутентификация](/docs/authentication).
</Callout>

## Доступные метрики

| Метрика | Метки | Значение |
| --- | --- | --- |
| `kapkan_ingest_flows_total` | `proto` | Общее число принятых потоков, с разбивкой по протоколу потоков. |
| `kapkan_ingest_packets_total` | `exporter`, `proto` | Общее число полученных телеметрических датаграмм, с разбивкой по исходному экспортёру (с ограниченной кардинальностью; см. [`flow_sources`](/docs/configuration)) и протоколу. |
| `kapkan_ingest_decode_errors_total` | `proto` | Телеметрические датаграммы, которые не удалось декодировать, с разбивкой по протоколу. |
| `kapkan_ingest_dropped_flows_total` | — | Потоки, отброшенные из-за того, что входная очередь движка была заполнена. |
| `kapkan_engine_active_attacks` | — | Gauge атак, находящихся в процессе в данный момент. |
| `kapkan_engine_attacks_total` | — | Счётчик атак, обнаруженных с момента запуска. |
| `kapkan_engine_process_latency_seconds` | — | Гистограмма задержки обработки на горячем пути, измеряемой один раз на каждую партию приёма (а не на каждый поток). |
| `kapkan_engine_tracked_hosts` | — | Gauge хостов, отслеживаемых движком в данный момент. |
| `kapkan_engine_boundary_debug_bytes_total` | `exporter`, `iface`, `dir` | Скорректированные на частоту выборки байты к защищаемым хостам (`dir=in`) или от них (`dir=out`), с разбивкой по `exporter` и интерфейсу (`iface` = ifIndex). Появляется только при [`sampling.boundary_debug: true`](/docs/configuration) — временное подспорье для поиска ваших граничных интерфейсов. Кардинальность НЕ ограничена; отключите метрику снова после того, как изучите разбивку. |
| `kapkan_mitigate_announced_routes` | `mode` | Gauge blackhole-маршрутов, анонсированных в данный момент, с разбивкой по режиму `real` / `dry_run`. |
| `kapkan_mitigate_flowspec_rules` | `mode` | Gauge правил [FlowSpec](/docs/flowspec), анонсированных в данный момент, с разбивкой по режиму `real` / `dry_run`. Следите за этим значением относительно лимита маршрутов FlowSpec на ваших маршрутизаторах. |
| `kapkan_mitigate_bans_rejected_total` | `reason` | Блокировки, отклонённые предохранителем безопасности, с разбивкой по `reason`: `max_active_bans`, `blast_radius_fraction`, `blast_radius_rate` или `max_active_prefix_bans` (ковровая бомбардировка). |
| `kapkan_mitigate_fallback_total` | `from`, `to` | Анонсы подавления, понизившиеся до запасного метода из-за того, что пир отклонил основной, с разбивкой по `from`/`to`. Ненулевая серия `from="flowspec"` указывает на аплинки, которые не поддерживают FlowSpec. |
| `kapkan_notify_notifications_total` | `channel`, `result` | Попытки отправки уведомлений, с разбивкой по каналу (telegram, slack, email, webhook, exec) и `result` (`ok`, `error` или `dropped`, когда слоты доставки были переполнены). |
| `kapkan_storage_rows_total` | `table`, `result` | Строки, переданные в опциональное [хранилище ClickHouse](/docs/storage), с разбивкой по таблице и значениям `written` / `dropped` / `error`. `dropped` = очередь хранилища была заполнена (хранилище никогда не блокирует обнаружение, поэтому потоки всё равно обрабатываются и атаки подавляются). |
| `kapkan_build_info` | `version`, `revision`, `goversion`, `goos`, `goarch` | Постоянный info-gauge со значением `1`, несущий данные о работающей сборке в своих метках (идиома `node_exporter`). Отслеживайте расхождение версий по парку с помощью `count by (version)(kapkan_build_info)` — без какого-либо phone-home. |
| `kapkan_update_available` | `latest_version`, `security` | `1`, когда включаемая опционально проверка [`update_check`](/docs/configuration) обнаруживает более новый релиз (иначе метрика отсутствует); `security="true"` отмечает релиз, связанный с безопасностью. |

Метка `mode` у `kapkan_mitigate_announced_routes` — самый быстрый способ подтвердить состояние
dry-run с одного взгляда: пока `dry_run: true`, каждый потенциальный маршрут учитывается под
`mode="dry_run"` и ничего под `mode="real"`. См. [Подавление атак](/docs/mitigation) и
[Модель безопасности](/docs/safety).

Метка `proto` — это протокол передачи, по которому пришёл поток: `sflow5`, `netflow5`,
`netflow9` или `ipfix` (например, `kapkan_ingest_flows_total{proto="sflow5"}`).

## Сбор метрик

Добавьте Kapkan как цель сбора (scrape target) в конфигурацию Prometheus:

```yaml
scrape_configs:
  - job_name: kapkan
    metrics_path: /metrics
    static_configs:
      - targets: ["127.0.0.1:8080"]
```

Используйте хост и порт из значения `api.listen` (по соглашению `127.0.0.1:8080`). Если вы открываете слушатель за пределами
localhost, задайте токен API — `/metrics` остаётся доступным без него, поэтому вашему сборщику
не нужны учётные данные.

Чтобы посмотреть сырой вывод напрямую:

```sh
curl -s localhost:8080/metrics | grep kapkan_
```

## Панель Grafana

Официальную панель управления Grafana для набора `kapkan_*` можно скачать напрямую по ссылке
[`kapkan-overview.json`](/kapkan-overview.json) (его исходник лежит в репозитории в каталоге
[`engine/deploy/grafana/`](https://github.com/fornex/kapkan/tree/main/engine/deploy/grafana)).
Он охватывает весь конвейер — скорости приёма потоков/датаграмм/декодирования и отброшенные потоки,
активные и начавшиеся атаки, отслеживаемые хосты, квантили задержки горячего пути, анонсированные
маршруты и правила FlowSpec по `mode`, блокировки, отклонённые по `reason`, запасные переключения
подавления, а также результаты уведомлений и хранилища.

В Grafana перейдите в **Dashboards → New → Import**, загрузите JSON и выберите ваш источник данных
Prometheus при запросе (он шаблонизирован как `DS_PROMETHEUS`). Панель управления рассчитана на
Grafana 10+ и проверен на импорт в Grafana 11. См.
[README](https://github.com/fornex/kapkan/blob/main/engine/deploy/grafana/README.md) рядом с
ним для информации о provisioning.
