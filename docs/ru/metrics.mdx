# Метрики

Kapkan экспортирует метрики Prometheus в пространстве имён `kapkan_` по адресу `GET /metrics` на
слушателе API (`api.listen`, по умолчанию `127.0.0.1:8080`). Они охватывают весь конвейер —
приём flow, движок детектирования, митигацию RTBH и уведомления — поэтому вы можете собирать
данные о пропускной способности, задержке, активных атаках и анонсированных маршрутах в свою
существующую систему мониторинга.

<Callout type="info" title="/metrics остаётся открытым">
Эндпоинт `/metrics` всегда доступен без аутентификации, даже когда вы задаёте токен API через
`api.token_env`. Только эндпоинты данных `/api/v1` требуют `Authorization: Bearer <token>`.
Привязывайте слушатель соответствующим образом — см. [Аутентификация](/docs/authentication).
</Callout>

## Доступные метрики

| Метрика | Метки | Значение |
| --- | --- | --- |
| `kapkan_ingest_flows_total` | `proto` | Общее число принятых flow, с разбивкой по протоколу flow. |
| `kapkan_ingest_packets_total` | `exporter`, `proto` | Общее число полученных телеметрических датаграмм, с разбивкой по исходному экспортёру (с ограниченной кардинальностью; см. [`flow_sources`](/docs/configuration)) и протоколу. |
| `kapkan_ingest_decode_errors_total` | `proto` | Телеметрические датаграммы, которые не удалось декодировать, с разбивкой по протоколу. |
| `kapkan_ingest_dropped_flows_total` | — | Flow, отброшенные из-за того, что входная очередь движка была заполнена. |
| `kapkan_engine_active_attacks` | — | Gauge атак, находящихся в процессе в данный момент. |
| `kapkan_engine_attacks_total` | — | Счётчик атак, обнаруженных с момента запуска. |
| `kapkan_engine_process_latency_seconds` | — | Гистограмма задержки обработки на горячем пути, измеряемой один раз на каждый батч приёма (а не на каждый flow). |
| `kapkan_engine_tracked_hosts` | — | Gauge хостов, отслеживаемых движком в данный момент. |
| `kapkan_mitigate_announced_routes` | `mode` | Gauge blackhole-маршрутов, анонсированных в данный момент, с разбивкой по режиму `real` / `dry_run`. |
| `kapkan_mitigate_flowspec_rules` | `mode` | Gauge правил [FlowSpec](/docs/flowspec), анонсированных в данный момент, с разбивкой по режиму `real` / `dry_run`. Следите за этим значением относительно лимита маршрутов FlowSpec на ваших маршрутизаторах. |
| `kapkan_mitigate_bans_rejected_total` | `reason` | Блокировки, отклонённые предохранителем безопасности, с разбивкой по `reason`: `max_active_bans`, `blast_radius_fraction`, `blast_radius_rate` или `max_active_prefix_bans` (carpet). |
| `kapkan_mitigate_fallback_total` | `from`, `to` | Анонсы митигации, понижавшиеся до резервного метода из-за того, что пир отклонил основной, с разбивкой по `from`/`to`. Ненулевая серия `from="flowspec"` указывает на аплинки, которые не поддерживают FlowSpec. |
| `kapkan_notify_notifications_total` | `channel`, `result` | Попытки отправки уведомлений, с разбивкой по каналу (telegram, slack, email, webhook, exec) и `result`. |
| `kapkan_storage_rows_total` | `table`, `result` | Строки, переданные в опциональное [хранилище ClickHouse](/docs/storage), с разбивкой по таблице и значениям `written` / `dropped` / `error`. |
| `kapkan_build_info` | `version`, `revision`, `goversion`, `goos`, `goarch` | Постоянный info-gauge со значением `1`, несущий данные о работающей сборке в своих метках (идиома `node_exporter`). Отслеживайте расхождение версий по парку с помощью `count by (version)(kapkan_build_info)` — без какого-либо phone-home. |
| `kapkan_update_available` | `latest_version`, `security` | `1`, когда включаемая опционально проверка [`update_check`](/docs/configuration) обнаруживает более новый релиз (иначе метрика отсутствует); `security="true"` отмечает релиз, связанный с безопасностью. |

Метка `mode` у `kapkan_mitigate_announced_routes` — самый быстрый способ подтвердить состояние
dry-run с одного взгляда: пока `dry_run: true`, каждый потенциальный маршрут учитывается под
`mode="dry_run"` и ничего под `mode="real"`. См. [Митигация](/docs/mitigation) и
[Модель безопасности](/docs/safety).

## Сбор метрик

Добавьте Kapkan как цель сбора (scrape target) в конфигурацию Prometheus:

```yaml
scrape_configs:
  - job_name: kapkan
    metrics_path: /metrics
    static_configs:
      - targets: ["127.0.0.1:8080"]
```

Используйте хост и порт из значения `api.listen`. Если вы открываете слушатель за пределами
localhost, задайте токен API — `/metrics` остаётся доступным без него, поэтому вашему сборщику
не нужны учётные данные.

Чтобы посмотреть сырой вывод напрямую:

```sh
curl -s localhost:8080/metrics | grep kapkan_
```

## Grafana dashboard

Официальный дашборд Grafana для набора `kapkan_*` можно скачать напрямую по ссылке
[`kapkan-overview.json`](/kapkan-overview.json) (его исходник лежит в репозитории в каталоге
[`engine/deploy/grafana/`](https://github.com/fornex/kapkan/tree/main/engine/deploy/grafana)).
Он охватывает весь конвейер — скорости приёма flow/датаграмм/декодирования и отброшенные flow,
активные и начавшиеся атаки, отслеживаемые хосты, квантили задержки горячего пути, анонсированные
маршруты и правила FlowSpec по `mode`, блокировки, отклонённые по `reason`, резервные переключения
митигации, а также результаты уведомлений и хранилища.

В Grafana перейдите в **Dashboards → New → Import**, загрузите JSON и выберите ваш источник данных
Prometheus при запросе (он шаблонизирован как `DS_PROMETHEUS`). Дашборд рассчитан на
Grafana 10+ и проверен на импорт в Grafana 11. См.
[README](https://github.com/fornex/kapkan/blob/main/engine/deploy/grafana/README.md) рядом с
ним для информации о provisioning.
