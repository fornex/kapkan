# Уведомления

Kapkan отправляет уведомление на каждое событие жизненного цикла атаки — когда атака
впервые обнаружена (`attack_started`) и когда она завершается (`attack_ended`). Каждое
событие доставляется во все настроенные каналы: Telegram, Slack, электронную почту,
универсальный JSON-webhook и exec-хук.

Он также отправляет однократное событие `update_available` через те же каналы, когда
опциональная проверка обновлений находит более новую версию и включён
`update_check.notify` — см. [Обновление](/docs/upgrading). Оно несёт поля версии вместо
полей атаки, а exec-хук в формате FastNetMon (у которого нет эквивалентного действия) его
пропускает.

Доставка выполняется по принципу best-effort и асинхронно. Медленный или сбоящий канал
никогда не задерживает обнаружение или подавление атаки, а каждый канал ограничен
собственным пулом обработчиков — при ковровой бомбардировке (атака, которая бьёт сразу по
множеству хостов в вашем диапазоне, поэтому Kapkan порождает по одному событию на хост)
избыточные уведомления отбрасываются и учитываются в счётчиках, а не накапливаются в виде
неограниченного числа соединений или процессов-хуков.

<Callout type="info" title="Секреты берутся из окружения">
Токены ботов и учётные данные SMTP читаются из именованных переменных окружения, а не
из файла конфигурации. В файле хранится только имя переменной (например,
`token_env: "KAPKAN_TG_TOKEN"`), поэтому конфигурацию можно держать в git.
</Callout>

## Каналы

### Telegram

Задайте в `notify.telegram.token_env` имя переменной окружения, содержащей токен вашего
бота, а в `notify.telegram.chat_id` — целевой чат. Токен читается из этой переменной в
момент отправки; в файле он никогда не хранится. Сообщения отправляются в формате HTML и
включают цель атаки, классификацию, сработавшую метрику и состояние блокировки.

<Callout type="info" title="Как узнать ID чата">
Чтобы узнать числовой ID чата, напишите боту [@getmyid_bot](https://t.me/getmyid_bot)
напрямую для личного чата или добавьте его в нужную группу, чтобы получить ID группы
(ID групп отрицательные, например `-1001234567890`).
</Callout>

### Slack

Задайте в `notify.slack.webhook_url` URL Slack
[incoming webhook](https://api.slack.com/messaging/webhooks). Kapkan отправляет на него
методом POST отформатированное сообщение при каждом событии. Оставьте поле пустым, чтобы
отключить канал.

### Электронная почта (SMTP)

Настройте `notify.email` для отправки почты по SMTP. Задайте `smtp_host` в виде
`host:port` (пустое значение отключает канал), `from` — отправитель в конверте, а `to` —
список получателей. Учётные данные читаются из окружения через `username_env` и
`password_env`.

| Ключ | Значение |
| --- | --- |
| `smtp_host` | SMTP-сервер в формате `host:port`; пустое значение отключает почту. |
| `from` | Адрес отправителя в конверте и заголовке. |
| `to` | Список адресов получателей. |
| `username_env` | Переменная окружения с именем пользователя SMTP (включает AUTH PLAIN). |
| `password_env` | Переменная окружения с паролем SMTP. |
| `require_tls` | Принудительно требовать STARTTLS даже без учётных данных. |

Kapkan использует STARTTLS всякий раз, когда сервер его анонсирует. STARTTLS
**обязателен**, если настроены учётные данные (непустой `username_env`) или если задано
`require_tls: true` — в этом случае сервер, не предлагающий STARTTLS, отклоняется с явной
ошибкой, вместо того чтобы откатиться на открытый текст; это превращает активную атаку
понижения уровня (downgrade) в заметный сбой. Когда TLS не предлагается и не требуется,
доставка на хост, не являющийся loopback, выполняется в открытом виде и явно
протоколируется.

<Callout type="warning" title="Открытый текст на удалённые хосты протоколируется, а не блокируется">
Если ваш SMTP-сервер не предлагает STARTTLS, а вы не задали ни учётных данных, ни
`require_tls`, Kapkan отправит почту в открытом виде на хост, не являющийся loopback, и
выдаст предупреждение. Установите `require_tls: true`, чтобы в этом случае возникал сбой.
</Callout>

### Webhook

Задайте в `notify.webhook.url` URL вашего приёмника (используйте HTTPS — payload может
содержать детали атаки). Kapkan отправляет HTTP-запрос `POST` с телом
`Content-Type: application/json` — [payload обратного вызова](#схема-payload-обратного-вызова),
описанный ниже — при каждом событии (начало атаки, завершение атаки и `update_available`).
Ответы с кодом, отличным от 2xx, считаются ошибкой доставки и учитываются в счётчиках.
Запрос имеет ограниченный таймаут, поэтому зависший получатель не сможет занять слот
доставки.

<Callout type="warning" title="Схема URL webhook не проверяется">
В отличие от Slack (для которого Kapkan требует HTTPS), схема URL универсального webhook не
проверяется. URL вида `http://` принимается, и payload — который может нести цели и детали
источника атаки — отправляется в открытом виде. Используйте конечную точку `https://`.
</Callout>

```http
POST /your/endpoint HTTP/1.1
Content-Type: application/json
```

### Exec-хук

Задайте в `notify.exec.command` абсолютный путь к существующему исполняемому файлу. На
каждом событии Kapkan запускает его напрямую — **без shell** — со следующими параметрами:

- JSON-[payload обратного вызова](#схема-payload-обратного-вызова) на **stdin**.
- Имя события (`attack_started`, `attack_ended` или `update_available`) в качестве **`argv[1]`**.
- **Минимальное окружение**: передаются только `PATH`, `HOME`, `TZ`, `LANG`, `USER` и
  `TMPDIR`. Секреты демона (токены ботов, учётные данные SMTP) никогда не наследуются
  хуком или его потомками.

Команда должна существовать и быть исполняемой на момент загрузки конфигурации.
`timeout_seconds` (по умолчанию `10`) ограничивает каждый запуск; при истечении таймаута
Kapkan завершает **всю группу процессов** хука, чтобы его «внуки» не выжили в виде сирот.
Хук получает ту же схему, что и webhook, поэтому один скрипт может обслуживать оба канала.

Соглашение о вызове выбирается через `notify.exec.format`: `kapkan` (по умолчанию,
описано выше) или `fastnetmon` (см. ниже).

<Callout type="danger" title="Хук выполняется от имени пользователя демона">
Exec-команда выполняется с привилегиями процесса Kapkan. Относитесь к скрипту как к части
вашей доверенной базы, держите его на пути, доступном для записи только root, и никогда не
передавайте недоверенные, контролируемые атакующим поля обратно в shell.
</Callout>

### Режим, совместимый с FastNetMon

Переходите с FastNetMon? Задайте `notify.exec.format: fastnetmon`, и Kapkan будет вызывать
хук точно так же, как это делает `notify_script` FastNetMon — поэтому ваши существующие
скрипты уведомлений работают без изменений:

- **`argv`** — это `<ip> <direction> <pps> <action>`.
- **stdin** несёт текстовую сводку об атаке (тело `attack_details` FastNetMon).

```yaml
notify:
  exec:
    command: "/usr/local/bin/notify_about_attack.sh"
    format: fastnetmon
```

`action` — одно из `ban`, `unban` или `attack_details`, отображённое из события Kapkan:

| Событие Kapkan | `action` |
| --- | --- |
| атака началась, хост отправлен в blackhole или отведён | `ban` |
| атака началась, только оповещение (без блокировки) | `attack_details` |
| атака завершилась, блокировка снята | `unban` |
| **любое событие уровня хоста, пока `dry_run` равно true** | `attack_details` |
| событие уровня группы (`calculation: total`) | *пропускается — хук не вызывается* |
| событие `update_available` | *пропускается — нет эквивалента в FastNetMon* |

<Callout type="warning" title="Dry-run никогда не порождает ban/unban">
В режиме dry-run Kapkan ничего не анонсирует — но скрипт блокировки FastNetMon *всё равно*
поставил бы хост в firewall. Поэтому в dry-run каждое событие понижается до информационного
`attack_details`, а не `ban`/`unban`, и скрипт FastNetMon не сможет заблокировать хост,
который вы лишь проверяете. Атаки уровня группы (`calculation: total`) не имеют единственного
хоста и в этом режиме пропускаются.
</Callout>

## Конфигурация

Полный блок `notify`. Каждый канал опционален — пустое значение `smtp_host`, `url`,
`webhook_url` или `command` отключает соответствующий канал.

```yaml
notify:
  telegram:
    token_env: "KAPKAN_TG_TOKEN"   # token is read from this env var, never the file
    chat_id: "-1001234567890"
  webhook:
    url: ""                        # generic JSON POST on every attack start/end
    # payload schema: docs/callback-schema.json (check schema_version)
  slack:
    webhook_url: ""                # Slack incoming webhook
  email:                           # SMTP; credentials from env vars
    smtp_host: ""                  # "mail.example.com:587"; empty disables
    from: ""
    to: []                         # ["ops@example.com"]
    username_env: "KAPKAN_SMTP_USER"
    password_env: "KAPKAN_SMTP_PASS"
    require_tls: false             # set true to fail (not send cleartext) if the server has no STARTTLS
  exec:                            # hook run on every attack event (no shell)
    command: ""                    # absolute path to an existing executable
    timeout_seconds: 10
    format: kapkan                 # or "fastnetmon" for FastNetMon-style argv/stdin
```

## Схема payload обратного вызова

Тело webhook и stdin exec-хука несут один и тот же JSON-объект. Он версионирован полем
`schema_version`, в настоящее время — `1`. Авторитетное определение находится в файле
[`docs/callback-schema.json`](https://github.com/fornex/kapkan/blob/main/docs/callback-schema.json).

<Callout type="info" title="Прямая совместимость">
Потребители ОБЯЗАНЫ проверять `schema_version` и ДОЛЖНЫ игнорировать неизвестные поля.
Версия повышается только при ломающих изменениях, поэтому новые поля могут добавляться,
ничего у вас не ломая.
</Callout>

| Поле | Тип | Примечания |
| --- | --- | --- |
| `schema_version` | integer | В этом релизе всегда `1`. |
| `event` | string | `attack_started`, `attack_ended` или `update_available` (существует более новая версия). |
| `scope` | string | `host` (атакуется одна `target`) или `group` (суммарный трафик hostgroup с `calculation: total`; задано `group`, `target` отсутствует). |
| `target` | string | Атакуемый хост — или, для исходящих атак, атакующий хост. Отсутствует для событий уровня группы. |
| `group` | string | Имя владеющей hostgroup; `global` для неявной резервной группы. |
| `direction` | string | `incoming` или `outgoing`, когда источником атаки является сама цель (вероятно, скомпрометирована). |
| `metric` | string | Сработавший порог (например, `pps`, `mbps`, `udp_pps`, `tcp_syn_pps`, `frag_pps`). |
| `rate` | number | Наблюдаемое значение сработавшей метрики, скорректированное на выборку. |
| `threshold` | number | Настроенный предел этой метрики на момент начала атаки. |
| `pps` | number | Пакеты в секунду с поправкой на выборку. |
| `mbps` | number | Мегабиты в секунду с поправкой на выборку. |
| `flows_per_sec` | number | Потоки в секунду с поправкой на выборку. |
| `ban_state` | string | `active`, `withdrawn` или `rejected`, когда выполнялась проверка подавления атаки. |
| `method` | string | `blackhole`, `flowspec` или `divert` — метод подавления, породивший блокировку; отсутствует, когда блокировки нет. |
| `route` | string | Анонс (в режиме dry-run: предполагаемый): маршрут blackhole или сводка `flowspec: ...` сгенерированных правил. |
| `dry_run` | boolean | Был ли механизм подавления в режиме dry-run для этого события. |
| `at` | string | Временная метка события в формате RFC 3339. |
| `classification` | object | Выведенный вектор атаки — только для `attack_started`. |
| `sample` | object | Выборка потоков, захваченная до срабатывания порога — только для `attack_started`, когда включён буфер трафика. |
| `current_version` | string | Текущая сборка — только для `update_available`. |
| `latest_version` | string | Тег более новой версии — только для `update_available`. |
| `security` | boolean | `true`, когда релиз помечен как связанный с безопасностью — только для `update_available`. |
| `release_url` | string | URL релиза — только для `update_available`. |

Объект `classification` несёт выведенный `type` (например, `ntp_amplification`,
`syn_flood`, `udp_flood` или `mixed`, когда ни одна сигнатура не совпала), долю
уверенности `confidence` от `0` до `1` и `src_port` для векторов усиления (amplification).
Объект `sample` несёт усечённый набор потоков `flows`, а также счётчики top-K — `top_sources`,
`top_src_ports`, `top_dst_ports` и `protocols` — и знаменатель `total_packets`.

    "type": "ntp_amplification",
