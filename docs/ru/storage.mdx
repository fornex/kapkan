# Хранилище (ClickHouse)

Укажите Kapkan на сервер ClickHouse, чтобы хранить историю атак и трафика — ответ на вопрос
«что нас атаковало в прошлый вторник». Без этого Kapkan работает полностью в памяти процесса
на «живых» данных.

Для этого вам нужен запущенный сервер ClickHouse (любой свежей версии). Kapkan обращается к его
HTTP-интерфейсу, который по умолчанию слушает порт `8123` — именно этот порт указан в примере
URL ниже. Установка ClickHouse выходит за рамки этого раздела; см. [clickhouse.com](https://clickhouse.com).

<Callout type="info" title="По принципу best-effort, никогда не блокирует обнаружение">
Сохранение работает через ограниченную очередь с неблокирующей отправкой. Медленный или
недоступный ClickHouse приводит к отбрасыванию строк (учитываются в `kapkan_storage_rows_total` со
значением `result="dropped"`), а не к задержке горячего пути движка.
</Callout>

## Включение

```yaml
storage:
  clickhouse:
    url: "http://127.0.0.1:8123"   # empty/absent disables persistence
    database: "kapkan"             # created if absent
    username_env: "KAPKAN_CH_USER" # optional; credentials come from the env
    password_env: "KAPKAN_CH_PASS"
    ttl_days: 7                    # rows auto-expire (per-row TTL)
    flush_interval_seconds: 5
    batch_size: 1000
    queue_size: 100000
    traffic_interval_seconds: 10
```

Учётные данные читаются из указанных переменных окружения (`username_env` / `password_env`),
а не из конфигурационного файла. Опустите оба параметра, если ваш ClickHouse принимает
локальные подключения без аутентификации (стандартный пользователь `default` часто так и
настроен) — задавайте их только тогда, когда ваш сервер требует аутентификации.

Изменение любой настройки хранилища требует полного перезапуска — перезагрузка конфигурации,
затрагивающая блок `storage`, отклоняется (`reload: storage settings cannot change at runtime (restart required)`).

## Проверка

После включения хранилища убедитесь, что строки действительно записываются — сохранение
работает по принципу best-effort и отказывает тихо. Две проверки:

- Следите за ростом метрики `kapkan_storage_rows_total{result="written"}` (в сравнении с
  `result="dropped"` или `result="error"`). См. [Метрики](/docs/metrics).
- Запросите ClickHouse напрямую:
  ```bash
  curl 'http://127.0.0.1:8123/?query=SELECT%20count()%20FROM%20kapkan.attack_events'
  ```

При неверных учётных данных или проблеме с правами движок записывает в журнал
`clickhouse schema init failed` при запуске и `clickhouse insert failed`, когда пакет
отклоняется — если счётчики остаются на нуле, сначала проверьте журналы.

## Как это работает

Kapkan обращается к **HTTP-интерфейсу** ClickHouse средствами стандартной библиотеки Go — без
зависимости от драйвера. Единственная внешняя зависимость — сам сервер ClickHouse.

При запуске он идемпотентно создаёт три таблицы MergeTree:

| Таблица | Содержимое |
| --- | --- |
| `attack_events` | Каждое начало и окончание атаки: тип, направление, скорости, топ источников из выборки, состояние блокировки и `reason` обнаружения. |
| `traffic` | Периодические снимки скорости и базового уровня по каждому хосту (каждые `traffic_interval_seconds`). |
| `audit_events` | Изменения с привязкой к оператору: кто инициировал каждую блокировку/разблокировку/перезагрузку конфигурации, когда и с каким результатом. См. [Журнал аудита](/docs/audit). |

Каждая таблица использует построчный TTL `ttl_days`, поэтому срок хранения ограничен без
вмешательства оператора.

Учётной записи для записи на первом запуске нужны права `CREATE` (база данных + таблицы) и
`INSERT`; в дальнейшем достаточно `INSERT`. Сбои при создании схемы записываются в журнал при
запуске и не являются фатальными.

Столбец `reason` таблицы `attack_events` содержит компактный JSON причины обнаружения
([reason](/docs/detection)) — почему сработала атака (происхождение порога,
прогрев, доли протоколов) — и пуст в строке окончания атаки. Существующие развёртывания получают
столбец автоматически: инициализация схемы выполняет `ADD COLUMN IF NOT EXISTS`, который молча
пропускается, если у учётной записи для записи нет прав `ALTER` (на новых установках он уже есть).

## Обратное давление (backpressure)

Строки помещаются в ограниченную очередь (`queue_size`) с неблокирующей отправкой и
сбрасываются пакетами (`batch_size` / `flush_interval_seconds`). Если ClickHouse работает
медленно или недоступен, строки отбрасываются — учитываются в `kapkan_storage_rows_total` со
значением `result="dropped"` — а движок продолжает работу. См. [Метрики](/docs/metrics).

## Что сохраняется, а что нет

Таблица `traffic` сейчас сохраняет только снимки по каждому хосту. Если настроен опциональный
блок `geoip` (база данных MaxMind `.mmdb`), Kapkan определяет ASN по источникам и сохраняет
топ ASN каждой атаки в `attack_events.top_asns` (объединённые через символ `|` записи вида
`"AS<n> <org>"`; пусто, когда geoip отключён). Временные ряды по ASN и итоги по hostgroup пока
не снимаются — оба варианта рассматриваются для последующей доработки.
