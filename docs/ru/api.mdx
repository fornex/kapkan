# REST API

Kapkan предоставляет небольшой JSON REST API для наблюдения за обнаружением и управления
ручным подавлением атак. Он обслуживается по адресу из `api.listen`, который по умолчанию равен
`127.0.0.1:8080`. Тот же слушатель также обслуживает встроенную [панель управления](/docs/dashboard) и
конечную точку Prometheus [`/metrics`](/docs/metrics).

API преимущественно работает на чтение: статус, активные и недавние атаки, снимок отслеживаемых
хостов и таблица блокировок — всё это `GET`-запросы. Три `POST`-конечные точки изменяют состояние: ручная
блокировка, ручное снятие блокировки и перезагрузка конфигурации.

<Callout type="warning" title="Настройте аутентификацию перед публикацией наружу">
По умолчанию `api.listen` привязывается к `127.0.0.1`, поэтому без аутентификации API безопасен
только на localhost. Прежде чем привязывать его за пределами loopback, установите bearer-токен.
См. [Аутентификация](/docs/authentication).
</Callout>

Все маршруты `/api/v1` проходят проверку токена, когда настроена аутентификация; после
настройки каждый запрос должен содержать `Authorization: Bearer <token>`. Токены несут
**роль**: `GET`-маршрутам нужна роль **viewer**, а изменяющим состояние маршрутам (`ban`,
`unban`, `config/reload`) — роль **operator**. `config/reload` идёт дальше: ему требуется
**непривязанный к тенанту (admin)** токен, поэтому даже токен operator, привязанный к тенанту,
отклоняется с `403`. Чтение с токеном viewer работает; изменение с
токеном viewer возвращает `403`; неизвестный токен возвращает `401`. `POST`-конечные точки
дополнительно требуют JSON-типа содержимого — отправляйте `Content-Type: application/json`.
`POST` без этого заголовка отклоняется с HTTP `415 Unsupported Media Type`, поэтому если блокировка
через `curl` падает с 415, то почти всегда причина именно в отсутствующем заголовке.
Обратите внимание, что `/metrics` и статическая оболочка панели управления обслуживаются без
токена; данные, которые загружает панель, запрашиваются через защищённый API. Роли токенов см. в
разделе [Аутентификация](/docs/authentication).

Токен также может быть **привязан к тенанту**: тогда на каждой конечной точке чтения он видит только
строки своего тенанта (`/status` пересобирается под его собственные хост-группы и счётчики) и
может выполнять `ban`/`unban` только в пределах своего тенанта — цель вне тенанта возвращает
единообразный `403`. Токен, привязанный к тенанту, вообще не может перезагрузить конфигурацию:
`config/reload` доступен только непривязанным к тенанту (admin) токенам и иначе возвращает
`403`. См. [Мультиарендность](/docs/multi-tenancy).

## Конечные точки

| Метод | Путь | Описание |
| --- | --- | --- |
| GET | `/api/v1/status` | Режим, время работы, защищаемые сети, пороги, хост-группы, счётчики активных атак/блокировок. |
| GET | `/api/v1/attacks` | Активные атаки плюс последние 100 завершившихся, с выборками и классификацией. |
| GET | `/api/v1/hosts` | Снимок отслеживаемых хостов: скорости по направлениям, выученные базовые уровни, состояние атаки. |
| GET | `/api/v1/bans` | Все блокировки — активные и исторические. |
| GET | `/api/v1/traffic` | Сохранённая история скоростей по хостам (viewer); `{available:false, points:[]}`, когда хранилище отключено. |
| GET | `/api/v1/audit` | Журнал аудита изменений с привязкой к оператору (кто заблокировал/перезагрузил, когда, результат). |
| POST | `/api/v1/ban` | Вручную заблокировать адрес. |
| POST | `/api/v1/unban` | Вручную снять блокировку. |
| POST | `/api/v1/config/reload` | Перечитать файл конфигурации (то же, что `SIGHUP`). |
| GET | `/metrics` | Метрики Prometheus. |
| GET | `/healthz` | Неаутентифицированная проверка liveness/readiness: `503` до полного запуска, затем `200`. |

Все ответы в формате JSON. Ошибки возвращают объект вида `{"error": "..."}` с соответствующим
HTTP-статусом.

### GET /api/v1/status

Возвращает текущий режим работы и сводку того, что защищает Kapkan. Чтобы прочитать его
(уберите заголовок `Authorization`, если токен не настроен):

```sh
curl -fsS localhost:8080/api/v1/status \
  -H "Authorization: Bearer $TOKEN" | jq
```

```json
{
  "dry_run": true,
  "uptime_seconds": 8123,
  "version": "v1.2.0 · a1b2c3d",
  "update_available": false,
  "role": "operator",
  "unscoped": true,
  "networks": ["203.0.113.0/24", "198.51.100.0/24"],
  "active_attacks": 1,
  "active_bans": 1,
  "thresholds": {
    "pps": 80000,
    "mbps": 1000,
    "flows_per_sec": 20000
  },
  "hostgroups": [
    { "name": "web", "calculation": "per_host", "mitigation": "blackhole", "ban": true }
  ]
}
```

`dry_run` сообщает глобальный режим: когда он `true`, ни один blackhole не анонсируется вашим
маршрутизаторам. `version` — это работающая сборка, всегда присутствует и не порождает исходящего
трафика. `update_available` равен `false`, если только не включён опциональный
[`update_check`](/docs/configuration) и он не находит более новый релиз — тогда он равен `true`, а
рядом с ним включаются `latest_version`, `latest_is_security` и `latest_url`. См.
[Обновление](/docs/upgrading). `thresholds` и `hostgroups`
отражают активную конфигурацию; полную их структуру см. в разделах
[Обнаружение](/docs/detection) и [Хост-группы](/docs/hostgroups).
`role` и `unscoped` присутствуют всегда — они сообщают панели управления, какой токен она
использует. Поля `networks`, `thresholds`, `bgp`, `scrubbing` и `notify` — это
общая для всего развёртывания конфигурация, и они возвращаются только непривязанному к тенанту
(admin) токену; привязанный к тенанту токен получает объект статуса без них (только `dry_run`,
`version`, `uptime_seconds`, `update_available`, `role`, `unscoped`, его собственные `hostgroups`
и счётчики). Admin-токен дополнительно получает объекты `bgp`, `scrubbing` и `notify` — `notify`
сообщает лишь о том, какие каналы включены, и никогда не раскрывает токены или URL. Поэтому если
поле кажется «пропавшим», проверьте, не привязан ли ваш токен к тенанту, прежде чем считать, что
Kapkan сломан.

### GET /api/v1/attacks

Возвращает текущие активные атаки плюс последние 100 завершившихся (новейшие первыми). Оба
массива содержат объекты `Attack`.

| Поле | Тип | Примечания |
| --- | --- | --- |
| `scope` | string | `host` или `group`. Атаки с областью group не содержат `target`. |
| `target` | string | Атакуемый адрес (область host). |
| `group` | string | Имя хост-группы (область group или группа хоста). |
| `tenant` | string | Тенант владеющей группы, когда он размечен (см. [Мультиарендность](/docs/multi-tenancy)). |
| `direction` | string | `incoming` или `outgoing`. |
| `metric` | string | Сработавшая метрика, например `pps`, `mbps`, `tcp_syn_pps`. |
| `rate` | number | Наблюдаемая скорость в момент обнаружения, в единицах метрики. |
| `threshold` | number | Пересечённый порог. |
| `rates` | object | Полная разбивка скоростей по протоколам (см. ниже). |
| `active` | bool | `true`, пока атака продолжается. |
| `ban_state` | string | `active`, `withdrawn` или `rejected`. Отсутствует при отсутствии блокировки. |
| `method` | string | Метод подавления: `blackhole`, `flowspec` или `divert`. Отсутствует при отсутствии блокировки. |
| `route` | string | Строка маршрута (`blackhole …` / `divert …`) или сводка `flowspec: ...`, когда блокировка существует. |
| `flowspec` | array | Сгенерированные правила [FlowSpec](/docs/flowspec), когда `method` равен `flowspec`. |
| `dry_run` | bool | Была ли блокировка виртуальной. |
| `started_at` | string | Временная метка RFC 3339. |
| `ended_at` | string | Временная метка RFC 3339; отсутствует, пока атака активна. |
| `sample` | object | Выборка потоков, захваченная при обнаружении; отсутствует, когда выборка отключена. |
| `classification` | object | Выведенный вектор атаки; отсутствует, когда атака не классифицирована. |
| `reason` | object | Почему сработало обнаружение — происхождение порога, прогрев, доли по протоколам. Прикрепляется в начале. |

```json
{
  "active": [
    {
      "scope": "host",
      "target": "203.0.113.66",
      "group": "web",
      "direction": "incoming",
      "metric": "pps",
      "rate": 412000,
      "threshold": 80000,
      "rates": {
        "pps": 412000,
        "mbps": 3100,
        "flows_per_sec": 9800,
        "udp_pps": 405000,
        "udp_mbps": 3080
      },
      "active": true,
      "ban_state": "active",
      "route": "blackhole 203.0.113.66/32 next-hop 192.0.2.1 community 65000:666",
      "dry_run": false,
      "started_at": "2026-06-13T09:41:07Z",
      "sample": {
        "top_sources": [
          { "key": "198.51.100.23", "packets": 1240000, "bytes": 1612000000 }
        ],
        "top_src_ports": [
          { "key": "123", "packets": 1180000, "bytes": 1534000000 }
        ],
        "top_dst_ports": [
          { "key": "443", "packets": 1240000, "bytes": 1612000000 }
        ],
        "protocols": [
          { "key": "udp", "packets": 1240000, "bytes": 1612000000 }
        ],
        "total_packets": 1240000
      },
      "classification": {
        "type": "ntp_amplification",
        "confidence": 0.95,
        "src_port": 123
      }
    }
  ],
  "recent": []
}
```

Объект `rates` содержит базовую тройку (`pps`, `mbps`, `flows_per_sec`) плюс ненулевые поля по
протоколам: `tcp_pps`, `tcp_mbps`, `udp_pps`, `udp_mbps`, `icmp_pps`, `icmp_mbps`,
`tcp_syn_pps`, `tcp_syn_mbps`, `frag_pps`, `frag_mbps`. Объект `sample` обобщает буферизованные
потоки, лежащие в основе обнаружения — `top_sources`, `top_src_ports`, `top_dst_ports`,
`protocols` (каждый представляет собой список счётчиков `{key, packets, bytes}`),
необязательный сырой список `flows` и `total_packets`. Поле `type` в `classification` — это
один из векторов, описанных в разделе [Обнаружение](/docs/detection); `confidence` — это
доля (`0..1`) трафика атаки, совпадающего с победившей сигнатурой, а `src_port` — отражённый
сервисный порт для векторов амплификации.

Объект `reason` объясняет, **почему** сработало обнаружение — фиксируется один раз в начале,
вне горячего пути. `threshold_source` равен `static` или `baseline`: пришёл ли пересечённый
предел из статической конфигурации или из прогретого [выученного базового уровня](/docs/baselines).
Когда он равен `baseline`, объект `baseline` содержит фактическую формулу
(`min(ceiling, max(floor, normal × factor))`). `baseline_configured`, `warming_up` и
`warmup_remaining_seconds` объясняют статический порог, который применился только потому, что
базовый уровень ещё не прогрелся. `shares` — это доля каждого протокола в общем PPS, которая
определила классификацию, а `dominant_share_gate` — доля, которую один протокол должен набрать,
чтобы выиграть вектор (иначе атака считается `mixed`). О том, как это читать, см.
[Обнаружение](/docs/detection).

```json
{
  "reason": {
    "threshold_source": "baseline",
    "baseline": { "normal": 1200, "factor": 8, "floor": 5000, "ceiling": 80000 },
    "baseline_configured": true,
    "shares": { "udp": 0.98, "syn": 0, "tcp": 0.01, "icmp": 0, "frag": 0 },
    "dominant_share_gate": 0.5
  }
}
```

### GET /api/v1/hosts

Возвращает снимок каждого отслеживаемого хоста — данные top-talkers. Каждая запись представляет
собой `HostStat`.

| Поле | Тип | Примечания |
| --- | --- | --- |
| `target` | string | Адрес хоста. |
| `group` | string | Имя хост-группы хоста. |
| `rates` | object | Текущие оконные входящие скорости. |
| `rates_out` | object | Исходящие скорости; ненулевые только при включённом обнаружении исходящего трафика. |
| `in_attack` | bool | Находится ли хост в какой-либо активной атаке. |
| `metric` | string | Метрика активной атаки; отсутствует, когда атаки нет. |
| `direction` | string | Направление активной атаки; отсутствует, когда атаки нет. |
| `baseline` | object | Выученный входящий базовый уровень; присутствует, когда базовые уровни настроены. |
| `baseline_out` | object | Выученный исходящий базовый уровень; присутствует, когда базовые уровни настроены. |

```json
{
  "hosts": [
    {
      "target": "203.0.113.66",
      "group": "web",
      "rates": {
        "pps": 412000,
        "mbps": 3100,
        "flows_per_sec": 9800,
        "udp_pps": 405000
      },
      "rates_out": {
        "pps": 120,
        "mbps": 2
      },
      "in_attack": true,
      "metric": "pps",
      "direction": "incoming",
      "baseline": {
        "pps": 950,
        "mbps": 7,
        "flows_per_sec": 60
      },
      "baseline_out": {
        "pps": 110,
        "mbps": 2,
        "flows_per_sec": 14
      }
    }
  ]
}
```

Объекты `baseline` и `baseline_out` содержат базовую тройку выученной нормы (`pps`, `mbps`,
`flows_per_sec`) и появляются только при настроенных EWMA-базовых уровнях для группы хоста.
См. [Базовые уровни](/docs/baselines).

### GET /api/v1/bans

Возвращает полную таблицу блокировок — активных и исторических. Каждая запись представляет собой `Ban`.

| Поле | Тип | Примечания |
| --- | --- | --- |
| `target` | string | Заблокированный адрес. |
| `prefix` | string | Префикс blackhole (`/32` или `/128`). |
| `metric` | string | Метрика, вызвавшая блокировку; отсутствует для ручных блокировок. |
| `rate` | number | Наблюдаемая скорость в момент блокировки; отсутствует, когда равна нулю. |
| `threshold` | number | Пересечённый порог; отсутствует, когда равен нулю. |
| `next_hop` | string | Next-hop для отбрасывания трафика. |
| `community` | string | Набор community, прикреплённый к маршруту, — community RTBH для blackhole, community отвода (очистки) для блокировки divert — соединяются через пробел, когда их больше одного. |
| `local_pref` | number | `LOCAL_PREF`, прикреплённый к маршруту; отсутствует, когда равен нулю. |
| `route` | string | Полная строка маршрута или сводка `flowspec: ...` для блокировок FlowSpec. |
| `state` | string | `active`, `withdrawn` или `rejected`. |
| `dry_run` | bool | Была ли блокировка виртуальной. |
| `manual` | bool | `true` для блокировок, запрошенных оператором. |
| `started_at` | string | Временная метка RFC 3339. |
| `expires_at` | string | Истечение TTL; блокировки никогда не бывают постоянными. |
| `withdrawn_at` | string | Когда маршрут был отозван; отсутствует, пока блокировка активна. |
| `reason` | string | Почему блокировка была отклонена или отозвана; иначе отсутствует. |
| `method` | string | Метод подавления: `blackhole`, `flowspec` или `divert`. |
| `flowspec` | array | Сгенерированные правила [FlowSpec](/docs/flowspec), когда `method` равен `flowspec`. |
| `escalation` | array | Настроенная [лестница эскалации](/docs/escalation), когда она задана. |
| `escalation_step` | number | Индекс текущей ступени лестницы. |

```json
{
  "bans": [
    {
      "target": "203.0.113.66",
      "prefix": "203.0.113.66/32",
      "metric": "pps",
      "rate": 412000,
      "threshold": 80000,
      "next_hop": "192.0.2.1",
      "community": "65000:666",
      "route": "blackhole 203.0.113.66/32 next-hop 192.0.2.1 community 65000:666",
      "state": "active",
      "dry_run": false,
      "manual": false,
      "started_at": "2026-06-13T09:41:07Z",
      "expires_at": "2026-06-13T10:41:07Z"
    }
  ]
}
```

См. [RTBH: подавление атак](/docs/mitigation) о том, как TTL, гистерезис и лимит блокировок формируют этот
жизненный цикл.

### GET /api/v1/audit

Возвращает журнал аудита с привязкой к оператору — кто инициировал каждый `ban`, `unban` или
`config_reload`, когда и каков результат (включая **отклонённые** действия). Записи идут
новейшими первыми, а конечная точка привязана к тенанту на стороне сервера. Он принимает необязательные
query-параметры `from`/`to` (RFC 3339), `action` (`ban`/`unban`/`config_reload`) и `target`
(IP-адрес), и по умолчанию возвращает данные за последний час:

```json
{
  "available": true,
  "events": [
    {
      "event_time": "2026-06-22 03:14:09",
      "action": "ban",
      "result": "rejected",
      "operator": "alice",
      "role": "operator",
      "tenant": "customerA",
      "target": "203.0.113.66",
      "target_type": "host",
      "reason": "whitelisted",
      "source": "api",
      "ban_state": "rejected",
      "dry_run": 0
    }
  ]
}
```

Когда хранилище отключено, ответ равен `{"available": false, "events": []}`. Обратите внимание,
что в записях аудита `dry_run` кодируется целым числом (`0`/`1`), а не JSON-булевым значением,
используемым в других местах на этой странице. Полный список полей, правила запросов, а также
модель идентификации оператора и привязки к тенанту см. на отдельной странице
[Журнал аудита](/docs/audit).

### POST /api/v1/ban

Вручную помещает адрес в blackhole. Тело запроса — один IP:

```http
POST /api/v1/ban
Content-Type: application/json

{"ip": "203.0.113.66"}
```

В виде готовой команды (уберите заголовок `Authorization`, если токен ещё не настроен):

```sh
curl -fsS -X POST localhost:8080/api/v1/ban \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"ip":"203.0.113.66"}'
```

Ответом является результирующий объект `Ban`. Ручная блокировка соблюдает все те же правила
безопасности, что и автоматическая:

- **Внесённая в белый список** цель отклоняется с HTTP `409 Conflict` и `reason: "whitelisted"`; она никогда не анонсируется.
- Цель **за пределами настроенных `networks`** отклоняется с HTTP `409` и `reason: "outside configured networks"`.
- Запрос, который превысил бы **`max_active_bans`**, отклоняется с HTTP `409` и `reason: "max_active_bans reached"`.

В каждом случае `409` телом по-прежнему является объект `Ban` со `state: "rejected"` и
заполненным полем `reason`. Недопустимый или неразбираемый IP возвращает HTTP `400`.

<Callout type="warning" title="Dry-run по-прежнему действует">
Ручная блокировка учитывает глобальный режим. Пока `dry_run` равен `true`, блокировка записывается и
возвращается с `dry_run: true`, но ни один маршрут не анонсируется. См. [Модель безопасности](/docs/safety).
</Callout>

### POST /api/v1/unban

Отзывает активную блокировку для заданного адреса. Тело имеет ту же структуру `{"ip": "..."}` и
требуется тот же JSON-тип содержимого:

```http
POST /api/v1/unban
Content-Type: application/json

{"ip": "203.0.113.66"}
```

В случае успеха возвращается отозванный `Ban`. Если для адреса нет активной блокировки, возвращается
HTTP `404`.

### POST /api/v1/config/reload

Перечитывает файл конфигурации с диска и применяет его — тот же эффект, что и при отправке
`SIGHUP`. Тело запроса пустое; JSON-тип содержимого всё равно требуется. В отличие от
`ban`/`unban`, этой конечной точке требуется **непривязанный к тенанту (admin)** токен:
перезагрузка подменяет всю конфигурацию целиком, поэтому токен, привязанный к тенанту,
отклоняется с `403` и сообщением `config reload is restricted to unscoped (admin) tokens`.

```json
{
  "reloaded": true,
  "dry_run": false,
  "thresholds": {
    "pps": 80000,
    "mbps": 1000,
    "flows_per_sec": 20000
  }
}
```

Если новая конфигурация не парсится или не проходит проверку, работающая конфигурация остаётся
нетронутой, а конечная точка возвращает HTTP `400` с ошибкой. См. [Конфигурация](/docs/configuration).

### GET /metrics

Отдаёт метрики Prometheus в стандартном текстовом формате экспозиции. Эта конечная точка не находится
под `/api/v1` и обслуживается без bearer-токена, чтобы до него мог достучаться сборщик метрик.
Полный список метрик см. в разделе [Метрики](/docs/metrics).

### GET /healthz

Проверка liveness/readiness, которая не находится под `/api/v1` и обслуживается **без**
bearer-токена (она ничего не раскрывает), чтобы супервизор или скрипт обновления
[`update.sh`](/docs/upgrading) мог убедиться, что демон поднялся после перезапуска. Она
возвращает `503 starting`, пока не запущены все компоненты, затем `200 ok`. Поскольку слушатель
API начинает принимать запросы только после запуска демона, любой `200` здесь означает, что
конфигурация распарсилась, компоненты подняты, и Kapkan работает.

```sh
curl -fsS localhost:8080/healthz   # exits non-zero (503) until ready, then prints "ok"
```
