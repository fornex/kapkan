# Аутентификация

По умолчанию API и панель управления Kapkan работают **без аутентификации**. Это безопасно
только потому, что значение `api.listen` по умолчанию привязано к `127.0.0.1` — слушатель
доступен лишь с самого хоста, поэтому для чтения статуса, просмотра списка атак или
выдачи ручных блокировок токен не требуется. Как только вы привяжете слушатель к маршрутизируемому
адресу, эта защита исчезает.

<Callout type="danger" title="Задайте токен, прежде чем открывать слушатель">
Если вы измените `api.listen` на что-либо отличное от localhost — например, на `0.0.0.0:8080`
или конкретный интерфейс — сначала настройте токен. Слушатель без аутентификации на
маршрутизируемом адресе позволяет любому обратиться к [API](/docs/api), включая конечные точки
ручной блокировки и разблокировки.
</Callout>

## Настройка токена

Самый простой вариант — единственный токен оператора. Добавьте `token_env` в блок `api` — он
указывает имя переменной окружения, в которой хранится токен; сам секрет считывается из
окружения процесса при каждом запросе, а не из файла конфигурации:

```yaml
api:
  listen: "0.0.0.0:8080"
  token_env: "KAPKAN_API_TOKEN"   # token read from this env var, never the file
```

Затем передайте значение вне конфигурации, например через окружение вашего менеджера служб
или файл секретов:

```sh
export KAPKAN_API_TOKEN="$(openssl rand -hex 32)"
./kapkan -config /etc/kapkan/config.yaml
```

Хранение токена в переменной окружения, а не в YAML, означает, что конфигурацию по-прежнему
безопасно коммитить в git и сравнивать вместе с остальной частью вашей [конфигурации](/docs/configuration).

Этого единственного токена достаточно для большинства развёртываний — остальная часть этой
страницы (роли, арендаторы) необязательна и может подождать.

## Роли

Для доступа на основе ролей используйте `tokens` вместо `token_env` — список, где каждая запись
указывает переменную окружения с секретом и роль:

```yaml
api:
  listen: "0.0.0.0:8080"
  tokens:
    - { name: dashboard,  token_env: "KAPKAN_API_RO", role: viewer }
    - { name: automation, token_env: "KAPKAN_API_RW", role: operator }
```

`name`, `token_env` и `role` обязательны для каждой записи; `role` должна быть строго
`viewer` или `operator`. Пропущенное или написанное с ошибкой значение — это жёсткая ошибка
проверки при запуске, а не молчаливое значение по умолчанию.

| Роль | Что разрешено |
| --- | --- |
| `viewer` | Только чтение: статус, атаки, хосты, блокировки, метрики. |
| `operator` | Всё, что может viewer, плюс ручная блокировка/разблокировка и перезагрузка конфигурации (перезагрузка доступна только администратору — см. ниже). |

Отдельной роли `admin`, которую можно назначить, не существует — *администратор* — это просто
токен оператора без заданного `tenant`. Единственные значения ролей, которые принимает движок, —
это `viewer` и `operator`; различие между администратором и ограниченным токеном относится к оси
[мультиарендности](/docs/multi-tenancy), рассмотренной ниже.

`token_env` и `tokens` **взаимоисключающи** — `token_env` задаёт ровно один токен оператора.
Запрос сопоставляется с каждым настроенным токеном (за постоянное время); применяется
наивысшая совпавшая роль. Чтение с токеном viewer проходит; мутация с токеном viewer
отклоняется с `403 Forbidden`; запрос без действительного токена получает `401 Unauthorized`.
Пустая или незаданная переменная окружения никогда не совпадает, поэтому неправильно
настроенный секрет отказывает в доступе по умолчанию.

Давайте каждому токену отдельный секрет — если одно и то же значение секрета настроено для двух
токенов с разными ролями или арендаторами, запрос отклоняется (`401 Unauthorized`), вместо того
чтобы угадывать, какой из них вы имели в виду. В этом случае демон записывает в журнал
`ambiguous API token`.

*Значение* секрета токена считывается из окружения при каждом запросе, поэтому его ротация
вступает в силу со следующего запроса. Добавление или удаление *записей* токенов в YAML меняет
набор токенов, что вступает в силу при следующей [перезагрузке конфигурации](/docs/api). Ни то,
ни другое не требует перезапуска.

Токен может также нести необязательный `tenant`, ограничивая его данными одного клиента. Это
функция [мультиарендности](/docs/multi-tenancy), наложенная поверх описанных здесь ролей.

Ручная блокировка/разблокировка требуют лишь токена оператора, но `POST /api/v1/config/reload`
дополнительно [доступен только администратору](/docs/multi-tenancy): токен должен быть без
ограничения (не нести `tenant`), потому что перезагрузка переписывает политику каждого
арендатора и сам набор токенов. Токен оператора, ограниченный арендатором, отклоняется с
`403 Forbidden` — `config reload is restricted to unscoped (admin) tokens`.

<Callout type="info" title="Имя токена — это идентификатор оператора">
Имя (`name`) каждого токена — это то, что [журнал аудита](/docs/audit) записывает как `operator`
для каждой блокировки, разблокировки и перезагрузки конфигурации, которые он инициирует, — поэтому давайте
токенам осмысленные имена. На открытом (без токена) API нет субъекта для атрибуции, и поле
`operator` в аудите пусто.
</Callout>

## Что это защищает

Когда токен задан, каждый запрос к `/api/v1` должен передавать его в заголовке `Authorization`
по схеме bearer:

```http
GET /api/v1/status HTTP/1.1
Host: kapkan.internal:8080
Authorization: Bearer <token>
```

Переданный токен сравнивается с настроенным за **постоянное время**, поэтому само сравнение
не раскрывает секрет через тайминг.

Что остаётся открытым, а что становится защищённым:

| Путь | Когда токен задан |
| --- | --- |
| `GET /api/v1/*` | Требует токен **viewer** (или operator) |
| `POST /api/v1/*` | Требует токен **operator** **и** content type JSON |
| `GET /metrics` | Открыт — сбор метрик Prometheus остаётся без аутентификации |
| Статическая оболочка UI (`/`) | Открыта — но данные за ней нет |

Статическая оболочка UI панели управления остаётся доступной, чтобы браузер мог её загрузить,
но собственных данных у неё нет: она опрашивает `/api/v1`, поэтому запрашивает токен и хранит
его в `sessionStorage`. `/metrics` точно так же остаётся открытым для вашего сборщика
Prometheus. О том, как UI обрабатывает запрос токена, см. на странице
[панели управления](/docs/dashboard).

## CSRF

Конечные точки POST требуют `Content-Type: application/json` в дополнение к токену. Браузер,
выполняющий межсайтовый запрос, не может задать собственный заголовок `Authorization` без
предварительного запроса CORS (preflight), а простая отправка формы не может передать content
type JSON. Требование обоих условий — токена в заголовке запроса и content type JSON —
блокирует межсайтовую подделку запросов (CSRF) против конечных точек ручной блокировки, разблокировки и
перезагрузки конфигурации.

<Callout type="info" title="Роли и арендаторы — это разные оси">
Роль ограничивает то, что токен может *делать* (чтение или мутация). Необязательный `tenant`
токена ограничивает то, какие данные он может *видеть и менять*. Они сочетаются: viewer может
быть без ограничения (все арендаторы) или ограничен одним. См. [Мультиарендность](/docs/multi-tenancy).
</Callout>
