# Журнал аудита

Каждое изменяющее состояние действие API записывается в **аудиторский след** — это ответ на вопрос «кто
забанил `203.0.113.66` в 03:14 и кто перезагрузил конфигурацию». В каждой записи указаны оператор
(API-токен, выполнивший вызов), действие, цель и результат — включая
**отклонённые** действия, поскольку отказ в бане сам по себе является решением, подлежащим аудиту.

След пишется из обработчиков мутаций [API](/docs/api) и доступен через единственную
конечную точку чтения, `GET /api/v1/audit`. Он сохраняется в [ClickHouse](/docs/storage) рядом с
историей атак и трафика и **ограничен областью tenant** на стороне сервера — так же, как и любое другое чтение.

<Callout type="info" title="Всегда логируется, доступно для запросов при включённом хранилище">
Каждая мутация логируется на уровне `INFO` независимо от хранилища. Для истории, доступной по запросу, нужен
[ClickHouse](/docs/storage): при отключённом хранилище конечная точка возвращает `available: false`
(она никогда не выдаёт ошибку), и у вас по-прежнему есть строки аудита в логе процесса.
</Callout>

## Что записывается

Аудиту подвергаются три действия, каждое — как при успехе, так и при ошибке:

| Действие | Когда | Записываемые результаты (`result`) |
| --- | --- | --- |
| `ban` | `POST /api/v1/ban` | `active` (либо состояние бана `withdrawn`/`dry_run`) при успехе; `rejected`, когда защитный механизм отказывает. |
| `unban` | `POST /api/v1/unban` | `withdrawn` при успехе. |
| `config_reload` | `POST /api/v1/config/reload` | `ok` при успехе; `error`, когда новую конфигурацию не удаётся разобрать или провалидировать. |

**Отклонённое** действие аудируется намеренно: бан, отклонённый whitelist, заданными
`networks` или [защитой по радиусу поражения (blast-radius)](/docs/safety), оставляет запись аудита с `result:
"rejected"` и причиной отказа в `reason`. Поэтому аудиторский след показывает как попытки изменений, так и
применённые изменения.

## Идентичность оператора

Поле `operator` — это **имя** сопоставленного [API-токена](/docs/authentication) — то самое
`name`, которое вы задаёте каждой записи в списке `api.tokens`. Именно так след привязывает
действие к человеку или системе:

```yaml
api:
  tokens:
    - { name: alice,      token_env: KAPKAN_ALICE, role: operator }
    - { name: automation, token_env: KAPKAN_AUTO,  role: operator }
```

Бан, выданный с токеном `alice`, записывает `operator: "alice"`. На открытом (без токенов) API
нет субъекта для привязки, поэтому `operator` пустой — ещё одна причина
[задать токены](/docs/authentication) перед тем, как выставлять listener наружу.

## GET /api/v1/audit

Возвращает записи аудита в порядке от новых к старым. Требует роль **viewer**, как и остальные операции чтения.

| Параметр запроса | По умолчанию | Примечания |
| --- | --- | --- |
| `from` | now − 1h | Начало окна в формате RFC 3339. |
| `to` | now | Конец окна в формате RFC 3339. Должен быть позже `from`; окно не может превышать **31 days**. |
| `action` | (все) | Фильтр по одному из `ban`, `unban`, `config_reload`. |
| `target` | (все) | Фильтр по одному IP. Должен быть виден tenant вызывающей стороны, иначе запрос — `403`. |

```http
GET /api/v1/audit?from=2026-06-22T00:00:00Z&action=ban HTTP/1.1
Authorization: Bearer <token>
```

```json
{
  "available": true,
  "events": [
    {
      "event_time": "2026-06-22 03:14:09",
      "action": "ban",
      "result": "rejected",
      "operator": "alice",
      "role": "operator",
      "tenant": "customerA",
      "target": "203.0.113.66",
      "target_type": "host",
      "reason": "whitelisted",
      "source": "api",
      "ban_state": "rejected",
      "dry_run": 0
    }
  ]
}
```

Каждая запись содержит следующие поля:

| Поле | Примечания |
| --- | --- |
| `event_time` | UTC, `YYYY-MM-DD HH:MM:SS`. |
| `action` | `ban`, `unban` или `config_reload`. |
| `result` | `active`, `rejected`, `withdrawn`, `ok` или `error`. |
| `operator` | Имя сопоставленного API-токена; пустое в открытом (без токенов) режиме. |
| `role` | Роль вызывающей стороны (`viewer` / `operator`). |
| `tenant` | Область tenant вызывающей стороны; пустая для неограниченного (admin) вызова. |
| `target` | IP для `ban`/`unban`; пустой для `config_reload`. |
| `target_type` | `host` для `ban`/`unban`; `global` для `config_reload`. |
| `reason` | Детали отказа или ошибки; пустое при успехе. |
| `source` | `api` (действие оператора). Внутренние автоматические действия движка зарезервированы за будущим источником `auto`. |
| `ban_state` | Итоговое состояние бана для `ban`/`unban`; пустое для `config_reload`. |
| `dry_run` | `1`, когда развёртывание находится в режиме [dry-run](/docs/safety). |

Запрос с окном вне допустимого диапазона возвращает `400`; неизвестное значение `action` возвращает `400`;
некорректный `target` возвращает `400`. При отключённом хранилище ответ —
`{"available": false, "events": []}` со статусом `200`.

## Ограничение областью tenant

Конечная точка аудита **ограничена областью tenant на стороне сервера**, точно как `/attacks` и `/bans`.
[Токен, ограниченный областью tenant](/docs/multi-tenancy), видит только те записи, чей `tenant` совпадает с его
собственной областью; область привязывается во время аутентификации и не может быть расширена никаким параметром
запроса. Неограниченный (admin) токен видит записи всех tenant. Фильтр `target`, который
называет адрес вне tenant вызывающей стороны, отклоняется с `403`, поэтому журнал аудита не является
оракулом существования между tenant.

## Хранение

Записи аудита живут в таблице ClickHouse `audit_events` и наследуют тот же построчный TTL `ttl_days`,
что и остальное [хранилище](/docs/storage) — срок хранения ограничен без вмешательства
оператора. Сохранение работает по принципу best-effort через ту же ограниченную очередь: медленный или недоступный ClickHouse
отбрасывает строки, а не блокирует бан, при этом строка лога уровня `INFO` выводится всегда.
