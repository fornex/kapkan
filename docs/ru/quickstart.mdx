# Быстрый старт

Это руководство проведёт вас от пустого места до работающего экземпляра Kapkan в режиме dry-run,
который обнаруживает атаки на тестовый префикс. Шага сборки нет — Kapkan поставляется как единый
готовый подписанный бинарник, — и ничто из описанного здесь не способно анонсировать маршрут,
потому что режим dry-run включён по умолчанию.

<Callout type="info" title="Что вам нужно">
Хост на Linux (`amd64` или `arm64`). Чтобы видеть *реальные* обнаружения, вам нужен
маршрутизатор, экспортирующий телеметрию потоков — NetFlow, IPFIX или sFlow (записи о том, кто
кому отправляет трафик; см. [глоссарий](/docs/glossary)). Без маршрутизатора, но с клоном
исходного кода вы всё равно можете убедиться, что обнаружение работает, запустив набор тестов —
см. [последний раздел](#5-опционально-нет-маршрутизатора-проверьте-обнаружение-из-исходного-кода). Go-инструментарий
нужен только если вы пойдёте этим путём или если вы
[собираете из исходного кода](/docs/deployment#building-from-source) вместо загрузки.
</Callout>

## 1. Установка Kapkan

Выберите тот вариант, который подходит вашему хосту, — каждая сборка поставляется готовой и
подписанной для `linux` `amd64`/`arm64`, поэтому вам ничего не нужно компилировать.

<Callout type="info" title="Используйте последнюю сборку — замените v1.0.0">
Прежде чем копировать команды, замените `v1.0.0` в каждой команде ниже на новейший тег со
[страницы релизов](https://github.com/fornex/kapkan/releases/latest) (если скопировать `v1.0.0`
дословно, вы получите 404, когда это уже не текущая сборка), а `amd64` — на `arm64` на хостах
ARM. Имена архивов и пакетов опускают ведущую `v` (например, `kapkan_1.0.0_linux_amd64.tar.gz`).
</Callout>

### Debian / Ubuntu (`.deb`)

```sh
VER=v1.0.0
curl -fLO "https://github.com/fornex/kapkan/releases/download/$VER/kapkan_${VER#v}_linux_amd64.deb"
sudo apt install "./kapkan_${VER#v}_linux_amd64.deb"
```

Пакет устанавливает бинарник в `/usr/local/bin/kapkan`, создаёт непривилегированного пользователя
`kapkan`, формирует структуру `/etc/kapkan` с конфигурацией `config.yaml` в режиме dry-run и
устанавливает защищённый юнит systemd — оставленный **остановленным**, чтобы вы могли сначала
просмотреть конфигурацию. Страница [Промышленное развёртывание](/docs/deployment) описывает работу
с сервисом дальше. На RHEL / Fedora всё то же самое с `.rpm`:
`sudo dnf install ./kapkan_<ver>_linux_amd64.rpm`.

### Tarball (любой дистрибутив)

```sh
VER=v1.0.0
curl -fLO "https://github.com/fornex/kapkan/releases/download/$VER/kapkan_${VER#v}_linux_amd64.tar.gz"
tar xzf "kapkan_${VER#v}_linux_amd64.tar.gz"
```

Это распаковывает бинарник `kapkan` плюс каталог `deploy/` (юнит systemd, пример конфигурации и
`update.sh`). Остальная часть этой страницы использует tarball. Прежде чем запускать его в
промышленной среде, стоит [проверить загрузку](/docs/upgrading#проверка-загрузки-вручную) —
подпись cosign и SHA-256.

## 2. Запуск в режиме dry-run

Из распакованного tarball запустите Kapkan со встроенным примером конфигурации и логами в читаемом
виде:

```sh
./kapkan -config deploy/config.example.yaml -log-format text
```

Он работает на переднем плане и печатает логи в терминал; нажмите Ctrl-C, чтобы корректно его
остановить.

`-config` указывает на YAML-конфигурацию, а `-log-format text` переключает логи с используемого по
умолчанию структурированного JSON на читаемый человеком вид — полный список флагов и допустимые для
них значения см. в [Справочнике CLI](/docs/cli).

В режиме dry-run Kapkan не требует особых привилегий и может работать от обычного пользователя.
(При вводе в эксплуатацию BGP привязывается к порту 179, для чего нужны root или возможность
`CAP_NET_BIND_SERVICE` — поставляемый юнит systemd берёт это на себя; см.
[Промышленное развёртывание](/docs/deployment).)

Встроенный [`deploy/config.example.yaml`](/docs/configuration) содержит `dry_run: true`, тестовые
защищаемые сети `203.0.113.0/24` и `2001:db8::/32`, `protected_whitelist` и слушает sFlow на
`:6343`, а NetFlow/IPFIX — на `:2055`. Поскольку режим dry-run включён, Kapkan записывает
потенциальные blackhole-маршруты, но никогда не анонсирует маршрут.

<Callout type="info" title="Установили пакет?">
Если вы установили `.deb`/`.rpm`, тот же демон вместо этого работает как сервис с
`/etc/kapkan/config.yaml` — `sudo systemctl start kapkan`, затем `journalctl -u kapkan -f`. См.
[Промышленное развёртывание](/docs/deployment).
</Callout>

## 3. Направьте экспортёры на Kapkan

Настройте экспортёры потоков ваших маршрутизаторов на отправку данных на порты, которые слушает Kapkan:

| Протокол | Порт по умолчанию |
| --- | --- |
| sFlow v5 | `:6343` |
| NetFlow v5/v9 + IPFIX | `:2055` |

NetFlow и IPFIX используют один и тот же UDP-сокет. Должен быть настроен хотя бы один слушатель.

## 4. Понаблюдайте за работой

По умолчанию Kapkan отдаёт свой REST API и метрики на `127.0.0.1:8080`:

```sh
# Current mode, protected networks, thresholds, active attack/ban counts
curl -s localhost:8080/api/v1/status | jq

# Active attacks plus the last 100 that ended, with samples and classification
curl -s localhost:8080/api/v1/attacks | jq

# Prometheus metrics
curl -s localhost:8080/metrics | grep kapkan_
```

Также можно открыть встроенную [панель управления](/docs/dashboard) по адресу
[http://localhost:8080](http://localhost:8080) в браузере.

## 5. Опционально: нет маршрутизатора? Проверьте обнаружение из исходного кода

<Callout type="info" title="Это путь для разработчика, а не путь бинарника">
Этот шаг **не** использует готовый бинарник, который вы загрузили выше. Он запускает набор тестов
Kapkan, поэтому ему нужны клон исходного кода и [Go-инструментарий](https://go.dev/dl/). Если у
вас есть только `.deb`/`.rpm`/tarball и пока нет маршрутизатора, пропустите этот шаг — направьте
маршрутизатор на порты прослушивания (шаг 3), когда он у вас появится. Инструментарий
синтетических атак (`pkg/flowgen`) — это пакет набора тестов, а не команда, которую предоставляет
поставляемый бинарник.
</Callout>

Что это доказывает: Kapkan обнаруживает атаку и заблокировал бы жертву — и всё это без отправки
чего-либо реальному маршрутизатору, поскольку dry-run лишь записывает потенциальную блокировку
(*виртуальная блокировка* — маршрут [blackhole](/docs/glossary), который Kapkan показывает через
API, но никогда не анонсирует).

Тест отправляет поддельный флуд типа NTP-amplification на экземпляр в режиме dry-run через реальный
UDP-сокет и проверяет, что атака появляется в API вместе с виртуальной блокировкой, которая затем
автоматически истекает по своему TTL. Запустите его из клона исходного кода:

```sh
git clone https://github.com/fornex/kapkan && cd kapkan/engine
go test ./internal/app -run TestEndToEndNTPAmplification -v
```
