# Мультиарендность

Один экземпляр Kapkan может обслуживать множество клиентов (сценарий MSP или IDC) и выдавать
каждому токен, который видит и затрагивает **только их собственные** атаки, баны и хосты.
Арендатор (tenant) — это просто необязательная метка на [хост-группе](/docs/hostgroups): нет
ни нового объекта верхнего уровня, ни отдельного реестра арендаторов, а горячий путь
детектирования остаётся нетронутым.

```yaml
tenant: "house"                 # optional: label the global/fallback group

hostgroups:
  - name: custA-web
    tenant: "customerA"         # this group belongs to customerA
    networks: ["203.0.113.0/26"]
  - name: custA-dns
    tenant: "customerA"         # a tenant can span several groups
    networks: ["203.0.113.64/26"]
  - name: custB
    tenant: "customerB"
    networks: ["198.51.100.0/24"]
  - name: shared-infra          # no tenant → visible only to admin tokens
    networks: ["192.0.2.0/24"]

api:
  tokens:
    - { name: admin,    token_env: KAPKAN_ADMIN, role: operator }
    - { name: a-portal, token_env: KAPKAN_A,     role: viewer,   tenant: "customerA" }
    - { name: b-ops,    token_env: KAPKAN_B,     role: operator, tenant: "customerB" }
```

## Модель

На вопрос «какому арендатору принадлежит этот IP» отвечает **тот же** поиск хост-группы по
наибольшему совпадающему префиксу, который движок и митигатор уже используют для атрибуции
трафика — есть один источник истины о принадлежности IP, и это разрешение никогда не
выполняется на пути детектирования отдельного flow. Арендатор может охватывать несколько
хост-групп (клиент с отдельными web- и DNS-группами). Неявная глобальная/резервная группа
может нести `tenant` верхнего уровня, чтобы трафик «по умолчанию» атрибутировался домашнему
(house) арендатору; хост-группа **без** `tenant` остаётся непомеченной и видна только
admin-токенам.

Метка арендатора **не** является единицей политики митигации — пороги, BGP-атрибуты и
эскалация остаются на уровне хост-группы. Это чисто тег принадлежности, используемый для
ограничения области API.

## Область видимости токена

[API-токен](/docs/authentication) получает необязательный `tenant`, который надстраивается над
его ролью:

| Токен | Видит | Может изменять |
| --- | --- | --- |
| без области (нет `tenant`) | всех арендаторов (admin) | согласно своей роли |
| `tenant: customerA`, `role: viewer` | только customerA | ничего |
| `tenant: customerA`, `role: operator` | только customerA | только customerA |

Токен без области — это admin: поведение по умолчанию, полностью обратно совместимое. Токен,
ограниченный арендатором, которого не использует ни одна хост-группа, отклоняется при загрузке
конфигурации (иначе опечатка молча не видела бы ничего).

## Применение правил

Ограничение области — это **запрет по умолчанию для ограниченных токенов** в единственной
контрольной точке: арендатор запроса определяется один раз при аутентификации, и один предикат
видимости решает судьбу каждой строки. Новый эндпоинт чтения, забывший про фильтрацию, для
ограниченного токена вернул бы пустоту — но никогда данные другого арендатора.

- **Чтение** — `/api/v1/status`, `/attacks`, `/hosts`, `/bans` возвращают только те строки, чья
  владеющая группа несёт метку арендатора вызывающей стороны. `/status` пересобирается под
  каждую область: ограниченный токен получает только свои хост-группы и счётчики, но никогда не
  глобальные защищаемые сети, глобальные пороги или резервную группу всего развёртывания — так
  арендатор никогда не узнаёт ни префиксов, ни BGP-позиции другого.
- **Мутации** — ограниченный operator может выполнять `ban`/`unban` только в пределах своих
  префиксов. Цель за пределами арендатора возвращает единообразный `403` независимо от того,
  существует ли бан, поэтому арендатор не может прощупать состояние банов другого.
  `POST /api/v1/config/reload` доступен **только admin** — перезагрузка переписывает политику
  каждого арендатора и сам набор токенов.
- **Неоднозначность приводит к отказу** — если один bearer-секрет совпадает с токенами разной
  роли или арендатора (повторно использованный секрет), запрос отклоняется, а не разрешается в
  пользу более широкой области.

<Callout type="info" title="/metrics и панель управления">
`/metrics` не ограничивается по арендатору — она остаётся поверхностью для сбора метрик
admin/operator. Оболочка панели управления общая (собственных данных она не хранит); каждый
запрос данных, который она делает, фильтруется вставленным в неё токеном, поэтому viewer-токен
арендатора показывает только данные этого арендатора.
</Callout>

<Callout type="warning" title="Принадлежность следует за текущей конфигурацией">
Видимость ограниченного токена вычисляется относительно живой конфигурации: смена арендатора
префикса (перезагрузка, доступная только admin) немедленно передаёт его текущие атаки и баны
**новому** владельцу. Это сделано намеренно — управление следует за текущей принадлежностью —
но это значит, что ошибочная перезагрузка может на короткое время переместить строки между
арендаторами.
</Callout>

## Обратная совместимость

Если арендатор нигде не настроен = поведение одного арендатора, байт в байт: каждая группа
непомечена, каждый токен — admin без области, и API возвращает всё как раньше.
Мультиарендность включается по желанию, и нельзя получить изоляцию на открытом (без токена)
API — там нет субъекта, область которого можно ограничить.
