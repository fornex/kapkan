# Мультиарендность

Один экземпляр Kapkan может обслуживать множество клиентов (сценарий поставщика управляемых
услуг (managed service provider, MSP) или дата-центра (internet data center, IDC)) и выдавать
каждому токен, который видит и затрагивает **только их собственные** атаки, блокировки и хосты.
Арендатор (tenant) — это просто необязательная метка на [хост-группе](/docs/hostgroups): нет
ни нового объекта верхнего уровня, ни отдельного реестра арендаторов, а горячий путь
обнаружения остаётся нетронутым.

```yaml
tenant: "house"                 # optional: label the global/fallback group

hostgroups:
  - name: custA-web
    tenant: "customerA"         # this group belongs to customerA
    networks: ["203.0.113.0/26"]
  - name: custA-dns
    tenant: "customerA"         # a tenant can span several groups
    networks: ["203.0.113.64/26"]
  - name: custB
    tenant: "customerB"
    networks: ["198.51.100.0/24"]
  - name: shared-infra          # no tenant → visible only to admin tokens
    networks: ["192.0.2.0/24"]

api:
  tokens:
    - { name: admin,    token_env: KAPKAN_ADMIN, role: operator }
    - { name: a-portal, token_env: KAPKAN_A,     role: viewer,   tenant: "customerA" }
    - { name: b-ops,    token_env: KAPKAN_B,     role: operator, tenant: "customerB" }
```

## Как это настроить

Три шага напрямую соответствуют YAML выше:

1. **Пометьте хост-группу(ы) каждого клиента** через `tenant: "customerA"` (одна и та же метка может повторяться в нескольких группах — скажем, web- и DNS-группы одного клиента).
2. **Добавьте API-токен** в `api.tokens` с той же `tenant:` плюс `role:` — `viewer` только читает, `operator` может ещё и блокировать/разблокировать.
3. **Передайте клиенту этот токен.** Теперь он видит и затрагивает только свои собственные префиксы; всё остальное (другие арендаторы, непомеченные группы) для него невидимо.

Это весь рабочий процесс. Остальная часть страницы объясняет, как разрешается принадлежность и что именно ограниченный токен может и не может делать.

## Модель

На вопрос «какому арендатору принадлежит этот IP» отвечает **тот же** поиск хост-группы по
наибольшему совпадающему префиксу, который движок и митигатор уже используют для атрибуции
трафика — при пересечении префиксов выигрывает наиболее специфичная совпадающая хост-группа
([поиск по наибольшему совпадающему префиксу](/docs/glossary)). Арендатор может охватывать
несколько хост-групп (клиент с отдельными web- и DNS-группами).

Любой IP, который не попадает ни в одну именованную хост-группу, обрабатывается неявной
**глобальной/резервной группой** (catch-all). Эта глобальная группа может нести `tenant`
верхнего уровня, чтобы трафик «по умолчанию» атрибутировался домашнему (house) арендатору;
именованная хост-группа **без** `tenant` остаётся непомеченной и видна только admin-токенам.

Метка арендатора **не** является единицей политики подавления атак — пороги, BGP-атрибуты и
эскалация остаются на уровне хост-группы. Это чисто тег принадлежности, используемый для
ограничения области API.

## Область видимости токена

[API-токен](/docs/authentication) получает необязательный `tenant`, который надстраивается над
его ролью: `viewer` может читать, но ничего не менять; `operator` может ещё и блокировать/разблокировать.

| Токен | Видит | Может изменять |
| --- | --- | --- |
| без области (нет `tenant`) | всех арендаторов (admin) | всё, что разрешает его роль (viewer читает всех арендаторов; operator может блокировать/разблокировать у всех арендаторов) |
| `tenant: customerA`, `role: viewer` | только customerA | ничего |
| `tenant: customerA`, `role: operator` | только customerA | только customerA |

Токен без области — это admin: поведение по умолчанию, полностью обратно совместимое. Токен,
ограниченный арендатором, которого не использует ни одна хост-группа, отклоняется при загрузке
конфигурации (иначе опечатка молча не видела бы ничего).

## Применение правил

Ограничение области — это **запрет по умолчанию: ограниченный токен всегда видит только своего
арендатора.** Арендатор определяется один раз при аутентификации и применяется к каждой строке,
поэтому ограниченный токен никогда не сможет провалиться к данным другого арендатора.

- **Чтение** — `/api/v1/status`, `/attacks`, `/hosts`, `/bans` и `/audit` возвращают только те
  строки, чья владеющая группа несёт метку арендатора вызывающей стороны. `/status`
  пересобирается под каждую область: ограниченный токен получает только свои хост-группы и
  счётчики, но никогда не глобальные защищаемые сети, глобальные пороги или строку конфигурации
  резервной группы всего развёртывания — так арендатор никогда не узнаёт ни префиксов, ни
  BGP-позиции другого. Домашний (house) арендатор (резервная группа, помеченная `tenant`) всё
  равно видит свои атаки, блокировки и хосты «по умолчанию»; скрывается лишь строка конфигурации этой
  группы. [Журнал аудита](/docs/audit) привязан к арендатору вызывающей стороны на стороне
  сервера, а фильтр `target`, называющий адрес за пределами арендатора, отклоняется — никакого
  межарендаторного оракула существования.
- **Мутации** — ограниченный operator может выполнять `ban`/`unban` только в пределах своих
  префиксов. Цель за пределами арендатора возвращает единообразный `403` независимо от того,
  существует ли блокировка, поэтому арендатор не может прощупать состояние блокировок другого.
  `POST /api/v1/config/reload` доступен **только admin** — перезагрузка переписывает политику
  каждого арендатора и сам набор токенов.
- **Неоднозначность приводит к отказу** — если один bearer-секрет совпадает с токенами разной
  роли или арендатора (повторно использованный секрет), запрос отклоняется, а не разрешается в
  пользу более широкой области.

### Проверка изоляции

Передайте ограниченный токен в `curl` и убедитесь, что он видит только своего арендатора.
`/api/v1/status` возвращает только хост-группы этого арендатора:

```bash
curl -H "Authorization: Bearer $KAPKAN_A" http://127.0.0.1:8080/api/v1/status
# → only customerA's hostgroups and counts; no other tenant, no global config
```

Затем убедитесь, что межарендаторная блокировка отклоняется — попытка заблокировать адрес за пределами
арендатора токена возвращает `403`:

```bash
curl -X POST -H "Authorization: Bearer $KAPKAN_A" \
  -d '{"ip":"198.51.100.5"}' http://127.0.0.1:8080/api/v1/ban
# → 403 {"error":"target is outside your tenant"}
```

<Callout type="info" title="/metrics и панель управления">
`/metrics` не ограничивается по арендатору — она остаётся открытой для сбора метрик Prometheus
и не требует токена, даже когда настроена аутентификация API. Оболочка панели управления общая (собственных данных она не хранит); каждый
запрос данных, который она делает, фильтруется вставленным в неё токеном, поэтому viewer-токен
арендатора показывает только данные этого арендатора.
</Callout>

<Callout type="warning" title="Принадлежность следует за текущей конфигурацией">
Видимость ограниченного токена вычисляется относительно живой конфигурации: смена арендатора
префикса (перезагрузка, доступная только admin) немедленно передаёт его текущие атаки и блокировки
**новому** владельцу. Это сделано намеренно — управление следует за текущей принадлежностью —
но это значит, что ошибочная перезагрузка может на короткое время переместить строки между
арендаторами.
</Callout>

## Обратная совместимость

Если арендатор нигде не настроен = поведение одного арендатора, байт в байт: каждая группа
непомечена, каждый токен — admin без области, и API возвращает всё как раньше.
Мультиарендность включается по желанию, и нельзя получить изоляцию на открытом (без токена)
API — там нет субъекта, область которого можно ограничить.
