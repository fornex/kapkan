# RTBH-митигация

Когда движок сообщает об атаке, Kapkan митигирует её, анонсируя для атакуемого хоста маршрут
**remotely-triggered blackhole (RTBH)**: префикс `/32` (IPv4) или `/128` (IPv6), передаваемый
через встроенный BGP-спикер [GoBGP](https://github.com/osrg/gobgp), помеченный вашим RTBH-community
и направленный на discard next-hop. Ваши граничные маршрутизаторы сопоставляют community и
отбрасывают весь трафик к этому хосту, гася флуд до того, как он достигнет ядра вашей сети.

Митигация основана на адресе назначения: blackhole отсекает трафик *к* цели. Для входящего флуда
это останавливает атаку; для [исходящей атаки](/docs/detection) со скомпрометированного хоста это
выводит хост из сети (что обычно прекращает злоупотребление) и останавливает исходящий флуд только
там, где ваша граница также отбрасывает источники в blackhole-префиксах, например с помощью uRPF.

<Callout type="warning" title="Dry-run по умолчанию">
Пока вы явно не зададите `dry_run: false`, каждый потенциальный blackhole вычисляется, логируется и
доступен через API — но **никогда не анонсируется** вашим маршрутизаторам. Проверьте детекцию и
BGP-пиринг на боевой телеметрии, прежде чем хоть один маршрут будет отправлен. См. [Запуск в боевой
режим](/docs/going-live).
</Callout>

## Методы митигации

`blackhole` (RTBH) — метод по умолчанию и предмет всей остальной части этой страницы. Через ключ
`mitigation` доступны и другие методы, каждый из которых переопределяется на уровне
[хостгруппы](/docs/hostgroups):

- **[FlowSpec](/docs/flowspec)** (`mitigation: flowspec`) — отбрасывать только вектор атаки с
  помощью правил BGP FlowSpec (RFC 8955/8956) вместо blackhole всей жертвы.
- **[Перенаправление трафика](/docs/scrubbing)** (`mitigation: divert`) — анонсировать жертву в
  сторону scrubbing-центра, чтобы её трафик очищался и реинжектировался, а не отбрасывался.
- **[Лестницы эскалации](/docs/escalation)** (`escalation:`) — усиливать реакцию по мере того, как
  атака продолжается дольше: от оповещения к FlowSpec, к divert и к blackhole.

Все они разделяют один и тот же жизненный цикл бана, TTL, гистерезис разбана и ограничение
`max_active_bans`, описанные ниже.

## Конфигурация BGP

Блок `bgp` определяет BGP-идентичность Kapkan, blackhole next-hop'ы, RTBH-community и eBGP-пиры,
которым он анонсирует маршруты.

```yaml
bgp:
  local_asn: 65000               # Kapkan's local ASN
  router_id: "198.51.100.1"      # must be a valid IPv4 address
  next_hop: "192.0.2.1"          # IPv4 discard next-hop
  next_hop6: "100::1"            # IPv6 discard next-hop (optional)
  community: "65000:666"         # RTBH community, ASN:value
  # communities: ["65000:666", "65000:777"]  # full set; overrides `community`
  # local_pref: 100              # optional LOCAL_PREF for iBGP peers; 0 = omit
  neighbors:
    - address: "198.51.100.254"
      remote_asn: 65001
      port: 179                  # optional; defaults to 179, override for testing
```

| Ключ | Значение |
| --- | --- |
| `local_asn` | Локальный ASN Kapkan для eBGP-сессий. Обязательный. |
| `router_id` | BGP router ID. Должен быть корректным адресом **IPv4**. |
| `next_hop` | IPv4 discard next-hop, анонсируемый с каждым blackhole `/32`. |
| `next_hop6` | IPv6 discard next-hop, анонсируемый с каждым blackhole `/128`. Необязательный; если не задан, Kapkan использует `100::1` (discard-префикс из RFC 6666). |
| `community` | RTBH-community в форме `ASN:value` (например `65000:666`). Разбирается один раз при загрузке в проводное значение, отправляемое с каждым маршрутом. |
| `communities` | Необязательный список community, который при задании **переопределяет** `community`, для аплинков, ожидающих полный набор. |
| `local_pref` | Необязательный атрибут пути `LOCAL_PREF`, прикрепляемый к blackhole. Учитывается на iBGP-сессиях, но отбрасывается через eBGP (он нетранзитивен), поэтому значим только для iBGP-соседей. Значение по умолчанию `0` опускает его. |
| `neighbors[]` | BGP-пиры, которым Kapkan анонсирует маршруты, каждый — либо eBGP, либо iBGP. Каждая запись принимает `address`, `remote_asn` и необязательный `port` (по умолчанию `179`, предназначен для тестирования). |

`community`, `communities`, `next_hop`, `next_hop6` и `local_pref` можно переопределить на уровне
хостгруппы — см. [BGP-атрибуты на уровне хостгруппы](#bgp-атрибуты-на-уровне-хостгруппы).

Blackhole-маршрут, который Kapkan строит для цели, читается как `blackhole <prefix> next-hop <next_hop>
community <community>` — ровно та строка, которая выводится в поле `route` [объекта
бана](#объект-бана). Для IPv6-целей вместо `next_hop` используется `next_hop6` (или запасной
`100::1`).

<Callout type="info" title="Пиринг работает в dry-run">
BGP-спикер устанавливает пиринг с вашими соседями даже при `dry_run` равном true — флагом
управляются только анонсы маршрутов. Это позволяет убедиться, что сессии достигают `ESTABLISHED`
(логируется как `bgp peer state`), и проверить связность до того, как вы анонсируете хоть один
маршрут.
</Callout>

## BGP-атрибуты на уровне хостгруппы

Глобальный блок `bgp` задаёт blackhole next-hop'ы и RTBH-community по умолчанию.
[Хостгруппа](/docs/hostgroups) может переопределить любой из них, чтобы разные клиенты
сигнализировали своим собственным аплинкам:

```yaml
bgp:
  next_hop: "192.0.2.1"
  community: "65000:666"                          # the default RTBH community

hostgroups:
  - name: customer-a
    networks: ["203.0.113.64/26"]
    bgp:
      communities: ["65000:100", "65001:200"]     # customer-A's own blackhole signal
      next_hop: "192.0.2.50"                       # and discard next-hop
      local_pref: 250
```

Каждое незаданное поле наследует глобальное значение `bgp`, так что группа может переопределить
только своё community, разделяя при этом глобальный next-hop. Разрешённые атрибуты **фиксируются на
каждом бане в момент его создания**: перезагрузка конфигурации меняет только будущие баны, но
никогда — маршрут, который уже анонсирован активным баном. Атрибуты `next_hop`, `community` и
`local_pref` каждого бана видны в `/api/v1/bans` и в поле `route`.

## Жизненный цикл бана

Бан — это единица, которую Kapkan отслеживает для каждого решения о blackhole. Его жизненный цикл на
каждом шаге ограничен [моделью безопасности](/docs/safety) — нет пути к перманентному или
неконтролируемому бану.

1. **Анонс при детекции.** Когда движок сообщает о новой атаке на хост и политика хоста допускает
   бан, Kapkan анонсирует blackhole-маршрут (или, в режиме dry-run, записывает потенциальный
   маршрут), и бан переходит в состояние `active`. Вторая атака — например, входящий и исходящий
   флуд на одном хосте одновременно — разделяет один RTBH-маршрут вместо повторного анонса.
2. **Авто-отзыв по TTL.** Каждый анонс несёт TTL. По истечении `ban.ttl_seconds` маршрут
   автоматически отзывается, даже если атака всё ещё продолжается; продолжающаяся атака обновляет
   TTL, но никогда дольше, чем свежий TTL от текущего момента. **Перманентных банов не существует.**
3. **Разбан с гистерезисом.** Бан, вызванный атакой, отзывается только после того, как трафик
   остаётся ниже порога в течение `ban.unban_hysteresis_seconds`. Эта задержка против дребезга
   предотвращает быстрый анонс и отзыв одного и того же маршрута при пограничной атаке.
4. **Жёсткое ограничение.** `ban.max_active_bans` — абсолютный потолок на число одновременно
   активных банов. После достижения лимита новые баны **отклоняются** (возвращаются как бан в
   состоянии `rejected` с причиной `max_active_bans reached`), и операторы получают оповещение — так
   что Kapkan никогда не сможет загнать в blackhole половину вашей сети при широкой атаке.

```yaml
ban:
  ttl_seconds: 1800            # every announcement auto-withdraws after this
  unban_hysteresis_seconds: 60 # traffic must stay below threshold this long before withdraw
  max_active_bans: 100         # hard cap on simultaneous bans
```

Бан также отзывается досрочно, если перезагрузка конфигурации удаляет его цель из защищаемых
`networks` — Kapkan не будет держать маршрут поднятым для адресного пространства, которое он больше
не защищает.

## Сохранение между перезапусками

Перезапуск — обновление ядра, новая версия Kapkan — обычно сбросил бы каждый активный маршрут и
оставил жертв незащищёнными до тех пор, пока движок заново не обнаружит каждую атаку. Два
взаимодополняющих, необязательных механизма поддерживают митигацию активной на этом промежутке:

- **Сохранение банов (`ban.state_file`).** Укажите его на путь, доступный для записи (например,
  `/var/lib/kapkan/bans.json`), и Kapkan записывает свои активные баны на диск. При следующем запуске
  он **восстанавливает** их и сразу же повторно анонсирует каждый маршрут, не дожидаясь, пока атака
  сработает снова. Пустое значение (по умолчанию) отключает сохранение. [Юнит systemd](/docs/deployment)
  предоставляет каталог через `StateDirectory=kapkan`.
- **BGP Graceful Restart (`bgp.graceful_restart`).** Включён по умолчанию. Пир, поддерживающий
  Graceful Restart (RFC 4724), **сохраняет** маршруты Kapkan как *устаревшие (stale)*, пока сессия
  не работает, вместо того чтобы сбросить их в момент разрыва BGP-соединения — так что blackhole
  остаётся в силе в течение короткого окна перезапуска. Анонсирование этой возможности безвредно
  для пира, который её не поддерживает.

```yaml
ban:
  state_file: /var/lib/kapkan/bans.json   # persist & re-announce active bans across a restart

bgp:
  graceful_restart:
    enabled: true              # default; set false to opt out
    restart_seconds: 120       # how long the peer holds routes as stale (1..4095)
    # long_lived: false        # LLGR: extend retention beyond restart_seconds
    # long_lived_stale_seconds: 3600  # LLGR stale time when long_lived is set (<= 86400)
```

При совместном использовании баны переживают перезапуск процесса (`state_file`), а маршруты
переживают разрыв сессии (graceful restart), так что обновление больше не снимает митигацию с
атаки, которая ещё идёт.

## Откат при отклонении метода

Аплинки различаются тем, что они принимают — пир может вовсе отклонить анонс FlowSpec или divert. По
умолчанию Kapkan не оставляет жертву без защиты в этом случае: `ban.fallback` равен `blackhole`, так
что отклонённый анонс `flowspec`/`divert` **деградирует до RTBH-маршрута**, а не проваливает бан.
Задайте `ban.fallback: none`, чтобы отключить это и вместо этого отклонять бан.

```yaml
ban:
  fallback: blackhole   # degrade a rejected flowspec/divert announce to RTBH (default); none disables
```

Когда происходит откат, бан записывает `fell_back_from` (метод, который отклонил пир), и [метрика](/docs/metrics)
`kapkan_mitigate_fallback_total` инкрементируется — ненулевая серия `from="flowspec"` сигнализирует об
аплинках, которые не поддерживают FlowSpec. Полный отказ BGP проваливает оба метода, так что бан всё
равно отклоняется как прежде; откат меняет только тот случай, когда не поддерживается именно
хирургический метод.

## Dry-run

В режиме dry-run (`dry_run: true`, по умолчанию) Kapkan делает всё, *кроме* отправки маршрута.
Целевой префикс, next-hop, community и полная строка `route` вычисляются, решение логируется как
`DRY-RUN: would announce blackhole route (not sent)`, а бан отслеживается в API с `dry_run: true` и
живыми состояниями `active`/`withdrawn`. Истечение TTL и гистерезис работают ровно так же, как в
бою, поэтому весь жизненный цикл наблюдаем до того, как любой маршрут достигнет маршрутизатора.

Ничего не анонсируется, пока вы не зададите `dry_run: false` и не выполните перезагрузку (`SIGHUP`
или `POST /api/v1/config/reload`). Поскольку [пиринг работает в dry-run](#конфигурация-bgp),
единственное поведение, которое меняется при переключении флага, — это сам анонс/отзыв.

<Callout type="info" title="Сначала проверьте">
Запустите систему на боевой телеметрии в режиме dry-run, убедитесь, что детекции срабатывают на
правильных префиксах и поля `route` корректны, затем выключите dry-run. Полная процедура описана в
[Запуске в боевой режим](/docs/going-live).
</Callout>

## Ручные баны

Операторы могут загнать хост в blackhole или освободить его напрямую через [REST API](/docs/api):

```http
POST /api/v1/ban
Content-Type: application/json

{"ip": "203.0.113.66"}
```

```http
POST /api/v1/unban
Content-Type: application/json

{"ip": "203.0.113.66"}
```

Ручной бан создаётся с `manual: true` и следует тем же TTL и гистерезису, что и автоматический. Он
соблюдает все правила безопасности без исключений: цель из белого списка отклоняется (возвращается
`409`, причина `whitelisted`), цель вне настроенных `networks` отклоняется (возвращается `409`,
причина `outside configured networks`), а превышение `max_active_bans` отклоняется (возвращается
`409`). Оба POST-эндпойнта требуют content type `application/json`. Полные формы запроса и ответа
см. в [справочнике по API](/docs/api).

## Объект бана

API возвращает каждый бан как JSON-объект. Это поля ровно в том виде, как их сериализует митигатор:

| Поле | Тип | Значение |
| --- | --- | --- |
| `target` | string | Адрес хоста в blackhole (IPv4 или IPv6). |
| `prefix` | string | Анонсируемый префикс хоста: `/32` для IPv4, `/128` для IPv6. |
| `metric` | string | Метрика, вызвавшая бан по атаке (опускается для ручных банов). |
| `rate` | number | Наблюдаемая скорость на момент бана, в единицах метрики (опускается при нуле). |
| `threshold` | number | Порог, который превысила скорость (опускается при нуле). |
| `next_hop` | string | Discard next-hop, анонсируемый с маршрутом. |
| `community` | string | RTBH-community в форме `ASN:value`. |
| `local_pref` | number | `LOCAL_PREF`, прикреплённый к маршруту (опускается при нуле). |
| `route` | string | Полный маршрут в виде `blackhole <prefix> next-hop <next_hop> community <community>` (или `divert ...` для divert-банов) или сводка `flowspec: ...` для FlowSpec-банов. |
| `state` | string | Одно из `active`, `withdrawn` или `rejected`. |
| `dry_run` | boolean | Был ли бан создан в режиме dry-run (маршрут фактически не отправлен). |
| `manual` | boolean | `true` для банов по запросу оператора, `false` для автоматических. |
| `started_at` | timestamp | Когда бан был создан. |
| `expires_at` | timestamp | Когда TTL автоматически отзовёт маршрут. |
| `withdrawn_at` | timestamp | Когда маршрут был отозван (опускается, пока бан активен). |
| `reason` | string | Почему бан был отозван или отклонён (опускается, пока бан активен). |
| `method` | string | Метод митигации, создавший бан: `blackhole`, `flowspec` или `divert`. |
| `flowspec` | array | Сгенерированные правила [FlowSpec](/docs/flowspec), когда метод — `flowspec`. |
| `escalation` | array | Настроенная [лестница эскалации](/docs/escalation), если она задана. |
| `escalation_step` | number | Индекс текущей ступени лестницы. |
| `fell_back_from` | string | Метод, чей анонс отклонил пир, что привело к деградации бана до его запасного варианта (опускается, если отката не было). |
| `fell_back_reason` | string | Человекочитаемая причина отката (опускается, если отката не было). |

Бан в состоянии `rejected` никогда не анонсируется — он фиксирует отказ (из белого списка, вне
`networks` или достигнут лимит), чтобы решение можно было проаудировать в API и уведомлениях.
