# RTBH: подавление атак

[BGP](/docs/glossary) (Border Gateway Protocol — то, как маршрутизаторы обмениваются маршрутами) —
основа всего подавления атак в Kapkan: Kapkan выступает как BGP-спикер, который сообщает вашим
маршрутизаторам, какие маршруты установить. Каждое решение о подавлении Kapkan отслеживает как
**блокировку**; для метода по умолчанию блокировка — это **blackhole-маршрут** ([RTBH](/docs/glossary)).

Когда движок сообщает об атаке, Kapkan подавляет её, анонсируя для атакуемого хоста маршрут
**remotely-triggered blackhole (RTBH)**: префикс `/32` (IPv4) или `/128` (IPv6), передаваемый
через встроенный BGP-спикер [GoBGP](https://github.com/osrg/gobgp), помеченный вашим RTBH-community
и направленный на [discard next-hop](/docs/glossary) — адрес, на который ваши маршрутизаторы
отправляют blackhole-трафик и затем отбрасывают (нулевой маршрут). Ваши граничные маршрутизаторы
сопоставляют community и отбрасывают весь трафик к этому хосту, гася флуд до того, как он достигнет
ядра вашей сети.

Подавление основано на адресе назначения: blackhole отбрасывает трафик *к* цели. Для входящего
флуда это останавливает атаку. Для [исходящей атаки](/docs/detection) со скомпрометированного хоста
это выводит хост из сети, что обычно прекращает злоупотребление, — но напрямую исходящие пакеты не
отбрасываются, если ваша граница также не фильтрует пакеты, *источник* которых попадает в
blackhole-префикс, например с помощью [uRPF](/docs/glossary) (unicast Reverse Path Forwarding —
функция маршрутизатора, отбрасывающая трафик с подменённым или несовпадающим обратным путём). Чтобы
получать оповещение без маршрута, задайте `ban: false` в [хост-группе](/docs/hostgroups).

<Callout type="warning" title="Dry-run по умолчанию">
Пока вы явно не зададите `dry_run: false`, каждый потенциальный blackhole вычисляется, логируется и
доступен через API — но **никогда не анонсируется** вашим маршрутизаторам. Проверьте обнаружение и
BGP-пиринг на боевой телеметрии, прежде чем хоть один маршрут будет отправлен. См. [Ввод в
эксплуатацию](/docs/going-live).
</Callout>

## Методы подавления

`blackhole` (RTBH) — метод по умолчанию и предмет всей остальной части этой страницы. Через ключ
`mitigation` доступны и другие методы, каждый из которых переопределяется на уровне
[хост-группы](/docs/hostgroups):

- **[FlowSpec](/docs/flowspec)** (`mitigation: flowspec`) — отбрасывать только вектор атаки с
  помощью правил BGP FlowSpec (RFC 8955/8956) вместо blackhole всей жертвы.
- **[Отвод трафика](/docs/scrubbing)** (`mitigation: divert`) — анонсировать жертву в сторону
  центра очистки, чтобы её трафик очищался и реинжектировался, а не отбрасывался.
- **[Лестницы эскалации](/docs/escalation)** (`escalation:`) — усиливать реакцию по мере того, как
  атака продолжается дольше: от оповещения к FlowSpec, к divert и к blackhole.

Все они разделяют один и тот же жизненный цикл блокировки, TTL, гистерезис снятия блокировки и ограничение
`max_active_bans`, описанные ниже.

## Конфигурация BGP

Блок `bgp` определяет BGP-идентичность Kapkan, blackhole next-hop'ы, RTBH-community и eBGP-пиры,
которым он анонсирует маршруты.

```yaml
bgp:
  local_asn: 65000               # Kapkan's local ASN
  router_id: "198.51.100.1"      # must be a valid IPv4 address
  next_hop: "192.0.2.1"          # IPv4 discard next-hop
  next_hop6: "100::1"            # IPv6 discard next-hop (optional)
  community: "65000:666"         # RTBH community, ASN:value
  # communities: ["65000:666", "65000:777"]  # full set; overrides `community`
  # local_pref: 100              # optional LOCAL_PREF for iBGP peers; 0 = omit
  neighbors:
    - address: "198.51.100.254"
      remote_asn: 65001
      port: 179                  # optional; defaults to 179, override for testing
```

| Ключ | Значение |
| --- | --- |
| `local_asn` | Локальный ASN Kapkan для eBGP-сессий. Обязательный. |
| `router_id` | BGP router ID. Должен быть корректным адресом **IPv4**. |
| `next_hop` | IPv4 discard next-hop, анонсируемый с каждым blackhole `/32`. |
| `next_hop6` | IPv6 discard next-hop, анонсируемый с каждым blackhole `/128`. Необязательный; если не задан, Kapkan использует запасной discard-адрес `100::1` из RFC 6666 (из discard-префикса `100::/64`). |
| `community` | RTBH-community в форме `ASN:value` (например `65000:666`). Разбирается один раз при загрузке в проводное значение, отправляемое с каждым маршрутом. |
| `communities` | Необязательный список community, который при задании **переопределяет** `community`, для вышестоящих провайдеров, ожидающих полный набор. |
| `local_pref` | Необязательный атрибут пути `LOCAL_PREF`, прикрепляемый к blackhole. Он распространяется только внутри вашей сети ([iBGP](/docs/glossary) — внутренний BGP), но не передаётся внешним пирам ([eBGP](/docs/glossary) — внешний BGP), поэтому значим только для ваших внутренних соседей. Значение по умолчанию `0` опускает его. |
| `neighbors[]` | BGP-пиры, которым Kapkan анонсирует маршруты, каждый — либо eBGP, либо iBGP. Каждая запись принимает `address`, `remote_asn` и необязательный `port` (по умолчанию `179`, предназначен для тестирования). |

`community`, `communities`, `next_hop`, `next_hop6` и `local_pref` можно переопределить на уровне
хост-группы — см. [BGP-атрибуты на уровне хост-группы](#bgp-атрибуты-на-уровне-хост-группы).

Blackhole-маршрут, который Kapkan строит для цели, читается примерно как `blackhole <prefix> next-hop <next_hop>
community <community>` — это та строка, которая выводится в поле `route` [объекта
блокировки](#объект-блокировки). Сегмент `community` опускается, если community не задано, а сегмент
`local-pref <n>` добавляется, когда настроен `local_pref`. Для IPv6-целей вместо `next_hop`
используется `next_hop6` (или запасной `100::1`).

<Callout type="info" title="Пиринг работает в dry-run">
BGP-спикер устанавливает пиринг с вашими соседями даже при `dry_run` равном true — флагом
управляются только анонсы маршрутов. Это позволяет убедиться, что сессии достигают `ESTABLISHED`
(логируется как `bgp peer state`), и проверить связность до того, как вы анонсируете хоть один
маршрут.
</Callout>

## BGP-атрибуты на уровне хост-группы

Глобальный блок `bgp` задаёт blackhole next-hop'ы и RTBH-community по умолчанию.
[Хост-группа](/docs/hostgroups) может переопределить любой из них, чтобы разные клиенты
сигнализировали своим собственным вышестоящим провайдерам:

```yaml
bgp:
  next_hop: "192.0.2.1"
  community: "65000:666"                          # the default RTBH community

hostgroups:
  - name: customer-a
    networks: ["203.0.113.64/26"]
    bgp:
      communities: ["65000:100", "65001:200"]     # customer-A's own blackhole signal
      next_hop: "192.0.2.50"                       # and discard next-hop
      local_pref: 250
```

Каждое незаданное поле наследует глобальное значение `bgp`, так что группа может переопределить
только своё community, разделяя при этом глобальный next-hop. Разрешённые атрибуты **фиксируются на
каждой блокировке в момент её создания**: перезагрузка конфигурации меняет только будущие блокировки, но
никогда — маршрут, который уже анонсирован активной блокировкой. Атрибуты `next_hop`, `community` и
`local_pref` каждой блокировки видны в `/api/v1/bans` и в поле `route`.

## Жизненный цикл блокировки

Блокировка — это единица, которую Kapkan отслеживает для каждого решения о blackhole. Её жизненный цикл на
каждом шаге ограничен [моделью безопасности](/docs/safety) — нет пути к перманентной или
неконтролируемой блокировке.

1. **Анонс при обнаружении.** Когда движок сообщает о новой атаке на хост и политика хоста допускает
   блокировку, Kapkan анонсирует blackhole-маршрут (или, в режиме dry-run, записывает потенциальный
   маршрут), и блокировка переходит в состояние `active`. Вторая атака — например, входящий и исходящий
   флуд на одном хосте одновременно — разделяет один RTBH-маршрут вместо повторного анонса.
2. **Авто-отзыв по TTL.** Каждый анонс несёт TTL. Пока атака всё ещё обнаруживается,
   каждое окно обнаружения обновляет TTL блокировки (но никогда дольше, чем свежий `ban.ttl_seconds`
   от текущего момента), поэтому маршрут остаётся анонсированным на всё время атаки. Как только
   обнаружение прекращается, маршрут автоматически отзывается в течение одного TTL. **Перманентных блокировок не существует.**
3. **Снятие блокировки с гистерезисом.** Блокировка, вызванная атакой, отзывается только после того, как трафик
   остаётся ниже порога в течение `ban.unban_hysteresis_seconds`. Эта задержка против дребезга
   предотвращает быстрый анонс и отзыв одного и того же маршрута при пограничной атаке.
4. **Жёсткое ограничение.** `ban.max_active_bans` — абсолютный потолок на число одновременно
   активных блокировок. После достижения лимита новые блокировки **отклоняются** (возвращаются как блокировка в
   состоянии `rejected` с причиной `max_active_bans reached`), и операторы получают оповещение — так
   что Kapkan никогда не сможет загнать в blackhole половину вашей сети при широкой атаке.

```yaml
ban:
  ttl_seconds: 1800            # every announcement auto-withdraws after this
  unban_hysteresis_seconds: 60 # traffic must stay below threshold this long before withdraw
  max_active_bans: 100         # hard cap on simultaneous bans
```

Блокировка также отзывается досрочно, если перезагрузка конфигурации удаляет её цель из защищаемых
`networks` — Kapkan не будет держать маршрут поднятым для адресного пространства, которое он больше
не защищает.

## Сохранение между перезапусками

Перезапуск — обновление ядра, новая версия Kapkan — обычно сбросил бы каждый активный маршрут и
оставил жертв незащищёнными до тех пор, пока движок заново не обнаружит каждую атаку. Два
взаимодополняющих, необязательных механизма поддерживают подавление активным на этом промежутке:

- **Сохранение блокировок (`ban.state_file`).** Укажите его на путь, доступный для записи (например,
  `/var/lib/kapkan/bans.json`), и Kapkan записывает свои активные блокировки на диск. При следующем запуске
  он **восстанавливает** их и сразу же повторно анонсирует каждый маршрут, не дожидаясь, пока атака
  сработает снова. Пустое значение (по умолчанию) отключает сохранение. [Юнит systemd](/docs/deployment)
  предоставляет каталог через `StateDirectory=kapkan`.
- **BGP Graceful Restart (`bgp.graceful_restart`).** Включён по умолчанию. Пир, поддерживающий
  Graceful Restart (RFC 4724), **сохраняет** blackhole-маршруты Kapkan как *устаревшие (stale)*, пока
  сессия не работает, вместо того чтобы сбросить их в момент разрыва BGP-соединения — так что
  blackhole остаётся в силе в течение короткого окна перезапуска. Анонсирование этой возможности
  безвредно для пира, который её не поддерживает.

```yaml
ban:
  state_file: /var/lib/kapkan/bans.json   # persist & re-announce active bans across a restart

bgp:
  graceful_restart:
    enabled: true              # default; set false to opt out
    restart_seconds: 120       # how long the peer holds routes as stale (0..4095; 0 = use the default 120)
    # long_lived: false        # LLGR (Long-Lived Graceful Restart): extend retention beyond restart_seconds
    # long_lived_stale_seconds: 3600  # LLGR stale time when long_lived is set (<= 86400)
```

Graceful Restart перекрывает разрыв сессии, но сам по себе пир очищает устаревшие маршруты, как
только Kapkan переподключается и сигнализирует [End-of-RIB](/docs/glossary) (маркер, который
говорит «я закончил повторную отправку своих маршрутов»). Используйте его вместе с `state_file`,
чтобы перезапущенный Kapkan повторно анонсировал блокировки до того, как это произойдёт: при совместном
использовании блокировки переживают перезапуск процесса (`state_file`), а маршруты переживают разрыв
сессии (graceful restart), так что обновление больше не снимает подавление с атаки, которая ещё
идёт.

## Откат при отклонении метода

Вышестоящие провайдеры различаются тем, что они принимают, — пир может вовсе отклонить анонс
FlowSpec или divert. По умолчанию Kapkan не оставляет жертву без защиты в этом случае: `ban.fallback`
равен `blackhole`, так что отклонённый анонс `flowspec`/`divert` **деградирует до RTBH-маршрута**, а
не проваливает блокировку. Задайте `ban.fallback: none`, чтобы отключить это и вместо этого отклонять блокировку.

```yaml
ban:
  fallback: blackhole   # degrade a rejected flowspec/divert announce to RTBH (default); none disables
```

Когда происходит откат, блокировка записывает `fell_back_from` (метод, который отклонил пир), и [метрика](/docs/metrics)
`kapkan_mitigate_fallback_total` инкрементируется — ненулевая серия `from="flowspec"` сигнализирует о
вышестоящих провайдерах, которые не поддерживают FlowSpec. Полный отказ BGP проваливает оба метода,
так что блокировка всё равно отклоняется как прежде; откат меняет только тот случай, когда не поддерживается
именно хирургический метод.

## Dry-run

В режиме dry-run (`dry_run: true`, по умолчанию) Kapkan делает всё, *кроме* отправки маршрута.
Целевой префикс, next-hop, community и полная строка `route` вычисляются, решение логируется как
`DRY-RUN: would announce mitigation (not sent)` (и соответствующая строка отзыва
`DRY-RUN: would withdraw mitigation (not sent)`), а блокировка отслеживается в API с `dry_run: true` и
живыми состояниями `active`/`withdrawn`. Истечение TTL и гистерезис работают ровно так же, как в
бою, поэтому весь жизненный цикл наблюдаем до того, как любой маршрут достигнет маршрутизатора.

Ничего не анонсируется, пока вы не зададите `dry_run: false` и не выполните перезагрузку (`SIGHUP`
или `POST /api/v1/config/reload`). Поскольку [пиринг работает в dry-run](#конфигурация-bgp),
единственное поведение, которое меняется при переключении флага, — это сам анонс/отзыв.

<Callout type="info" title="Сначала проверьте">
Запустите систему на боевой телеметрии в режиме dry-run, убедитесь, что обнаружения срабатывают на
правильных префиксах и поля `route` корректны, затем выключите dry-run. Полная процедура описана в
разделе [Ввод в эксплуатацию](/docs/going-live).
</Callout>

## Ручные блокировки

Операторы могут загнать хост в blackhole или освободить его напрямую через [REST API](/docs/api).
Если вы прямо сейчас под атакой, начните с [руководства «Атака прямо
сейчас»](/docs/under-attack), которое пошагово разбирает последовательность блокировки хоста.

```http
POST /api/v1/ban
Content-Type: application/json

{"ip": "203.0.113.66"}
```

```http
POST /api/v1/unban
Content-Type: application/json

{"ip": "203.0.113.66"}
```

Ручная блокировка создаётся с `manual: true` и следует тому же TTL (поэтому она всё равно автоматически
истекает — перманентных блокировок не существует), но снимается только явным запросом на снятие или истечением
TTL, никогда — падением трафика ниже порога. Она соблюдает все правила безопасности без исключений:
цель из белого списка отклоняется (возвращается `409`, причина `whitelisted`), цель вне настроенных
`networks` отклоняется (возвращается `409`, причина `outside configured networks`), и любое
ограничение радиуса поражения — `max_active_bans`, `max_banned_fraction` или `max_bans_per_window` —
отклоняется (возвращается `409`). Поле `reason` называет сработавшее правило; см. [модель
безопасности](/docs/safety). Оба POST-эндпойнта требуют content type `application/json`. Полные формы
запроса и ответа см. в [справочнике по API](/docs/api).

## Объект блокировки

API возвращает каждую блокировку как JSON-объект. Это поля ровно в том виде, как их сериализует митигатор:

| Поле | Тип | Значение |
| --- | --- | --- |
| `target` | string | Адрес хоста в blackhole (IPv4 или IPv6). |
| `prefix` | string | Анонсируемый префикс хоста: `/32` для IPv4, `/128` для IPv6. |
| `metric` | string | Метрика, вызвавшая блокировку по атаке (опускается для ручных блокировок). |
| `rate` | number | Наблюдаемая скорость на момент блокировки, в единицах метрики (опускается при нуле). |
| `threshold` | number | Порог, который превысила скорость (опускается при нуле). |
| `next_hop` | string | Discard next-hop, анонсируемый с маршрутом. |
| `community` | string | Набор community, прикреплённый к маршруту (RTBH-community для blackhole, divert-community для divert-блокировки), объединяется через пробел при наличии нескольких. |
| `local_pref` | number | `LOCAL_PREF`, прикреплённый к маршруту (опускается при нуле). |
| `route` | string | Полный маршрут в виде `blackhole <prefix> next-hop <next_hop> community <community>` (или `divert ...` для divert-блокировок) или сводка `flowspec: ...` для FlowSpec-блокировок. |
| `state` | string | Одно из `active`, `withdrawn` или `rejected`. |
| `dry_run` | boolean | Была ли блокировка создана в режиме dry-run (маршрут фактически не отправлен). |
| `manual` | boolean | `true` для блокировок по запросу оператора, `false` для автоматических. |
| `started_at` | timestamp | Когда блокировка была создана. |
| `expires_at` | timestamp | Когда TTL автоматически отзовёт маршрут. |
| `withdrawn_at` | timestamp | Когда маршрут был отозван (опускается, пока блокировка активна). |
| `reason` | string | Почему блокировка была отозвана или отклонена (опускается, пока блокировка активна). |
| `method` | string | Метод подавления, создавший блокировку: `blackhole`, `flowspec` или `divert`. |
| `flowspec` | array | Сгенерированные правила [FlowSpec](/docs/flowspec), когда метод — `flowspec`. |
| `escalation` | array | Настроенная [лестница эскалации](/docs/escalation), если она задана. |
| `escalation_step` | number | Индекс текущей ступени лестницы. |
| `fell_back_from` | string | Метод, чей анонс отклонил пир, что привело к деградации блокировки до её запасного варианта (опускается, если отката не было). |
| `fell_back_reason` | string | Человекочитаемая причина отката (опускается, если отката не было). |

Блокировка в состоянии `rejected` никогда не анонсируется — она фиксирует отказ (из белого списка, вне
`networks` или достигнут лимит), чтобы решение можно было проаудировать в API и уведомлениях.
