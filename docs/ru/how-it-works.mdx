# Как это работает

Kapkan — это единый статически слинкованный бинарный файл на Go с модульным устройством внутри. Нет ни сопутствующих сервисов (sidecar), ни базы данных, ни отдельного веб-сервиса для запуска: приём телеметрии потоков, движок обнаружения, подавление атак через BGP (BGP, [Border Gateway Protocol](/docs/glossary), — это протокол, по которому маршрутизаторы обмениваются маршрутами; Kapkan выступает как BGP-спикер, который сообщает вашим маршрутизаторам, какие маршруты установить), REST API, панель управления и средства уведомлений — всё это пакеты, скомпилированные в один процесс. Вы указываете ему YAML-конфигурацию, направляете на него экспортёры своих маршрутизаторов — и он работает.

Всё, что вы наблюдаете, — атаки, основные источники трафика (top talkers), выученные базовые уровни (baselines), баны — хранится в памяти процесса и отдаётся прямо оттуда. Для работы не требуется ничего внешнего; единственные зависимости — это опциональные интеграции (сервер ClickHouse для истории, SMTP-релей для электронной почты), и без них Kapkan работает полностью внутри процесса.

## Конвейер

Бинарный файл организован как единый однонаправленный конвейер. Каждая стадия передаёт нормализованные данные следующей, а движок распределяет свои результаты по трём стадиям-потребителям:

1. **Ingest (приём)** — UDP-слушатели декодируют датаграммы sFlow v5, NetFlow v5/v9 и IPFIX (с помощью библиотеки [goflow2](https://github.com/netsampler/goflow2) в библиотечном режиме) в единое нормализованное представление `Flow`. Частота выборки считывается из пакета, когда экспортёр её сообщает, иначе — из `sampling.default_rate`.
2. **Engine (горячий путь)** — каждый поток сворачивается в шардированные счётчики по хостам на скользящем окне, корректируется с учётом выборки до реальных единиц трафика и оценивается относительно активных порогов (глобальных, по протоколам, по hostgroup и выученных базовых уровней). Это критичная по производительности стадия; см. [Производительность](#производительность).
3. **Mitigate / notify / api** — когда порог превышен, движок генерирует событие атаки. Стадия mitigate анонсирует или отзывает маршруты [RTBH](/docs/glossary) (Remotely Triggered Black Hole — BGP-маршрут, который сообщает вашим маршрутизаторам отбрасывать весь трафик к атакуемому IP) через встроенный BGP-спикер; каждое такое решение Kapkan отслеживает как *бан*. Стадия notify распределяет событие по вашим каналам, а стадия api предоставляет актуальное состояние панели управления и вызывающим сторонам.

Поток данных одной строкой:

```text
ingest → engine (hot path) → [mitigate, notify, api]
```

<Callout type="info" title="Только в одном направлении">
Потоки движутся вперёд по конвейеру; стадии-потребители никогда не возвращают данные обратно в горячий путь. Движок — единственная стадия, которая работает с состоянием на уровне отдельных потоков, и именно это позволяет горячему пути обходиться без аллокаций и почти без блокировок.
</Callout>

## Компоненты

Внутри Kapkan следует стандартной раскладке проекта на Go. Каждый пакет владеет одной стадией конвейера:

| Пакет | Зона ответственности |
| --- | --- |
| `cmd/kapkan/` | main, разбор флагов, обработка сигналов |
| `internal/app/` | связывание всех компонентов; сквозной (end-to-end) тест |
| `internal/config/` | загрузка YAML, проверка, горячая перезагрузка по SIGHUP |
| `internal/ingest/` | приём через goflow2 в библиотечном режиме в нормализованный `Flow` |
| `internal/engine/` | шардированные счётчики по хостам, скользящее окно, оценка порогов |
| `internal/mitigate/` | встроенный GoBGP: анонс/отзыв RTBH, TTL, лимиты, dry-run |
| `internal/notify/` | уведомления в Telegram, Slack, по email, через webhook и exec-хуки |
| `internal/api/` | REST API + метрики Prometheus |
| `pkg/flowgen/` | синтетический генератор NetFlow/sFlow для тестов и нагрузки |

Та же раскладка в виде дерева:

```text
cmd/kapkan/        main, flag parsing, signal handling
internal/app/      wiring of all components; end-to-end test
internal/config/   YAML load, validation, SIGHUP hot-reload
internal/ingest/   goflow2 library-mode ingestion -> normalized Flow
internal/engine/   sharded per-host counters, sliding window, threshold eval
internal/mitigate/ embedded GoBGP: RTBH announce/withdraw, TTL, caps, dry-run
internal/notify/   Telegram, Slack, email, webhook and exec-hook notifications
internal/api/      REST API + Prometheus metrics
pkg/flowgen/       synthetic NetFlow/sFlow generator for tests and load
```

Ключевые сторонние библиотеки — это [goflow2](https://github.com/netsampler/goflow2) для декодирования потоков и [GoBGP](https://github.com/osrg/gobgp) для BGP-спикера — обе используются в библиотечном режиме, поэтому рядом с Kapkan не нужно разворачивать ни внешний коллектор, ни маршрутизирующий демон. Для HTTP и структурированного логирования используется стандартная библиотека Go.

## Поток данных

```text
ingest → engine (hot path) → [mitigate, notify, api]
```

Каждый переход по порядку:

- **Экспортёр маршрутизатора → ingest.** Ваши маршрутизаторы отправляют записи потоков с выборкой по UDP на настроенные порты прослушивания (`listen.sflow`, `listen.netflow`). Пакет ingest декодирует формат передачи и нормализует каждую запись — адреса, порты, протокол, счётчики байтов/пакетов, частоту выборки — в единый внутренний тип `Flow`, независимо от того, каким протоколом она была порождена.
- **Ingest → engine.** Нормализованные потоки попадают в горячий путь. Движок относит каждый поток к хосту-назначению внутри `networks` (защищаемые префиксы, которые вы настраиваете, — см.
  [Обнаружение и пороги](/docs/detection)), умножает его счётчики на эффективную частоту выборки (частоту экспортёра, скорректированную на дедупликацию на границе там, где это настроено), чтобы скорости выражались в реальном (без выборки) трафике, и обновляет счётчики этого хоста на скользящем окне. Назначения вне `networks` учитываются в метриках, но никогда не приводят к действиям.
- **Engine → mitigate.** Когда хост пересекает порог, движок поднимает событие атаки. Стадия mitigate решает, анонсировать ли RTBH-маршрут — с учётом dry-run, `protected_whitelist`, политики `ban` для hostgroup и лимита банов — и планирует его отзыв по истечении TTL.
- **Engine → notify.** То же событие доставляется в каждый настроенный канал (Telegram, Slack, email, webhook, exec-хук) с приложенными классификацией атаки и образцом потока. Движок классифицирует каждую атаку по вектору (SYN-флуд, UDP/NTP/DNS-усиление, ICMP-флуд, флуд фрагментами и так далее) на основе соотношения трафика по протоколам и образца потока, и именно этот вектор отображается как `type` атаки в уведомлении.
- **Engine → api.** Актуальное состояние — активные и недавние атаки, отслеживаемые хосты, баны — считывается из памяти движка через REST API и встроенную панель управления.

Поскольку недавние потоки буферизуются непрерывно (`samples.*`), в момент превышения порога доминирующие источники, порты и протоколы атаки уже приложены к событию — задержки на захват данных после обнаружения нет.

## Производительность

Горячий путь — это обработка отдельных потоков в `internal/engine`, и он построен так, чтобы оставаться быстрым при частоте потоков масштаба атаки:

- **Шардированные счётчики по хостам.** Состояние хостов разбито по шардам с ключом по хэшу IP (256 шардов), поэтому конкурентные потоки редко конкурируют за одну и ту же блокировку.
- **Скользящее окно.** Каждый хост хранит оконное представление своего недавнего трафика, поэтому пороги оцениваются относительно скользящей скорости, а не мгновенных всплесков.
- **Горячий путь без аллокаций.** Буферы преаллоцируются, а счётчики атомарны; путь обработки отдельного потока избегает аллокаций в куче, чтобы сборщик мусора не мешал работе.

Целевые показатели описываются двумя цифрами в разных масштабах — представляйте их именно как то, чем они являются:

- В README указано, что движок выдерживает **≥20M flows/sec/core** на горячем пути — пропускную способность шага свёртки в памяти на одно ядро в изоляции.
- Инженерная цель проекта — **≥200k flows/sec на 8 ядрах** для сквозной обработки отдельных потоков — планка, которую изменение горячего пути должно преодолеть в бенчмарках `make bench`, прежде чем оно будет считаться завершённым.

<Callout type="info" title="Сначала измерьте бенчмарком, потом доверяйте">
Движок поставляется с бенчмарками `go test -bench` (`make bench`). Правило проекта — запускать их, прежде чем заявлять, что изменение горячего пути завершено, чтобы приведённые выше цифры оставались честными от релиза к релизу.
</Callout>

## Конфигурация и перезагрузка

Kapkan настраивается одним YAML-файлом, передаваемым через `-config`. Здесь нет ни многослойной конфигурации, ни слияния с окружением, о которых пришлось бы помнить, — то, что лежит в файле (плюс несколько секретов, считываемых из именованных переменных окружения), и есть рабочее состояние. Хранение его в git даёт вам поддающуюся сравнению (diff) и проверке (review) запись каждого порога и каждого пира.

Конфигурация **перезагружается на лету** без перезапуска демона. Запустить перезагрузку можно двумя способами:

- Отправить процессу `SIGHUP` (например, `sudo systemctl reload kapkan` или `kill -HUP <pid>` на установке без systemd либо при ручном запуске).
- Вызвать `POST /api/v1/config/reload` через API.

Оба способа перечитывают и повторно проверяют файл на месте. Почти всё вступает в силу при перезагрузке — переключатель `dry_run`, пороги обнаружения, `networks`, hostgroup, базовые уровни, настройки уведомлений и `sampling.default_rate` перезагружаются на лету. Несколько структурных настроек отклоняют перезагрузку и требуют полного перезапуска: адреса прослушивания, идентичность BGP (`local_asn`, `router_id`), `api.listen`, а также настройки `samples`, `storage` и `geoip`. См. [Справочник конфигурации](/docs/configuration) с полным списком того, какие ключи перезагружаются, а какие нет.

<Callout type="warning" title="Перезагрузки проходят проверку">
Перезагрузка повторно проверяет весь файл. Если новая конфигурация некорректна, перезагрузка отклоняется, а продолжает работать предыдущая исправная конфигурация — плохое изменение не положит демона, но проверьте логи, чтобы убедиться, что ваше изменение действительно применилось.
</Callout>
