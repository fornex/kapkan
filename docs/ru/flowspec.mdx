# FlowSpec-митигация

Blackhole-фильтрация через RTBH полностью выводит жертву из строя — она меняет атаку на простой. BGP
FlowSpec (RFC 8955 для IPv4, RFC 8956 для IPv6), напротив, распространяет правило, которое отбрасывает только
совпадающий трафик атаки, поэтому жертва продолжает обслуживать всё остальное.

<Callout type="info" title="Сначала dry-run">
Сгенерированные правила появляются в `/api/v1/attacks` (`method`, `flowspec`), в `/api/v1/bans`
и в уведомлениях **до** того, как вы установите `dry_run: false`, — так что вы можете сначала сверить их
с поддержкой FlowSpec на вашем апстриме. См. [Запуск в продакшен](/docs/going-live).
</Callout>

## Включение FlowSpec

Установите метод митигации в `flowspec`. Метод по умолчанию — `blackhole` (RTBH).

```yaml
mitigation: flowspec            # default method for all groups (default: blackhole)
flowspec:
  action: discard               # or rate_limit
  rate_mbps: 100                # required for rate_limit
hostgroups:
  - name: web
    networks: ["203.0.113.0/26"]
    mitigation: blackhole       # per-group override
```

`mitigation` можно переопределить на уровне отдельной [хостгруппы (hostgroup)](/docs/hostgroups), поэтому вы можете глобально использовать FlowSpec
и откатываться на blackhole для конкретного набора префиксов (или наоборот).

## Сгенерированные правила

При атаке Kapkan выводит **минимальный набор правил** (не более 8) из
[классификации](/docs/detection) атаки и выборки flow, сопоставляя жертву как назначение плюс
вектор:

| Атака | Сгенерированное совпадение FlowSpec |
| --- | --- |
| Усиление (amplification) NTP / DNS / CLDAP / memcached / SSDP / chargen | `dst=victim, proto=udp, src-port=<reflected port>` |
| SYN-флуд | `dst=victim, proto=tcp, tcp-flags=SYN` |
| Флуд фрагментами | `dst=victim, fragment` |
| ICMP- / UDP- / TCP-флуд | `dst=victim, proto=<icmp/udp/tcp>` |
| смешанная / неизвестная | `dst=victim` (плюс правило на каждый доминирующий отражённый порт источника в выборке) |

Жертва всегда сопоставляется как `/32` (IPv4) или `/128` (IPv6). **IPv6 FlowSpec полностью на
паритете с IPv4** — тогда как в дорожной карте самого FastNetMon IPv6 FlowSpec до сих пор числится как неподдерживаемый.

## Исходящие атаки

Для [исходящей атаки](/docs/detection) — скомпрометированного хоста, генерирующего флуд наружу, — правило
сопоставляет хост как **источник** (source-префикс по RFC 8955/8956), поэтому оно действительно отбрасывает
исходящий флуд. Это отличается от blackhole RTBH на основе назначения, который уничтожает только трафик
*к* хосту.

## Привязка к источнику

По умолчанию каждое сгенерированное правило сопоставляет жертву как **назначение**, отбрасывая вектор атаки
от *всех* источников — включая легитимных клиентов, отправляющих тот же протокол. При
`flowspec.source_anchored: true`, когда выборка атаки показывает **концентрированный** набор источников
атакующих, Kapkan вместо этого выдаёт составные правила, закрепляющие **и** жертву (назначение), **и** каждого
доминирующего атакующего (источник), поэтому отбрасываются только атакующие, а легитимный трафик жертвы
не затрагивается.

```yaml
flowspec:
  action: discard
  source_anchored: true          # drop the attackers, not the protocol from everyone
  min_source_concentration: 0.8  # ...only when the top sources cover >= this share (default 0.8)
```

Механизм включается только тогда, когда доминирующие источники покрывают не менее `min_source_concentration` от выборки
атаки в пределах бюджета из 8 правил. **Рассеянная** атака — отражение или спуфинг, распределённые по множеству
источников, — никогда не проходит этот порог и откатывается на правило с привязкой к жертве. Выбранные источники
сопоставляются только как host-префиксы (`/32`, `/128`): расширения до более крупных префиксов не происходит, поэтому правило
никогда не может задеть невинного соседа атакующего.

## Особенности

- Совпадение по `tcp-flags` для SYN-флуда — это битовая маска, которая также совпадает с SYN-ACK, поэтому
  действие `discard` отбрасывает и исходящие соединения жертвы, инициированные ею самой, — для TCP-векторов
  предпочтительнее `rate_limit`.
- `max_active_bans` ограничивает **баны**, а не правила: FlowSpec-бан может нести до 8 правил, поэтому N
  банов могут означать до 8N правил в RIB вашего апстрима. Следите за [метрикой](/docs/metrics) `kapkan_mitigate_flowspec_rules`
  относительно лимита FlowSpec-маршрутов на ваших маршрутизаторах.

## Действие правила и жизненный цикл

Каждое правило несёт extended community traffic-rate: `discard` (rate 0) или потолок `rate_limit`.
FlowSpec-баны используют тот же TTL, тот же гистерезис разбана и тот же жизненный цикл `max_active_bans`,
что и blackhole-баны, — см. [RTBH-митигация](/docs/mitigation).

FlowSpec работает через **те же BGP-соседи**, что и RTBH: AFI/SAFI для FlowSpec анонсируется
дополнительно, а пир, который его не поддерживает, просто не будет его согласовывать. FlowSpec
**недопустим для хостгрупп с `calculation: total`** — в них нет единого префикса жертвы для
сопоставления.
