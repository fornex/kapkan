# CLI reference

Kapkan is a single binary. Everything it does is selected by command-line flags at startup тАФ
there are no subcommands beyond the utility modes below. All logs go to **stderr**; the REST
API and Prometheus metrics are served over HTTP on `api.listen` (default `127.0.0.1:8080`), while
the utility flags (`-version`, `-dump-schema`) print their output to **stdout**.

```sh
kapkan [flags]
```

## Flags

| Flag | Default | Description |
| --- | --- | --- |
| `-config <path>` | `configs/dev.yaml` | Path to the YAML [configuration file](/docs/configuration). |
| `-log-format <fmt>` | `json` | Log output format: `json` (structured, for log collectors) or `text` (human-readable, for local runs). Any value other than `text` is treated as `json`. |
| `-log-level <lvl>` | `info` | Minimum log level: `debug`, `info`, `warn`, `error`. An unrecognized value falls back to `info`. |
| `-check-config <path>` | тАФ | Validate the config file at `<path>` and exit тАФ does **not** start the daemon. See [below](#validating-a-config). |
| `-dump-schema` | `false` | Print the configuration JSON schema to stdout and exit. See [below](#dumping-the-schema). |
| `-version` | `false` | Print the build version and exit. The same version is exposed at runtime via `/api/v1/status` and the `kapkan_build_info` [metric](/docs/metrics). |
| `-check-update` | `false` | Run the opt-in [update check](/docs/configuration) once and exit тАФ `0` up to date, `10` an update is available, `1` on error. Polls the GitHub Releases API; sends only the request itself, never node identity or config. |

<Callout type="info" title="json or text?">
Use `-log-format text` for local development тАФ it is the format the Quickstart and the
[`make run-dev`](/docs/quickstart) target use. Keep the default `json` in production so a log
collector can parse the structured fields; the [systemd unit](/docs/deployment) ships with
`-log-format json -log-level info`.
</Callout>

## Running the daemon

With no utility flag, Kapkan opens its flow listeners, REST API and (when not in dry-run) BGP
speaker, then runs until it receives `SIGINT` or `SIGTERM`, at which point it shuts down
cleanly. It exits non-zero only on a fatal startup or runtime error.

```sh
# Local, human-readable logs against the bundled dev config:
kapkan -config configs/dev.yaml -log-format text

# Production-style: structured logs at info level:
kapkan -config /etc/kapkan/config.yaml -log-format json -log-level info
```

The config file is reloadable at runtime without a restart тАФ send `SIGHUP`
(`systemctl reload kapkan`) or `POST /api/v1/config/reload`. See the
[configuration reference](/docs/configuration) for what reloads and what is fixed at startup.

## Utility flags

These flags exit before the daemon starts тАФ they never open a listener or send a route
announcement, so they are safe to run on a production host. (`-check-update` makes a single
outbound request to the releases API; the others are fully offline.)

### Validating a config

```sh
kapkan -check-config /etc/kapkan/config.yaml
```

Runs the engine's exact parse and validation тАФ including cross-field rules a static schema
cannot express тАФ then prints the resolved configuration (mode, protected networks, and the
effective mitigation per [hostgroup](/docs/hostgroups)) and exits:

| Exit code | Meaning |
| --- | --- |
| `0` | The config is valid. |
| `1` | The config is invalid; the exact error is printed to stderr. |

The clean 0/1 split drops straight into CI or a pre-deploy gate. See
[Validating the configuration](/docs/configuration#validating-the-configuration).

### Dumping the schema

```sh
kapkan -dump-schema
```

Prints the configuration's JSON schema to stdout and exits `0` (or `1` on an internal error).
The [config builder](/config) consumes this schema to validate fields in your browser.

### Printing the version

```sh
kapkan -version
```

Prints the build version (release tag plus short VCS revision when known) to stdout and exits.
The same string is served at runtime on `/api/v1/status` and carried by the `kapkan_build_info`
[metric](/docs/metrics) тАФ so you can confirm a binary's version locally with zero egress.

### Checking for updates

```sh
kapkan -check-update
```

Runs the [`update_check`](/docs/configuration) once on demand and exits: `0` if up to date, `10`
if a newer release is available, `1` on error. It queries the GitHub Releases API and sends only
the request itself (your IP and a generic User-Agent) тАФ never node identity, config or attack
data. The periodic in-process check is **off by default**; this flag works regardless of that
setting.
