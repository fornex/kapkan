# Schnellstart

Diese Anleitung führt Sie von null zu einer laufenden Kapkan-Instanz im Dry-Run, die Angriffe
gegen ein Test-Präfix erkennt. Es gibt keinen Build-Schritt — Kapkan wird als ein einzelnes
vorgefertigtes, signiertes Binary ausgeliefert — und nichts hier kann eine Route ankündigen, denn
Dry-Run ist die Voreinstellung.

<Callout type="info" title="Voraussetzungen">
Ein Linux-Host (`amd64` oder `arm64`). Um *Live*-Erkennungen zu sehen, benötigen Sie außerdem
einen Router, der NetFlow / IPFIX / sFlow exportiert — aber auch ohne einen solchen können Sie die
Erkennung durchgängig mit dem mitgelieferten Werkzeug für synthetische Angriffe validieren (der
[letzte Abschnitt](#5-kein-router-synthetische-angriffe-erzeugen), der aus einem Source-Checkout läuft).
Eine Go-Toolchain benötigen Sie nur, wenn Sie sich für das
[Bauen aus dem Quellcode](/docs/deployment#aus-dem-quellcode-bauen) statt für den Download entscheiden.
</Callout>

## 1. Kapkan installieren

Wählen Sie, was zu Ihrem Host passt — jedes Release wird vorgefertigt und signiert für `linux`
`amd64`/`arm64` ausgeliefert, sodass Sie nie etwas kompilieren.

### Debian / Ubuntu (`.deb`)

```sh
VER=v1.0.0
curl -fLO "https://github.com/fornex/kapkan/releases/download/$VER/kapkan_${VER#v}_linux_amd64.deb"
sudo apt install "./kapkan_${VER#v}_linux_amd64.deb"
```

Das Paket installiert das Binary nach `/usr/local/bin/kapkan`, legt einen unprivilegierten
Benutzer `kapkan` an, richtet `/etc/kapkan` mit einer Dry-Run-`config.yaml` ein und installiert die
gehärtete systemd-Unit — die **gestoppt** belassen wird, damit Sie die Konfiguration zuerst prüfen
können. Die Seite [Produktiv-Deployment](/docs/deployment) behandelt den Dienst von dort an. RHEL /
Fedora funktioniert genauso mit dem `.rpm`: `sudo dnf install ./kapkan_<ver>_linux_amd64.rpm`.

### Tarball (jede Distribution)

```sh
VER=v1.0.0
curl -fLO "https://github.com/fornex/kapkan/releases/download/$VER/kapkan_${VER#v}_linux_amd64.tar.gz"
tar xzf "kapkan_${VER#v}_linux_amd64.tar.gz"
```

Das entpackt das `kapkan`-Binary plus ein Verzeichnis `deploy/` (die systemd-Unit, eine
Beispielkonfiguration und `update.sh`). Der Rest dieser Seite verwendet den Tarball. Bevor Sie ihn
im Produktivbetrieb ausführen, lohnt es sich, [den Download zu
verifizieren](/docs/upgrading#einen-download-von-hand-verifizieren) — die cosign-Signatur und den SHA-256.

<Callout type="info" title="Das neueste Release verwenden">
Ersetzen Sie `v1.0.0` durch den neuesten Tag von der
[Releases-Seite](https://github.com/fornex/kapkan/releases/latest) und `amd64` durch `arm64` auf
ARM-Hosts. Archiv- und Paketnamen lassen das führende `v` weg (z. B. `kapkan_1.0.0_linux_amd64.tar.gz`).
</Callout>

## 2. Im Dry-Run betreiben

Starten Sie Kapkan aus dem entpackten Tarball mit der mitgelieferten Beispielkonfiguration und
menschenlesbaren Logs:

```sh
./kapkan -config deploy/config.example.yaml -log-format text
```

`-config` verweist auf die YAML-Konfiguration und `-log-format text` stellt die Logs vom
voreingestellten strukturierten JSON auf eine menschenlesbare Form um — siehe die
[CLI-Referenz](/docs/cli) für jeden Flag und seine zulässigen Werte.

Die mitgelieferte [`deploy/config.example.yaml`](/docs/configuration) hat `dry_run: true`,
geschützte Testnetze von `203.0.113.0/24` und `2001:db8::/32`, eine `protected_whitelist` und
lauscht auf sFlow an `:6343` sowie auf NetFlow/IPFIX an `:2055`. Da der Dry-Run aktiv ist,
zeichnet Kapkan potenzielle Blackholes auf, kündigt aber niemals eine Route an.

<Callout type="info" title="Das Paket installiert?">
Wenn Sie das `.deb`/`.rpm` installiert haben, läuft derselbe Daemon stattdessen als Dienst gegen
`/etc/kapkan/config.yaml` — `sudo systemctl start kapkan`, dann `journalctl -u kapkan -f`. Siehe
[Produktiv-Deployment](/docs/deployment).
</Callout>

## 3. Richten Sie Ihre Exporter auf Kapkan aus

Konfigurieren Sie die Flow-Exporter Ihrer Router so, dass sie an die Listen-Ports von Kapkan senden:

| Protokoll | Standard-Port |
| --- | --- |
| sFlow v5 | `:6343` |
| NetFlow v5/v9 + IPFIX | `:2055` |

NetFlow und IPFIX teilen sich denselben UDP-Socket. Mindestens ein Listener muss konfiguriert sein.

## 4. Beim Arbeiten zusehen

Kapkan stellt seine REST-API und Metriken standardmäßig unter `127.0.0.1:8080` bereit:

```sh
# Current mode, protected networks, thresholds, active attack/ban counts
curl -s localhost:8080/api/v1/status | jq

# Active attacks plus the last 100 that ended, with samples and classification
curl -s localhost:8080/api/v1/attacks | jq

# Prometheus metrics
curl -s localhost:8080/metrics | grep kapkan_
```

Sie können auch das eingebettete [Dashboard](/docs/dashboard) unter
[http://localhost:8080](http://localhost:8080) im Browser öffnen.

## 5. Kein Router? Synthetische Angriffe erzeugen

Dieser letzte Schritt läuft aus einem **Source-Checkout** (er nutzt die Test-Suite), benötigt also
das Repository und eine Go-Toolchain. Kapkan wird mit `pkg/flowgen` ausgeliefert, das echte
NetFlow- / sFlow-Datagramme im Wire-Format erzeugt — einschließlich Amplification-, SYN-Flood- und
UDP-Flood-Mustern. Es wird in der gesamten Test-Suite verwendet, um die Erkennung durchgängig zu
validieren, ohne einen echten Router anzufassen. Der Integrationstest in `internal/app` spielt einen
NTP-Amplification-Flood über einen echten UDP-Socket gegen eine Dry-Run-Instanz ab und prüft, dass
der Angriff und sein (automatisch ablaufender) virtueller Bann — das aufgezeichnete potenzielle
Blackhole, das der Dry-Run über die API offenlegt, ohne es je anzukündigen — in der API erscheinen:

```sh
git clone https://github.com/fornex/kapkan && cd kapkan/engine
go test ./internal/app -run TestEndToEnd -v
```
