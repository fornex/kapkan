# Benachrichtigungen

Kapkan löst bei jedem Ereignis im Lebenszyklus eines Angriffs eine Benachrichtigung
aus — wenn ein Angriff erstmals erkannt wird (`attack_started`) und wenn er abklingt
(`attack_ended`). Jedes Ereignis wird an jeden konfigurierten Kanal zugestellt:
Telegram, Slack, E-Mail, einen generischen JSON-Webhook und einen Exec-Hook.

Außerdem löst es ein einmaliges `update_available`-Ereignis über dieselben Kanäle aus, wenn
die optionale Update-Prüfung eine neuere Version findet und `update_check.notify` aktiviert
ist — siehe [Upgrade](/docs/upgrading). Es trägt die Versionsfelder anstelle der
Angriffsfelder, und der Exec-Hook im FastNetMon-Format (der keine entsprechende Aktion hat)
überspringt es.

Die Zustellung erfolgt nach dem Best-Effort-Prinzip und asynchron. Ein langsamer oder
fehlschlagender Kanal blockiert niemals die Erkennung oder Mitigation, und jeder Kanal
ist durch seinen eigenen Worker-Pool begrenzt — bei einem Carpet-Bomb-Angriff, der pro
Host ein Ereignis erzeugt, werden überzählige Benachrichtigungen verworfen und gezählt,
anstatt sich unbegrenzt als Verbindungen oder Hook-Prozesse anzusammeln.

<Callout type="info" title="Geheimnisse stammen aus der Umgebung">
Bot-Tokens und SMTP-Zugangsdaten werden aus benannten Umgebungsvariablen gelesen,
niemals aus der Konfigurationsdatei. Die Datei enthält nur den Namen der Variable (zum
Beispiel `token_env: "KAPKAN_TG_TOKEN"`), sodass Sie die Konfiguration in git halten
können.
</Callout>

## Kanäle

### Telegram

Setzen Sie `notify.telegram.token_env` auf den Namen der Umgebungsvariable, die Ihr
Bot-Token enthält, und `notify.telegram.chat_id` auf den Ziel-Chat. Das Token wird zum
Sendezeitpunkt aus dieser Variable gelesen; es wird niemals in der Datei gespeichert.
Nachrichten werden als HTML gesendet und enthalten das Angriffsziel, die Klassifizierung,
die auslösende Metrik und den Sperrstatus.

<Callout type="info" title="Chat-ID herausfinden">
Um die numerische Chat-ID zu ermitteln, schreiben Sie [@getmyid_bot](https://t.me/getmyid_bot)
für einen persönlichen Chat direkt an oder fügen Sie ihn der Zielgruppe hinzu, um die
Gruppen-ID auszulesen (Gruppen-IDs sind negativ, z. B. `-1001234567890`).
</Callout>

### Slack

Setzen Sie `notify.slack.webhook_url` auf die URL eines Slack-
[Incoming-Webhooks](https://api.slack.com/messaging/webhooks). Kapkan sendet bei jedem
Ereignis per POST eine formatierte Nachricht dorthin. Lassen Sie das Feld leer, um den
Kanal zu deaktivieren.

### E-Mail (SMTP)

Konfigurieren Sie `notify.email`, um SMTP-Mail zu versenden. Setzen Sie `smtp_host` auf
`host:port` (ein leerer Wert deaktiviert den Kanal), `from` auf den Envelope-Absender und
`to` auf eine Liste von Empfängern. Die Zugangsdaten werden über `username_env` und
`password_env` aus der Umgebung gelesen.

| Schlüssel | Bedeutung |
| --- | --- |
| `smtp_host` | SMTP-Server als `host:port`; leer deaktiviert E-Mail. |
| `from` | Absenderadresse für Envelope und Header. |
| `to` | Liste der Empfängeradressen. |
| `username_env` | Umgebungsvariable mit dem SMTP-Benutzernamen (aktiviert AUTH PLAIN). |
| `password_env` | Umgebungsvariable mit dem SMTP-Passwort. |
| `require_tls` | STARTTLS auch ohne Zugangsdaten erzwingen. |

Kapkan verwendet STARTTLS, sobald der Server es anbietet. STARTTLS ist **erforderlich**,
wenn Zugangsdaten konfiguriert sind (ein nicht leeres `username_env`) oder wenn
`require_tls: true` gesetzt ist — in diesem Fall wird ein Server, der kein STARTTLS
anbietet, mit einem deutlichen Fehler abgelehnt, anstatt auf Klartext zurückzufallen, was
einen aktiven Downgrade-Angriff in einen sichtbaren Fehler verwandelt. Wird TLS weder
angeboten noch verlangt, läuft die Zustellung an einen Nicht-Loopback-Host im Klartext ab
und wird deutlich protokolliert.

<Callout type="warning" title="Klartext an entfernte Hosts wird protokolliert, nicht blockiert">
Wenn Ihr SMTP-Server kein STARTTLS anbietet und Sie weder Zugangsdaten noch `require_tls`
setzen, versendet Kapkan Mail im Klartext an einen Nicht-Loopback-Host und gibt eine
Warnung aus. Setzen Sie `require_tls: true`, damit dieser Fall stattdessen fehlschlägt.
</Callout>

### Webhook

Setzen Sie `notify.webhook.url` auf einen beliebigen HTTPS-Endpunkt. Kapkan sendet bei
jedem Angriffsbeginn und -ende ein HTTP-`POST` mit einem `Content-Type: application/json`-Body
— dem unten beschriebenen [Callback-Payload](#callback-payload-schema). Nicht-2xx-Antworten
werden als Zustellfehler behandelt und gezählt. Die Anfrage trägt ein begrenztes Timeout,
sodass ein hängender Empfänger keinen Zustell-Slot blockieren kann.

```http
POST /your/endpoint HTTP/1.1
Content-Type: application/json
```

### Exec-Hook

Setzen Sie `notify.exec.command` auf den absoluten Pfad einer vorhandenen ausführbaren
Datei. Bei jedem Ereignis führt Kapkan diese direkt aus — **ohne Shell** — mit:

- Dem [Callback-Payload](#callback-payload-schema) als JSON auf **stdin**.
- Dem Ereignisnamen (`attack_started`, `attack_ended` oder `update_available`) als **`argv[1]`**.
- Einer **minimalen Umgebung**: nur `PATH`, `HOME`, `TZ`, `LANG`, `USER` und `TMPDIR`
  werden durchgereicht. Die Geheimnisse des Daemons (Bot-Tokens, SMTP-Zugangsdaten) werden
  niemals an den Hook oder dessen Kindprozesse vererbt.

Der Befehl muss beim Laden der Konfiguration vorhanden und ausführbar sein.
`timeout_seconds` (Standard `10`) begrenzt jeden Lauf; bei einem Timeout beendet Kapkan
die **gesamte Prozessgruppe** des Hooks, damit Enkelprozesse nicht als Waisen
weiterleben können. Der Hook erhält dasselbe Schema wie der Webhook, sodass ein einziges
Skript beide Kanäle bedienen kann.

Die Aufrufkonvention wird über `notify.exec.format` gewählt: `kapkan` (der Standard,
oben beschrieben) oder `fastnetmon` (siehe unten).

<Callout type="danger" title="Der Hook läuft als Daemon-Benutzer">
Der Exec-Befehl läuft mit den Rechten des Kapkan-Prozesses. Behandeln Sie das Skript als
Teil Ihrer vertrauenswürdigen Basis, halten Sie es auf einem Pfad, in den nur root
schreiben kann, und lesen Sie niemals nicht vertrauenswürdige, von Angreifern
kontrollierte Felder zurück in eine Shell.
</Callout>

### FastNetMon-kompatibler Modus

Sie migrieren von FastNetMon? Setzen Sie `notify.exec.format: fastnetmon`, und Kapkan ruft
den Hook genau so auf, wie es FastNetMons `notify_script` tut — sodass Ihre vorhandenen
Notify-Skripte unverändert laufen:

- **`argv`** ist `<ip> <direction> <pps> <action>`.
- **stdin** trägt eine Klartext-Angriffszusammenfassung (FastNetMons `attack_details`-Body).

```yaml
notify:
  exec:
    command: "/usr/local/bin/notify_about_attack.sh"
    format: fastnetmon
```

`action` ist einer von `ban`, `unban` oder `attack_details`, abgeleitet aus dem
Kapkan-Ereignis:

| Kapkan-Ereignis | `action` |
| --- | --- |
| Angriff begonnen, Host blackholed oder umgeleitet | `ban` |
| Angriff begonnen, nur Alarm (keine Sperre) | `attack_details` |
| Angriff beendet, eine Sperre wurde zurückgenommen | `unban` |
| **jedes Host-scoped-Ereignis, während `dry_run` true ist** | `attack_details` |
| Group-scoped-Ereignis (`calculation: total`) | *übersprungen — Hook wird nicht aufgerufen* |
| `update_available`-Ereignis | *übersprungen — kein FastNetMon-Äquivalent* |

<Callout type="warning" title="Dry-Run gibt niemals ban/unban aus">
Im Dry-Run kündigt Kapkan nichts an — aber ein FastNetMon-Ban-Skript *würde* den Host
per Firewall sperren. Daher wird im Dry-Run jedes Ereignis auf das informative
`attack_details` herabgestuft, niemals auf `ban`/`unban`, und ein FastNetMon-Skript kann
keinen Host sperren, den Sie nur validieren. Group-scoped-Angriffe (`calculation: total`)
haben keinen einzelnen Host und werden in diesem Modus übersprungen.
</Callout>

## Konfiguration

Ein vollständiger `notify`-Block. Jeder Kanal ist optional — ein leeres `smtp_host`,
`url`, `webhook_url` oder `command` deaktiviert den jeweiligen Kanal.

```yaml
notify:
  telegram:
    token_env: "KAPKAN_TG_TOKEN"   # token is read from this env var, never the file
    chat_id: "-1001234567890"
  webhook:
    url: ""                        # generic JSON POST on every attack start/end
    # payload schema: docs/callback-schema.json (check schema_version)
  slack:
    webhook_url: ""                # Slack incoming webhook
  email:                           # SMTP; credentials from env vars
    smtp_host: ""                  # "mail.example.com:587"; empty disables
    from: ""
    to: []                         # ["ops@example.com"]
    username_env: "KAPKAN_SMTP_USER"
    password_env: "KAPKAN_SMTP_PASS"
    require_tls: false             # STARTTLS is auto-required when credentials are set
  exec:                            # hook run on every attack event (no shell)
    command: ""                    # absolute path to an existing executable
    timeout_seconds: 10
    format: kapkan                 # or "fastnetmon" for FastNetMon-style argv/stdin
```

## Callback-Payload-Schema

Der Webhook-Body und die stdin des Exec-Hooks tragen dasselbe JSON-Objekt. Es ist über
`schema_version` versioniert, derzeit `1`. Die maßgebliche Definition befindet sich in
[`docs/callback-schema.json`](https://github.com/fornex/kapkan/blob/main/docs/callback-schema.json).

<Callout type="info" title="Vorwärtskompatibilität">
Konsumenten MÜSSEN `schema_version` prüfen und SOLLTEN unbekannte Felder ignorieren. Die
Version wird nur bei Breaking Changes erhöht, sodass neue Felder hinzugefügt werden können,
ohne Sie zu beeinträchtigen.
</Callout>

| Feld | Typ | Hinweise |
| --- | --- | --- |
| `schema_version` | integer | In diesem Release immer `1`. |
| `event` | string | `attack_started`, `attack_ended` oder `update_available` (eine neuere Version existiert). |
| `scope` | string | `host` (ein einzelnes `target` wird angegriffen) oder `group` (der summierte Traffic einer Hostgroup mit `calculation: total`; `group` gesetzt, `target` fehlt). |
| `target` | string | Angegriffener Host — oder, bei ausgehenden Angriffen, der angreifende Host. Bei Ereignissen mit Group-Scope nicht vorhanden. |
| `group` | string | Name der zugehörigen Hostgroup; `global` für die implizite Fallback-Gruppe. |
| `direction` | string | `incoming`, oder `outgoing`, wenn das Ziel den Angriff selbst ausgeht (wahrscheinlich kompromittiert). |
| `metric` | string | Der überschrittene Schwellwert (zum Beispiel `pps`, `mbps`, `udp_pps`, `tcp_syn_pps`, `frag_pps`). |
| `rate` | number | Beobachteter Wert der auslösenden Metrik, sampling-korrigiert. |
| `threshold` | number | Konfiguriertes Limit dieser Metrik bei Angriffsbeginn. |
| `pps` | number | Sampling-korrigierte Pakete pro Sekunde. |
| `mbps` | number | Sampling-korrigierte Megabit pro Sekunde. |
| `flows_per_sec` | number | Sampling-korrigierte Flows pro Sekunde. |
| `ban_state` | string | `active`, `withdrawn` oder `rejected`, wenn die Mitigation konsultiert wurde. |
| `method` | string | `blackhole` oder `flowspec` — die Mitigationsmethode, die die Sperre erzeugt hat; entfällt, wenn keine Sperre vorliegt. |
| `route` | string | Die (im Dry-Run: potenzielle) Ankündigung: eine Blackhole-Route oder eine `flowspec: ...`-Zusammenfassung der generierten Regeln. |
| `dry_run` | boolean | Ob die Mitigation für dieses Ereignis im Dry-Run-Modus war. |
| `at` | string | RFC-3339-Zeitstempel des Ereignisses. |
| `classification` | object | Abgeleiteter Angriffsvektor — nur bei `attack_started`. |
| `sample` | object | Flow-Sample, das erfasst wurde, bevor der Schwellwert überschritten wurde — nur bei `attack_started`, wenn der Traffic-Puffer aktiviert ist. |
| `current_version` | string | Laufender Build — nur bei `update_available`. |
| `latest_version` | string | Das Tag der neueren Version — nur bei `update_available`. |
| `security` | boolean | `true`, wenn die Version als sicherheitsrelevant markiert ist — nur bei `update_available`. |
| `release_url` | string | Die URL der Version — nur bei `update_available`. |

Das `classification`-Objekt trägt den abgeleiteten `type` (zum Beispiel `ntp_amplification`,
`syn_flood`, `udp_flood` oder `mixed`, wenn keine Signatur passte), einen `confidence`-Anteil
von `0` bis `1` sowie `src_port` für Amplification-Vektoren. Das `sample`-Objekt trägt
gekürzte `flows` sowie Top-K-Zähler `top_sources`, `top_src_ports`, `top_dst_ports` und
`protocols` und einen Nenner `total_packets`.
