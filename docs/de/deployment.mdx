# Produktiv-Deployment

Eine gehärtete systemd-Unit und ein Beispiel für eine Produktivkonfiguration liegen im
Verzeichnis [`engine/deploy/`](https://github.com/fornex/kapkan/tree/main/engine/deploy) des Repositorys bei:
`engine/deploy/kapkan.service` und `engine/deploy/config.example.yaml`. Diese Seite führt durch die
Installation des Binaries als Systemdienst, zeigt, wo die einzelnen Dateien liegen, wie
Geheimnisse aus der YAML-Datei herausgehalten werden und wie sich die Konfiguration ohne
Neustart neu laden lässt.

<Callout type="info" title="Kein Bauen erforderlich">
Der schnellste Weg ist ein vorgefertigtes Paket: `apt install ./kapkan_*.deb` (oder das `.rpm`)
erledigt alles in diesem Abschnitt für Sie — siehe [Das Paket installieren](#das-paket-installieren)
unten. Die manuellen Schritte danach gelten für den [Release-Tarball](/docs/quickstart#1-kapkan-installieren).
So oder so gibt es nichts zu kompilieren; [Aus dem Quellcode bauen](#aus-dem-quellcode-bauen) ist
nur dann nötig, wenn Sie es möchten.
</Callout>

## Installation

### Das Paket installieren

Unter Debian/Ubuntu (`.deb`) oder RHEL/Fedora (`.rpm`) erledigt das Paket alles, was die manuellen
Schritte unten tun: Es installiert das Binary nach `/usr/local/bin/kapkan`, legt den unprivilegierten
Benutzer `kapkan` an, richtet `/etc/kapkan` mit einer aus dem Beispiel erzeugten Dry-Run-`config.yaml`
ein, erstellt das beschreibbare Zustandsverzeichnis und installiert die gehärtete systemd-Unit
(nach `/usr/lib/systemd/system`).

```sh
VER=v1.0.0
curl -fLO "https://github.com/fornex/kapkan/releases/download/$VER/kapkan_${VER#v}_linux_amd64.deb"
sudo apt install "./kapkan_${VER#v}_linux_amd64.deb"   # RHEL/Fedora: dnf install ./kapkan_<ver>_linux_amd64.rpm
```

Der Dienst ist installiert, wird aber **gestoppt belassen** — ein Mitigation-Daemon sollte nicht
starten, bevor Sie seine Konfiguration angesehen haben. Die mitgelieferte `config.yaml` hat
`dry_run: true`. Prüfen Sie sie, (empfohlen) [setzen Sie einen API-Token](#geheimnisse) in
`/etc/kapkan/kapkan.env` und starten Sie ihn dann beim Boot und jetzt:

```sh
sudoedit /etc/kapkan/config.yaml          # adapt networks, BGP and thresholds
sudo systemctl enable --now kapkan
```

Aktualisieren Sie mit einem neueren Paket (`apt install ./kapkan_<newer>.deb`) oder mit
[`update.sh`](/docs/upgrading) — beide behalten Ihre bearbeitete `/etc/kapkan/config.yaml`.
`apt purge kapkan` entfernt jede Spur, einschließlich Konfiguration und Zustand.

### Manuelle Installation (Tarball)

Möchten Sie die Dateien lieber selbst platzieren, oder sind Sie auf einer Distribution ohne
`apt`/`dnf`? Nach dem Entpacken des [Release-Tarballs](/docs/quickstart#1-kapkan-installieren) (sodass
das Verzeichnis `deploy/` neben dem `kapkan`-Binary liegt) führen Sie das Äquivalent als root aus.
Es installiert das Binary, legt einen unprivilegierten Systembenutzer an, richtet `/etc/kapkan` ein,
installiert die Produktivkonfiguration, schreibt die Geheimnis-Env-Datei mit restriktiven
Berechtigungen, installiert die systemd-Unit und aktiviert den Dienst.

```sh
sudo install -m 0755 kapkan /usr/local/bin/kapkan
sudo useradd --system --no-create-home --shell /usr/sbin/nologin kapkan
sudo install -d -o kapkan -g kapkan /etc/kapkan
sudo install -m 0640 -o kapkan -g kapkan deploy/config.example.yaml /etc/kapkan/config.yaml
echo 'KAPKAN_API_TOKEN=a-long-random-string' | sudo install -m 0600 /dev/stdin /etc/kapkan/kapkan.env
sudo install -m 0644 deploy/kapkan.service /etc/systemd/system/kapkan.service
sudo systemctl daemon-reload && sudo systemctl enable --now kapkan
```

Die Env-Datei oben enthält nur `KAPKAN_API_TOKEN`, da ein API-Token empfohlen wird, bevor Sie
den Listener nach außen öffnen (siehe [Authentifizierung](/docs/authentication)). Das Befüllen
der Env-Datei aktiviert die Authentifizierung jedoch nicht von selbst: Die mitgelieferte
Konfiguration bindet `127.0.0.1` mit auskommentiertem `api.token_env`, sodass der Token
wirkungslos bleibt. Kommentieren Sie `api.token_env` ein (oder fügen Sie eine `api.tokens`-Liste
hinzu), um die Authentifizierung zu aktivieren. Alle Benachrichtigungskanäle sind optional —
fügen Sie `KAPKAN_TG_TOKEN`, SMTP- oder ClickHouse-Zugangsdaten nur hinzu, wenn Sie diese
Funktionen tatsächlich nutzen. Die vollständige Liste der `*_env`-Schlüssel finden Sie unten
unter [Geheimnisse](#geheimnisse).

Die Unit läuft als Benutzer `kapkan` mit `ExecStart=/usr/local/bin/kapkan -config
/etc/kapkan/config.yaml -log-format json -log-level info`, startet bei Fehlern neu und ist
stark abgeschottet: `NoNewPrivileges`, `ProtectSystem=strict`, `PrivateTmp`,
`ReadOnlyPaths=/etc/kapkan`, `RestrictAddressFamilies=AF_INET AF_INET6`,
`MemoryDenyWriteExecute` und mehr. Sie deklariert außerdem `StateDirectory=kapkan` (Modus
`0700`) und stellt dem Daemon damit ein beschreibbares `/var/lib/kapkan` für
[`ban.state_file`](/docs/mitigation) bereit, während `ProtectSystem=strict` den Rest des
Dateisystems schreibgeschützt hält. Ein Crash-Loop-Schutz (`StartLimitIntervalSec=60`,
`StartLimitBurst=4`) versetzt die Unit in den Zustand `failed`, wenn das Binary innerhalb
einer Minute viermal nicht startet, sodass ein fehlerhaftes Upgrade sichtbar wird, statt zu
flattern — das [Upgrade-Skript](/docs/upgrading) stützt sich darauf zusammen mit der
`/healthz`-Readiness-Probe.

<Callout type="info" title="BGP an Port 179 binden">
Kapkan benötigt im dry-run oder beim Peering von einem hohen lokalen Port keine besonderen
Privilegien. Wenn Sie den BGP-Speaker an den well-known Port 179 binden, erteilen Sie die
nötige Capability, indem Sie `AmbientCapabilities=CAP_NET_BIND_SERVICE` zum Abschnitt
`[Service]` der Unit hinzufügen.
</Callout>

Sobald der Dienst läuft, prüfen Sie den Status und verfolgen Sie die Logs:

```sh
sudo systemctl status kapkan
sudo journalctl -u kapkan -f
```

## Dateilayout

| Pfad | Modus | Zweck |
| --- | --- | --- |
| `/usr/local/bin/kapkan` | `0755` | Das einzelne statische Daemon-Binary. |
| `/etc/kapkan/config.yaml` | `0640` | Die YAML-Konfiguration, im Besitz von `kapkan:kapkan`. |
| `/etc/kapkan/kapkan.env` | `0600` | Geheimnis-Env-Datei (Tokens, SMTP-Zugangsdaten). |
| `/etc/systemd/system/kapkan.service` | `0644` | Die gehärtete systemd-Unit. Das `.deb`/`.rpm` installiert sie stattdessen nach `/usr/lib/systemd/system/kapkan.service` (Hersteller-Verzeichnis). |
| `/var/lib/kapkan/` | `0700` | Beschreibbares Zustandsverzeichnis (`StateDirectory=kapkan`); enthält `ban.state_file`, wenn die ban-Persistenz aktiviert ist. |

Das gesamte Verzeichnis `/etc/kapkan` wird dem Prozess über `ReadOnlyPaths=/etc/kapkan`
schreibgeschützt eingehängt, sodass der Daemon seine Konfiguration und Geheimnisse lesen,
aber nicht verändern kann.

## Geheimnisse

Keine Zugangsdaten gehören in `config.yaml`. Die YAML-Datei verweist auf jedes Geheimnis über
den Namen einer **Umgebungsvariablen**, und die Unit lädt diese Variablen aus der Env-Datei
über `EnvironmentFile=-/etc/kapkan/kapkan.env`. Behalten Sie für diese Datei den Modus `0600` bei.

Die Konfigurationsschlüssel, die aus der Umgebung lesen, sind die `*_env`-Schlüssel:

- `notify.telegram.token_env` — z. B. `KAPKAN_TG_TOKEN`, der Telegram-Bot-Token.
- `notify.email.username_env` / `notify.email.password_env` — SMTP-Zugangsdaten.
- `api.token_env` — der API-Bearer-Token (siehe [Authentifizierung](/docs/authentication)).
- `storage.clickhouse.username_env` / `password_env` — ClickHouse-Zugangsdaten, falls Sie
  den optionalen Verlaufsspeicher aktivieren.

Zum Beispiel benennt die YAML-Datei die Variable, und die Env-Datei liefert ihren Wert:

```yaml
notify:
  telegram:
    token_env: "KAPKAN_TG_TOKEN"   # name of the env var, never the token itself
    chat_id: "-1001234567890"
```

```text
# /etc/kapkan/kapkan.env  (mode 0600)
KAPKAN_TG_TOKEN=123456:abcdef
KAPKAN_API_TOKEN=a-long-random-string
```

<Callout type="success" title="Warum das wichtig ist">
Da Geheimnisse ausschließlich in der Prozessumgebung leben, können Sie Ihre `config.yaml`
gefahrlos in git aufbewahren und diffen. Die systemd-Sandbox hält `/etc/kapkan`
schreibgeschützt und betreibt den Daemon als unprivilegierten Benutzer, sodass die Env-Datei
für den Dienst niemals beschreibbar ist.
</Callout>

## Konfigurations-Reload

Kapkan lädt seine Konfiguration bei `SIGHUP` im laufenden Betrieb neu — ohne Neustart, ohne
verlorenen flow-Status. Die systemd-Unit verdrahtet dies mit
`ExecReload=/bin/kill -HUP $MAINPID`, sodass Sie folgendermaßen neu laden:

```sh
sudo systemctl reload kapkan
```

Gleichwertig können Sie denselben Reload über die API auslösen:

```sh
curl -s -X POST localhost:8080/api/v1/config/reload \
  -H 'Content-Type: application/json' \
  -H "Authorization: Bearer $KAPKAN_API_TOKEN"
```

`Content-Type: application/json` ist bei jedem verändernden `POST` zwingend (andernfalls
liefert der Server `415`), und wenn die Authentifizierung aktiviert ist, benötigt der Reload
einen ungescopeten (Admin-)Bearer-Token — ein mandantengebundener Token erhält `403`.

Die meisten Schlüssel werden live neu geladen, darunter `dry_run`, Schwellenwerte,
`networks`, hostgroups, Baselines, Benachrichtigungseinstellungen und `sampling.default_rate`.
Sechs Dinge sind beim Start fest und ein Reload, der eines davon ändert, wird abgelehnt
(Neustart erforderlich): die `listen`-Adressen, die BGP-Identität (`bgp.local_asn` /
`router_id`), `api.listen`, die `samples`-Einstellungen, die `storage`-Einstellungen und die
`geoip`-Einstellungen. Siehe die [Konfigurationsreferenz](/docs/configuration) und, um das
Binary zu wechseln, ohne die Mitigation zu unterbrechen, [Upgrade](/docs/upgrading).

## Aus dem Quellcode bauen

Zum Bauen wird **Go 1.22 oder neuer** benötigt. Das `Makefile` stellt die üblichen Targets bereit:

```sh
make build  # static binary
make test   # go test -race ./...
make lint   # golangci-lint run
make bench  # engine hot-path benchmarks
```

`make build` erzeugt das einzelne statische `kapkan`-Binary, das Sie in den obigen Schritten
installieren. Die Tests verwenden synthetische NetFlow-/sFlow-Datagramme, die von
`pkg/flowgen` erzeugt werden (echtes Wire-Format), sowie einen In-Process-GoBGP-Peer, sodass
niemals echte Router kontaktiert werden.

<Callout type="warning" title="Bevor Sie Kapkan exponieren">
Behalten Sie `dry_run: true` bei, bis Sie die Erkennung gegen Live-Telemetrie validiert haben
— siehe [In Betrieb gehen](/docs/going-live). Und setzen Sie einen API-Token, bevor Sie den
Listener über `127.0.0.1` hinaus binden, sonst sind API und Dashboard unauthentifiziert
erreichbar — siehe [Authentifizierung](/docs/authentication).
</Callout>
