# Metriken

Kapkan stellt Prometheus-Metriken im Namespace `kapkan_` unter `GET /metrics` auf dem
API-Listener (`api.listen`, Standard `127.0.0.1:8080`) bereit. Sie decken die gesamte Pipeline ab —
Flow-Ingest, die Detection-Engine, RTBH-Mitigation und Benachrichtigungen — sodass Sie Durchsatz,
Latenz, aktive Angriffe und angekündigte Routen in Ihr bestehendes Monitoring übernehmen können.

<Callout type="info" title="/metrics bleibt offen">
Der Endpunkt `/metrics` ist immer unauthentifiziert, selbst wenn Sie mit `api.token_env` ein
API-Token festlegen. Nur die Daten-Endpunkte unter `/api/v1` erfordern `Authorization: Bearer <token>`.
Binden Sie den Listener entsprechend — siehe [Authentifizierung](/docs/authentication).
</Callout>

## Verfügbare Metriken

| Metrik | Labels | Bedeutung |
| --- | --- | --- |
| `kapkan_ingest_flows_total` | `proto` | Insgesamt aufgenommene Flows, nach Flow-Protokoll. |
| `kapkan_ingest_packets_total` | `exporter`, `proto` | Insgesamt empfangene Telemetrie-Datagramme, nach Quell-Exporter (kardinalitätsbegrenzt; siehe [`flow_sources`](/docs/configuration)) und Protokoll. |
| `kapkan_ingest_decode_errors_total` | `proto` | Telemetrie-Datagramme, deren Dekodierung fehlgeschlagen ist, nach Protokoll. |
| `kapkan_ingest_dropped_flows_total` | — | Flows, die verworfen wurden, weil die Eingangs-Queue der Engine voll war. |
| `kapkan_engine_active_attacks` | — | Gauge der aktuell laufenden Angriffe. |
| `kapkan_engine_attacks_total` | — | Zähler der seit dem Start erkannten Angriffe. |
| `kapkan_engine_process_latency_seconds` | — | Histogramm der Verarbeitungslatenz auf dem Hot Path, einmal pro Ingest-Batch erfasst (nicht pro Flow). |
| `kapkan_engine_tracked_hosts` | — | Gauge der aktuell von der Engine verfolgten Hosts. |
| `kapkan_mitigate_announced_routes` | `mode` | Gauge der aktuell angekündigten Blackhole-Routen, aufgeteilt nach den Modi `real` / `dry_run`. |
| `kapkan_mitigate_flowspec_rules` | `mode` | Gauge der aktuell angekündigten [FlowSpec](/docs/flowspec)-Regeln, aufgeteilt nach den Modi `real` / `dry_run`. Behalten Sie diesen Wert im Verhältnis zum FlowSpec-Routenlimit Ihrer Router im Auge. |
| `kapkan_mitigate_bans_rejected_total` | `reason` | Sperren, die durch eine Sicherheitsregel abgelehnt wurden, nach `reason`: `max_active_bans`, `blast_radius_fraction`, `blast_radius_rate` oder `max_active_prefix_bans` (carpet). |
| `kapkan_mitigate_fallback_total` | `from`, `to` | Mitigation-Ankündigungen, die auf eine fallback-Methode heruntergestuft wurden, weil der Peer die primäre Methode abgelehnt hat, nach `from`/`to`. Eine Reihe mit `from="flowspec"` und einem Wert ungleich null kennzeichnet Upstreams, die FlowSpec nicht berücksichtigen. |
| `kapkan_notify_notifications_total` | `channel`, `result` | Versuchte Benachrichtigungen, nach Kanal (telegram, slack, email, webhook, exec) und `result`. |
| `kapkan_storage_rows_total` | `table`, `result` | Zeilen, die an die optionale [ClickHouse-Speicherung](/docs/storage) übergeben wurden, nach Tabelle und `written` / `dropped` / `error`. |
| `kapkan_build_info` | `version`, `revision`, `goversion`, `goos`, `goarch` | Konstante Info-Gauge mit dem Wert `1`, die den laufenden Build in ihren Labels trägt (das `node_exporter`-Idiom). Fragen Sie die Versionsdivergenz der Flotte mit `count by (version)(kapkan_build_info)` ab — ohne Phone-home. |
| `kapkan_update_available` | `latest_version`, `security` | `1`, wenn der per Opt-in aktivierte [`update_check`](/docs/configuration) eine neuere Version findet (andernfalls nicht vorhanden); `security="true"` kennzeichnet eine sicherheitsrelevante Version. |

Das Label `mode` an `kapkan_mitigate_announced_routes` ist der schnellste Weg, den Dry-run-Zustand
auf einen Blick zu bestätigen: Solange `dry_run: true` gilt, wird jede potenzielle Route unter
`mode="dry_run"` gezählt und nichts unter `mode="real"`. Siehe [Mitigation](/docs/mitigation) und das
[Sicherheitsmodell](/docs/safety).

## Scraping

Fügen Sie Kapkan als Scrape-Target in Ihrer Prometheus-Konfiguration hinzu:

```yaml
scrape_configs:
  - job_name: kapkan
    metrics_path: /metrics
    static_configs:
      - targets: ["127.0.0.1:8080"]
```

Verwenden Sie den Host und den Port aus Ihrem Wert für `api.listen`. Wenn Sie den Listener über
localhost hinaus verfügbar machen, legen Sie ein API-Token fest — `/metrics` bleibt auch ohne dieses
erreichbar, sodass Ihr Scraper keine Anmeldedaten benötigt.

Um die Rohausgabe direkt zu inspizieren:

```sh
curl -s localhost:8080/metrics | grep kapkan_
```

## Grafana dashboard

Ein offizielles Grafana-Dashboard für die `kapkan_*`-Sammlung steht direkt zum Download bereit unter
[`kapkan-overview.json`](/kapkan-overview.json) (sein Quellcode liegt im Repository unter
[`engine/deploy/grafana/`](https://github.com/fornex/kapkan/tree/main/engine/deploy/grafana)).
Es deckt die gesamte Pipeline ab — Ingest-Raten für Flows/Datagramme/Dekodierung und
verworfene Flows, aktive und gestartete Angriffe, verfolgte Hosts, Latenz-Quantile auf dem Hot Path,
angekündigte Routen und FlowSpec-Regeln nach `mode`, nach `reason` abgelehnte Sperren,
Mitigation-Fallbacks sowie Benachrichtigungs- und Speicherergebnisse.

Gehen Sie in Grafana zu **Dashboards → New → Import**, laden Sie die JSON hoch und wählen Sie bei der
Aufforderung Ihre Prometheus-Datenquelle aus (sie ist als `DS_PROMETHEUS` templatisiert). Das Dashboard
ist auf Grafana 10+ ausgerichtet und wurde gegen Grafana 11 importverifiziert. Siehe die
[README](https://github.com/fornex/kapkan/blob/main/engine/deploy/grafana/README.md) daneben für die
Bereitstellung.
