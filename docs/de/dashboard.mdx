# Dashboard

Öffnen Sie **http://localhost:8080** in einem Browser, der auf dem Kapkan-Host läuft. Wenn Sie
unter Angriff stehen, schauen Sie zuerst auf das Banner **Live-Modus** (kündigt Kapkan tatsächlich
an, oder beobachtet es nur?) und auf die Liste **Aktive Angriffe** (was wird getroffen, und wie
heftig).

Die Web-Oberfläche ist direkt in die `kapkan`-Binärdatei eingebaut, sodass nichts zusätzlich zu
installieren oder bereitzustellen ist — derselbe Prozess stellt das Dashboard, die JSON-API und die
Prometheus-Metriken bereit.

<Callout type="info" title="Eine Binärdatei, ein Listener">
Das Dashboard, die JSON-API und `/metrics` werden alle vom selben Prozess unter derselben Adresse
bereitgestellt. Es gibt keinen separaten Webdienst, der bereitgestellt oder synchron gehalten
werden müsste.
</Callout>

## Wo es zu finden ist

Das Dashboard wird unter `/` auf derselben Adresse wie die REST-API bereitgestellt — dem Wert
`api.listen` aus Ihrer Konfiguration. Mit der Standardeinstellung `127.0.0.1:8080` öffnen Sie es
unter [http://localhost:8080](http://localhost:8080).

Da die Standardeinstellung an `127.0.0.1` bindet, ist das Dashboard nur vom Kapkan-Host selbst aus
erreichbar — wenn Sie `localhost:8080` auf Ihrem eigenen Laptop öffnen, sehen Sie nichts. Um es von
Ihrer Arbeitsstation aus anzuzeigen, leiten Sie den Port über SSH weiter, anstatt `api.listen` auf
eine öffentliche Adresse umzubinden:

```bash
ssh -L 8080:127.0.0.1:8080 user@kapkan-host
```

Öffnen Sie dann [http://localhost:8080](http://localhost:8080) auf Ihrer Arbeitsstation. (Das
Umbinden des Listeners auf eine öffentliche Adresse erfordert zuerst ein API-Token — siehe den
Hinweis unter [Authentifizierung](#authentifizierung) weiter unten.)

Um nur die JSON-API und die Metriken ohne die Oberfläche bereitzustellen, deaktivieren Sie es:

```yaml
api:
  listen: "127.0.0.1:8080"
  dashboard: false   # serve the JSON API and /metrics only
```

Mit `api.dashboard: false` funktionieren die `/api/v1`-Endpunkte und `/metrics` weiterhin — nur
die eingebettete UI-Shell wird nicht mehr bereitgestellt.

## Was es anzeigt

Das Dashboard fragt die JSON-API ab und stellt alles dar, was Kapkan im Speicher verfolgt:

- **Live-Modus** — ob die Instanz im Dry-run- oder im Live-Betrieb läuft, sodass Sie auf einen
  Blick sehen, ob ein erkannter Angriff tatsächlich angekündigt würde.
- **Aktive und kürzliche Angriffe** — aktuell laufende Angriffe sowie die kürzlich beendeten,
  jeweils mit ihrer Klassifizierung des Angriffstyps (zum Beispiel NTP/DNS-Amplification oder ein
  SYN-Flood — siehe die Seite [Erkennung](/docs/detection) für die Bedeutung jedes Typs) und den
  dominierenden Quellen, Ports und Protokollen, die aus den Flow-Samples erfasst wurden.
- **Top-Talker** — der Snapshot der verfolgten Hosts mit Raten je Richtung und gelernten
  Baselines, sodass Sie sehen können, wer am lautesten ist und wie sich das im Vergleich zum
  Normalzustand verhält.
- **Hostgroups** — Ihre benannten Präfixgruppen und deren Zustand.
- **Sperrtabelle** — jede Sperre, aktiv und historisch.
- **Traffic / Reports** — Live-Diagramme für ein- und ausgehenden Verkehr, die aus den
  In-Memory-Ratenpuffern erstellt werden, sowie ein Abschnitt **Historische Berichte**, der die
  persistierten ClickHouse-Tabellen liest, wenn die Speicherung aktiviert ist.
- **Steuerungen** — dieselben Aktionen, die auch die API bietet: **manuelles Sperren / Entsperren**
  einer Adresse und **Konfigurations-Reload** (das Äquivalent zum Senden von `SIGHUP`).

Der Abschnitt **Historische Berichte** in der Ansicht Traffic / Reports ist der einzige Teil, der
Speicherung benötigt: Er wird nur befüllt, wenn die [ClickHouse-Speicherung](/docs/storage)
aktiviert ist, und rendert andernfalls einen beschrifteten Platzhalter, der erklärt, wie man sie
aktiviert.

<Callout type="success" title="Keine Datenbank erforderlich">
Das Dashboard wird vollständig aus dem In-Memory-Zustand der Engine aufgebaut. Die historische
Flow-Speicherung ist eine separate, optionale Funktion — sie ist nicht erforderlich, um
Live-Angriffe, Top-Talker oder Sperren zu sehen.
</Callout>

## Angriffsdetails

Klicken Sie in der Liste **Aktive Angriffe** (aktive und kürzliche Angriffe) auf einen Angriff, um
dessen Detail-Drawer zu öffnen. Neben der Klassifizierung, der Aufschlüsselung „warum er ausgelöst
wurde", der Eskalationsleiter und der Mitigationsroute ordnen zwei Abschnitte den Datenverkehr zu:

- **Erfasste Stichprobe** — die dominierenden **Top-Quellen** (bei einem ausgehenden Angriff in
  **Top-Ziele** umbenannt), **Protokolle** und **Top-Quell- / Ziel-Ports**, destilliert aus der
  Flow-Stichprobe, sowie die insgesamt erfassten Pakete. Wenn GeoIP aktiviert ist, wird zusätzlich
  eine Zeile **Top-ASNs** angezeigt (bei ausgehenden Angriffen **Top-Ziel-ASNs**): die aktivsten
  autonomen Systeme nach Stichproben-korrigierten Paketen, aufgeführt als
  `AS<Nummer> <Organisation>`.
- **Rohe Flüsse** — eine Tabelle einzelner erfasster Flows (Quelle, Protokoll, Ziel, Flags,
  Fragmentierung, Pakete). Wenn eine Länderdatenbank geladen ist, trägt jede Quelle ein kleines
  Ländercode-Abzeichen wie `US` oder `DE`; fahren Sie mit der Maus darüber, um die Organisation
  (das Netz) der Quelle zu sehen.

Die Zeile **Top-ASNs** und die Länder-Abzeichen pro Flow erscheinen nur, wenn die GeoIP-Zuordnung
aktiviert ist. Verweisen Sie `geoip.asn_database` auf eine MaxMind-GeoLite2/GeoIP2-ASN-Datenbank für
die ASNs und Organisationsnamen und `geoip.country_database` auf eine Länderdatenbank für die
Länder-Abzeichen — siehe den `geoip`-Block unter [Konfiguration](/docs/configuration). Ohne sie wird
der Drawer trotzdem gerendert, nur ohne die ASN-Zeile und die Länder-Tags.

## Authentifizierung

Standardmäßig sind die API und das Dashboard nicht authentifiziert, was nur deshalb sicher ist,
weil die Standardeinstellung `api.listen` an `127.0.0.1` bindet. Wenn Sie ein API-Token setzen,
passt sich das Dashboard an: Die statische UI-Shell wird weiterhin geladen, die Daten dahinter
jedoch erst, wenn Sie das Token bereitstellen.

Wenn die Authentifizierung aktiviert ist, fragt das Dashboard nach dem Token und behält es im
`sessionStorage` des Browsers, wobei es bei jeder API-Anfrage als Bearer-Header angehängt wird.
Jede `/api/v1`-Anfrage wird mit `Authorization: Bearer <token>` autorisiert; `/metrics` und die
statische Shell bleiben offen, die Live-Daten sowie die Steuerungen für Sperre / Reload jedoch
nicht.

<Callout type="warning" title="Setzen Sie ein Token, bevor Sie den Listener freigeben">
Bevor Sie `api.listen` an etwas anderes als `127.0.0.1` binden, konfigurieren Sie ein API-Token.
Unter [Authentifizierung](/docs/authentication) erfahren Sie, wie Sie `api.token_env` setzen und
wie Anfragen verifiziert werden.
</Callout>
