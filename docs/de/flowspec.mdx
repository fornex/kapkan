# FlowSpec-Mitigation

Ein [RTBH](/docs/glossary)-Blackhole (Remotely-Triggered Blackhole — eine BGP-Route, die
allen Datenverkehr zum Opfer verwirft) nimmt das gesamte Opfer vom Netz — es tauscht den
Angriff gegen einen Ausfall ein. BGP [FlowSpec](/docs/glossary) (RFC 8955 für IPv4, RFC 8956
für IPv6) verteilt stattdessen eine Regel, die nur den passenden Angriffsverkehr verwirft,
sodass das Opfer alles Übrige weiter ausliefert.

<Callout type="info" title="Erst der Trockenlauf">
Die generierten Regeln erscheinen in `/api/v1/attacks` (`method`, `flowspec`), in `/api/v1/bans`
und in Benachrichtigungen, **bevor** Sie jemals `dry_run: false` setzen — so können Sie sie zunächst
gegen die FlowSpec-Unterstützung Ihres Upstreams überprüfen. Siehe [Inbetriebnahme](/docs/going-live).
</Callout>

## FlowSpec aktivieren

Setzen Sie die Mitigation-Methode auf `flowspec`. Die Standardmethode ist `blackhole` (RTBH).

```yaml
mitigation: flowspec            # default method for all groups (default: blackhole)
flowspec:
  action: discard               # or rate_limit
  rate_mbps: 100                # only used when action: rate_limit (inert under discard)
hostgroups:
  - name: web
    networks: ["203.0.113.0/26"]
    mitigation: blackhole       # per-group override
```

`mitigation` ist pro [Hostgroup](/docs/hostgroups) überschreibbar, sodass Sie FlowSpec global
betreiben und für eine bestimmte Präfixmenge auf blackhole zurückfallen können (oder umgekehrt).

## Generierte Regeln

Bei einem Angriff leitet Kapkan aus der [Klassifizierung](/docs/detection) des Angriffs und der
Flow-Stichprobe einen **minimalen Regelsatz** (höchstens 8) ab, der das Opfer als Ziel sowie den
Vektor abgleicht:

| Angriff | Generierter FlowSpec-Match |
| --- | --- |
| NTP- / DNS- / CLDAP- / memcached- / SSDP- / chargen-Amplification | `dst=victim, proto=udp, src-port=<reflected port>` |
| SYN-Flood | `dst=victim, proto=tcp, tcp-flags=SYN` |
| Fragment-Flood | `dst=victim, fragment` |
| ICMP- / UDP- / TCP-Flood | `dst=victim, proto=<icmp/udp/tcp>` |
| gemischt / unbekannt | `dst=victim` (plus eine Regel pro dominantem reflektiertem Quellport in der Stichprobe) |

Das Opfer wird stets als `/32` (IPv4) bzw. `/128` (IPv6) abgeglichen. **IPv6-FlowSpec funktioniert
identisch zu IPv4** — es werden dieselben Regeln generiert, wobei das Opfer als `/128` abgeglichen wird.

## Ausgehende Angriffe

Bei einem [ausgehenden Angriff](/docs/detection) — einem kompromittierten Host, der nach außen flutet —
gleicht die Regel den Host als **Quelle** ab (das RFC-8955/8956-Quellpräfix), sodass die ausgehende
Flut tatsächlich verworfen wird. Das unterscheidet sich von einem zielbasierten RTBH-Blackhole, das nur
den Datenverkehr *zum* Host vernichtet.

## Quellverankerung

Standardmäßig gleicht jede generierte Regel das Opfer als **Ziel** ab und verwirft den Angriffsvektor von *allen* Quellen — einschließlich legitimer Clients, die dasselbe Protokoll senden. Mit `flowspec.source_anchored: true` gibt Kapkan, wenn die Angriffsstichprobe eine **konzentrierte** Menge von Angreifer-Quellen zeigt, stattdessen zusammengesetzte Regeln aus, die **sowohl** das Opfer (Ziel) **als auch** jeden dominanten Angreifer (Quelle) festnageln, sodass nur die Angreifer verworfen werden und der legitime Datenverkehr des Opfers verschont bleibt.

```yaml
flowspec:
  action: discard
  source_anchored: true          # drop the attackers, not the protocol from everyone
  min_source_concentration: 0.8  # ...only when the top sources cover >= this share (default 0.8)
```

Sie greift nur, wenn die dominanten Quellen innerhalb des 8-Regeln-Budgets mindestens `min_source_concentration` des stichprobenartig erfassten Angriffs abdecken. Ein **diffuser** Angriff — Reflection oder Spoofing, über viele Quellen verteilt — erreicht diese Schwelle nie und fällt auf die opferverankerte Regel zurück. Ausgewählte Quellen werden ausschließlich als Host-Präfixe (`/32`, `/128`) abgeglichen: Es gibt keine Aufweitung zu größeren Präfixen, sodass eine Regel niemals auf einen unschuldigen Nachbarn eines Angreifers übergreifen kann.

## Einschränkungen

- Der `tcp-flags`-Match für SYN-Floods ist eine Bitmaske, die auch auf SYN-ACK passt, sodass eine
  `discard`-Aktion auch die vom Opfer ausgehend initiierten Verbindungen verwirft — bevorzugen Sie
  `rate_limit` für TCP-Vektoren. Um es zu nutzen, setzen Sie `action: rate_limit` und eine Obergrenze
  in `rate_mbps` (die obigen Beispiele verwenden `discard`):

  ```yaml
  flowspec:
    action: rate_limit            # rate-cap the vector instead of dropping it
    rate_mbps: 100                # ceiling, megabits/s
  ```
- `max_active_bans` begrenzt **Bans**, nicht Regeln: Ein FlowSpec-Ban kann bis zu 8 Regeln tragen,
  sodass N Bans bis zu 8N Regeln in der RIB Ihres Upstreams bedeuten können. Beobachten Sie die
  Metrik `kapkan_mitigate_flowspec_rules` ([Metric](/docs/metrics)) gegen das FlowSpec-Routenlimit Ihrer Router.

## Regelaktion und Lebenszyklus

Jede Regel trägt eine Traffic-Rate-Extended-Community: `discard` (Rate 0) oder eine `rate_limit`-Obergrenze.
FlowSpec-Bans teilen sich denselben TTL-, Unban-Hysterese- und `max_active_bans`-Lebenszyklus wie
Blackhole-Bans — siehe [RTBH-Mitigation](/docs/mitigation).

FlowSpec läuft über **dieselben BGP-Nachbarn** wie RTBH: Es wird als zusätzliche Capability über
dieselben BGP-Sessions ausgehandelt (die FlowSpec-AFI/SAFI), und ein Peer, der sie nicht unterstützt,
wird sie schlicht nicht aktivieren. FlowSpec ist **nicht gültig für Hostgroups mit
`calculation: total`** — es gibt kein einzelnes Opfer-Präfix zum Abgleichen.

Um zu bestätigen, dass Ihr Upstream die Regeln tatsächlich akzeptiert: Prüfen Sie im Trockenlauf, dass das
Feld `flowspec` in `/api/v1/attacks` befüllt ist. Nach der Inbetriebnahme sollte die Metrik
`kapkan_mitigate_flowspec_rules` ([Metric](/docs/metrics)) mit `mode="real"` ungleich null sein. Lehnt
ein Peer FlowSpec ab, fällt Kapkan den Ban auf blackhole zurück und macht dies in `/api/v1/bans` (die Felder
`fell_back_from` und `fell_back_reason`) sowie in Benachrichtigungen sichtbar — sodass ein stillschweigend
nicht unterstützender Upstream sichtbar bleibt.
