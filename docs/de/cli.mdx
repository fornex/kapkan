# CLI-Referenz

Kapkan ist ein einzelnes Binary. Alles, was es tut, wird beim Start über Kommandozeilen-Flags
ausgewählt — es gibt keine Subcommands über die Hilfsmodi unten hinaus. Alle Logs gehen
nach **stderr**; die REST-API und die Prometheus-Metriken werden über HTTP auf `api.listen`
(Standard `127.0.0.1:8080`) bereitgestellt, während die Hilfs-Flags (`-version`, `-dump-schema`)
ihre Ausgabe nach **stdout** schreiben.

```sh
kapkan [flags]
```

## Flags

| Flag | Standard | Beschreibung |
| --- | --- | --- |
| `-config <path>` | `configs/dev.yaml` | Pfad zur YAML-[Konfigurationsdatei](/docs/configuration). |
| `-log-format <fmt>` | `json` | Log-Ausgabeformat: `json` (strukturiert, für Log-Collectors) oder `text` (menschenlesbar, für lokale Läufe). Jeder andere Wert als `text` wird als `json` behandelt. |
| `-log-level <lvl>` | `info` | Minimales Log-Level: `debug`, `info`, `warn`, `error`. Ein unbekannter Wert fällt auf `info` zurück. |
| `-check-config <path>` | — | Validiert die Konfigurationsdatei unter `<path>` und beendet sich — startet den Daemon **nicht**. Siehe [unten](#eine-konfiguration-validieren). |
| `-dump-schema` | `false` | Gibt das JSON-Schema der Konfiguration auf stdout aus und beendet sich. Siehe [unten](#das-schema-ausgeben). |
| `-version` | `false` | Gibt die Build-Version aus und beendet sich. Dieselbe Version wird zur Laufzeit über `/api/v1/status` und die `kapkan_build_info`-[Metrik](/docs/metrics) bereitgestellt. |
| `-check-update` | `false` | Führt die optionale [Update-Prüfung](/docs/configuration) einmalig aus und beendet sich — `0` aktuell, `10` ein Update ist verfügbar, `1` bei einem Fehler. Fragt die GitHub-Releases-API ab; sendet nur die Anfrage selbst, niemals Knotenidentität oder Konfiguration. |
| `-pid-file <path>` | `/run/kapkan/kapkan.pid` | Wohin der Daemon beim Start seine Prozess-ID schreibt (bei sauberem Herunterfahren entfernt); wird von `-s` gelesen, um den laufenden Daemon zu finden. Ein Schreibfehler ist nicht fatal — der Daemon läuft trotzdem, nur `-s` ist nicht verfügbar. |
| `-s <signal>` | — | Signalisiert einen laufenden Daemon und beendet sich, indem `-pid-file` gelesen wird: `reload`, `stop` oder `quit`. Startet **keinen** Daemon. Siehe [unten](#einen-laufenden-daemon-steuern). |

<Callout type="info" title="json oder text?">
Verwenden Sie `-log-format text` für die lokale Entwicklung — es ist das Format, das der
[Schnellstart](/docs/quickstart) nutzt. Behalten Sie im
Produktivbetrieb den Standard `json` bei, damit ein Log-Collector die strukturierten Felder
parsen kann; die [systemd-Unit](/docs/deployment) wird mit `-log-format json -log-level info`
ausgeliefert.
</Callout>

## Den Daemon betreiben

Ohne Hilfs-Flag öffnet Kapkan seine Flow-Listener, die REST-API und (sofern nicht im Dry-Run) den
BGP-Speaker und läuft dann, bis es `SIGINT` oder `SIGTERM` empfängt, woraufhin es sich sauber
herunterfährt. Es endet nur bei einem fatalen Start- oder Laufzeitfehler mit einem Wert ungleich
null.

```sh
# Local, human-readable logs against the bundled dev config:
kapkan -config configs/dev.yaml -log-format text

# Production-style: structured logs at info level:
kapkan -config /etc/kapkan/config.yaml -log-format json -log-level info
```

Die Konfigurationsdatei ist zur Laufzeit ohne Neustart neu ladbar — führen Sie
[`kapkan -s reload`](#einen-laufenden-daemon-steuern) aus, senden Sie `SIGHUP` direkt
(`systemctl reload kapkan`) oder `POST /api/v1/config/reload`. Siehe die
[Konfigurationsreferenz](/docs/configuration) dazu, was neu geladen wird und was beim Start
fixiert ist.

## Einen laufenden Daemon steuern

```sh
kapkan -s reload    # re-read the config file (SIGHUP)
kapkan -s stop      # graceful shutdown (SIGTERM)
kapkan -s quit      # alias of stop
```

`-s` liest die Prozess-ID aus `-pid-file` (Standard `/run/kapkan/kapkan.pid`, beim Start vom
Daemon geschrieben) und stellt das passende Signal an diesen Prozess zu — dieselbe nginx-artige
lokale Steuerung wie `nginx -s reload`. Es benötigt keine Netzwerkverbindung und kein API-Token,
funktioniert also auch, wenn die REST-API nicht erreichbar ist; es benötigt jedoch die Berechtigung,
den Daemon zu signalisieren, führen Sie es daher als `root` oder als `kapkan`-Benutzer aus.

| `-s`-Wert | Signal | Wirkung |
| --- | --- | --- |
| `reload` | `SIGHUP` | Lädt die Konfigurationsdatei im laufenden Betrieb neu. Wie `systemctl reload kapkan` oder `POST /api/v1/config/reload`. |
| `stop` / `quit` | `SIGTERM` | Sauberes Herunterfahren. Wie `systemctl stop kapkan`. Die beiden sind Aliase — Kapkans Herunterfahren ist immer graceful (es bittet BGP-Peers, Mitigation-Routen via [Graceful Restart](/docs/mitigation) beizubehalten). |

Wenn die pid-Datei fehlt oder veraltet ist (kein solcher Prozess) oder Ihnen die Berechtigung zum
Signalisieren fehlt, gibt `-s` den Grund auf stderr aus und beendet sich mit `1`.

<Callout type="info" title="Wo die pid-Datei liegt">
Die mitgelieferte [systemd-Unit](/docs/deployment) setzt `RuntimeDirectory=kapkan`, sodass
`/run/kapkan` existiert und für den `kapkan`-Benutzer beschreibbar ist; der Daemon schreibt dort
`/run/kapkan/kapkan.pid` und entfernt es bei einem sauberen Stopp. In einem lokalen Dev-Lauf
existiert dieses Verzeichnis möglicherweise nicht — der Daemon protokolliert eine Warnung und fährt
fort, und `-s` meldet „no pid file". Übergeben Sie `-pid-file /tmp/kapkan.pid` sowohl an den Daemon
als auch an `-s`, wenn Sie die Abkürzung lokal nutzen möchten.
</Callout>

## Hilfs-Flags

Diese Flags beenden sich, bevor der Daemon startet — sie öffnen niemals einen Listener und
senden keine Route-Ankündigung, sodass sie sich gefahrlos auf einem Produktivhost ausführen
lassen. (`-check-update` stellt eine einzelne ausgehende Anfrage an die Releases-API; die
übrigen sind vollständig offline.)

### Eine Konfiguration validieren

```sh
kapkan -check-config /etc/kapkan/config.yaml
```

Führt exakt dasselbe Parsen und dieselbe Validierung der Engine aus — einschließlich der
feldübergreifenden Regeln, die ein statisches Schema nicht ausdrücken kann — gibt dann die
aufgelöste Konfiguration aus (Modus, geschützte Netze, die konfigurierten Flow-Listener, die
Gruppenanzahl und die effektive Mitigation pro [Hostgroup](/docs/hostgroups)) und beendet sich:

| Exit-Code | Bedeutung |
| --- | --- |
| `0` | Die Konfiguration ist gültig. |
| `1` | Die Konfiguration ist ungültig; der genaue Fehler wird auf stderr ausgegeben. |

Die saubere 0/1-Aufteilung fügt sich direkt in CI oder eine Pre-Deploy-Prüfung ein. Siehe
[Konfiguration validieren](/docs/configuration#konfiguration-validieren).

### Das Schema ausgeben

```sh
kapkan -dump-schema
```

Gibt das JSON-Schema der Konfiguration auf stdout aus und endet mit `0` (oder `1` bei einem
internen Fehler). Der [Konfigurations-Builder](/config) nutzt dieses Schema, um Felder direkt in
Ihrem Browser zu prüfen.

### Die Version ausgeben

```sh
kapkan -version
```

Gibt die Build-Version (Release-Tag plus kurze VCS-Revision, sofern bekannt) auf stdout aus und
beendet sich. Dieselbe Zeichenkette wird zur Laufzeit auf `/api/v1/status` bereitgestellt und von
der `kapkan_build_info`-[Metrik](/docs/metrics) getragen — so können Sie die Version eines Binarys
lokal ohne jeglichen Egress bestätigen.

### Nach Updates suchen

```sh
kapkan -check-update
```

Führt die [`update_check`](/docs/configuration) einmalig auf Anfrage aus und beendet sich: `0`,
wenn aktuell, `10`, wenn ein neueres Release verfügbar ist, `1` bei einem Fehler. Es fragt die
GitHub-Releases-API ab und sendet nur die Anfrage selbst (Ihre IP und einen generischen
User-Agent) — niemals Knotenidentität, Konfiguration oder Angriffsdaten. Die periodische
In-Process-Prüfung ist **standardmäßig deaktiviert**; dieses Flag funktioniert unabhängig von
dieser Einstellung.
