# Eskalationsstufen

Eine einzelne `mitigation`-Methode löst in dem Moment, in dem ein Angriff erkannt wird, immer dieselbe
Reaktion aus. Eine Eskalationsstufenleiter hingegen **verstärkt die Reaktion, je länger ein Angriff
andauert** — deklarativ, ohne Callback-Skripte.

Die meisten Deployments kommen mit einer einzelnen `mitigation:`-Methode aus. Greifen Sie nur dann zu
einer Leiter, wenn Sie zunächst eine sanftere Reaktion wünschen (z. B. erst eine Warnung, dann gezieltes
Filtern), bevor Sie zu einem vollständigen [Blackhole](/docs/glossary) greifen.

```yaml
escalation:                         # supersedes `mitigation` when present
  - { after_seconds: 0,   action: none }       # alert only at first
  - { after_seconds: 30,  action: flowspec }   # still under attack after 30s → surgical drop
  - { after_seconds: 90,  action: divert }     # still under attack after 90s → scrub
  - { after_seconds: 300, action: blackhole }  # still under attack after 300s → blackhole
flowspec:
  action: discard
scrubbing:
  next_hop: "192.0.2.100"   # scrubbing center; see Traffic diversion
  community: "65000:200"
```

Ein [`bgp`-Block](/docs/mitigation#bgp-konfiguration) ist erforderlich, damit überhaupt eine Stufe
ankündigen kann; die `flowspec`- und `scrubbing`-Blöcke oben werden nur für die `flowspec`- und
`divert`-Stufen benötigt.

## So greifen die Stufen

Das `after_seconds` jeder Stufe wird ab dem Beginn des Angriffs gemessen. Eine Stufe greift, sobald so
viel Zeit verstrichen ist **und der Ban noch aktiv ist**. (Ein Ban wird erst zurückgenommen, nachdem der
Datenverkehr für die konfigurierte Abkühlphase, `ban.unban_hysteresis_seconds`, unter dem Schwellenwert
geblieben ist — siehe [Konfiguration](/docs/configuration).) Die Stufenuhr läuft währenddessen durchgehend
ab dem Beginn des Angriffs weiter.

- Die erste Stufe muss bei `0` Sekunden liegen.
- Eine Leiter darf höchstens **5 Stufen** haben — das ist eine harte Grenze; der Daemon verweigert den
  Start bei einer längeren Leiter.
- `action` ist `none` (nur Warnung), `flowspec`, `divert` ([Scrubbing](/docs/scrubbing)) oder
  `blackhole`.
- Eine Leiter darf die Reaktion nur **halten oder verstärken**: `none` < `flowspec` < `divert` <
  `blackhole`. Eine Deeskalation zwischen den Stufen ist ein Konfigurationsfehler.

## Make-before-break

Der Aufstieg zu einer Stufe erfolgt nach dem Make-before-break-Prinzip: Die neue Stufe wird zuerst
angekündigt, und die vorherige wird erst zurückgezogen, nachdem dies erfolgreich war — sodass das Opfer
während des Umschaltens nie kurzzeitig ungeschützt ist. Schlägt die Ankündigung fehl, hält der Ban die
funktionierende Stufe und wiederholt den Versuch beim nächsten Tick. Werden mehrere Stufen gleichzeitig
fällig (ein lang andauernder Angriff oder der Daemon, der nach einer Pause aufholt), springt der Ban
direkt zur höchsten fälligen Stufe und kündigt die übersprungenen Stufen nie an.

Eine `flowspec`- oder `divert`-Stufe, deren Ankündigung der Upstream-Peer ablehnt, fällt auf Blackhole
zurück (es sei denn, Sie setzen `ban.fallback: none` in der [Konfiguration](/docs/configuration)); die API
und Benachrichtigungen weisen dies über das Feld `fell_back_from` aus. Blackhole- und reine
Warnungs-Stufen haben kein Fallback.

Die einzige Ausnahme vom Ablauf „erst ankündigen, dann zurückziehen“ ist `divert → blackhole`: Beide
kündigen dieselbe Host-Route an (das `/32` oder `/128` des Opfers), sodass die Blackhole-Ankündigung die
Divert-Route atomar ersetzt — kein separates Zurückziehen, keine Lücke. Wechsel zwischen Adressfamilien
(FlowSpec-Regeln vs. eine Host-Route) folgen dem oben beschriebenen Make-before-break-Fall.

## Lebenszyklus und Kompatibilität

Die Leiter ist **pro Hostgroup überschreibbar** und teilt sich den restlichen Ban-Lebenszyklus: das
automatische TTL-Zurückziehen, das `max_active_bans`-Limit, die Blast-Radius-Limits
(`max_banned_fraction`, `max_bans_per_window`), die absolute Whitelist und den Dry-run (der die Leiter
voranschreiten lässt und jede Stufe protokolliert, aber nie ankündigt).

Ist kein `escalation`-Block gesetzt, verhält sich die einzelne `mitigation`-Methode exakt wie eine
einstufige Leiter bei `0` Sekunden — vollständige Abwärtskompatibilität. Jeder Ban in `/api/v1/bans`
gibt die vollständige konfigurierte Leiter (`escalation`), den Index der aktuellen Stufe
(`escalation_step`) und die Methode der aktiven Stufe (`method`) preis.
