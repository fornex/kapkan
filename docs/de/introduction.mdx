# Einführung

**Kapkan ist eine kostenlose, quelloffene DDoS-Erkennung (Distributed Denial of Service) und [RTBH](/docs/glossary)-Mitigation für ISPs und Hosting-Anbieter.**

Es ist ein einzelnes Go-Binary, das Flow-Telemetrie — NetFlow v5/v9, IPFIX und sFlow v5 —
von Ihren Routern aufnimmt, volumetrische Angriffe auf die von Ihnen geschützten [Präfixe](/docs/glossary) (IP-Bereiche
in CIDR-Form, z. B. `203.0.113.0/24`) in Sekunden erkennt und eine automatisierte
[BGP](/docs/glossary)- (Border Gateway Protocol — wie Router Routen austauschen)
[RTBH](/docs/glossary)-Mitigation (Remotely-Triggered Blackhole) auslöst. Es wird mit einer
REST-API, einem integrierten Web-Dashboard, Prometheus-Metriken und Benachrichtigungen über
Telegram / Slack / E-Mail / Webhook / Exec-Hook ausgeliefert.

Kapkan ist ein kostenloser Ersatz für die Funktionen, für die kommerzielle Flow-DDoS-Produkte Geld verlangen.

<Callout type="info" title="Neu bei BGP, RTBH oder NetFlow?">
Kein Problem — der [Schnellstart](/docs/quickstart) führt Sie Schritt für Schritt durch, das
[Glossar](/docs/glossary) erklärt jeden Begriff, und Kapkan rührt keine einzige Route an, bevor
Sie es nicht erlauben. Kapkan behandelt *volumetrische* Angriffe (Floods, die versuchen, Ihre
Leitung oder Ihren Host mit roher Paket- und Bitmenge zu sättigen), keine Angriffe auf
Anwendungsebene.
</Callout>

<Callout type="warning" title="Standardmäßig im Dry-run">
Bis Sie explizit `dry_run: false` setzen, wird jeder potenzielle Blackhole protokolliert und
über die API bereitgestellt — aber **niemals** an Ihre Router angekündigt. Sie können Kapkan
gegen Produktivtelemetrie laufen lassen und die Erkennung validieren, bevor sie auch nur eine
einzige Route berühren kann. Siehe das [Sicherheitsmodell](/docs/safety).
</Callout>

## Was es leistet

- **Aufnehmen** — sFlow v5, NetFlow v5/v9 und IPFIX über UDP, im Bibliotheksmodus (ohne Sidecar),
  über [goflow2](https://github.com/netsampler/goflow2).
- **Erkennen** — volumetrische Angriffe pro Ziel anhand sampling-korrigierter pps- / Mbps- /
  flows/sec-Schwellenwerte über ein gleitendes Fenster, zuzüglich optionaler Limits pro Protokoll,
  [gelernter Baselines](/docs/baselines) und Erkennung ausgehender Angriffe.
- **Klassifizieren** — jeder Angriff wird anhand seines Flow-Samples gekennzeichnet: Amplification
  (NTP / DNS / CLDAP / memcached / SSDP / chargen), SYN- / UDP- / TCP- / ICMP- / Fragment-Floods.
- **Mitigieren** — Ankündigung von `/32`- und `/128`-Blackhole-Routen über einen eingebetteten
  [GoBGP](https://github.com/osrg/gobgp)-Speaker, gezieltes Verwerfen ausschließlich des
  Angriffsvektors mit chirurgisch präzisen [BGP-FlowSpec](/docs/flowspec)-Regeln (RFC 8955/8956)
  oder Umleitung des Opfers in ein [Scrubbing-Center](/docs/scrubbing), statt seinen Verkehr zu verwerfen —
  optional über die Zeit hochgestuft mit [Eskalationsstufen](/docs/escalation). Kapkan verfolgt jede
  Mitigation-Entscheidung als **Ban**; bei der Standardmethode *ist* ein Ban eine Blackhole-Route (RTBH).
- **Beobachten** — eine REST-API, ein eingebettetes Web-Dashboard, Prometheus `/metrics` und
  Benachrichtigungen über Telegram, Slack, E-Mail, Webhook und einen Exec-Hook.
- **Persistieren (optional)** — Angriffs- und Datenverkehrshistorie in
  [ClickHouse](/docs/storage) für Forensik und Reporting aufbewahren.

## Für wen es gedacht ist

Netzwerkbetreiber — ISPs, Hosting-Anbieter und alle, die BGP betreiben — die eine automatisierte
Mitigation volumetrischer Angriffe ohne kommerzielle Lizenz benötigen (einschließlich Nutzern von
FastNetMon Community, die eine quelloffene Alternative suchen).

## Warum Kapkan

| | Kapkan |
| --- | --- |
| **Lizenz** | Apache 2.0, vollständig quelloffen — einschließlich der Erkennungslogik |
| **Footprint** | Ein statisches Go-Binary; kein MongoDB, kein separater Webdienst |
| **Web-UI** | Integriertes Dashboard inklusive, kein kostenpflichtiges Add-on |
| **Konfiguration** | Deklaratives YAML, das Sie in Git ablegen und diffen können |
| **Sicherheit** | Standardmäßig Dry-run; TTLs, Hysterese und Ban-Caps im Code durchgesetzt |

## Was es (noch) nicht leistet

Diese Punkte stehen auf der Roadmap,
sind aber **nicht** im aktuellen Release enthalten:

- Integrierte Hochverfügbarkeits-Deduplizierung über Instanzen hinweg
- Port-Mirror-Erfassung (SPAN / AF_PACKET / XDP) als Aufnahmequelle
- Benachrichtigungs-Routing pro Mandant (heute sind Benachrichtigungen global; das Scoping pro Mandant gilt nur für die [API](/docs/multi-tenancy))

## Nächste Schritte

**Neu hier? Beginnen Sie mit dem [Schnellstart](/docs/quickstart)** — laden Sie Kapkan herunter, richten
Sie einen Router darauf aus und sehen Sie Ihre erste Erkennung. (Noch kein Router? Sie können
synthetischen Angriffsverkehr auch direkt von der Quelle erzeugen — der Schnellstart zeigt, wie.)

Dann:

- [Funktionsweise](/docs/how-it-works) — die Pipeline Aufnahme → Engine → Mitigation.
- [Konfigurationsreferenz](/docs/configuration) — jeder Schlüssel in der YAML-Datei.
- [In Betrieb gehen](/docs/going-live) — wie Sie Dry-run sicher abschalten.
