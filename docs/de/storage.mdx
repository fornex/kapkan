# Speicherung (ClickHouse)

Verweisen Sie Kapkan auf einen ClickHouse-Server, um den Angriffs- und Verkehrsverlauf
festzuhalten — die Antwort auf die Frage "Was hat uns letzten Dienstag getroffen?". Ohne ihn
läuft Kapkan vollständig prozessintern auf Live-Daten.

Dafür benötigen Sie einen laufenden ClickHouse-Server (jede aktuelle Version). Kapkan
kommuniziert mit dessen HTTP-Schnittstelle, die standardmäßig auf Port `8123` lauscht — das ist
der Port in der Beispiel-URL weiter unten. Die Installation von ClickHouse wird hier nicht
behandelt; siehe [clickhouse.com](https://clickhouse.com).

<Callout type="info" title="Best-Effort, blockiert nie die Erkennung">
Die Persistenz arbeitet über eine begrenzte Warteschlange mit nicht blockierendem Senden. Ein
langsames oder ausgefallenes ClickHouse verwirft Zeilen (gezählt in `kapkan_storage_rows_total` mit
`result="dropped"`), anstatt den Hot Path der Engine zu blockieren.
</Callout>

## Aktivierung

```yaml
storage:
  clickhouse:
    url: "http://127.0.0.1:8123"   # empty/absent disables persistence
    database: "kapkan"             # created if absent
    username_env: "KAPKAN_CH_USER" # optional; credentials come from the env
    password_env: "KAPKAN_CH_PASS"
    ttl_days: 7                    # rows auto-expire (per-row TTL)
    flush_interval_seconds: 5
    batch_size: 1000
    queue_size: 100000
    traffic_interval_seconds: 10
```

Die Anmeldedaten werden aus den benannten Umgebungsvariablen (`username_env` / `password_env`)
gelesen, niemals aus der Konfigurationsdatei. Lassen Sie beide weg, wenn Ihr ClickHouse
nicht authentifizierte lokale Verbindungen akzeptiert (der voreingestellte Benutzer `default`
tut das oft) — setzen Sie sie nur, wenn Ihr Server eine Authentifizierung verlangt.

Das Ändern einer beliebigen Speicherungseinstellung erfordert einen vollständigen Neustart — ein
Konfigurations-Neuladen, das den `storage`-Block berührt, wird abgelehnt
(`reload: storage settings cannot change at runtime (restart required)`).

## Überprüfung

Bestätigen Sie nach dem Aktivieren der Speicherung, dass tatsächlich Zeilen ankommen — die
Persistenz arbeitet nach dem Best-Effort-Prinzip und schlägt stillschweigend fehl. Zwei Prüfungen:

- Beobachten Sie, wie die Metrik `kapkan_storage_rows_total{result="written"}` steigt (im
  Gegensatz zu `result="dropped"` oder `result="error"`). Siehe [Metriken](/docs/metrics).
- Fragen Sie ClickHouse direkt ab:
  ```bash
  curl 'http://127.0.0.1:8123/?query=SELECT%20count()%20FROM%20kapkan.attack_events'
  ```

Bei falschen Anmeldedaten oder einem Berechtigungsproblem protokolliert die Engine beim Start
`clickhouse schema init failed` und `clickhouse insert failed`, wenn ein Batch abgelehnt wird —
prüfen Sie zuerst die Logs, wenn die Zählwerte bei null bleiben.

## Funktionsweise

Kapkan kommuniziert über die **HTTP-Schnittstelle** von ClickHouse mit der Go-Standardbibliothek —
ohne Treiberabhängigkeit. Die einzige externe Abhängigkeit ist der ClickHouse-Server selbst.

Beim Start erstellt Kapkan idempotent drei MergeTree-Tabellen:

| Tabelle | Inhalt |
| --- | --- |
| `attack_events` | Jeder Angriffsbeginn und jedes Angriffsende: Typ, Richtung, Raten, die Top-Quellen des Samples, der Sperrstatus und der Erkennungs-`reason`. |
| `traffic` | Periodische Momentaufnahmen von Rate und Baseline pro Host (alle `traffic_interval_seconds`). |
| `audit_events` | Dem Betreiber zugeordnete Änderungen: wer jede Sperre/Entsperrung/Konfigurations-Neuladung ausgelöst hat, wann und mit welchem Ergebnis. Siehe [Audit-Log](/docs/audit). |

Jede trägt eine `ttl_days`-TTL pro Zeile, sodass die Aufbewahrung ohne Eingriff des Betreibers
begrenzt bleibt.

Die Schreib-Anmeldedaten benötigen beim ersten Lauf `CREATE`-Rechte (Datenbank + Tabellen) und
`INSERT`-Rechte; danach genügt `INSERT`. Fehler bei der Schema-Erstellung werden beim Start
protokolliert, sind aber nicht fatal.

Die `reason`-Spalte der `attack_events`-Tabelle enthält das kompakte JSON des Erkennungs-
[reason](/docs/detection) — warum der Angriff ausgelöst wurde (Herkunft des Schwellenwerts,
Warm-up, Protokollanteile) — und ist in der Zeile zum Angriffsende leer. Bestehende Installationen
erhalten die Spalte automatisch: Die Schema-Initialisierung führt ein `ADD COLUMN IF NOT EXISTS`
aus, das stillschweigend übersprungen wird, wenn die Schreib-Anmeldedaten keine `ALTER`-Rechte
haben (Neuinstallationen verfügen bereits darüber).

## Backpressure

Zeilen gelangen in eine begrenzte Warteschlange (`queue_size`) mit nicht blockierendem Senden und
werden in Batches (`batch_size` / `flush_interval_seconds`) geleert. Wenn ClickHouse langsam ist
oder ausfällt, werden Zeilen verworfen — gezählt als `kapkan_storage_rows_total` mit
`result="dropped"` — und die Engine läuft weiter. Siehe [Metriken](/docs/metrics).

## Was gespeichert wird und was nicht

Die `traffic`-Tabelle speichert derzeit nur Momentaufnahmen pro Host. Wenn der optionale `geoip`-
Block konfiguriert ist (eine MaxMind-`.mmdb`-Datenbank), löst Kapkan ASNs pro Quelle auf und
speichert die Top-ASNs jedes Angriffs in `attack_events.top_asns` (durch Pipe verbunden `"AS<n> <org>"`,
leer wenn geoip deaktiviert ist). Zeitreihen pro ASN und Summen pro Hostgroup werden noch nicht
als Momentaufnahme erfasst — beides sind Kandidaten für eine Folgeversion.
