# Déploiement en production

Une unité systemd renforcée et un exemple de configuration de production sont fournis dans le
répertoire [`engine/deploy/`](https://github.com/fornex/kapkan/tree/main/engine/deploy) du dépôt : `engine/deploy/kapkan.service`
et `engine/deploy/config.example.yaml`. Cette page explique comment installer le binaire en tant que
service système, où réside chaque fichier, comment garder les secrets hors du YAML et comment
recharger la configuration sans redémarrage.

<Callout type="info" title="Aucune compilation requise">
Le chemin le plus rapide est un paquet précompilé : `apt install ./kapkan_*.deb` (ou le `.rpm`)
fait tout ce que décrit cette section à votre place — voir [Installer le paquet](#installer-le-paquet)
ci-dessous. Les étapes manuelles qui suivent concernent le [tarball de release](/docs/quickstart#1-installer-kapkan).
Dans les deux cas, il n'y a rien à compiler ; [Compilation depuis les sources](#compilation-depuis-les-sources)
n'est utile que si vous le souhaitez.
</Callout>

## Installation

### Installer le paquet

Sur Debian/Ubuntu (`.deb`) ou RHEL/Fedora (`.rpm`), le paquet fait tout ce que font les étapes
manuelles ci-dessous : il installe le binaire dans `/usr/local/bin/kapkan`, crée l'utilisateur
non privilégié `kapkan`, met en place `/etc/kapkan` avec un `config.yaml` en dry-run initialisé à
partir de l'exemple, crée le répertoire d'état accessible en écriture et installe l'unité systemd
renforcée (dans `/usr/lib/systemd/system`).

```sh
VER=v1.0.0
curl -fLO "https://github.com/fornex/kapkan/releases/download/$VER/kapkan_${VER#v}_linux_amd64.deb"
sudo apt install "./kapkan_${VER#v}_linux_amd64.deb"   # RHEL/Fedora: dnf install ./kapkan_<ver>_linux_amd64.rpm
```

Le service est installé mais **laissé arrêté** — un démon de mitigation ne devrait pas démarrer
avant que vous n'ayez examiné sa configuration. Le `config.yaml` fourni a `dry_run: true` (Kapkan
calcule les routes blackhole mais ne les annonce pas à votre routeur). Examinez-le, (recommandé)
[définissez un jeton d'API](#secrets) dans `/etc/kapkan/kapkan.env`, puis démarrez-le au boot et
maintenant :

```sh
sudoedit /etc/kapkan/config.yaml          # adapt networks, BGP and thresholds
sudo systemctl enable --now kapkan
```

Mettez à niveau avec un paquet plus récent (`apt install ./kapkan_<newer>.deb`) ou avec
[`update.sh`](/docs/upgrading) — les deux conservent votre `/etc/kapkan/config.yaml` modifié.
`apt purge kapkan` en supprime toute trace, y compris la configuration et l'état.

### Installation manuelle (tarball)

Vous préférez placer les fichiers vous-même, ou êtes sur une distribution sans `apt`/`dnf` ?
Après avoir décompressé le [tarball de release](/docs/quickstart#1-installer-kapkan) (de sorte que
le répertoire `deploy/` se trouve à côté du binaire `kapkan`), exécutez l'équivalent en tant que
root :

```sh
sudo install -m 0755 kapkan /usr/local/bin/kapkan
sudo groupadd --system kapkan
sudo useradd --system --gid kapkan --no-create-home --home-dir /var/lib/kapkan --shell /usr/sbin/nologin kapkan
sudo install -d -o kapkan -g kapkan /etc/kapkan
sudo install -m 0640 -o kapkan -g kapkan deploy/config.example.yaml /etc/kapkan/config.yaml
# generate a real random API token, never deploy the literal placeholder
echo "KAPKAN_API_TOKEN=$(openssl rand -hex 32)" | sudo install -m 0600 /dev/stdin /etc/kapkan/kapkan.env
sudo install -m 0644 deploy/kapkan.service /etc/systemd/system/kapkan.service
sudo systemctl daemon-reload && sudo systemctl enable --now kapkan
```

Le fichier d'environnement ci-dessus ne renseigne que `KAPKAN_API_TOKEN`, car un jeton d'API
est recommandé avant d'exposer l'écouteur (voir [Authentification](/docs/authentication)).
Renseigner le fichier d'environnement n'active pas l'authentification à lui seul : la configuration
fournie lie `127.0.0.1` avec `api.token_env` mis en commentaire, de sorte que le jeton reste inerte.
Décommentez `api.token_env` (ou ajoutez une liste `api.tokens`) pour activer l'authentification.
Tous les canaux de notification sont optionnels — n'ajoutez `KAPKAN_TG_TOKEN`, les identifiants SMTP
ou ClickHouse que si vous utilisez réellement ces fonctionnalités. La liste complète des clés
`*_env` se trouve ci-dessous dans [Secrets](#secrets).

L'unité s'exécute sous l'utilisateur `kapkan` avec `ExecStart=/usr/local/bin/kapkan -config
/etc/kapkan/config.yaml -log-format json -log-level info -pid-file /run/kapkan/kapkan.pid`,
redémarre en cas d'échec et est
fortement cloisonnée : `NoNewPrivileges`, `ProtectSystem=strict`, `PrivateTmp`,
`ReadOnlyPaths=/etc/kapkan`, `RestrictAddressFamilies=AF_INET AF_INET6`,
`MemoryDenyWriteExecute` et bien plus encore. Elle déclare également `StateDirectory=kapkan`
(mode `0700`), offrant au démon un `/var/lib/kapkan` accessible en écriture pour
[`ban.state_file`](/docs/mitigation), et `RuntimeDirectory=kapkan`, lui offrant un `/run/kapkan`
accessible en écriture pour le pid file, tandis que `ProtectSystem=strict` maintient le reste
du système de fichiers en lecture seule. Un garde-fou contre les boucles de crash
(`StartLimitIntervalSec=60`, `StartLimitBurst=4`) fait passer l'unité à l'état `failed` si le
binaire échoue à démarrer quatre fois en moins d'une minute, de sorte qu'une mauvaise mise à
niveau est observable plutôt que de s'emballer — le [script de mise à niveau](/docs/upgrading)
s'appuie sur ce mécanisme conjointement avec la sonde de disponibilité `/healthz`.

<Callout type="info" title="Lier BGP au port 179">
Kapkan n'a besoin d'aucun privilège particulier en dry-run ou lorsqu'il établit un peering depuis
un port local élevé. Si vous liez le locuteur [BGP](/docs/glossary) au port bien connu 179, accordez la capacité en
ajoutant `AmbientCapabilities=CAP_NET_BIND_SERVICE` à la section `[Service]` de l'unité.
</Callout>

Une fois le service en cours d'exécution, vérifiez son état et suivez les journaux :

```sh
sudo systemctl status kapkan
sudo journalctl -u kapkan -f
```

## Organisation des fichiers

| Chemin | Mode | Rôle |
| --- | --- | --- |
| `/usr/local/bin/kapkan` | `0755` | Le binaire unique et statique du démon. |
| `/etc/kapkan/config.yaml` | `0640` | La configuration YAML, appartenant à `kapkan:kapkan`. |
| `/etc/kapkan/kapkan.env` | `0600` | Fichier d'environnement des secrets (jetons, identifiants SMTP). |
| `/etc/systemd/system/kapkan.service` | `0644` | L'unité systemd renforcée. Le `.deb`/`.rpm` l'installe plutôt dans `/usr/lib/systemd/system/kapkan.service` (emplacement fournisseur). |
| `/var/lib/kapkan/` | `0700` | Répertoire d'état accessible en écriture (`StateDirectory=kapkan`) ; contient `ban.state_file` lorsque la persistance des bans est activée. |
| `/run/kapkan/` | `0755` | Répertoire d'exécution (`RuntimeDirectory=kapkan`), créé au démarrage et supprimé à l'arrêt ; contient `kapkan.pid` pour que [`kapkan -s reload`](/docs/cli#contrôler-un-démon-en-cours-dexécution) puisse trouver le démon. |

L'ensemble du répertoire `/etc/kapkan` est monté en lecture seule pour le processus via
`ReadOnlyPaths=/etc/kapkan`, de sorte que le démon peut lire sa configuration et ses secrets mais
ne peut pas les modifier.

## Secrets

Aucun identifiant ne doit figurer dans `config.yaml`. Le YAML référence chaque secret par le nom
d'une **variable d'environnement**, et l'unité charge ces variables depuis le fichier d'environnement
via `EnvironmentFile=-/etc/kapkan/kapkan.env`. Conservez ce fichier en mode `0600`.

Les clés de configuration qui lisent depuis l'environnement sont les clés `*_env` :

- `notify.telegram.token_env` — par ex. `KAPKAN_TG_TOKEN`, le jeton du bot Telegram.
- `notify.email.username_env` / `notify.email.password_env` — identifiants SMTP.
- `api.token_env` — le jeton bearer de l'API (voir [Authentification](/docs/authentication)).
- `storage.clickhouse.username_env` / `password_env` — identifiants ClickHouse, si vous
  activez le magasin d'historique optionnel.

Par exemple, le YAML nomme la variable et le fichier d'environnement en fournit la valeur :

```yaml
notify:
  telegram:
    token_env: "KAPKAN_TG_TOKEN"   # name of the env var, never the token itself
    chat_id: "-1001234567890"
```

```text
# /etc/kapkan/kapkan.env  (mode 0600)
KAPKAN_TG_TOKEN=123456:abcdef
KAPKAN_API_TOKEN=...   # a real random token, e.g. from: openssl rand -hex 32
```

<Callout type="success" title="Pourquoi c'est important">
Comme les secrets ne résident que dans l'environnement du processus, votre `config.yaml` peut être
conservé et comparé (diff) en toute sécurité dans git. Le cloisonnement systemd maintient
`/etc/kapkan` en lecture seule et exécute le démon sous un utilisateur non privilégié, de sorte que
le fichier d'environnement n'est jamais accessible en écriture par le service.
</Callout>

## Rechargement de la configuration

Kapkan recharge sa configuration à chaud sur `SIGHUP` — sans redémarrage, sans perte de l'état des
flow. L'unité systemd met cela en place avec `ExecReload=/bin/kill -HUP $MAINPID`, de sorte que vous
rechargez avec :

```sh
sudo systemctl reload kapkan
```

En dehors du gestionnaire de services de la machine, le démon répond aussi au drapeau de
contrôle local de style nginx, qui lit le pid file que l'unité écrit dans
`/run/kapkan/kapkan.pid` :

```sh
sudo kapkan -s reload    # re-read the config (SIGHUP)
sudo kapkan -s stop      # graceful shutdown (SIGTERM)
```

Consultez [Contrôler un démon en cours d'exécution](/docs/cli#contrôler-un-démon-en-cours-dexécution) pour le comportement complet et
les codes de sortie. De manière équivalente, vous pouvez déclencher le même rechargement via l'API :

```sh
curl -s -X POST localhost:8080/api/v1/config/reload \
  -H 'Content-Type: application/json' \
  -H "Authorization: Bearer $KAPKAN_API_TOKEN"
```

`Content-Type: application/json` est obligatoire sur chaque `POST` qui modifie l'état (le serveur
renvoie `415` sinon), et lorsque l'authentification est activée le rechargement nécessite un jeton
Bearer non restreint (admin) — un [jeton restreint à un locataire (tenant)](/docs/multi-tenancy) reçoit `403`.

La plupart des clés se rechargent à chaud, notamment `dry_run`, les seuils, `networks`, les
hostgroups, les références (baselines), les paramètres de notification et `sampling.default_rate`.
Six éléments sont figés au démarrage, et un rechargement qui en modifie un est rejeté (redémarrage
requis) : les adresses `listen`, l'identité BGP (`bgp.local_asn` / `router_id`), `api.listen`, les
paramètres `samples`, les paramètres `storage` et les paramètres `geoip`. Consultez la
[référence de configuration](/docs/configuration) et, pour changer le binaire sans interrompre la
mitigation, [Mise à niveau](/docs/upgrading).

## Compilation depuis les sources

La compilation requiert **Go 1.26 ou une version plus récente**. Le `Makefile` fournit les cibles
standard :

```sh
make build  # static binary
make test   # go test -race ./...
make lint   # golangci-lint run
make bench  # engine hot-path benchmarks
```

`make build` produit le binaire `kapkan` unique et statique que vous installez aux étapes
ci-dessus. Les tests utilisent des datagrammes NetFlow/sFlow synthétiques générés par `pkg/flowgen`
(format de transmission réel) et un peer GoBGP intra-processus, de sorte qu'aucun routeur réel n'est
jamais contacté.

<Callout type="warning" title="Avant d'exposer Kapkan">
Conservez `dry_run: true` jusqu'à ce que vous ayez validé la détection face à de la télémétrie en
direct — voir [Mise en production](/docs/going-live). Et définissez un jeton d'API avant de lier
l'écouteur au-delà de `127.0.0.1`, faute de quoi l'API et le tableau de bord seront accessibles sans
authentification — voir [Authentification](/docs/authentication).
</Callout>
