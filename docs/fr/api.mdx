# API REST

Kapkan expose une petite API REST JSON pour observer la détection et piloter la mitigation
manuelle. Elle est servie sur l'adresse définie dans `api.listen`, qui vaut par défaut
`127.0.0.1:8080`. Le même écouteur sert également le [tableau de bord](/docs/dashboard) intégré
et le point de terminaison Prometheus [`/metrics`](/docs/metrics).

L'API est essentiellement en lecture : le statut, les attaques actives et récentes, l'instantané
des hôtes suivis et la table des bannissements sont tous des `GET`. Trois points de terminaison
`POST` modifient l'état — ban manuel, déban manuel et rechargement de la configuration.

<Callout type="warning" title="Authentifiez avant de l'exposer">
La valeur par défaut de `api.listen` se lie à `127.0.0.1`, de sorte que l'API n'est sûre sans
authentification que sur localhost. Avant de vous lier au-delà de la boucle locale, définissez
un jeton bearer. Voir [Authentification](/docs/authentication).
</Callout>

Toutes les routes `/api/v1` passent par la vérification du jeton lorsque l'authentification est
configurée ; une fois configurée, chaque requête doit porter `Authorization: Bearer <token>`. Les
jetons portent un **rôle** : les routes `GET` nécessitent le rôle **viewer**, et les routes
mutables (`ban`, `unban`, `config/reload`) nécessitent le rôle **operator**. `config/reload` va
plus loin : il exige un jeton **non rattaché (admin)**, de sorte que même un jeton operator
rattaché à un tenant est refusé avec un `403`. Une lecture avec un jeton viewer fonctionne ; une
mutation avec un jeton viewer renvoie `403` ; un jeton inconnu renvoie `401`. Les points de
terminaison `POST` exigent en outre un type de contenu JSON —
envoyez `Content-Type: application/json`. Notez que `/metrics` et l'enveloppe statique du tableau
de bord sont servis sans jeton ; les données chargées par le tableau de bord sont, elles,
récupérées via l'API protégée. Voir [Authentification](/docs/authentication) pour les rôles des
jetons.

Un jeton peut aussi être **rattaché à un tenant** : il ne voit alors que les lignes de son
tenant sur chaque point de terminaison de lecture (`/status` est reconstruit pour ses propres
hostgroups et compteurs), et ne peut effectuer `ban`/`unban` qu'au sein de son tenant — une cible
hors tenant renvoie un `403` uniforme. Un jeton rattaché à un tenant ne peut pas du tout recharger
la configuration : `config/reload` est réservé aux jetons non rattachés (admin) et renvoie `403`
sinon. Voir [Multi-tenant](/docs/multi-tenancy).

## Points de terminaison

| Méthode | Chemin | Description |
| --- | --- | --- |
| GET | `/api/v1/status` | Mode, durée de fonctionnement, réseaux protégés, seuils, hostgroups, nombre d'attaques/bannissements actifs. |
| GET | `/api/v1/attacks` | Attaques actives ainsi que les 100 dernières terminées, avec échantillons et classification. |
| GET | `/api/v1/hosts` | Instantané des hôtes suivis : débits par direction, baselines apprises, état d'attaque. |
| GET | `/api/v1/bans` | Tous les bannissements, actifs et historiques. |
| GET | `/api/v1/traffic` | Historique des débits par hôte persisté (viewer) ; `{available:false, points:[]}` lorsque le stockage est désactivé. |
| GET | `/api/v1/audit` | Piste d'audit des mutations attribuée aux opérateurs (qui a banni/rechargé, quand, résultat). |
| POST | `/api/v1/ban` | Bannir manuellement une adresse. |
| POST | `/api/v1/unban` | Retirer manuellement un bannissement. |
| POST | `/api/v1/config/reload` | Relire le fichier de configuration (équivalent à `SIGHUP`). |
| GET | `/metrics` | Métriques Prometheus. |
| GET | `/healthz` | Sonde de vivacité/disponibilité non authentifiée : `503` jusqu'au démarrage complet, puis `200`. |

Toutes les réponses sont au format JSON. Les erreurs renvoient un objet de la forme
`{"error": "..."}` accompagné d'un statut HTTP approprié.

### GET /api/v1/status

Renvoie le mode de fonctionnement actuel et un résumé de ce que Kapkan protège.

```json
{
  "dry_run": true,
  "uptime_seconds": 8123,
  "version": "v1.2.0 · a1b2c3d",
  "update_available": false,
  "networks": ["203.0.113.0/24", "198.51.100.0/24"],
  "active_attacks": 1,
  "active_bans": 1,
  "thresholds": {
    "pps": 80000,
    "mbps": 1000,
    "flows_per_sec": 20000
  },
  "hostgroups": [
    { "name": "web", "calculation": "per_host", "mitigation": "blackhole", "ban": true }
  ]
}
```

`dry_run` indique le mode global : lorsqu'il vaut `true`, aucun blackhole n'est annoncé à vos
routeurs. `version` est le build en cours d'exécution, toujours présent et sans aucune sortie
réseau. `update_available` vaut `false` à moins que la vérification optionnelle
[`update_check`](/docs/configuration) ne soit activée et ne détecte une version plus récente —
il vaut alors `true` et `latest_version`, `latest_is_security` et `latest_url` sont inclus à ses côtés. Voir [Mise à niveau](/docs/upgrading).
`thresholds` et `hostgroups` reflètent la configuration active ; voir
[Détection](/docs/detection) et [Hostgroups](/docs/hostgroups) pour leur forme complète.

### GET /api/v1/attacks

Renvoie les attaques actuellement actives ainsi que les 100 dernières terminées (la plus récente
en premier). Les deux tableaux contiennent des objets `Attack`.

| Champ | Type | Notes |
| --- | --- | --- |
| `scope` | string | `host` ou `group`. Les attaques à portée de groupe ne portent pas de `target`. |
| `target` | string | L'adresse attaquée (portée hôte). |
| `group` | string | Nom du hostgroup (portée groupe, ou groupe de l'hôte). |
| `tenant` | string | Le tenant du groupe propriétaire, lorsqu'il est étiqueté (voir [Multi-tenant](/docs/multi-tenancy)). |
| `direction` | string | `incoming` ou `outgoing`. |
| `metric` | string | La métrique qui a déclenché, p. ex. `pps`, `mbps`, `tcp_syn_pps`. |
| `rate` | number | Le débit observé à la détection, dans l'unité de la métrique. |
| `threshold` | number | Le seuil franchi. |
| `rates` | object | Ventilation complète des débits par protocole (voir ci-dessous). |
| `active` | bool | `true` tant que l'attaque est en cours. |
| `ban_state` | string | `active`, `withdrawn` ou `rejected`. Omis lorsqu'il n'y a pas de bannissement. |
| `method` | string | Méthode de mitigation : `blackhole`, `flowspec` ou `divert`. Omis lorsqu'il n'y a pas de bannissement. |
| `route` | string | La chaîne de route (`blackhole …` / `divert …`), ou un résumé `flowspec: ...`, lorsqu'un bannissement existe. |
| `flowspec` | array | Les règles [FlowSpec](/docs/flowspec) générées, lorsque `method` vaut `flowspec`. |
| `dry_run` | bool | Indique si le bannissement était virtuel. |
| `started_at` | string | Horodatage RFC 3339. |
| `ended_at` | string | Horodatage RFC 3339 ; omis tant que l'attaque est active. |
| `sample` | object | Échantillon de flow capturé à la détection ; omis lorsque l'échantillonnage est désactivé. |
| `classification` | object | Vecteur d'attaque inféré ; omis lorsqu'il n'est pas classifié. |
| `reason` | object | Pourquoi la détection a déclenché — provenance du seuil, warm-up, parts de protocole. Attaché au démarrage. |

```json
{
  "active": [
    {
      "scope": "host",
      "target": "203.0.113.66",
      "group": "web",
      "direction": "incoming",
      "metric": "pps",
      "rate": 412000,
      "threshold": 80000,
      "rates": {
        "pps": 412000,
        "mbps": 3100,
        "flows_per_sec": 9800,
        "udp_pps": 405000,
        "udp_mbps": 3080
      },
      "active": true,
      "ban_state": "active",
      "route": "blackhole 203.0.113.66/32 next-hop 192.0.2.1 community 65000:666",
      "dry_run": false,
      "started_at": "2026-06-13T09:41:07Z",
      "sample": {
        "top_sources": [
          { "key": "198.51.100.23", "packets": 1240000, "bytes": 1612000000 }
        ],
        "top_src_ports": [
          { "key": "123", "packets": 1180000, "bytes": 1534000000 }
        ],
        "top_dst_ports": [
          { "key": "443", "packets": 1240000, "bytes": 1612000000 }
        ],
        "protocols": [
          { "key": "udp", "packets": 1240000, "bytes": 1612000000 }
        ],
        "total_packets": 1240000
      },
      "classification": {
        "type": "ntp_amplification",
        "confidence": 0.95,
        "src_port": 123
      }
    }
  ],
  "recent": []
}
```

L'objet `rates` porte le trio de base (`pps`, `mbps`, `flows_per_sec`) plus les champs
par protocole qui sont non nuls : `tcp_pps`, `tcp_mbps`, `udp_pps`, `udp_mbps`,
`icmp_pps`, `icmp_mbps`, `tcp_syn_pps`, `tcp_syn_mbps`, `frag_pps`, `frag_mbps`. L'objet
`sample` résume les flows mis en tampon à l'origine de la détection — `top_sources`,
`top_src_ports`, `top_dst_ports`, `protocols` (chacun une liste de compteurs
`{key, packets, bytes}`), une liste `flows` brute facultative et `total_packets`. Le `type`
de `classification` est l'un des vecteurs documentés dans [Détection](/docs/detection) ;
`confidence` est la part (`0..1`) du trafic d'attaque correspondant à la signature gagnante, et
`src_port` est le port de service réfléchi pour les vecteurs d'amplification.

L'objet `reason` explique **pourquoi** la détection a déclenché — capturé une seule fois au
démarrage, hors du chemin critique. `threshold_source` vaut `static` ou `baseline` : selon que la
limite franchie provient de la configuration statique ou d'une [baseline apprise](/docs/baselines)
échauffée. Lorsqu'il vaut `baseline`, un objet `baseline` porte le calcul effectif
(`min(ceiling, max(floor, normal × factor))`). `baseline_configured`, `warming_up` et
`warmup_remaining_seconds` expliquent un seuil statique qui ne s'est appliqué que parce que la
baseline n'était pas encore échauffée. `shares` est la fraction par protocole du total des PPS
qui a guidé la classification, et `dominant_share_gate` est la part qu'un protocole doit atteindre
pour remporter un vecteur (sinon l'attaque est `mixed`). Voir [Détection](/docs/detection) pour
savoir comment le lire.

```json
{
  "reason": {
    "threshold_source": "baseline",
    "baseline": { "normal": 1200, "factor": 8, "floor": 5000, "ceiling": 80000 },
    "baseline_configured": true,
    "shares": { "udp": 0.98, "syn": 0, "tcp": 0.01, "icmp": 0, "frag": 0 },
    "dominant_share_gate": 0.5
  }
}
```

### GET /api/v1/hosts

Renvoie un instantané de chaque hôte suivi — les données des principaux émetteurs. Chaque entrée
est un `HostStat`.

| Champ | Type | Notes |
| --- | --- | --- |
| `target` | string | L'adresse de l'hôte. |
| `group` | string | Le nom du hostgroup de l'hôte. |
| `rates` | object | Débits entrants courants sur la fenêtre glissante. |
| `rates_out` | object | Débits sortants ; non nuls uniquement lorsque la détection sortante est activée. |
| `in_attack` | bool | Indique si l'hôte est impliqué dans une attaque active. |
| `metric` | string | La métrique de l'attaque active ; omise lorsque l'hôte n'est pas attaqué. |
| `direction` | string | La direction de l'attaque active ; omise lorsque l'hôte n'est pas attaqué. |
| `baseline` | object | Baseline entrante apprise ; présente lorsque des baselines sont configurées. |
| `baseline_out` | object | Baseline sortante apprise ; présente lorsque des baselines sont configurées. |

```json
{
  "hosts": [
    {
      "target": "203.0.113.66",
      "group": "web",
      "rates": {
        "pps": 412000,
        "mbps": 3100,
        "flows_per_sec": 9800,
        "udp_pps": 405000
      },
      "rates_out": {
        "pps": 120,
        "mbps": 2
      },
      "in_attack": true,
      "metric": "pps",
      "direction": "incoming",
      "baseline": {
        "pps": 950,
        "mbps": 7,
        "flows_per_sec": 60
      },
      "baseline_out": {
        "pps": 110,
        "mbps": 2,
        "flows_per_sec": 14
      }
    }
  ]
}
```

Les objets `baseline` et `baseline_out` portent le trio de base de la normale apprise (`pps`,
`mbps`, `flows_per_sec`) et n'apparaissent que lorsque des baselines EWMA sont configurées pour
le groupe de l'hôte. Voir [Baselines](/docs/baselines).

### GET /api/v1/bans

Renvoie la table complète des bannissements — actifs et historiques. Chaque entrée est un `Ban`.

| Champ | Type | Notes |
| --- | --- | --- |
| `target` | string | L'adresse bannie. |
| `prefix` | string | Le préfixe de blackhole (`/32` ou `/128`). |
| `metric` | string | La métrique qui a déclenché le bannissement ; omise pour les bannissements manuels. |
| `rate` | number | Débit observé au moment du bannissement ; omis lorsqu'il est nul. |
| `threshold` | number | Seuil franchi ; omis lorsqu'il est nul. |
| `next_hop` | string | Le next-hop de rejet. |
| `community` | string | La community RTBH. |
| `local_pref` | number | Le `LOCAL_PREF` attaché à la route ; omis lorsqu'il est nul. |
| `route` | string | La chaîne de route complète, ou un résumé `flowspec: ...` pour les bannissements FlowSpec. |
| `state` | string | `active`, `withdrawn` ou `rejected`. |
| `dry_run` | bool | Indique si le bannissement était virtuel. |
| `manual` | bool | `true` pour les bannissements demandés par l'opérateur. |
| `started_at` | string | Horodatage RFC 3339. |
| `expires_at` | string | Expiration du TTL ; les bannissements ne sont jamais permanents. |
| `withdrawn_at` | string | Moment où la route a été retirée ; omis tant que le bannissement est actif. |
| `reason` | string | Raison pour laquelle un bannissement a été rejeté ou retiré ; omise sinon. |
| `method` | string | Méthode de mitigation : `blackhole`, `flowspec` ou `divert`. |
| `flowspec` | array | Les règles [FlowSpec](/docs/flowspec) générées, lorsque `method` vaut `flowspec`. |
| `escalation` | array | L'[échelle d'escalade](/docs/escalation) configurée, lorsqu'une est définie. |
| `escalation_step` | number | Index du barreau courant de l'échelle. |

```json
{
  "bans": [
    {
      "target": "203.0.113.66",
      "prefix": "203.0.113.66/32",
      "metric": "pps",
      "rate": 412000,
      "threshold": 80000,
      "next_hop": "192.0.2.1",
      "community": "65000:666",
      "route": "blackhole 203.0.113.66/32 next-hop 192.0.2.1 community 65000:666",
      "state": "active",
      "dry_run": false,
      "manual": false,
      "started_at": "2026-06-13T09:41:07Z",
      "expires_at": "2026-06-13T10:41:07Z"
    }
  ]
}
```

Voir [Mitigation](/docs/mitigation) pour comprendre comment le TTL, l'hystérésis et le plafond de
bannissements façonnent ce cycle de vie.

### GET /api/v1/audit

Renvoie la piste d'audit attribuée aux opérateurs — qui a émis chaque `ban`, `unban` ou
`config_reload`, quand, et le résultat (y compris les actions **refusées**). Les enregistrements
sont triés du plus récent au plus ancien et le point de terminaison est rattaché au tenant côté
serveur. Il accepte des paramètres de requête optionnels `from`/`to` (RFC 3339), `action`
(`ban`/`unban`/`config_reload`) et `target` (une IP), et porte par défaut sur la dernière heure :

```json
{
  "available": true,
  "events": [
    {
      "event_time": "2026-06-22 03:14:09",
      "action": "ban",
      "result": "rejected",
      "operator": "alice",
      "role": "operator",
      "tenant": "customerA",
      "target": "203.0.113.66",
      "target_type": "host",
      "reason": "whitelisted",
      "source": "api",
      "ban_state": "rejected",
      "dry_run": 0
    }
  ]
}
```

Lorsque le stockage est désactivé, la réponse est `{"available": false, "events": []}`. Notez que
les enregistrements d'audit encodent `dry_run` sous forme d'entier (`0`/`1`), et non sous forme du
booléen JSON utilisé ailleurs sur cette page. Voir la page dédiée [Journal d'audit](/docs/audit)
pour la liste complète des champs, les règles de requête, ainsi que le modèle d'identité des
opérateurs et de rattachement aux tenants.

### POST /api/v1/ban

Place manuellement une adresse en blackhole. Le corps est une seule adresse IP :

```http
POST /api/v1/ban
Content-Type: application/json

{"ip": "203.0.113.66"}
```

La réponse est l'objet `Ban` résultant. Un bannissement manuel respecte toutes les règles de
sécurité qu'un bannissement automatique respecte :

- Une cible figurant sur la **liste blanche** est refusée avec un statut HTTP `409 Conflict` et `reason: "whitelisted"` ; elle n'est jamais annoncée.
- Une cible **en dehors des `networks` configurés** est refusée avec un statut HTTP `409` et `reason: "outside configured networks"`.
- Une requête qui dépasserait **`max_active_bans`** est refusée avec un statut HTTP `409` et `reason: "max_active_bans reached"`.

Dans chaque cas de `409`, le corps reste un objet `Ban` avec `state: "rejected"` et le champ
`reason` renseigné. Une IP invalide ou non analysable renvoie un statut HTTP `400`.

<Callout type="warning" title="Le dry-run s'applique toujours">
Un bannissement manuel respecte le mode global. Tant que `dry_run` vaut `true`, le bannissement
est enregistré et renvoyé avec `dry_run: true`, mais aucune route n'est annoncée. Voir le
[modèle de sécurité](/docs/safety).
</Callout>

### POST /api/v1/unban

Retire un bannissement actif pour l'adresse donnée. Le corps a la même forme `{"ip": "..."}` et
le même type de contenu JSON est requis :

```http
POST /api/v1/unban
Content-Type: application/json

{"ip": "203.0.113.66"}
```

En cas de succès, renvoie le `Ban` retiré. S'il n'existe aucun bannissement actif pour l'adresse,
renvoie un statut HTTP `404`.

### POST /api/v1/config/reload

Relit le fichier de configuration depuis le disque et l'applique — le même effet que l'envoi d'un
`SIGHUP`. Le corps est vide ; le type de contenu JSON reste requis. Contrairement à
`ban`/`unban`, ce point de terminaison exige un jeton **non rattaché (admin)** : un rechargement
remplace l'intégralité de la configuration, de sorte qu'un jeton rattaché à un tenant est refusé
avec un `403` et `config reload is restricted to unscoped (admin) tokens`.

```json
{
  "reloaded": true,
  "dry_run": false,
  "thresholds": {
    "pps": 80000,
    "mbps": 1000,
    "flows_per_sec": 20000
  }
}
```

Si la nouvelle configuration échoue à l'analyse ou à la validation, la configuration en cours
d'exécution reste intacte et le point de terminaison renvoie un statut HTTP `400` avec l'erreur.
Voir [Configuration](/docs/configuration).

### GET /metrics

Sert les métriques Prometheus au format d'exposition texte standard. Ce point de terminaison n'est
pas sous `/api/v1` et est servi sans le jeton bearer afin qu'un scraper puisse l'atteindre. Voir
[Métriques](/docs/metrics) pour la liste complète des métriques.

### GET /healthz

Une sonde de vivacité/disponibilité, non sous `/api/v1` et servie **sans** jeton bearer (elle ne
divulgue rien), afin qu'un superviseur ou le script de mise à niveau [`update.sh`](/docs/upgrading)
puisse confirmer que le démon est opérationnel après un redémarrage. Elle renvoie `503 starting`
jusqu'à ce que chaque composant ait démarré, puis `200 ok`. Comme l'écouteur de l'API ne commence à
accepter des connexions qu'une fois le démon démarré, tout `200` ici signifie que la configuration a
été analysée, que les composants sont opérationnels et que Kapkan répond.

```sh
curl -fsS localhost:8080/healthz   # exits non-zero (503) until ready, then prints "ok"
```
