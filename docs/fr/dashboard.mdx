# Tableau de bord

Ouvrez **http://localhost:8080** dans un navigateur exécuté sur l'hôte Kapkan. Lorsque vous êtes
attaqué, regardez d'abord la bannière **Mode actif** (Kapkan annonce-t-il réellement, ou se
contente-t-il d'observer ?) et la liste des **Attaques actives** (qu'est-ce qui est touché, et avec quelle intensité).

L'interface web est intégrée directement dans le binaire `kapkan` lui-même, il n'y a donc rien de
supplémentaire à installer ou à déployer — le même processus sert le tableau de bord, l'API JSON et les
métriques Prometheus.

<Callout type="info" title="Un seul binaire, un seul écouteur">
Le tableau de bord, l'API JSON et `/metrics` sont tous servis par le même processus à la même
adresse. Il n'y a aucun service web distinct à déployer ou à maintenir synchronisé.
</Callout>

## Où il se trouve

Le tableau de bord est servi sur `/` à la même adresse que l'API REST — la valeur `api.listen`
de votre configuration. Avec la valeur par défaut `127.0.0.1:8080`, vous l'ouvrez à l'adresse
[http://localhost:8080](http://localhost:8080).

Comme la valeur par défaut se lie à `127.0.0.1`, le tableau de bord n'est accessible que depuis l'hôte
Kapkan lui-même — ouvrir `localhost:8080` sur votre propre portable n'affichera rien. Pour le consulter depuis
votre poste de travail, redirigez le port via SSH plutôt que de relier `api.listen` à une adresse publique :

```bash
ssh -L 8080:127.0.0.1:8080 user@kapkan-host
```

Ouvrez ensuite [http://localhost:8080](http://localhost:8080) sur votre poste de travail. (Relier
l'écouteur à une adresse publique nécessite d'abord un jeton d'API — voir la mise en garde
[Authentification](#authentification) ci-dessous.)

Pour ne servir que l'API JSON et les métriques sans l'interface, désactivez-la :

```yaml
api:
  listen: "127.0.0.1:8080"
  dashboard: false   # serve the JSON API and /metrics only
```

Avec `api.dashboard: false`, les points de terminaison `/api/v1` et `/metrics` continuent de fonctionner —
seule l'enveloppe de l'interface embarquée cesse d'être servie.

## Ce qu'il affiche

Le tableau de bord interroge l'API JSON et restitue tout ce que Kapkan suit en mémoire :

- **Mode actif** — indique si l'instance fonctionne en dry-run ou en mode actif, afin que vous puissiez voir d'un
  coup d'œil si une attaque détectée serait réellement annoncée.
- **Attaques actives et récentes** — les attaques en cours ainsi que celles récemment terminées,
  chacune avec sa classification du type d'attaque (par exemple, amplification NTP/DNS ou une
  inondation SYN — voir la page [Détection](/docs/detection) pour la signification de chaque type) ainsi que les
  sources, ports et protocoles dominants capturés à partir des échantillons de flow.
- **Principaux émetteurs (top talkers)** — l'instantané des hôtes suivis avec les débits par direction et les
  références (baselines) apprises, afin que vous puissiez voir qui est le plus bruyant et comment cela se compare à la normale.
- **Hostgroups** — vos groupes de préfixes nommés et leur état.
- **Table de bannissement** — chaque bannissement, actif et historique.
- **Trafic / Rapports** — des graphiques de trafic entrant/sortant en direct construits à partir des
  tampons de débit en mémoire, ainsi qu'une section **Rapports historiques** qui lit les tables
  ClickHouse persistées lorsque le stockage est activé.
- **Contrôles** — les mêmes actions que celles offertes par l'API : **bannissement / débannissement manuel**
  d'une adresse et **rechargement de la configuration** (l'équivalent de l'envoi de `SIGHUP`).

La section **Rapports historiques** de la vue Trafic / Rapports est la seule partie qui nécessite
du stockage : elle n'est renseignée que lorsque le [stockage ClickHouse](/docs/storage) est activé, et
affiche sinon un espace réservé étiqueté expliquant comment l'activer.

<Callout type="success" title="Aucune base de données requise">
Le tableau de bord est entièrement construit à partir de l'état en mémoire du moteur. Le stockage historique des flows
est une fonctionnalité distincte et optionnelle — il n'est pas nécessaire pour voir les attaques en direct, les principaux émetteurs ou les bannissements.
</Callout>

## Authentification

Par défaut, l'API et le tableau de bord ne sont pas authentifiés, ce qui n'est sûr que parce que la valeur par défaut
de `api.listen` se lie à `127.0.0.1`. Lorsque vous définissez un jeton d'API, le tableau de bord s'adapte : l'enveloppe
statique de l'interface se charge toujours, mais les données qui la sous-tendent ne s'affichent qu'une fois le jeton fourni.

Lorsque l'authentification est activée, le tableau de bord demande le jeton et le conserve dans le
`sessionStorage` du navigateur, en l'attachant comme en-tête bearer sur chaque requête API qu'il effectue. Chaque
requête `/api/v1` est autorisée avec `Authorization: Bearer <token>` ; `/metrics` et l'enveloppe
statique restent ouverts, mais pas les données en direct ni les contrôles de bannissement / rechargement.

<Callout type="warning" title="Définissez un jeton avant d'exposer l'écouteur">
Avant de lier `api.listen` à autre chose que `127.0.0.1`, configurez un jeton d'API. Consultez
[Authentification](/docs/authentication) pour savoir comment définir `api.token_env` et comment les requêtes sont
vérifiées.
</Callout>
