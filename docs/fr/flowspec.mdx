# Atténuation FlowSpec

Le blackholing [RTBH](/docs/glossary) (remotely-triggered blackhole — une route BGP qui supprime
tout le trafic vers la victime) met toute la victime hors ligne : il échange l'attaque contre une
coupure de service. BGP [FlowSpec](/docs/glossary) (RFC 8955 pour IPv4, RFC 8956 pour IPv6) diffuse
au contraire une règle qui ne supprime que le trafic d'attaque correspondant, de sorte que la victime
continue de servir tout le reste.

<Callout type="info" title="Le dry-run d'abord">
Les règles générées apparaissent dans `/api/v1/attacks` (`method`, `flowspec`), dans `/api/v1/bans`
et dans les notifications **avant** même que vous ne définissiez `dry_run: false` — vous pouvez
ainsi d'abord les vérifier par rapport à la prise en charge de FlowSpec par votre upstream. Voir
[Mise en production](/docs/going-live).
</Callout>

## Activer FlowSpec

Définissez la méthode d'atténuation sur `flowspec`. La méthode par défaut est `blackhole` (RTBH).

```yaml
mitigation: flowspec            # default method for all groups (default: blackhole)
flowspec:
  action: discard               # or rate_limit
  rate_mbps: 100                # only used when action: rate_limit (inert under discard)
hostgroups:
  - name: web
    networks: ["203.0.113.0/26"]
    mitigation: blackhole       # per-group override
```

`mitigation` est surchargeable par [hostgroup](/docs/hostgroups), ce qui vous permet d'utiliser
FlowSpec de manière globale et de basculer sur le blackhole pour un ensemble de préfixes précis
(ou l'inverse).

## Règles générées

Lors d'une attaque, Kapkan dérive un **ensemble minimal de règles** (au plus 8) à partir de la
[classification](/docs/detection) de l'attaque et de l'échantillon de flow, en faisant correspondre
la victime comme destination ainsi que le vecteur :

| Attaque | Correspondance FlowSpec générée |
| --- | --- |
| Amplification NTP / DNS / CLDAP / memcached / SSDP / chargen | `dst=victim, proto=udp, src-port=<reflected port>` |
| SYN flood | `dst=victim, proto=tcp, tcp-flags=SYN` |
| Fragment flood | `dst=victim, fragment` |
| Flood ICMP / UDP / TCP | `dst=victim, proto=<icmp/udp/tcp>` |
| mixte / inconnu | `dst=victim` (plus une règle par port source réfléchi dominant dans l'échantillon) |

La victime est toujours mise en correspondance en `/32` (IPv4) ou `/128` (IPv6). **FlowSpec IPv6
fonctionne de manière identique à IPv4** — les mêmes règles sont générées, la victime étant mise en
correspondance en `/128`.

## Attaques sortantes

Pour une [attaque sortante](/docs/detection) — un hôte compromis qui inonde vers l'extérieur — la
règle fait correspondre l'hôte comme **source** (le préfixe source de la RFC 8955/8956), de sorte
qu'elle supprime réellement le flood sortant. Cela diffère d'un blackhole RTBH basé sur la
destination, qui ne supprime que le trafic *vers* l'hôte.

## Ancrage par la source

Par défaut, chaque règle générée fait correspondre la victime comme **destination**, supprimant le vecteur d'attaque depuis *toutes* les sources — y compris les clients légitimes qui envoient le même protocole. Avec `flowspec.source_anchored: true`, lorsque l'échantillon d'attaque révèle un ensemble **concentré** de sources attaquantes, Kapkan émet à la place des règles composites qui épinglent **à la fois** la victime (destination) **et** chaque attaquant dominant (source), de sorte que seuls les attaquants sont supprimés et que le trafic légitime de la victime est épargné.

```yaml
flowspec:
  action: discard
  source_anchored: true          # drop the attackers, not the protocol from everyone
  min_source_concentration: 0.8  # ...only when the top sources cover >= this share (default 0.8)
```

Cela ne s'active que lorsque les sources dominantes couvrent au moins `min_source_concentration` de l'attaque échantillonnée dans le budget de 8 règles. Une attaque **diffuse** — réflexion ou usurpation répartie sur de nombreuses sources — n'atteint jamais ce seuil et revient à la règle ancrée sur la victime. Les sources sélectionnées sont mises en correspondance uniquement comme préfixes d'hôte (`/32`, `/128`) : il n'y a aucun élargissement vers des préfixes plus larges, de sorte qu'une règle ne peut jamais déborder sur un voisin innocent d'un attaquant.

## Mises en garde

- La correspondance `tcp-flags` pour les SYN floods est un masque de bits qui correspond aussi à
  SYN-ACK, de sorte qu'une action `discard` supprime également les connexions sortantes initiées par
  la victime — préférez `rate_limit` pour les vecteurs TCP. Pour l'utiliser, définissez
  `action: rate_limit` et un plafond dans `rate_mbps` (les exemples ci-dessus utilisent `discard`) :

  ```yaml
  flowspec:
    action: rate_limit            # rate-cap the vector instead of dropping it
    rate_mbps: 100                # ceiling, megabits/s
  ```
- `max_active_bans` plafonne les **bannissements**, pas les règles : un bannissement FlowSpec peut
  porter jusqu'à 8 règles, donc N bannissements peuvent représenter jusqu'à 8N règles dans la RIB de
  votre upstream. Surveillez la [métrique](/docs/metrics) `kapkan_mitigate_flowspec_rules` par rapport à la
  limite de routes FlowSpec de vos routeurs.

## Action de règle et cycle de vie

Chaque règle porte une communauté étendue de débit de trafic (traffic-rate) : `discard` (débit 0) ou
un plafond `rate_limit`. Les bannissements FlowSpec partagent le même TTL, la même hystérésis de
débannissement et le même cycle de vie `max_active_bans` que les bannissements blackhole — voir
[Atténuation RTBH](/docs/mitigation).

FlowSpec circule sur les **mêmes voisins BGP** que RTBH : il est négocié comme une capacité
supplémentaire sur les mêmes sessions BGP (l'AFI/SAFI FlowSpec), et un pair qui ne le prend pas en
charge ne l'activera tout simplement pas. FlowSpec n'est **pas valide pour les hostgroups
`calculation: total`** — il n'y a pas de préfixe de victime unique à faire correspondre.

Pour confirmer que votre upstream accepte réellement les règles : en dry-run, vérifiez que le champ
`flowspec` est renseigné dans `/api/v1/attacks`. Une fois en production, la [métrique](/docs/metrics)
`kapkan_mitigate_flowspec_rules` avec `mode="real"` devrait être non nulle. Si un pair rejette
FlowSpec, Kapkan replie le bannissement sur le blackhole et le signale dans `/api/v1/bans` (les champs
`fell_back_from` et `fell_back_reason`) ainsi que dans les notifications — de sorte qu'un upstream
silencieusement non pris en charge reste visible.
