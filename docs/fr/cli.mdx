# Référence CLI

Kapkan est un binaire unique. Tout ce qu'il fait est sélectionné par des options de ligne de commande
au démarrage — il n'existe aucune sous-commande au-delà des modes utilitaires ci-dessous. Tous les
journaux vont vers **stderr** ; l'API REST et les métriques Prometheus sont servies via HTTP sur
`api.listen` (par défaut `127.0.0.1:8080`), tandis que les options utilitaires (`-version`,
`-dump-schema`) écrivent leur sortie sur **stdout**.

```sh
kapkan [flags]
```

## Options

| Option | Valeur par défaut | Description |
| --- | --- | --- |
| `-config <path>` | `configs/dev.yaml` | Chemin du [fichier de configuration](/docs/configuration) YAML. |
| `-log-format <fmt>` | `json` | Format de sortie des journaux : `json` (structuré, pour les collecteurs de journaux) ou `text` (lisible par un humain, pour les exécutions locales). Toute valeur autre que `text` est traitée comme `json`. |
| `-log-level <lvl>` | `info` | Niveau de journalisation minimal : `debug`, `info`, `warn`, `error`. Une valeur non reconnue retombe sur `info`. |
| `-check-config <path>` | — | Valide le fichier de configuration situé à `<path>` puis se termine — ne démarre **pas** le démon. Voir [ci-dessous](#valider-une-configuration). |
| `-dump-schema` | `false` | Affiche le schéma JSON de la configuration sur stdout puis se termine. Voir [ci-dessous](#exporter-le-schéma). |
| `-version` | `false` | Affiche la version de build puis se termine. La même version est exposée à l'exécution via `/api/v1/status` et la [métrique](/docs/metrics) `kapkan_build_info`. |
| `-check-update` | `false` | Exécute une fois la [vérification de mise à jour](/docs/configuration) optionnelle puis se termine — `0` à jour, `10` une mise à jour est disponible, `1` en cas d'erreur. Interroge l'API GitHub Releases ; n'envoie que la requête elle-même, jamais l'identité du nœud ni la configuration. |

<Callout type="info" title="json ou text ?">
Utilisez `-log-format text` pour le développement local — c'est le format utilisé par le
[Démarrage rapide](/docs/quickstart). Conservez la valeur par défaut `json` en
production afin qu'un collecteur de journaux puisse analyser les champs structurés ; l'[unité
systemd](/docs/deployment) est livrée avec `-log-format json -log-level info`.
</Callout>

## Exécuter le démon

Sans option utilitaire, Kapkan ouvre ses écouteurs de flow, son API REST et (hors dry-run) son
locuteur BGP, puis tourne jusqu'à ce qu'il reçoive `SIGINT` ou `SIGTERM`, moment auquel il s'arrête
proprement. Il ne se termine avec un code non nul qu'en cas d'erreur fatale au démarrage ou à
l'exécution.

```sh
# Local, human-readable logs against the bundled dev config:
kapkan -config configs/dev.yaml -log-format text

# Production-style: structured logs at info level:
kapkan -config /etc/kapkan/config.yaml -log-format json -log-level info
```

Le fichier de configuration est rechargeable à l'exécution sans redémarrage — envoyez `SIGHUP`
(`systemctl reload kapkan`) ou `POST /api/v1/config/reload`. Voir la
[référence de configuration](/docs/configuration) pour ce qui se recharge et ce qui est figé au
démarrage.

## Options utilitaires

Ces options se terminent avant le démarrage du démon — elles n'ouvrent jamais d'écouteur ni
n'envoient d'annonce de route, elles sont donc sûres à exécuter sur un hôte de production.
(`-check-update` effectue une seule requête sortante vers l'API des releases ; les autres sont
entièrement hors ligne.)

### Valider une configuration

```sh
kapkan -check-config /etc/kapkan/config.yaml
```

Effectue exactement la même analyse et la même validation que le moteur — y compris les règles
inter-champs qu'un schéma statique ne peut pas exprimer — puis affiche la configuration résolue (mode,
réseaux protégés et l'atténuation effective par [hostgroup](/docs/hostgroups)) et se termine :

| Code de sortie | Signification |
| --- | --- |
| `0` | La configuration est valide. |
| `1` | La configuration est invalide ; l'erreur exacte est affichée sur stderr. |

Cette séparation nette 0/1 s'intègre directement dans une CI ou une vérification de pré-déploiement.
Voir [Valider la configuration](/docs/configuration#valider-la-configuration).

### Exporter le schéma

```sh
kapkan -dump-schema
```

Affiche le schéma JSON de la configuration sur stdout et se termine avec le code `0` (ou `1` en cas
d'erreur interne). Le [générateur de configuration](/config) utilise ce schéma pour valider les champs
dans votre navigateur.

### Afficher la version

```sh
kapkan -version
```

Affiche la version de build (le tag de release plus la révision VCS courte lorsqu'elle est connue) sur
stdout puis se termine. La même chaîne est servie à l'exécution sur `/api/v1/status` et portée par la
[métrique](/docs/metrics) `kapkan_build_info` — vous pouvez ainsi confirmer la version d'un binaire
localement sans aucune sortie réseau.

### Vérifier les mises à jour

```sh
kapkan -check-update
```

Exécute une fois à la demande la [`update_check`](/docs/configuration) puis se termine : `0` si à jour,
`10` si une release plus récente est disponible, `1` en cas d'erreur. Elle interroge l'API GitHub
Releases et n'envoie que la requête elle-même (votre adresse IP et un User-Agent générique) — jamais
l'identité du nœud, la configuration ni les données d'attaque. La vérification périodique en cours de
processus est **désactivée par défaut** ; cette option fonctionne indépendamment de ce réglage.
