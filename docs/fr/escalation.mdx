# Paliers d'escalade

Une méthode `mitigation` unique déclenche la même réponse à l'instant même où une attaque est
détectée. Un palier d'escalade, lui, **renforce progressivement la réponse à mesure qu'une
attaque persiste** — de manière déclarative, sans aucun script de rappel.

La plupart des déploiements se contentent d'une seule méthode `mitigation:`. N'utilisez un
palier que lorsque vous voulez d'abord une réponse plus douce (par exemple une alerte, puis un
filtrage chirurgical) avant de recourir à un [blackhole](/docs/glossary) complet.

```yaml
escalation:                         # supersedes `mitigation` when present
  - { after_seconds: 0,   action: none }       # alert only at first
  - { after_seconds: 30,  action: flowspec }   # still under attack after 30s → surgical drop
  - { after_seconds: 90,  action: divert }     # still under attack after 90s → scrub
  - { after_seconds: 300, action: blackhole }  # still under attack after 300s → blackhole
flowspec:
  action: discard
scrubbing:
  next_hop: "192.0.2.100"   # scrubbing center; see Traffic diversion
  community: "65000:200"
```

Un [bloc `bgp`](/docs/mitigation#configuration-bgp) est requis pour qu'un palier puisse
annoncer ; les blocs `flowspec` et `scrubbing` ci-dessus ne sont nécessaires que pour les
paliers `flowspec` et `divert`.

## Application des paliers

Le `after_seconds` de chaque palier est mesuré à partir du début de l'attaque. Un palier
s'applique une fois ce délai écoulé **et tant que le bannissement est encore actif**. (Un
bannissement n'est levé qu'après que le trafic est resté sous le seuil pendant toute la période
de refroidissement configurée, `ban.unban_hysteresis_seconds` — voir
[Configuration](/docs/configuration).) L'horloge du palier continue de tourner depuis le début
de l'attaque tout du long.

- Le premier palier doit être à `0` seconde.
- Un palier peut comporter au plus **5 échelons** — c'est une limite stricte ; le démon refuse
  de démarrer avec un palier plus long.
- `action` vaut `none` (alerte seule), `flowspec`, `divert` ([scrubbing](/docs/scrubbing)) ou
  `blackhole`.
- Un palier ne peut que **maintenir ou renforcer** la réponse : `none` < `flowspec` < `divert` <
  `blackhole`. Une désescalade entre paliers constitue une erreur de configuration.

## Make-before-break

La montée vers un palier suit le principe make-before-break : le nouveau palier est annoncé en
premier, et le précédent n'est retiré qu'une fois cette annonce réussie — de sorte que la
victime n'est jamais momentanément sans protection pendant le basculement. Si l'annonce échoue,
le bannissement conserve le palier opérationnel et réessaie au tick suivant. Si plusieurs
paliers arrivent à échéance en même temps (attaque de longue durée, ou démon qui rattrape son
retard après une pause), le bannissement saute directement au palier dû le plus élevé et
n'annonce jamais les paliers qu'il franchit.

Un palier `flowspec` ou `divert` dont l'annonce est rejetée par le pair amont bascule vers le
blackhole (sauf si vous définissez `ban.fallback: none` dans
[Configuration](/docs/configuration)) ; l'API et les notifications le signalent via le champ
`fell_back_from`. Les paliers blackhole et alerte seule n'ont pas de repli.

La seule exception au ballet annonce-puis-retrait est `divert → blackhole` : tous deux
annoncent la même route d'hôte (le `/32` ou le `/128` de la victime), de sorte que l'annonce du
blackhole remplace atomiquement la route de divert — sans retrait distinct, sans interruption.
Les basculements entre familles (règles FlowSpec vs route d'hôte) relèvent du cas
make-before-break ci-dessus.

## Cycle de vie et compatibilité

Le palier est **surchargeable par hostgroup** et partage le reste du cycle de vie du
bannissement : retrait automatique au TTL, plafond `max_active_bans`, les plafonds de rayon
d'impact (`max_banned_fraction`, `max_bans_per_window`), la liste blanche absolue et le dry-run
(qui fait avancer le palier et journalise chaque échelon, mais sans jamais annoncer).

Lorsqu'aucun bloc `escalation` n'est défini, la méthode `mitigation` unique se comporte
exactement comme un palier à un seul échelon à `0` seconde — compatibilité ascendante totale.
Chaque bannissement dans `/api/v1/bans` expose le palier complet configuré (`escalation`),
l'indice de l'échelon courant (`escalation_step`) et la méthode de l'échelon actif (`method`).
