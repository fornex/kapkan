# Métriques

Kapkan expose des métriques Prometheus sous l'espace de noms `kapkan_` via `GET /metrics` sur
l'écouteur de l'API (`api.listen`, par convention `127.0.0.1:8080` — la valeur de la configuration d'exemple ; ce champ est obligatoire, alors définissez-le). Elles couvrent l'ensemble du
pipeline — l'ingestion des flows, le moteur de détection, l'atténuation RTBH et les notifications —
afin que vous puissiez collecter le débit, la latence, les attaques actives et les routes annoncées
dans votre supervision existante.

<Callout type="info" title="/metrics reste ouvert">
Le point de terminaison `/metrics` n'est jamais authentifié, même lorsque vous définissez un jeton
d'API avec `api.token_env`. Seuls les points de terminaison de données `/api/v1` exigent
`Authorization: Bearer <token>`. Configurez l'écouteur en conséquence — voir
[Authentification](/docs/authentication).
</Callout>

## Métriques disponibles

| Métrique | Labels | Signification |
| --- | --- | --- |
| `kapkan_ingest_flows_total` | `proto` | Total des flows ingérés, par protocole de flow. |
| `kapkan_ingest_packets_total` | `exporter`, `proto` | Total des datagrammes de télémétrie reçus, par exporteur source (cardinalité bornée ; voir [`flow_sources`](/docs/configuration)) et par protocole. |
| `kapkan_ingest_decode_errors_total` | `proto` | Datagrammes de télémétrie dont le décodage a échoué, par protocole. |
| `kapkan_ingest_dropped_flows_total` | — | Flows abandonnés parce que la file d'entrée du moteur était pleine. |
| `kapkan_engine_active_attacks` | — | Jauge des attaques actuellement en cours. |
| `kapkan_engine_attacks_total` | — | Compteur des attaques détectées depuis le démarrage. |
| `kapkan_engine_process_latency_seconds` | — | Histogramme de la latence de traitement sur le chemin critique, observée une fois par lot d'ingestion (et non par flow). |
| `kapkan_engine_tracked_hosts` | — | Jauge des hôtes actuellement suivis par le moteur. |
| `kapkan_engine_boundary_debug_bytes_total` | `exporter`, `iface`, `dir` | Octets corrigés de l'échantillonnage vers (`dir=in`) ou depuis (`dir=out`) les hôtes protégés, ventilés par `exporter` et par interface (`iface` = ifIndex). N'apparaît que lorsque [`sampling.boundary_debug: true`](/docs/configuration) — une aide temporaire pour identifier vos interfaces de bordure. Sa cardinalité n'est PAS bornée ; désactivez-la de nouveau après avoir lu la ventilation. |
| `kapkan_engine_events_dropped_total` | `kind` | Événements de cycle de vie du moteur abandonnés parce que le canal d'événements était plein, par `kind` (`attack_started`, `attack_ongoing`, `attack_ended`). Un `attack_started` ou `attack_ended` abandonné est une perte réelle — la mitigation ou la notification de cet épisode est manquée ; un heartbeat `attack_ongoing` abandonné se rétablit de lui-même à la fenêtre de détection suivante. Devrait rester à zéro ; un compteur qui augmente signifie que le consommateur d'événements ne suit pas. |
| `kapkan_mitigate_announced_routes` | `mode` | Jauge des routes de blackhole actuellement annoncées, ventilées par mode `real` / `dry_run`. |
| `kapkan_mitigate_flowspec_rules` | `mode` | Jauge des règles [FlowSpec](/docs/flowspec) actuellement annoncées, ventilées par mode `real` / `dry_run`. Surveillez-la par rapport à la limite de routes FlowSpec de vos routeurs. |
| `kapkan_mitigate_bans_rejected_total` | `reason` | Bannissements refusés par une garde de sécurité, par `reason` : `max_active_bans`, `blast_radius_fraction`, `blast_radius_rate` ou `max_active_prefix_bans` (carpet). |
| `kapkan_mitigate_fallback_total` | `from`, `to` | Annonces d'atténuation ayant basculé vers une méthode de repli parce que le pair a rejeté la méthode principale, par `from`/`to`. Une série `from="flowspec"` non nulle signale les opérateurs en amont qui n'honorent pas FlowSpec. |
| `kapkan_notify_notifications_total` | `channel`, `result` | Notifications tentées, par canal (telegram, slack, email, webhook, exec) et par `result` (`ok`, `error` ou `dropped` lorsque les créneaux de livraison étaient saturés). |
| `kapkan_storage_rows_total` | `table`, `result` | Lignes transmises au [stockage ClickHouse](/docs/storage) optionnel, par table et selon `written` / `dropped` / `error`. `dropped` = la file du stockage était pleine (le stockage ne bloque jamais la détection, les flows sont donc tout de même traités et atténués). |
| `kapkan_build_info` | `version`, `revision`, `goversion`, `goos`, `goarch` | Jauge d'information constante à `1` portant le build en cours d'exécution dans ses labels (l'idiome `node_exporter`). Interrogez la dérive de version de votre parc avec `count by (version)(kapkan_build_info)` — sans aucun phone-home. |
| `kapkan_update_available` | `latest_version`, `security` | `1` lorsque la vérification optionnelle [`update_check`](/docs/configuration) détecte une version plus récente (absente sinon) ; `security="true"` signale une version liée à la sécurité. |

Le label `mode` sur `kapkan_mitigate_announced_routes` est le moyen le plus rapide de confirmer
l'état dry-run d'un coup d'œil : tant que `dry_run: true`, chaque route qui aurait été annoncée est
comptabilisée sous `mode="dry_run"` et aucune sous `mode="real"`. Voir [Atténuation](/docs/mitigation)
et le [Modèle de sécurité](/docs/safety).

Le label `proto` est le protocole filaire sur lequel le flow est arrivé : `sflow5`, `netflow5`,
`netflow9` ou `ipfix` (par ex. `kapkan_ingest_flows_total{proto="sflow5"}`).

## Collecte (scraping)

Ajoutez Kapkan comme cible de collecte dans votre configuration Prometheus :

```yaml
scrape_configs:
  - job_name: kapkan
    metrics_path: /metrics
    static_configs:
      - targets: ["127.0.0.1:8080"]
```

Utilisez l'hôte et le port de votre valeur `api.listen` (par convention `127.0.0.1:8080`). Si vous exposez l'écouteur au-delà de
localhost, définissez un jeton d'API — `/metrics` reste accessible sans celui-ci, de sorte que
votre collecteur n'a besoin d'aucun identifiant.

Pour inspecter directement la sortie brute :

```sh
curl -s localhost:8080/metrics | grep kapkan_
```

## Grafana dashboard

Un dashboard Grafana officiel pour l'ensemble `kapkan_*` est disponible en téléchargement direct à
[`kapkan-overview.json`](/kapkan-overview.json) (sa source se trouve dans le dépôt sous
[`engine/deploy/grafana/`](https://github.com/fornex/kapkan/tree/main/engine/deploy/grafana)).
Il couvre l'ensemble du pipeline — taux d'ingestion des flows/datagrammes/décodage et flows abandonnés,
attaques actives et démarrées, hôtes suivis, quantiles de latence du chemin critique, routes annoncées
et règles FlowSpec par `mode`, bannissements refusés par `reason`, replis d'atténuation, ainsi que les
résultats des notifications et du stockage.

Dans Grafana, allez dans **Dashboards → New → Import**, téléversez le JSON et sélectionnez votre source
de données Prometheus lorsqu'elle est demandée (elle est paramétrée sous `DS_PROMETHEUS`). Le dashboard
cible Grafana 10+ et est vérifié à l'import avec Grafana 11. Voir le
[README](https://github.com/fornex/kapkan/blob/main/engine/deploy/grafana/README.md) situé à côté de
celui-ci pour le provisionnement.
