# Multi-tenant

Une seule instance Kapkan peut servir de nombreux clients (cas d'usage fournisseur de services
managés (MSP) ou centre de données internet (IDC)) et donner à chacun un token qui ne voit et ne
touche **que les siens** : attaques, bannissements et hôtes. Un tenant n'est qu'une étiquette
optionnelle sur un [hostgroup](/docs/hostgroups) — il n'y a pas de nouvel objet de premier niveau,
pas de registre de tenants distinct, et le chemin critique de détection reste intact.

```yaml
tenant: "house"                 # optional: label the global/fallback group

hostgroups:
  - name: custA-web
    tenant: "customerA"         # this group belongs to customerA
    networks: ["203.0.113.0/26"]
  - name: custA-dns
    tenant: "customerA"         # a tenant can span several groups
    networks: ["203.0.113.64/26"]
  - name: custB
    tenant: "customerB"
    networks: ["198.51.100.0/24"]
  - name: shared-infra          # no tenant → visible only to admin tokens
    networks: ["192.0.2.0/24"]

api:
  tokens:
    - { name: admin,    token_env: KAPKAN_ADMIN, role: operator }
    - { name: a-portal, token_env: KAPKAN_A,     role: viewer,   tenant: "customerA" }
    - { name: b-ops,    token_env: KAPKAN_B,     role: operator, tenant: "customerB" }
```

## Comment le mettre en place

Trois étapes correspondent directement au YAML ci-dessus :

1. **Étiquetez le ou les hostgroups de chaque client** avec `tenant: "customerA"` (la même étiquette peut se répéter sur plusieurs groupes — par exemple les groupes web et DNS d'un client).
2. **Ajoutez un token API** sous `api.tokens` portant ce même `tenant:` plus un `role:` — `viewer` est en lecture seule, `operator` peut aussi bannir/débannir.
3. **Remettez ce token au client.** Il ne voit et ne touche alors que ses propres préfixes ; tout le reste (les autres tenants, les groupes non étiquetés) lui est invisible.

C'est tout le workflow. Le reste de cette page explique comment la propriété est résolue et précisément ce qu'un token cadré peut et ne peut pas faire.

## Le modèle

« Quel tenant possède cette IP » est résolu par **la même** recherche de hostgroup par préfixe le
plus long que le moteur et le mitigateur utilisent déjà pour attribuer le trafic — quand des
préfixes se chevauchent, le hostgroup correspondant le plus spécifique l'emporte
([longest-prefix match](/docs/glossary)). Un tenant peut s'étendre sur plusieurs hostgroups (un
client avec des groupes web et DNS séparés).

Toute IP qui tombe en dehors de tout hostgroup nommé est gérée par le **groupe global/de repli**
implicite (le fourre-tout). Ce groupe global peut porter un `tenant` de premier niveau afin que ce
trafic fourre-tout soit attribué à un tenant « house » ; un hostgroup nommé **sans** `tenant` est
non étiqueté et visible uniquement par les tokens admin.

Une étiquette de tenant n'est **pas** une unité de politique de mitigation — les seuils, les
attributs BGP et l'escalade restent par hostgroup. C'est purement une étiquette de propriété
utilisée pour cadrer l'API.

## Portée du token

Un [token API](/docs/authentication) gagne un `tenant` optionnel, qui s'appuie sur son rôle : un
`viewer` peut lire mais ne rien modifier ; un `operator` peut aussi bannir/débannir.

| Token | Voit | Peut modifier |
| --- | --- | --- |
| sans portée (pas de `tenant`) | tous les tenants (admin) | tout ce que son rôle autorise (un viewer lit tous les tenants ; un operator peut bannir/débannir dans tous les tenants) |
| `tenant: customerA`, `role: viewer` | uniquement customerA | rien |
| `tenant: customerA`, `role: operator` | uniquement customerA | uniquement customerA |

Un token sans portée est un admin — le comportement par défaut, entièrement rétrocompatible. Un
token cadré sur un tenant qu'aucun hostgroup n'utilise est rejeté au chargement de la config (une
faute de frappe ne verrait sinon silencieusement rien).

## Application

Le cadrage est **par défaut en refus : un token cadré ne voit jamais que son propre tenant.** Le
tenant est dérivé une seule fois lors de l'authentification et appliqué à chaque ligne, de sorte
qu'un token cadré ne peut jamais retomber sur les données d'un autre tenant.

- **Lectures** — `/api/v1/status`, `/attacks`, `/hosts`, `/bans` et `/audit` ne renvoient que les
  lignes dont le groupe propriétaire porte le tenant de l'appelant. `/status` est reconstruit par
  portée : un token cadré n'obtient que ses propres hostgroups et compteurs, jamais les réseaux
  protégés globaux, les seuils globaux ni la ligne de config du groupe de repli à l'échelle du
  déploiement — ainsi un tenant n'apprend jamais les préfixes ou la posture BGP d'un autre. Un
  tenant house (le groupe de repli étiqueté d'un `tenant`) voit toujours ses attaques, bannissements
  et hôtes fourre-tout ; seule la ligne de config de ce groupe est masquée. Le [journal d'audit](/docs/audit)
  est lié au tenant de l'appelant côté serveur, et un filtre `target` nommant une adresse hors du
  tenant est refusé — pas d'oracle d'existence inter-tenant.
- **Mutations** — un operator cadré ne peut `ban`/`unban` que dans ses propres préfixes. Une cible
  hors du tenant renvoie un `403` uniforme, qu'un bannissement existe ou non, de sorte qu'un
  tenant ne peut pas sonder l'état de bannissement d'un autre. `POST /api/v1/config/reload` est
  **réservé aux admins** — un reload réécrit la politique de chaque tenant ainsi que l'ensemble
  des tokens lui-même.
- **L'ambiguïté échoue de façon sûre** — si un même secret bearer correspond à des tokens de rôle
  ou de tenant différents (un secret réutilisé), la requête est refusée plutôt que résolue vers la
  portée la plus large.

### Vérifier l'isolation

Remettez le token cadré à `curl` et confirmez qu'il ne voit que son propre tenant. `/api/v1/status`
ne renvoie que les hostgroups de ce tenant :

```bash
curl -H "Authorization: Bearer $KAPKAN_A" http://127.0.0.1:8080/api/v1/status
# → only customerA's hostgroups and counts; no other tenant, no global config
```

Confirmez ensuite qu'un bannissement inter-tenant est refusé — bannir une adresse hors du tenant du
token renvoie `403` :

```bash
curl -X POST -H "Authorization: Bearer $KAPKAN_A" \
  -d '{"ip":"198.51.100.5"}' http://127.0.0.1:8080/api/v1/ban
# → 403 {"error":"target is outside your tenant"}
```

<Callout type="info" title="/metrics et le tableau de bord">
`/metrics` n'est pas cadré par tenant — il reste ouvert au scraping Prometheus et ne requiert aucun token,
même lorsque l'authentification de l'API est configurée. La coquille
du tableau de bord est partagée (elle ne détient aucune donnée propre) ; chaque appel de données
qu'elle effectue est filtré par le token qui y est collé, de sorte que le token viewer d'un tenant
n'affiche que les données de ce tenant.
</Callout>

<Callout type="warning" title="La propriété suit la config courante">
La visibilité d'un token cadré est calculée par rapport à la config en direct : re-tenanter un
préfixe (un reload réservé aux admins) transfère immédiatement ses attaques et bannissements en
cours au **nouveau** propriétaire. C'est délibéré — le contrôle suit la propriété courante — mais
cela signifie qu'un reload erroné peut brièvement déplacer des lignes entre tenants.
</Callout>

## Rétrocompatibilité

Aucun tenant configuré nulle part = comportement mono-tenant, à l'octet près : chaque groupe est
non étiqueté, chaque token est un admin sans portée, et l'API renvoie tout comme avant. La
multi-tenancy est opt-in et vous ne pouvez pas obtenir d'isolation sur une API ouverte (sans
token) — il n'y a aucun principal à cadrer.
