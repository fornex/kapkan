# Démarrage rapide

Ce guide vous accompagne de zéro jusqu'à une instance Kapkan en cours d'exécution, en dry-run, qui
détecte les attaques visant un préfixe de test. Il n'y a aucune étape de compilation — Kapkan est
livré sous forme d'un unique binaire précompilé et signé — et rien ici ne peut annoncer une route,
car le mode dry-run est activé par défaut.

<Callout type="info" title="Prérequis">
Un hôte Linux (`amd64` ou `arm64`). Pour observer des détections *en conditions réelles*, vous avez
besoin d'un routeur exportant de la télémétrie de flow — NetFlow, IPFIX ou sFlow (des
enregistrements de qui envoie du trafic à qui ; voir le [glossaire](/docs/glossary)). Sans routeur et
avec un checkout des sources, vous pouvez tout de même confirmer que la détection fonctionne en
exécutant la suite de tests — voir la
[dernière section](#5-optionnel-pas-de-routeur-valider-la-détection-depuis-les-sources). Vous n'avez besoin d'un
toolchain Go que si vous suivez cette voie, ou si vous
[compilez depuis les sources](/docs/deployment#compilation-depuis-les-sources) plutôt que de télécharger.
</Callout>

## 1. Installer Kapkan

Choisissez l'option qui convient à votre hôte — chaque release est livrée précompilée et signée pour
`linux` `amd64`/`arm64`, vous n'avez donc jamais rien à compiler.

<Callout type="info" title="Utilisez la dernière release — remplacez v1.0.0">
Avant de copier-coller, remplacez `v1.0.0` dans chacune des commandes ci-dessous par le tag le plus
récent depuis la [page des releases](https://github.com/fornex/kapkan/releases/latest) (copier
`v1.0.0` tel quel renverra une erreur 404 si ce n'est pas la release actuelle), et `amd64` par
`arm64` sur les hôtes ARM. Les noms d'archives et de paquets omettent le `v` initial (par ex.
`kapkan_1.0.0_linux_amd64.tar.gz`).
</Callout>

### Debian / Ubuntu (`.deb`)

```sh
VER=v1.0.0
curl -fLO "https://github.com/fornex/kapkan/releases/download/$VER/kapkan_${VER#v}_linux_amd64.deb"
sudo apt install "./kapkan_${VER#v}_linux_amd64.deb"
```

Le paquet installe le binaire dans `/usr/local/bin/kapkan`, crée un utilisateur `kapkan` non
privilégié, met en place `/etc/kapkan` avec un `config.yaml` en dry-run, et installe l'unité systemd
renforcée — laissée **arrêtée** afin que vous puissiez d'abord examiner la configuration. La page
[Déploiement en production](/docs/deployment) couvre le service à partir de là. Sur RHEL / Fedora,
c'est identique avec le `.rpm` : `sudo dnf install ./kapkan_<ver>_linux_amd64.rpm`.

### Tarball (toute distribution)

```sh
VER=v1.0.0
curl -fLO "https://github.com/fornex/kapkan/releases/download/$VER/kapkan_${VER#v}_linux_amd64.tar.gz"
tar xzf "kapkan_${VER#v}_linux_amd64.tar.gz"
```

Cela décompresse le binaire `kapkan` ainsi qu'un répertoire `deploy/` (l'unité systemd, un exemple de
configuration et `update.sh`). Le reste de cette page utilise le tarball. Avant de le lancer en
production, il vaut la peine de [vérifier le téléchargement](/docs/upgrading#vérifier-un-téléchargement-à-la-main)
— la signature cosign et le SHA-256.

## 2. Lancer en dry-run

Depuis le tarball décompressé, démarrez Kapkan avec l'exemple de configuration fourni et des journaux
lisibles par un humain :

```sh
./kapkan -config deploy/config.example.yaml -log-format text
```

Il s'exécute au premier plan et affiche les journaux dans votre terminal ; appuyez sur Ctrl-C pour
l'arrêter proprement.

`-config` désigne la configuration YAML et `-log-format text` bascule les journaux du JSON structuré
par défaut vers une forme lisible par un humain — voir la [Référence CLI](/docs/cli) pour toutes les
options et leurs valeurs acceptées.

En dry-run, Kapkan n'a besoin d'aucun privilège particulier et peut s'exécuter en tant qu'utilisateur
normal. (La mise en production lie le BGP au port 179, ce qui nécessite root ou la capacité
`CAP_NET_BIND_SERVICE` — l'unité systemd fournie s'en charge ; voir
[Déploiement en production](/docs/deployment).)

L'exemple de configuration fourni ([`deploy/config.example.yaml`](/docs/configuration)) a
`dry_run: true`, des réseaux protégés de test `203.0.113.0/24` et `2001:db8::/32`, un
`protected_whitelist`, et écoute le sFlow sur `:6343` et le NetFlow/IPFIX sur `:2055`. Comme le mode
dry-run est activé, Kapkan enregistre les blackholes potentiels mais n'annonce jamais de route.

<Callout type="info" title="Vous avez installé le paquet ?">
Si vous avez installé le `.deb`/`.rpm`, le même démon s'exécute plutôt comme un service face à
`/etc/kapkan/config.yaml` — `sudo systemctl start kapkan`, puis `journalctl -u kapkan -f`. Voir
[Déploiement en production](/docs/deployment).
</Callout>

## 3. Orienter vos exporteurs vers Kapkan

Configurez les exporteurs de flow de vos routeurs pour qu'ils envoient les données aux ports
d'écoute de Kapkan :

| Protocole | Port par défaut |
| --- | --- |
| sFlow v5 | `:6343` |
| NetFlow v5/v9 + IPFIX | `:2055` |

NetFlow et IPFIX partagent le même socket UDP. Au moins un écouteur doit être configuré.

## 4. L'observer en action

Kapkan expose son API REST et ses métriques sur `127.0.0.1:8080` par défaut :

```sh
# Current mode, protected networks, thresholds, active attack/ban counts
curl -s localhost:8080/api/v1/status | jq

# Active attacks plus the last 100 that ended, with samples and classification
curl -s localhost:8080/api/v1/attacks | jq

# Prometheus metrics
curl -s localhost:8080/metrics | grep kapkan_
```

Vous pouvez aussi ouvrir le [tableau de bord](/docs/dashboard) intégré sur
[http://localhost:8080](http://localhost:8080) dans un navigateur.

## 5. Optionnel : pas de routeur ? Valider la détection depuis les sources

<Callout type="info" title="C'est une voie de développeur, pas la voie du binaire">
Cette étape n'utilise **pas** le binaire précompilé que vous avez téléchargé ci-dessus. Elle exécute
la suite de tests de Kapkan, elle nécessite donc un checkout des sources et un
[toolchain Go](https://go.dev/dl/). Si vous n'avez que le `.deb`/`.rpm`/tarball et pas encore de
routeur, ignorez cette étape — orientez un routeur vers les ports d'écoute (étape 3) lorsque vous en
aurez un. L'outillage d'attaques synthétiques (`pkg/flowgen`) est un paquet de la suite de tests, pas
une commande exposée par le binaire livré.
</Callout>

Ce que cela prouve : Kapkan détecte une attaque et aurait mis la victime sur liste noire — le tout
sans rien envoyer à un véritable routeur, puisque le dry-run se contente d'enregistrer le blocage
potentiel (un *ban virtuel* — une route [blackhole](/docs/glossary) que Kapkan expose via l'API mais
n'annonce jamais).

Le test envoie un faux flood d'amplification NTP vers une instance en dry-run sur un socket UDP réel
et vérifie que l'attaque apparaît dans l'API, avec un ban virtuel qui expire ensuite automatiquement
selon son TTL. Exécutez-le depuis un checkout des sources :

```sh
git clone https://github.com/fornex/kapkan && cd kapkan/engine
go test ./internal/app -run TestEndToEndNTPAmplification -v
```
