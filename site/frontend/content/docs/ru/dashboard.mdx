# Дашборд

Kapkan поставляется с автономным веб-интерфейсом, встроенным прямо в бинарный файл через `go:embed`.
Здесь нет ни этапа сборки, ни сборщика (bundler), ни внешних ресурсов, которые нужно подгружать — один и тот же
статический бинарный файл `kapkan` обслуживает дашборд вместе с JSON API и метриками Prometheus. Это обычный
JavaScript, который опрашивает REST API, а не приложение на React, поэтому он работает целиком на актуальных
данных и не требует базы данных.

<Callout type="info" title="Один бинарный файл, один слушатель">
Дашборд, JSON API и `/metrics` обслуживаются одним и тем же процессом по одному и тому же
адресу. Нет отдельного веб-сервиса, который нужно разворачивать или поддерживать в синхронном состоянии.
</Callout>

## Где он находится

Дашборд обслуживается по пути `/` по тому же адресу, что и REST API — это значение `api.listen`
из вашей конфигурации. При значении по умолчанию `127.0.0.1:8080` вы открываете его по адресу
[http://localhost:8080](http://localhost:8080).

Чтобы обслуживать только JSON API и метрики без интерфейса, отключите его:

```yaml
api:
  listen: "127.0.0.1:8080"
  dashboard: false   # serve the JSON API and /metrics only
```

При `api.dashboard: false` эндпоинты `/api/v1` и `/metrics` продолжают работать — перестаёт обслуживаться
только оболочка встроенного интерфейса.

## Что он показывает

Дашборд опрашивает JSON API и отображает всё, что Kapkan отслеживает в памяти:

- **Режим работы (Live mode)** — запущен ли экземпляр в режиме dry-run или в боевом режиме (live), чтобы вы могли
  с первого взгляда понять, будет ли обнаруженная атака действительно анонсирована.
- **Активные и недавние атаки** — атаки, происходящие в данный момент, а также недавно завершившиеся,
  каждая со своей классификацией (векторы усиления (amplification), такие как NTP / DNS / CLDAP / memcached /
  SSDP / chargen, а также SYN / UDP / TCP / ICMP / fragment-флуды) и доминирующими источниками,
  портами и протоколами, зафиксированными по сэмплам потоков (flow).
- **Топ источников (Top talkers)** — снимок отслеживаемых хостов с показателями по каждому направлению и
  обученными базовыми уровнями (baselines), чтобы вы могли видеть, кто шумит громче всех и как это соотносится с нормой.
- **Hostgroups** — ваши именованные группы префиксов и их состояние.
- **Таблица банов** — все баны, активные и исторические.

Он также предоставляет те же управляющие эндпоинты, что и API: **ручной бан / разбан**
адреса и **перезагрузку конфигурации** (эквивалент `SIGHUP`). Всё это представление работает только на актуальных
данных — ни ClickHouse, ни какая-либо другая база данных для этого не нужна.

<Callout type="success" title="База данных не требуется">
Дашборд полностью строится на состоянии движка, хранящемся в памяти. Историческое хранение потоков (flow) —
это отдельная, опциональная возможность; она не нужна для просмотра текущих атак, топа источников или банов.
</Callout>

## Аутентификация

По умолчанию API и дашборд не требуют аутентификации, что безопасно только потому, что значение `api.listen`
по умолчанию привязано к `127.0.0.1`. Когда вы задаёте API-токен, дашборд адаптируется: статическая
оболочка интерфейса по-прежнему загружается, но данные за ней не появляются, пока вы не предоставите токен.

Когда аутентификация включена, дашборд запрашивает токен и хранит его в `sessionStorage`
браузера, добавляя его в виде заголовка bearer к каждому запросу к API, который он выполняет. Каждый
запрос `/api/v1` авторизуется с помощью `Authorization: Bearer <token>`; `/metrics` и
статическая оболочка остаются открытыми, а актуальные данные и элементы управления банами / перезагрузкой — нет.

<Callout type="warning" title="Задайте токен перед тем, как открывать слушатель">
Прежде чем привязывать `api.listen` к чему-либо за пределами `127.0.0.1`, настройте API-токен. См.
[Аутентификация](/docs/authentication) о том, как задать `api.token_env` и как проверяются запросы.
</Callout>
